## `edited-files` is `Resolvable` — `[]` means *measured, and clean*

The payload is `Resolvable<EditedFile[]>`, not a bare `EditedFile[]`. That
distinction is decision-grade: `deriveExitMode` reads `files.value.length === 0`
to arm the destructive **Drop & Close**, so an empty list must only ever mean a
worktree we actually measured and found clean.

A conversation with no `worktreePath`, and one whose worktree is reaped
mid-compute (`WorktreeGoneError`), are the same determinate state — no
trustworthy worktree to read. Both collapse onto `unresolved("worktree
unavailable")` through the local `onWorktree` helper, and `revalidate` returns
the constant `"no-worktree"` ETag **from the same branch**, so the value and its
signature stay one co-produced pair (see
`research/2026-07-09-global-etag-value-coproduction.md`). Consumers cannot reach
`.value` without narrowing on `.resolved`, so the destructive default is
unreachable by construction.

Every *other* git failure still **throws**: that is a transient failure, and
live-state's readiness gate (`pending` = "no trustworthy value") makes it
un-absorbable. Do not convert one into an `unresolved(…)` — a retryable failure
that settles would stop retrying and would render as a permanent answer.
