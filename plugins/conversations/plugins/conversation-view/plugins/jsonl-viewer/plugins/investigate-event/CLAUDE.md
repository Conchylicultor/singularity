The copy says so throughout â€” "Add a renderer", not "Investigate". The action's
value is the fix, not the diagnosis: a reader who hits a raw-JSON row wants that
row to render properly next time, and everything needed to build that is already
known at the click. Keep any future wording pointed at the outcome.

**The entire gate is `useDispatchOutcome()?.matched === false`.** The action reads
the outcome of the *nearest* enclosing `.Dispatch` and renders `null` whenever a
contribution matched. That single rule covers every fallback surface â€” an
unhandled event kind, an unhandled tool name (`SendMessage`, `Grep`, an
un-rendered `mcp__*`), an unhandled attachment subtype â€” and any fallback added
later, with zero wiring. Nesting resolves correctly by construction: a `Bash`
card's inner tool dispatch matched, so no button; a `SendMessage` card's did not,
so one appears on that card and not on the event row wrapping it.

Consequently the plugin exports **nothing** and is imported by nobody. It is not
a presentational button any more: no caller passes it a `label`, a `json` blob,
or a `className`, and no fallback renderer may hand-place it (see the contributor
rule in the parent `jsonl-viewer/CLAUDE.md`). The label is `outcome.key`
verbatim â€” deliberately not prettied up per kind, since reconstructing a nicer
one means re-coupling to the very fallbacks this replaced.
