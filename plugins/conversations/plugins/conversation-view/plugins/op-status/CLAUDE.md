# op-status

Banner rendered above the prompt input showing the worktree's in-flight
long-running operation: `Build in progress`, `Push in progress`, or
`Push queued — waiting for lock`, each with a live-ticking elapsed timer.

Fills the gap where a build/push (or a push stuck behind the global push lock)
was indistinguishable from the agent merely "working".

## How it works

- The build/push CLI write a per-worktree op marker at
  `~/.singularity/worktrees/<slug>/ops/{build,push}.json` (owned by the
  `worktree` primitive). The push marker carries a `phase`:
  `waiting-for-lock` while it queues for the global push lock, flipped to
  `running` the instant the lock is granted. Builds only ever write `running`.
- This plugin's server watches that marker tree with `createFileWatcher`
  (mirrors `@plugins/infra/git-watcher`) and pushes the full
  `{ slug → op }` map to the `worktree-ops` live-state resource on every
  change. No polling — markers change via discrete filesystem events.
- The banner reads the resource, keys off `basename(conversation.worktreePath)`
  (the same key the status poller uses), and renders the matching op. The
  elapsed `mm:ss` is a presentational 1s ticker — the op state itself is
  push-driven.

## Boundary notes

- The marker read/write primitive (`markWorktreeOpStart`, `setWorktreeOpPhase`,
  `listActiveWorktreeOps`, `worktreesDir`) lives in
  `@plugins/infra/plugins/worktree/server`. This plugin only consumes it; it
  owns the resource, the watcher, and the banner.
- Renders through the `Conversation.AbovePromptInput` slot, alongside the
  turn-summary card.
