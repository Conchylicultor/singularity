# transcript-retention

Keeps **active** conversations' Claude session JSONL transcripts from being
deleted by Claude Code's own cleanup.

## Why this exists

Claude Code deletes transcript files under `~/.claude/projects/<dir>/<session>.jsonl`
whose **mtime** is older than `cleanupPeriodDays` (default 30). Singularity reads
those files directly as the *sole* source of truth for conversation content — it
never copies them into Postgres. A conversation's mtime only refreshes when it
takes a new turn, so an **active-but-idle** conversation (one a user could still
resume) would silently lose its transcript after 30 idle days.

## What it does

A single scheduled job (`conversations.transcript-touch`) runs daily and
`utimes()`-touches the JSONL of every active conversation — `status <> 'done'`,
which includes the resumable `gone` state — via
[`listActiveConversations`](../../../tasks-core/server) +
[`findTranscriptPath`](../transcript-watcher/server). Touching is also exactly
what `claude --resume` needs, since resume reads the same file.

Closed (`done`) conversations are deliberately left alone, so this is selective —
it never keeps dead transcripts around or bloats disk.

The recurrence is provided by the jobs **`schedule`** primitive
([`@plugins/infra/plugins/jobs`](../../../infra/plugins/jobs)); there is no timer
or self-reschedule here. graphile-worker's native cron dedups fleet-wide, so the
touch runs once per tick regardless of how many worktrees are up.
