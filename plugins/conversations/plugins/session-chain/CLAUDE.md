# session-chain

The append-only record of **which Claude session ids a conversation has run
under, in first-seen order**.

A Claude conversation is not one transcript file. Claude Code relocates a live
session across session ids over the conversation's life (fork, compaction,
daemon handoff after a CLI self-update), and **none of Claude's own on-disk
artifacts records that ordering** â€” transcripts carry no forward/back pointer,
and `~/.claude/history.jsonl` is keyed by cwd, which a worktree shares between
several conversations. So we record the chain ourselves, as the poller observes
each id change. See `research/2026-07-09-conversations-session-chain.md`.

## Boundary

This plugin knows **nothing about file paths or transcripts** â€” only
`conversationId â†’ [claudeSessionId]`. That is what keeps the import graph a DAG:
`transcript-watcher` composes `listSessionChain` with its own
`findTranscriptPath`, rather than this plugin reaching back into it.

Server-only. No `core/`: nothing crosses the runtime boundary, so the field
record lives next to the table it defines.

## Model

`conversation_sessions(id pk, conversation_id, claude_session_id, seen_at)`.

- **Append-only.** Never updated, never deleted. `conversations.claude_session_id`
  keeps its own meaning â€” the live tail, what `claude --resume` uses; this table
  is the history behind it.
- **Soft FK** to `conversations` (plain `text`, no cascade), like the
  `conversation_summaries` precedent: the chain has its own lifecycle and
  outlives the conversation row, so a deleted conversation's transcript files
  are still locatable.
- **`UNIQUE (conversation_id, claude_session_id)`** is the load-bearing
  constraint. `recordSessionId` runs on a 1s poller tick, so the append must be
  idempotent under concurrency; the guard is `ON CONFLICT DO NOTHING` against
  this index rather than a read-then-write check, which would be a TOCTOU. The
  tail probe in front of it is a cost optimization (no write in the steady
  state), not the correctness mechanism.
- A consequence worth stating: a session id that flaps away and comes back
  (`A â†’ B â†’ A`) does **not** re-append. Its position stays pinned at first-seen.
  That is what the transcript merge wants â€” chain order is file order, first
  occurrence wins, and a consumer never reads the same file twice.
- `seen_at` defaults to the transaction's `now()`. Ordering is by `seen_at`
  alone; the poller appends at most one row per tick per conversation, so
  distinct entries are always separated by a real transaction boundary.
