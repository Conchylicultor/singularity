# asset-mirror

Generic server-side mirror for remote binary assets. A plugin declares a remote
source; the primitive lazily downloads each requested file once, caches it under
`~/.singularity/asset-mirror/<id>/`, and serves it same-origin thereafter. After
one online warm-up the assets are fully offline and the external source is never
touched again. The disk cache lives outside any worktree, so the download
happens at most once per machine regardless of how many worktrees are running.

## API

```ts
// server — declare a mirror (add the token to the plugin's `register` array)
import { defineAssetMirror } from "@plugins/infra/plugins/asset-mirror/server";

export default {
  name: "My Plugin",
  register: [
    defineAssetMirror({
      id: "my-assets",
      remoteBaseUrl: "https://cdn.example.com/assets",
    }),
  ],
} satisfies ServerPluginDefinition;
```

```ts
// web — point a consumer at the local mirror
import { assetMirrorUrl } from "@plugins/infra/plugins/asset-mirror/core";

const base = assetMirrorUrl("my-assets"); // "/api/asset-mirror/my-assets"
// fetch(`${base}/logo.png`) → mirrored from https://cdn.example.com/assets/logo.png
```

## Caveats

- **Fail-loud is server-side.** A cache miss that cannot reach the remote source
  logs an error and returns **502** — it never writes a partial/placeholder file.
  Callers that swallow non-200 responses (e.g. smplr's sample loader) will
  therefore degrade silently on the client; the loud signal is the server log +
  the browser Network tab, not a client exception.
- **First request only.** Files are fetched lazily, on demand — there is no
  prefetch. The first (online) use of a consumer populates the cache.
- **Flat file names.** `:file` is a single path segment; mirrored names must not
  contain `/`. Names are URL-encoded by the consumer and decoded server-side.
