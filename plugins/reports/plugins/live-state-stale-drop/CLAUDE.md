# live-state-stale-drop

The report kind for a **live-state resource wedged on a stale HTTP body** ‚Äî the
"Close (state unknown)" bug (`research/2026-07-15-global-live-state-http-cache-poisoning-class-fix.md`).

When `fetchOverHttp` (`primitives/live-state`) fetches a resource over HTTP and its
version/epoch guard judges the body stale, it drops the body. If the query has never been
settled by a server-vouched value, dropping the body cannot fall back to a trustworthy value ‚Äî
so the primitive throws `ResourceStaleReadError` and the query stays `pending`, leaving any
consumer of that resource rendering a placeholder the server never vouched for. The two drop
reasons, discriminated by `reason` in the payload:

- **`stale-version`** ‚Äî same-boot strict-`<` guard: the body's `version` was below the version
  the client already held. Within one boot this is a legitimate race (an invalidate bumped the
  entry past a GET that returned the older number) that heals on retry.
- **`stale-epoch`** ‚Äî cross-boot epoch guard (case 3 of the matrix): the body's boot epoch was
  the *stale* one relative to the WS channel's current server identity. Post-restart this should
  be a transient that heals once the socket re-subscribes.

Either reason is benign as a transient; a **repeating** same-key drop with the query never
settling is the wedge.

## Signal source and threshold

`live-state` is a primitive and must not import `reports`, so `fetchOverHttp` emits a neutral
`HttpStaleDropReport` into a module-level `httpStaleDropReportSink` on **every** drop, carrying a
running `consecutiveDrops` count (reset to zero on any successful apply ‚Äî WS or HTTP).
`LiveStateStaleDropCollector` (mounted via `Core.Root`) drains that sink; this is the same
inversion `reports/crash` uses for `error-boundary`'s `boundaryReportSink` and
`reports/optimistic-divergence` uses for the optimistic-mutation sink.

**The threshold lives in the collector, not the primitive** (policy-free emitter, consumer owns
policy ‚Äî the slow-resource-reporter split). It files a report only at the exact instant
`consecutiveDrops === 3 && neverApplied`. Exactly-equals (not `>=`) so one wedge episode files
exactly one report; the counter reset on apply re-arms it for the next genuine wedge. Three
consecutive never-applied drops is a resource that dropped, retried, dropped, retried, dropped
and still holds only its placeholder ‚Äî a wedge, not a one-off race.

## Fingerprint

`sha256("live-state-stale-drop" + key + reason)`, first 16 hex chars.

`key` + `reason` only: one wedged resource is one bug. Everything volatile is **excluded** ‚Äî
`params` (the same wedge fires for every param combination), `consecutiveDrops`,
`bodyVersion`/`haveVersion`, the three epochs, `source`, and `neverApplied`. Any of these varies
between otherwise-identical repeats of the same wedge, so including it would split one bug across
many `_reports` rows.

## Re-arm

`notifCooldownMs: 6h`, like `optimistic-divergence` / `render-loop`. A still-wedged resource keeps
dropping on every retry; the bell resurfaces it periodically rather than collapsing forever onto
the first sighting.
