# slow-ops

## The `element` metric is cold-start UX truth — don't suppress it

The client `element` signal measures a live-state resource's **mount → first-data
settle** — i.e. user-perceived *time-to-content*. On cold start that is exactly
what the user waits through, so a slow boot wave is a **real UX regression, not
noise**. Resist the temptation to filter out the burst of slow settles that
follows a deploy.

That burst should not happen, because the gateway hot-swaps to a new backend only
once `GET /api/health/ready` returns `200` — which flips after the
`onReadyBlocking` barrier applies migrations and pre-warms the DB pool
(`warmPool`). "Ready" is therefore a contract that the next request is *fast*, not
just answered. So slow `element` settles right after a swap mean readiness flipped
before the backend could serve fast — a **readiness/hot-swap defect to fix at the
source**, never to mute here. (Gateway swap mechanics: `gateway/CLAUDE.md` →
Backend Contract; readiness barrier: `plugins/database/CLAUDE.md` → Bootstrap.)
