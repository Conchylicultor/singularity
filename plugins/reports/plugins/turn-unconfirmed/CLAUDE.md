# turn-unconfirmed

The report kind for **a sent conversation turn that was POSTed and acked but never
confirmed in the transcript** — the silent paste-race symptom.

The conversations pending-turn state machine
(`research/2026-07-22-conversations-optimistic-turn-send-feedback.md`) POSTs a turn, gets a
2xx ack, and then watches the transcript for the message text to land. If it never does
within the confirmation window, the turn enters `unconfirmed`: the user's text is kept on
the pending-turn card with a Retry affordance, and the owner tab files exactly ONE report
(latched via `reported`) with kind `"turn-unconfirmed"`, source `"client-turn-unconfirmed"`,
message `"Turn not confirmed in transcript"`, and payload
`{ conversationId, textPreview, elapsedMs }`.

Unlike `optimistic-divergence`, there is **no collector component here**: the emitter calls
the reports web `report()` API directly, so this plugin is only the kind spec (server) plus
the Debug → Reports summary line (web). `renderTask` walks an agent through comparing the
disk transcript with the endpoint's `resolvedText` to separate a one-off paste race (Retry
covers it) from a systematically broken delivery path.

## Fingerprint

`sha256(conversationId)`, first 16 hex chars.

`textPreview` and `elapsedMs` are **excluded**: every unconfirmed turn on one conversation
is one broken delivery path, not N distinct findings — repeats on the same conversation must
collapse onto a single `_reports` row rather than filing one row per attempted message.

## Re-arm

`notifCooldownMs: 6h`, like `optimistic-divergence`. A conversation that keeps dropping
turns re-fires on every send: it is a recurring warning, not a one-shot crash, so the bell
resurfaces it periodically rather than collapsing forever onto the first sighting.
