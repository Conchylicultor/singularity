# noise-rules

Built-in noise classification rules for the reports plugin.

Some reports are structurally benign and high-frequency — the canonical example
is the `ResizeObserver loop completed with undelivered notifications` browser
warning. These are recorded and visible like any other report, but classified as
"noise" so the notification they file is `muted` (it stays out of the bell's
unread surface). The deduped report is still recorded — noise classification governs
notification loudness, not whether the report is tracked.

This plugin contributes one `ReportNoiseRule` per known low-signal pattern via
the generic `ReportNoiseRule` slot owned by the parent `reports` plugin.
`recordReport` enumerates all rules and stamps a `noise` boolean on the report
row — it never names a specific rule.

## How to add a rule

Add another `ReportNoiseRule({ id, matches })` entry to the `contributions` array
in `server/index.ts`:

```ts
ReportNoiseRule({
  id: "my-benign-pattern",
  matches: ({ message, errorType, source, stack }) =>
    message.includes("…"),
}),
```

`matches` receives the report's `source`, `errorType`, `message`, and `stack`.
Return `true` to mark the report as noise. A thrown error in `matches` is caught
and treated as "no match", so a buggy rule never breaks the report pipeline.
