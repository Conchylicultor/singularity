# engine

The run job (`server/internal/run-job.ts`) walks the step graph as a single
linear data pipeline: each step's input is the previous step's output
(`lastOutput`). The two threading decisions live as pure, unit-tested helpers in
`server/internal/step-flow.ts`:

- **`resolveStepOutput`** — a step that omits `output` from its `StepResult` is
  *transparent*: its input flows through unchanged. This is what keeps a routing
  step (`branch`, which returns only `{ branchKey }`) from nulling the pipeline.
  Setting `output` explicitly — even to `null` — overwrites the value.
- **`resolveNextStepId`** — a matching `branchKey` routes via `nextStepMapping`;
  otherwise the step's default `next` edge is followed.
