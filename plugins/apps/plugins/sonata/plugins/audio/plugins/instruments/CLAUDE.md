# instruments

The **Instrument** extension axis for Sonata's audio — the slot a plugin
contributes a playable timbre to (`SonataAudio.Instrument`), plus the two voice
contracts every contributor and consumer speaks in (`InstrumentVoices`,
`ScheduledNote`).

## Why this is its own leaf

An instrument is an **audio contract**, not an app-shell extension point. A
contributor hands back an `InstrumentVoices` — a voice manager bound to a Web
Audio `AudioContext` (`createVoices(ctx, destination)`), able to `schedule` timed
notes, `play` sustaining hand-played keys, and `allOff` / `dispose`. Nothing
about that needs the Sonata app surface: any host with an `AudioContext` can
register or resolve an instrument.

This axis used to live in `sonata/shell`, alongside the app's other slots
(`Source`, `Display`, `Overlay`, `Transport`, …) **and** the barrel's
`Apps.App({ id: "sonata" })` registration. That co-location meant any instrument
contributor — or any host resolving the sampled grand generically — had to import
`shell/web`, dragging a second `Apps.App` into its bundle. The website's
app-gallery vignette embeds a real, playable Sonata keyboard + sampled piano; on
the released public site that stray `Apps.App` would make bare `/` redirect to
`/sonata` instead of `/website`.

Splitting only the Instrument axis into this leaf cuts that edge: contributors
(`audio/piano`, `audio/soundfont`) and consumers (the engine, metronome,
live-play, track-mixer, the website vignette) import `SonataAudio.Instrument`
from here and never reach the app shell. The slot id string is unchanged
(`"sonata.instrument"`), so no contribution data or docs churn — only the group
symbol moved (`Sonata.Instrument` → `SonataAudio.Instrument`). Sonata's other
slot axes genuinely render inside the app surface and stay with `shell`.

The barrel default-exports an empty-contributions plugin: this leaf only *owns*
the slot; it registers no instrument itself.
