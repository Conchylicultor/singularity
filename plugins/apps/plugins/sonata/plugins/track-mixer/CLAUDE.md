# track-mixer

Per-track view-state for the Sonata player: each track of the open song gets a
**categorical color**, a **mute** toggle (silences it in the audio scheduler),
and a **hide** toggle (drops its notes from the piano-roll). The compact
`Sonata.Section` panel ("Tracks", area `player`) lists every track with its
name (MIDI track name → instrument hint → `Track N`), instrument hint, and note
count, plus a per-song reset.

## Design

- **State is DB-persisted per (song, track)** in `sonata_track_view` — a plain
  1:many table (compound PK `(songId, trackId)`, FK-cascade on song delete), not
  an entity-extension (those are 1:1). `color` is nullable: null means "use the
  palette default for the track's index"; `muted`/`hidden` default to false so an
  absent row reads as audible + visible.
- **One reactive rollup, three consumers.** The `trackViewResource` live-state
  rollup is the single source of truth. The panel reads + writes it; the
  **piano-roll** imports `useTrackColorMap` + `useHiddenTrackIds` to color and
  filter notes; the **audio engine** imports `useMutedTrackIds` to drop muted
  tracks' notes before scheduling. The narrow hooks all derive from
  `useTrackMixerEntries`, which joins `score.tracks` (order → default color) with
  the persisted overrides and a per-track note tally.
- **Writes are fire-and-forget** (`void fetchEndpoint`): the UI never reads the
  response — state refreshes via the live-state push the upsert/reset emits.
- Default colors come from a fixed categorical palette (`palette.ts`), cycled by
  track index.
