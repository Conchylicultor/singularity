# rhythm-circle

A generic, rotating **rhythm necklace** — concentric rings of beads that spin
with a playhead. `<RhythmCircle>` (from the `web` barrel) draws one ring per
track (outermost = `tracks[0]`), a `<circle>` bead per pulse (index 0 at 12
o'clock, increasing clockwise), filling the beads whose index is an onset, with a
needle sweeping the current phase.

**It is domain-agnostic by construction.** It imports nothing from Sonata —
not `shell`, not `rhythm/core`, not `voicing` — and speaks only plain numbers
(`subdivisions`, `onsets`, `phase`). That is the entire point of the plugin: the
same primitive can later drive a drum machine, a metronome visualiser, or any
other onset surface. Boundary-enforced by `./singularity check plugin-boundaries`.

## The zero-render spin

The consumer owns the clock. `<RhythmCircle ref>` exposes a
`RhythmCircleHandle.setPhase(phase)` that, imperatively and with **no** React
render:

- rotates the needle `<g>` about the circle centre via the SVG `transform`
  attribute (`rotate(deg cx cy)` — explicit centre, never the bounding box), and
- flips `data-active` off the previously-lit bead and onto the bead now under the
  needle (`floor(phase × subdivisions)`), through cached per-ring element refs.

Only an *onset* bead swept by the needle "strikes" (a bright primary rim, scoped
CSS keyed on `[data-active][data-onset]`); a rest under the needle stays quiet.
Bead refs are rebuilt only when the track shape changes, so `setPhase` is
`O(tracks)` per frame with no allocation. The `phase` prop covers the
controlled / static case; the imperative handle wins after mount. The primitive
runs no `requestAnimationFrame` loop — drive `setPhase` from your transport.

Pass `onToggleOnset` to make beads interactive (a `button` role, pointer cursor,
hover feedback, keyboard activation); omit it for a purely presentational circle.
Bead transitions respect `prefers-reduced-motion`; the per-frame needle is never
CSS-transitioned.
