# closure-tree

The composition's closure as a plugin tree, tinted by membership state — inline
in the detail pane.

This section is why the detail pane exists. Compositions used to visualize their
closure by auto-pushing the standalone Explorer pane beside the list, casting a
peer plugin browser as its detail view. The tree belongs to the composition, so
it lives here as one section among several; the Explorer goes back to being an
independent browser.

**The tint is free and must stay that way.** `PluginTree` renders the
`Explorer.TreeRowAccent` slot, and `explorer/membership` paints it off
`useActiveComposition()` — the same module-level store this pane seeds. There is
no tinting code here, and adding any would duplicate a contribution that already
works.

**Its own `defineDataView` marker is deliberate.** `PluginTree` defaults its
`storageKey` to the Explorer's `studio.explorer.tree`; sharing it would mean
filtering the Explorer silently filters this closure (and vice versa) — exactly
the surprise this restructure exists to remove. The marker must be declared in
this plugin's own `web/**`: codegen attributes `defineDataView` to the *defining*
plugin, which is what puts the views config under this plugin's config path.

The `max-h-[60vh]` bound is load-bearing, not cosmetic: unbounded, a full plugin
tree renders taller than every sibling section combined and the collapsible
layout stops reading as a list of peers.
