# auto-serve

The **serve capability** for a composition. No longer a detail-pane section of
its own — it exports the two pieces the unified **Build & serve** section (in the
`release` plugin) and the Compositions **list** consume:

- `ServeTargetPanel({ item })` — the **Serve live** target panel: the
  `Serve`/`Serving` `ToggleChip` plus the live serve-URL `LinkChip` once on.
- `useServeComposition()` — `{ serve, stop }`. `serve(id)` persists the intent
  (`setAutoBuild(id, true)`) **and** POSTs `serveCompositionEndpoint`
  (`@plugins/build/core`) to kick an immediate main build, so the live URL is
  ready without waiting for the next full build. `stop(id)` is flag-only
  (`setAutoBuild(id, false)`); the composition is simply not served on the next
  full build.

`autoBuild` is engine-opaque config metadata (dropped by `manifestItemToManifest`).
The CLI `compose-serve` stage reads it from **main's** (`singularity`) resolved
config and, for each activated composition, composes a per-composition frontend
dist + empty DB served at `http://<id>.localhost:9000` on every main build. Because
the stage only reads main's config, toggling from a non-main worktree is inert
until it lands on main (the documented authoritative-worktree caveat).

To keep the import graph a DAG, this plugin never imports `compositions` or
`release`; the edges run the other way (`release → auto-serve`,
`compositions → auto-serve`).
