# thread-list

The Mail app's thread-list column — Miller column 1. Owns `mailboxViewPane`
(`segment: "v/:view"`), a live, windowed list of Gmail-style thread rows for the
selected mailbox view.

## Server

- **`queryThreadsEndpoint`** (`POST /api/mail/threads`) — one keyset page of
  threads for a view. Resolves the (single) account, compiles the view to SQL via
  `mailViewFilterSql` (from `mail-core/server`), orders by
  `COALESCE(last_message_at, created_at) DESC, id DESC`, seeks past the cursor,
  and fetches `limit + 1` to detect a further page. The cursor is an opaque
  base64url token `${sortMs}:${id}` (`server/internal/cursor.ts`, with a
  co-located `bun:test` for the round-trip); the sort key is materialized as
  epoch-millis so the SQL seek and the JS cursor compare the exact same integer.
- **`mailThreadsRevisionResource`** — a coarse `${count}:${maxUpdatedAtMillis}`
  revision over `mail_threads`, push-scoped + debounced. Byte-identical payloads
  are suppressed, so it only pulses on a genuine change.

## Web

- **`mailboxViewPane`** — the list column. `useThreadList(view)`
  (`@tanstack/react-query` `useInfiniteQuery`) owns the keyset pages; a
  subscription to `mailThreadsRevisionResource` triggers an in-place `refetch()`
  on any real thread change (live, no polling, no scroll reset). Rows render
  through `VirtualRows`; an IntersectionObserver sentinel pulls the next page.
- **`ThreadRow`** — leading star (display-only in this read-only phase), senders
  (`senderSummary`) + important/attachment markers + relative date on line 1,
  subject + inline snippet on line 2. Unread threads bold their senders +
  subject.

## Boundaries

Consumes only `mail-core` (schema/types, view model, `mailViewFilterSql`,
`resolveMailAccountId`) and the endpoint/live-state/pane primitives, plus
`reading-pane/web` for `threadPane` (a row opens the reading pane as a pushed
column). It must **never** import `mailbox` — `mailbox → thread-list` (the
sidebar opens `mailboxViewPane`), so the reverse edge would cycle. The label
name for a `label:<id>` view's pane title is therefore unavailable here (the
labels resource is `mailbox`'s); the title falls back to the label id, while the
sidebar — which owns the labels — shows the friendly name.
