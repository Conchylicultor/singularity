# remote-images

The **SSRF-guarded remote-image proxy** for email HTML. Remote images embedded
in an email (`<img src="https://tracker.example/pixel.gif">`, CSS
`background:url(http…)`) are a classic **privacy leak**: loading one directly
tells the sender your IP, rough location, and the exact moment you opened the
mail (tracking pixels). So the reading pane **never** points the browser at a
remote email image directly.

Instead:

- `core` exports `MAIL_IMAGE_PROXY_ROUTE` (`GET /api/mail/image`) and
  `mailImageProxyUrl(remote)` (`/api/mail/image?url=<encoded>`).
- `<MailHtml>` (`mail-html`) rewrites each remote image `src` to that same-origin
  proxy URL — **and only when the user has opted into "Display images"** for
  that message. Until then the src is stripped entirely, so nothing loads.

## Why this is safe (not a tracking-pixel leak, not an open proxy)

The server handler (`server/internal/handle-image.ts`) is a **raw** streaming
route (not `implement()` — it streams bytes, no JSON codec), with three
invariants:

1. **Same-origin.** The browser only ever talks to our own origin; the real
   remote host is contacted server-side. The sender never sees the user's IP
   or open-time from the request path (the fetch is anonymous — no cookies).
2. **SSRF-guarded.** The upstream is fetched through `safeFetch`, which blocks
   loopback / private / link-local / cloud-metadata targets and closes the
   DNS-rebinding TOCTOU. `SsrfError` → 400.
3. **Image-content-type-restricted.** The upstream `content-type` must start
   with `image/`, else **415**. This is what stops the route from being a
   generic open proxy for arbitrary bytes — it can only ever return images.

Response headers are rebuilt fresh from an allowlist (`content-type`,
`content-length`, `cache-control: private, max-age=86400`) — upstream headers
are never copied wholesale, so no `set-cookie` or upstream security/caching
header leaks through. Network/timeout failures map to 502; genuinely unexpected
errors propagate (fail loudly).
