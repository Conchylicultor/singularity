# attachments

Lazy, on-demand download of Gmail attachment blobs for the reading pane.

The sync engine stores attachment **metadata** at hydration time
(`filename` / `mimeType` / `sizeBytes` / `gmailAttachmentId` on `mail_attachments`)
but never fetches the bytes — `stored_attachment_id` stays null. This plugin is
the "fetch the bytes when the user actually wants them" half: a chip click or an
inline `cid:` image resolution triggers one download, which is cached forever.

## Server — `POST /api/mail/attachment`

`mailAttachmentDownloadEndpoint` (contract in `core`), implemented in
`server/internal/handlers.ts`:

- Load the `mail_attachments` row by `attachmentRowId`. Missing → `HttpError(404)`.
- **Cache hit** (`storedAttachmentId` already set): return its same-origin URL
  with no Gmail round-trip.
- **Cache miss**: resolve a Gmail token (`requireGmailToken()` from `mail-core`),
  fetch the bytes (`getAttachment` from `gmail-api`), mint an `infra/attachments`
  blob (`createAttachment`), stamp `stored_attachment_id`, and return the URL.

A stored attachment id is minted once and never changes, so the URL
(`mailAttachmentUrl(id)` → `/api/attachments/:id`, served by `infra/attachments`)
is immutable — the whole flow is a write-once cache. A Gmail/token failure
propagates loudly; there is no silent fallback.

## Web

- `useMailAttachment()` → `{ download(attachmentRowId) => Promise<url> }`. A
  **module-level** promise cache dedupes every download of a row across cards,
  re-mounts, and repeated `cid:` resolutions (the id is immutable, so caching the
  resolved URL forever is correct). A failed download is evicted so a later call
  retries.
- `<AttachmentChip attachment={…} />` — a pill chip (MIME icon + filename + human
  size). Click downloads on demand (spinner while in flight) and opens the URL in
  a new tab. A already-stored attachment resolves with no round-trip.

## Boundaries

Consumes only barrel APIs: `mail-core` (`_mailAttachments`, `requireGmailToken`),
`gmail-api` (`getAttachment`), `infra/attachments` (`createAttachment`),
`infra/endpoints`. The reading-pane plugin is the sole web consumer.
