# mailbox

The Mail app's left-rail navigation and the live data behind it. Renders the
single `Mail.Sidebar` nav — a **Mailboxes** section (the fixed system views:
Inbox, Starred, Important, Sent, Drafts, All Mail, Spam, Trash) above a
**Labels** section (the account's user labels, live). Each row navigates the
thread-list column, is highlighted when it is the open view, and carries a live
unread-count badge.

## What it owns

- **`mailLabelsResource`** (server) — the account's `type="user"` labels, ordered
  by name; push-scoped to `mail_labels`.
- **`mailViewCountsResource`** (server) — unread thread counts per view (system
  ids plus `label:<id>`), computed in one scan with `count(*) filter (where …)`;
  push-scoped to `mail_threads`, debounced.

The client resource descriptors live in `mailbox/core`; the server halves
(`defineResource` + `Resource.Declare`) live in `mailbox/server`.

## Boundaries

The **view model** (`MAIL_SYSTEM_VIEWS`, `MailViewFilter`, `parseMailView`,
`labelViewId`, `DEFAULT_MAIL_VIEW`, …) and the **view→SQL compiler**
(`mailViewFilterSql`) live in `mail-core` (the leaf both this plugin and
`thread-list` consume), NOT here — so the counts resource and the thread query
share one definition of what each view means. The counts loader imports
`mailViewFilterSql` from `mail-core/server`.

The sidebar nav navigates by opening `mailboxViewPane` (from `thread-list/web`),
so `mailbox → thread-list`. `thread-list` must never import `mailbox` — that
would cycle. Anything both need lives down in `mail-core`.

The web icon map (view id → Material icon) is web-only (`web/internal`); the core
view model stays icon-free plain data.
