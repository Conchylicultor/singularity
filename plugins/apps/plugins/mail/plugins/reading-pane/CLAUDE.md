# reading-pane

The Mail app's reading pane — the second Miller column that shows one thread's
messages, oldest→newest.

## Server

`threadMessagesResource` (descriptor in `core`) is declared as a `push` resource
scoped to the `mail_messages` table (`server/internal/resource.ts`). The DB
change-feed auto-pushes whenever a message in the open thread is inserted or
updated — a new reply, a flag flip, or a body hydration — so the pane stays live
with no polling. The loader returns the thread's message **envelopes** ordered by
`internal_date ASC NULLS FIRST, id ASC`. Bodies are null on the stubs; they are
hydrated on demand (below).

## Web

`threadPane` (`Pane.define`, `segment: "thread/:threadId"`, width 640) is **exported
from `web/index.ts`** so the thread-list plugin can open + track it, and
registered via `Pane.Register` in the plugin definition. Open it with
`openPane(threadPane, { threadId }, { mode: "push" })`.

`ThreadPaneView` reads the live `threadMessagesResource` and renders a scrolling
list of collapsible message cards inside `PaneChrome` (title = thread subject):

- **Collapse model** (`useCollapsible`): the newest message is expanded by
  default; older ones collapse to a one-line summary (sender · snippet · date)
  and expand on click.
- **On-demand body hydration** (`internal/use-hydrated-message.ts`): a message's
  body is fetched via `sync`'s `mailHydrateMessageEndpoint` only when its card is
  expanded (the collapsible unmounts closed content, so a React-Query call keyed
  by message id fires on first open and is cached forever after).
- **Body rendering**: `message.bodyHtml` → `<MailHtml>` (from the `mail-html`
  plugin — sanitize + privacy-safe remote images + `cid:` resolution + quote
  collapse); `message.bodyText` fallback → pre-wrapped escaped text. A per-message
  "Display images" bar (driven by `MailHtml`'s `onRemoteImagesDetected`) flips
  remote images on for that message only.
- **Inline `cid:` images** (`internal/use-resolve-cid.ts`): `MailHtml`'s
  `resolveCid` maps a Content-ID to the matching attachment, lazily downloading it
  (via the `attachments` plugin's `useMailAttachment`) and returning the URL once
  resolved (`undefined` until then — `MailHtml` re-renders on change).
- **Attachments**: non-inline attachments render as `AttachmentChip`s below the
  body.

## Boundaries

Consumes only barrels: `mail-core` (types + `_mailMessages`), `sync`
(`mailHydrateMessageEndpoint`), `mail-html` (`<MailHtml>`), `attachments`
(`useMailAttachment` + `AttachmentChip`), and the shared primitives. The
`mail-html` plugin owns all HTML sanitization/transform — this plugin never
parses MIME or HTML itself.
