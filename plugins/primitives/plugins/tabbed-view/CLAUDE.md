# tabbed-view

The **bespoke-only opaque-view host** factory. `defineTabbedView<ViewProps>(id)`
returns a `{ View, Host }` pair: each consumer contributes its own set of opaque
view components to the per-instance `View` slot, and `Host` renders the active
one beneath the shared switcher chrome. Use it when a surface has a *small, fixed,
bespoke* set of views that pass through caller-specific `ViewProps` (e.g.
conversations' Queue/Grouped/History, debug slow-ops' Local/Cluster) — each tab
host having a *different* set of tabs is exactly why this is a factory, not a
global slot.

Post-`unified-view-switcher` roles:

- The **generic named-instance** role (config-authored N instances per type,
  per-view sort/filter/search) lives in `data-view` + its `view-core` child — not
  here. `tabbed-view` is deliberately the opaque-view escape hatch, not a
  data-surface engine.
- The **chrome** (`<ViewSwitcher>`) and the **device-local active-id**
  (`useActiveViewId`) both come from the `view-switcher` primitive. `tabbed-view`
  owns no persistence logic of its own — it just wires `useActiveViewId(id)` into
  the shared switcher.
