# inline-text

Renders a raw string with inline interactive widgets spliced in — file-path
links and active-data chips (`conv-…`, `att-…`, `task-…`, `<ui-context>`) — on
**non-markdown** surfaces (sent user messages, task descriptions). For markdown,
use `<Markdown>` instead; it runs the same walkers via its enhancer slot.

## Why this exists

The widgets are applied by independent `(ReactNode) => ReactNode` walkers
(active-data's `useActiveDataLinkify`, file-links' `linkifyChildren`). They are
**not symmetric**: seeded with a raw string, active-data yields composable
Fragment-wrapped text, but file-links yields a single opaque `<FileLinkText>`
custom component. Both walkers leave custom components opaque, so file-links
**must** run after active-data. Hand-composing them got this backwards and
silently rendered raw text (no type/runtime error).

`<InlineText text={…}/>` removes the footgun: it owns an **ordered walker
registry** (`InlineTextWalkerSlot`) and always seeds with the raw **string**, so
a custom-component root — the wrong-order signature — is structurally
impossible. Consumers never name or order a walker; adding a new inline widget
type is one registration with zero consumer changes.

## How it works

Mirrors the `markdown` enhancer pipeline. Walkers contribute a Component to
`InlineTextWalkerSlot` (`{ id, order, Component }`, lower `order` runs first).
The Component reads any context it needs (e.g. file-links' `onFileOpen`) and
pushes a transform via `useInlineTextWalker`. `<InlineText>` sorts the walkers,
nests their Components outermost-first, and the innermost `InlineTextSeed`
reduces the stacked transforms over the raw string.

Walkers register elsewhere (the primitive itself contributes none):

- active-data → `order: 0` (`plugins/active-data/web`).
- file-links → `order: 10` (`…/conversation-view/markdown-extensions/web`,
  where the conversation/task `onFileOpen` resolution lives).
