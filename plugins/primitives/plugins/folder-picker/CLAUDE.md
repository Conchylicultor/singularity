# folder-picker

A reusable widget for picking a directory on the **host filesystem**, plus the
server endpoint that powers it.

The browser cannot read the host filesystem, so this primitive owns its full
stack across three runtimes:

- `core` — the `browseHostDir` endpoint contract
  (`GET /api/primitives/folder-picker/browse?path=`). One endpoint serves both
  navigation (list a directory's subdirectories, with `parent` for breadcrumb
  nav) and validation (`exists` / `isDirectory`).
- `server` — `implement`s the endpoint over `node:fs`. Defaults to the user's
  home directory, rejects relative paths (400), surfaces `EACCES`/`EPERM` as
  403, and treats a missing path as a legitimate `exists: false` result (not an
  error). Anything unexpected propagates (fail loudly).
- `web` — `useHostDir` (a thin `useEndpoint` wrapper), the `FolderPicker`
  browser block (breadcrumb + subdirectory rows + confirm), and
  `FolderPickerPopover` (a typeable path input with a live green-check /
  red-X validity indicator and a "Browse" popover).

It depends only on lower-level primitives (popover, filepath-breadcrumb, row,
icon-button, spinner, placeholder, truncating-text) and `infra/endpoints`. It
has **no** config_v2 dependency — the `directory-path` config field type wraps
`FolderPickerPopover`, not the other way around.

> Security: the browse endpoint exposes host directory **names** (never file
> contents) for any absolute path the server process can read. This is
> intentional for a local-first tool whose premise is operating on the user's
> machine; it adds no sandbox because the feature is to browse anywhere. The OS
> still gates access — unreadable directories surface as 403.
