# rank

## DO NOT use floats or integers for ordered lists

Use this plugin. `fractional-indexing` strings ("a0", "V0m", "Zz9ŌĆ”") can always
insert between any two neighbors, sort correctly under the `rank_text` PostgreSQL
domain (C collation = byte order), and never need a full-list rewrite.

## When to use this vs tree's `computeDrop`

- **`computeDrop` (`@plugins/primitives/plugins/tree/shared`)** ŌĆö client-side DnD:
  computes the rank for an item *moved between two existing neighbors*. Call it in
  `onDragEnd` / `onDrop` handlers.

- **`nextRankIn` / `nextRankUnder` (this plugin, server barrel)** ŌĆö server-side
  *inserts*: generates the rank for a new item appended at the end of a list.
  Both accept an optional `executor` (a `tx` from `db.transaction`) so they
  compose safely inside transactions.

- **`rankAfterSibling` (this plugin, server barrel)** ŌĆö server-side *positional*
  inserts: the rank that places a new row immediately after an anchor row among
  its true siblings (`afterId === null` prepends at the start). The positional
  twin of `nextRankUnder`, and likewise `executor`-composable.

  **Rank arithmetic is only valid over the COMPLETE sibling set**, which is why
  this is a server API. A client usually holds a *filtered projection* of a
  shared ordering space (a searched tree, a status-filtered list): a rank it
  mints between two *visible* neighbours can collide with an invisible row
  sitting between them (this really happened ŌĆö two `page_blocks` siblings both
  at `"a0"`). So a client must send **positional intent** ŌĆö an anchor id ŌĆö and
  never a client-minted rank. A bad anchor throws; it never silently degrades
  into an append.

## Import paths

```ts
// Server ŌĆö inserting a new row at the end of a flat table

// Server ŌĆö inserting under a parent (parentId, groupId, etc.)

// Server ŌĆö inserting immediately after an anchor sibling (positional intent)

// rankAfterSibling(table, parentCol, parentId, afterId, idCol, executor?)
// afterId === null ŌåÆ prepend at the start of the sibling list

// Server ŌĆö declaring a rank column on a new table schema

// Shared / client ŌĆö computing a rank between two neighbors
import { Rank } from "@plugins/primitives/plugins/rank/shared";
// Rank.between(prev, next) ŌĆö generates a key between two Rank values (or null for open ends)
// Rank.compare(a, b)       ŌĆö byte-order sort comparator (-1 | 0 | 1)
// Rank.equals(a, b)        ŌĆö identity check (use instead of ===)
// Rank.from(str)           ŌĆö wrap a raw string (e.g. from HTTP body)

// Shared / client ŌĆö Zod schema that accepts string | Rank and always outputs Rank

```

## Anti-patterns

- `rank: integer` / `rank: float` ŌĆö can't insert between adjacent integers; floats lose precision
- `ORDER BY createdAt` as a proxy for user order
- `MAX(rank) + 1` ŌĆö the integer trap
- Calling `generateKeyBetween` directly ŌĆö use `Rank.between()` instead
- Sorting with `.localeCompare()` ŌĆö TypeScript now prevents it; `Rank` has no such method
- Comparing `Rank` objects with `===` ŌĆö use `Rank.equals(a, b)`
