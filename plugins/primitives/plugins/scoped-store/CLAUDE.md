# scoped-store

Per-`<Provider>`-instance external store primitive. `defineScopedStore<S>(initial)`
is the sanctioned replacement for the ad-hoc
`let x; const listeners = new Set(); useSyncExternalStore(...)` module-singleton
pattern. The **factory** is module-level and stable, but the **state** is created
once per `<Provider>` mount — so each React subtree (each app surface) gets its
own isolated store. A module-singleton store works only while exactly one
consumer mounts at a time; the moment two surfaces are visible (multi-tab /
multi-window) the singleton tears. Scoping the state to the provider fixes that
without giving up the external-store wins.

```ts
import { defineScopedStore } from "@plugins/primitives/plugins/scoped-store/web";

const CursorStore = defineScopedStore<number>(0);

// At the surface root:
<CursorStore.Provider>{children}</CursorStore.Provider>;

// Imperative (rAF loop / synchronous read — ZERO React renders):
const store = CursorStore.useStoreApi();
store.setState((beat) => beat + 1, { meta: { seek: true } });

// Reactive whole-state:
const beat = CursorStore.useStore();

// Derived-with-bailout — re-renders only when the selected slice changes:
const key = CursorStore.useSelector((beat) => keyAt(beat), [score]);
```

## Read paths

Generalizes the three-read-path shape of Sonata's `cursor-store`:

- `useStoreApi()` — imperative, in-subtree. Drive a scene handle or DOM
  transform directly; throws when used outside its `<Provider>`. Use for rAF
  loops and synchronous reads/writes that must not trigger a React render.
- `useStore()` — raw reactive whole-state. Re-renders the caller on every
  change (`useSyncExternalStore`).
- `useSelector(selector, deps, isEqual?)` — a hand-rolled
  `useSyncExternalStoreWithSelector`. Re-renders ONLY when `selector(state)`
  changes per `isEqual` (default `Object.is`). `deps` invalidate the cache
  exactly like `useMemo`'s deps — pass everything the selector reads besides
  `state`. For selectors that build a fresh object each call, pass a
  value-comparing `isEqual`.

> A component that BOTH renders a `<Provider>` and needs the store in its own
> body (e.g. a rAF loop that writes it) can't use the hooks — they'd resolve
> outside the Provider its own JSX renders. Put the `<Provider>` one level up
> (in a parent/layout component) so the writer sits inside it and uses the
> normal hooks. (Sonata renders the cursor/key-mode providers in `SonataLayout`
> wrapping `SonataProvider` for exactly this reason.)

## Invariants

- `setState` resolves an updater function or a value and **bails on an
  `Object.is`-equal next state**, keeping the snapshot identity referentially
  stable so `useSyncExternalStore`'s tearing check never loops. The `useSelector`
  cache is keyed on this state identity, which is what makes the selector
  bailout correct.
- `setState(next, { meta })` forwards `meta` to every `subscribe` listener —
  the escape hatch for "this change was a seek/jump, not a smooth advance"
  signals that imperative subscribers branch on (the `cursor-store`'s `seek`
  flag generalized).
- The plugin has **no contributions** — it is a pure library primitive.
