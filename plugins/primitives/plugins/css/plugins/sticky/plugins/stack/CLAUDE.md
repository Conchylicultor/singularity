# stack

The **sticky stack** â€” N sticky siblings sharing one containing block, each pinned
*below* the ones before it. The accumulating section-header behaviour, as opposed
to plain `<Sticky>`'s swap hand-off, where every header pins at the same `top` and
the arriving one simply covers the pinned one.

```tsx
<Stack gap="none">                                {/* the real containing block */}
  <StickyStack keys={sections.map((s) => s.key)} base="var(--dv-header-offset, 0px)">
    {sections.map((s) => (
      <Fragment key={s.key}>
        <StickyStackItem itemKey={s.key} mask layer="raised">{s.header}</StickyStackItem>
        {s.body}
      </Fragment>
    ))}
  </StickyStack>
</Stack>
```

## `StickyStack` renders no DOM â€” that is the whole point

`position: sticky` is bounded by the element's containing block: an item un-pins
the moment that block scrolls away. Classic per-section sticky gets that for free
because each header sits inside its own section wrapper â€” which is exactly why the
headers *can't* stack: header B's own block is below header A's, so A is already
gone by the time B arrives.

A stack therefore needs every item to share **one** containing block spanning the
whole list, and each item to pin at its own computed `top`. So `StickyStack` is a
context provider and nothing else: the items stay direct children of the caller's
own real block (the list's `<Stack>`, the table's subgrid). A wrapper `<div>` here
would re-bound each item to the wrapper and silently reintroduce the bug the
primitive exists to fix â€” and in the subgrid table it would additionally break
column alignment by displacing the `col-span-full` rows out of the grid.

The corollary for callers: since the provider contributes no box, the items and
whatever sits between them are laid out by the *caller's* container. In the list
view that also means the per-section `<Collapsible>` wrapper had to go â€” hence
`CollapsibleProvider` (the DOM-less collapsible), so the header and the content
are siblings in the one shared block.

## Offsets are measured, not assumed

Each `StickyStackItem` measures its own height (`element-size`, whose initial
measure is synchronous inside a layout effect) and reports it to the provider,
which sums the rounded heights of the keys *preceding* an item to derive its
`top`: `calc(${base} + ${sum}px)`. Heights are rounded once at the source, so
sub-pixel `getBoundingClientRect` jitter can never re-render the stack, and the
provider only writes state when a rounded value actually changed â€” a
ResizeObserver feeding an unconditional `setState` is an observeâ†’renderâ†’observe
loop. An item whose height hasn't landed yet contributes `0`; the synchronous
measure closes that before first paint, so it is not a visible jump.

This is what the `queue-view` hand-rolled with a hardcoded `SECTION_H = 28` â€” a
constant that is wrong the instant a header's font, padding, or density token
changes.

## The cap (`DEFAULT_MAX_STACKED = 5`)

Stacking is only free while there are few items. Every pinned item costs its own
height of viewport **permanently** â€” twenty pinned group headers would squeeze the
content they head off the screen entirely. So the stack is capped: at
`keys.length <= maxStacked` items stack; above it every item pins at `base` and
the primitive degrades to plain sticky's swap hand-off. The threshold is a prop
(`maxStacked`) because the right number depends on the item's height and the
surface's viewport.

## Props

- **`StickyStack`** â€” `keys` (ordered, topâ†’bottom in DOM order; its **length**
  decides whether the stack stacks), `base` (CSS length the *first* item pins at â€”
  any expression, incl. `var()`/`calc()`; default `"0px"`), `maxStacked`.
- **`StickyStackItem`** â€” `itemKey` (must be in `keys`; absent â‡’ throws, it is a
  wiring bug, not a transient state), plus every `<Sticky>` prop except `edge` /
  `offset`. `top` is stack-owned and wins over a caller `style.top` (a caller
  `top` would silently break the stack); a caller `ref` composes with the
  primitive's measurement ref.
