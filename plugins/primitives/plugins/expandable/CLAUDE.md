# expandable

`Expandable` clamps tall content to a max height and reveals a **Show more / Show
less** toggle — but only when the content **actually overflows** that height.

## Why it exists (measure, don't guess)

The collapse decision is made by **measuring the content's real rendered height**
with a `ResizeObserver` (mirroring `collapsible-wrap` / `responsive-overflow` —
deferred via `requestAnimationFrame`, no timers, no polling), NOT by guessing
from character counts or hard-newline counts.

A char/line heuristic is blind to soft-wrapping, font size, and viewport width: a
long single-paragraph prose block with no hard line-breaks wraps into many visual
lines and can fill most of the pane while staying under any character threshold —
so it would never offer a toggle. Measuring the rendered box fixes the entire
class of bug at once, and also handles async-loaded media (an image that loads and
grows the content re-fires the observer).

## How the measurement stays state-independent

The measured `contentRef` child is **never height-constrained**. The clamp lives
on its *parent* (`max-height` + `overflow:hidden` + a fade mask), so the child's
`offsetHeight` always reflects the full natural content height in BOTH collapsed
and expanded states. The overflow check (`offsetHeight > collapsedHeight + 1`)
therefore never depends on the current expand state — no flapping at the boundary.

## API

- `children` — the content to clamp.
- `collapsedHeight?` — collapsed clamp height in px (default `192` = `12rem`).
- `onToggle?(expanded)` — notified with the next expanded state on user toggle
  (e.g. so a streaming surface can re-pin its scroll).
- `className?` — applied to the root.
