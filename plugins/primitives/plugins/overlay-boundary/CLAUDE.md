# overlay-boundary

A React-only leaf primitive that gives **transient overlay content** its own
error boundary, so a crash inside an open overlay (popover / dialog / dropdown /
select / tooltip / floating-surface) stays contained to that overlay rather than
bubbling up to the slot-level `PluginErrorBoundary` that wraps the launching
chrome (e.g. the action-bar trigger).

`OverlayBoundary` is a plain React `Component` class. When healthy it renders
`children` with no DOM node of its own (transparent, like `SingleLineProvider`).
When a child throws during render it renders a fallback. Isolation is a pure
consequence of React resolving to the *nearest* boundary: the inner
`OverlayBoundary` catches before the crash can reach the outer per-slot boundary
around the trigger.

## The fallback-injection seam (why the boundary lives here)

ui-kit's base-ui `*Content` components are the single shared choke point for
every overlay, so that is where the boundary must be wrapped. But ui-kit sits at
the bottom of the plugin DAG and `error-boundary` transitively depends on it
(`error-boundary â†’ css/text â†’ ui-kit`). ui-kit therefore **cannot** import
`error-boundary`'s rich `CrashFallback` â€” that would close the cycle
`ui-kit â†’ error-boundary â†’ css/text â†’ ui-kit`, which the boundary checker
rejects (the same constraint ui-kit documents for `Frame`).

The resolution is a module-level fallback registry. This leaf depends on **only
`react`** (terminal edge â€” no cycle). ui-kit imports the leaf and wraps its
`*Content` children in `OverlayBoundary`. `error-boundary`, which sits *above*
ui-kit, calls `registerOverlayFallback(...)` at boot to inject the existing
`CrashFallback` (routing overlay crashes through `boundaryReportSink` â†’ reports
and the `ErrorBoundary.Action` "Fix" button, unchanged). The boundary logic
lives low; the fallback UI is filled from above.

The built-in fallback (a text-only Retry `<button>`) exists only for the
pre-registration edge; in the real app the injected `CrashFallback` always wins.

## The `kind` prop

Each `*Content` passes its overlay kind (`"popover"`, `"dialog"`, `"dropdown"`,
`"select"`, `"tooltip"`, `"floating"`). It flows into the injected fallback's
`report.slot`, so a crashed popover's tag reads `popover crashed`.

## Reset semantics

base-ui Popups unmount their content on close, so closing and reopening a
crashed overlay remounts `OverlayBoundary` fresh and clears the error for free.
The fallback's **Retry** button covers residual cases (`keepMounted` surfaces).
