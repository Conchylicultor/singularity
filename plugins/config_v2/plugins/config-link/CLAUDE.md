# config-link

Affordances that link an in-context surface to its backing config. The premise:
a `config_v2` `ConfigDescriptor` already self-identifies its settings location
through the `ConfigV2.WebRegister` registry (`useConfigRegistrations()` maps every
registered descriptor → `storePath`, and the `settings` plugin's
`configDetailPane` renders one descriptor's editor). So the descriptor *is* the
link — surfacing a "configure this" control needs no new registry and no
per-surface wiring beyond passing the descriptor the surface already reads with
`useConfig()`.

## What it exposes

- **`useOpenConfig()`** — returns `(descriptor) => void` that opens the config
  nav + detail chain focused on that descriptor (via `openPane(configDetailPane,
  …, { mode: "root" })`). Throws loudly if the descriptor was never registered.
- **`<ConfigGearButton descriptor />`** — a compact gear `IconButton` that calls
  `useOpenConfig()`. Drop in anywhere (e.g. the top-left of an expanded panel).
- **`<ConfigPopoverHeader label descriptor />`** — the standard header for a
  config-backed popover: an eyebrow `SectionLabel` + trailing `ConfigGearButton`.
  Swap a bare `<SectionLabel>` for this and the affordance comes for free.

## Convention

Any chip/popover whose body is driven by config should use
`ConfigPopoverHeader` (or drop a `ConfigGearButton`) instead of a plain section
label. This keeps "where do I change this?" one click away from the thing it
configures, consistently, without each surface re-deriving navigation.

This plugin only *navigates* to config — the `settings` plugin owns the panes
and the editor.
