# fields

Slot-based field type registry for config_v2. Each field type is a sub-plugin
with a `core/` factory and a `web/` renderer — no central switch-case, no
registration step beyond the plugin barrel.

See [config v2 vision](../../../research/2026-05-16-config-v2-vision.md) for
the full list of planned field types and the overall config design.

## Adding a new field type

Create `plugins/fields/plugins/<name>/` with two runtimes:

### 1. Core factory (`core/internal/<name>.ts`)

```ts
import { z } from "zod";
import { defineFieldType, type FieldDef, type FieldMeta } from "@plugins/config_v2/core";

export const fooFieldType = defineFieldType<MyType>("foo");

export interface FooFieldDef extends FieldDef<MyType> {
  readonly type: typeof fooFieldType;
}

export function fooField(opts?: FieldMeta & { default?: MyType }): FooFieldDef {
  return Object.freeze({
    type: fooFieldType,
    schema: z.object({ /* ... */ }),
    defaultValue: opts?.default ?? /* ... */,
    meta: { label: opts?.label, description: opts?.description, placeholder: opts?.placeholder },
  });
}
```

`defineFieldType<T>(id)` returns a frozen token used for reference-equality
dispatch. Call it exactly once per field type — both the factory and the
renderer must import the same singleton.

### 2. Web renderer (`web/components/<name>-renderer.tsx`)

```tsx
import type { FieldRendererComponent } from "@plugins/config_v2/plugins/fields/web";
import { fooFieldType } from "../../core";

const FooRenderer: FieldRendererComponent<MyType> = ({ field, value, onChange }) => {
  return (
    <div className="flex items-start justify-between gap-4 py-3">
      {/* label + description */}
      {field.meta.label && <label className="text-sm font-medium">{field.meta.label}</label>}
      {/* your editor widget */}
    </div>
  );
};
FooRenderer.type = fooFieldType;  // critical — dispatch matches on this

export { FooRenderer };
```

### 3. Web barrel (`web/index.ts`)

```ts
import { Fields } from "@plugins/config_v2/plugins/fields/web";
import { FooRenderer } from "./components/foo-renderer";

export default {
  id: "config-v2-fields-foo",
  contributions: [Fields.Renderer(FooRenderer)],
} satisfies PluginDefinition;
```

The settings pane auto-discovers the renderer via the `Fields.Renderer` slot.
No other file needs editing.

### Notes

- **`FieldHeader` is not exported** from the primitives plugin. Inline the
  label/description markup (it's just two lines of JSX) rather than importing
  via a deep path.
- **No server barrel needed** unless the field type has server-side logic. The
  Zod schema in the `FieldDef` handles serialization automatically.
- **Complex value types** that don't exist in a `core/` barrel elsewhere should
  be defined locally in your field's `core/` (see `avatar` and `list` for
  examples).

## Plugin reference

- Description: Field type registry. Sub-plugins contribute field types with core factories and web renderers.
- Exports (web):
  - Types: `FieldRendererComponent`, `FieldRendererProps`
  - Values: `FieldRenderer`, `Fields`
- Sub-plugins:
  - **`avatar`** — Avatar field type (icon + color picker).
  - **`enum`** — Enum field type: single-choice from a fixed set of options.
  - **`list`** — Sortable list field type with stable UUID identity and fractional-index ordering.
  - **`multiline-text`** — Multi-line text field type.
  - **`primitives`** — Basic field types: bool, text, int, float.
