# host

Mixing host for apps that use **both** layout renderers. Most apps mount a
single renderer (`<MillerColumns/>` for pure column apps, `<FullPane/>` for
pure full-surface apps). An app that genuinely mixes the two — some routes as
columns, some full-surface — mounts `<PaneLayoutHost full={[…]}/>` instead.

## How it picks

```tsx
import { PaneLayoutHost } from "@plugins/layouts/plugins/host/web";
import { settingsPane } from "./panes";

function MyAppSurface() {
  // active pane in `full` → FullPane; otherwise → MillerColumns.
  return <PaneLayoutHost full={[settingsPane]} />;
}
```

The host resolves the route once (`usePaneRoute`), looks at the **active pane**
(`match.panes.at(-1)`), and renders `<FullPane/>` if that pane is in `full`,
otherwise `<MillerColumns/>`. It provides `PaneMatchContext` and passes the
resolved `match` to whichever renderer it picks, so the inner renderer skips its
own self-resolve — the registry is synced exactly once per frame.

`full` lists **this app's own** `PaneObject`s — not a cross-plugin contributor.
That keeps layout ownership in the layout layer: panes never declare their
layout, and naming a pane "full-surface" is a decision the app makes here, where
it mounts its surface. Matching is by `pane.id` (the public identifier on both
the listed `PaneObject` and the active route entry), so the host never reaches
into pane internals.

## Public API

- `<PaneLayoutHost full={PaneObject[]}/>` — the mixing renderer.
