# stall-monitor

The **alert side of an event-loop stall**. The main-backend health sampler already
**detects** and **stack-attributes** a frozen loop ‚Äî it drains the JSC sampling
profiler and aggregates the stacks into a `StallSection` ‚Äî but before this plugin
that evidence only reached the trace store (Debug ‚Üí Slow Events). No report kind,
no bell, no investigation path.

Consequence observed Jul 7: the stall stacks named the culprit verbatim (a ccusage
transcript-parse burning 52‚Äì77 % of samples, repeatedly, over days), while the
`reports` table filled with **thousands of *victim* slow-op reports**
(flushNotifies 40 s, page-loads, delivers) that could never name the cause. The
signal existed in full; only a manual JSONL/trace read found it.

This plugin adds the missing half. It is modeled on `debug/op-rate` and
`debug/boot-budget` (config + `ReportKind` + web `KindView` renderer) and, like
`op-rate`'s `op-time` kind, owns **both** the trace (evidence) and the report
(alert) for a stall ‚Äî so a frozen backend reaches the bell + Debug ‚Üí Reports,
linked to its trace.

## Push-fed by a direct call, not a job

health-monitor calls `recordEventLoopStall(section, durationMs, thresholdMs)`
directly (a barrel import), and that function fires `captureTrace` + `recordReport`
inline. There is **no scheduled drain job and no accumulator** ‚Äî the mirror-image
of boot-budget's documented "pull-only, no accumulator" deviation. The consumer set
is closed and singular (the trace + the report both live inside this one plugin),
so a direct call is correct ‚Äî not a `defineReportSink` fan-out (that primitive is
for error-path reporters where a low-level primitive must not hard-depend on
`reports`; not this case). This matches `op-rate` / `slow-ops`, which call
`captureTrace()` and `recordReport()` directly. The detector (health-monitor) owns
"what counts as a stall" ‚Äî the threshold constant stays there; only an `enabled`
mute lives in this plugin's config.

## Fingerprint on the STACK, not the leaf

`deriveCulprit` (pure, unit-tested) keys the report dedup fingerprint on
`topStacks[0].stack` (the dominant caller path), **not** `topLeaves[0].key`. A real
flaw caught in review: the hottest leaf on the Jul-7 stall is `JSON.parse [native]`
‚Äî a generic native frame shared by every JSON caller ‚Äî so a leaf-based fingerprint
misattributes and collapses unrelated JSON-heavy freezes into one row. The top
*stack* (`parseTranscript ‚Üê readEntries ‚Üê ‚Ä¶`) names the actual caller path, is
already line-free (names only, robust to edits), and distinguishes callers sharing
a native leaf. The same stack is passed as `captureTrace`'s `label` so the
trace-side admission/cooldown key stays stable across the ticks of one freeze.
`hotFrame` is a human hint for the summary + task title only ‚Äî never the
fingerprint.

## The LABEL comes from the dominant stack's own frames

`hotFrame` is derived from `topStacks[0].frames` ‚Äî **that same stack's** own
frames, innermost ‚Üí outermost ‚Äî and never from the independent `topLeaves`
histogram. The leaf leads (it burned the samples); if it is native (`spawn`,
`JSON.parse` ‚Äî a mechanism, not a subsystem) we walk outward for the first frame
carrying a ` @ path:line` source and append it as the *where*, giving
`spawn ‚Üê listPanes @ ‚Ä¶/tmux-runtime.ts:499`. An already-attributable leaf is both
halves at once, so it is returned bare.

Why structural rather than a filter tweak ‚Äî the incident, Jul 16: a `spawn`-rooted
freeze (7 of 15 samples, `spawn ‚Üê listPanes ‚Üê ‚Ä¶ ‚Üê collectLive`, the conversations
poller's 1s tick spawning `tmux`/`ps`) filed a report titled
`is @ .../drizzle-orm/entity.js:7` ‚Äî a **1-of-15** frame with nothing to do with
the freeze. Two defects, one structural. Proximate: the old scan filtered
`topLeaves` for ` @ `, which skips *unattributed* frames, so the dominant native
`spawn` was passed over and the scan landed on an arbitrary cold tie. Structural:
`hotFrame` came from `topLeaves` while `culpritStack` came from `topStacks` ‚Äî two
**separately-sorted populations**, so even with the filter fixed the label could
describe a minority stall while the fingerprint described the dominant one.
Reading the dominant stack's own frames makes label/fingerprint coherence hold by
construction; they cannot drift apart. This is the same failure mode the plugin
exists to prevent (Jul-7: real evidence buried under misleading reports) ‚Äî a title
that lies sends triage into the wrong subsystem entirely.

`frames` is additive evidence on `StallStack`, optional so pre-existing traces
still parse; without it `deriveCulprit` falls back to the legacy `topLeaves[0]`.

## Load-bearing literals

The config `name` `stall-monitor` and the report kind `event-loop-stall` are
explicit literals ‚Äî persisted config and report dedup depend on them; do not
rename.
