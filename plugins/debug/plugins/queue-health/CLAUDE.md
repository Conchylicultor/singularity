# queue-health

The graphile-worker job queue degrades **silently** тАФ nothing in the UI signals
when it breaks. This plugin makes two failure modes **loud** by filing them into
the existing reports engine (the same surface that captures crashes), modeled
byte-for-byte on `debug/slow-ops` (durable signal тЖТ `ReportKind` тЖТ deduped task).
It reads the queue **read-only** through the jobs plugin's public introspection API (`queryDeadJobStats` / `queryQueueBacklog`), which owns the graphile-internals coupling тАФ the `jobs.run` task literal, the `payload->>'jobName'` encoding, and the terminally-dead predicate тАФ so this monitor can never drift from how the queue is actually encoded.

## What it monitors

One cheap scheduled monitor job (`debug.queue-health-monitor`) runs two aggregate
queries per tick and files reports only when a threshold trips:

- **`queue-dead-job`** (variant `error`) тАФ terminally-failed jobs
  (`attempts >= max_attempts AND locked_at IS NULL`, the same predicate
  `reconcileDeadJobs` uses), grouped by `payload->>'jobName'`. **One report per
  distinct jobName** (fingerprint `queue-dead-job:<jobName>`), so a retry-storm of
  one broken job collapses to a single task while distinct broken jobs get
  distinct tasks.
- **`queue-backlog`** (variant `warning`, escalates to STALLED) тАФ depth/stall.
  Trips when `readyCount > backlogDepthThreshold` **or** the oldest ready job is
  overdue past `oldestOverdueMinutes`. `stalled = lockedCount === 0 && overdue`
  (the worker is making no progress). **One rolling report per worktree**
  (fingerprint `queue-backlog:rollup`; the reports unique index is
  `(fingerprint, worktree)`, so worktrees never collide).

## Thresholds (config_v2, mirroring slowOpConfig)

`enabled = true`, `backlogDepthThreshold = 200`, `oldestOverdueMinutes = 10`.
Read live each tick via `getConfig`, editable in Settings тЖТ Config.

## Why per-worktree, singleton, cheap

- **`perWorktree: true`** тАФ every worktree backend runs its own graphile worker
  against its own DB fork, so dead/backlog state accumulates per-DB and must be
  sampled per-DB (inverse of the usual main-only default, same as `jobs.dead-gc`).
- **`dedup: "singleton"`** тАФ the monitor itself can never pile up.
- **`maxAttempts: 3`** тАФ a transiently-broken monitor doesn't become a dead-job
  storm of its own.
- **Two aggregate queries, no row fetches** тАФ negligible cost; reports fire only
  on a tripped threshold (silent when healthy), and the engine's velocity limiter
  + dedup absorb bursts. Both kinds set `notifCooldownMs тЙИ 10 min` so a persistent
  problem re-alerts the bell periodically without spamming.
