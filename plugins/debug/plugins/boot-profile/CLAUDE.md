# boot-profile

The **Boot Profile** Debug page renders the real browser boot timeline as a
Gantt, reusing the Gantt primitives from `debug/profiling`. It reads the
per-page-load trace from the `primitives/perfs/boot-trace` client store
(`getBootTrace()`) and decomposes boot into these phases:

- `navigation` ‚Äî Navigation Timing, decomposed across the 0‚Üífirst-byte window:
  unload/redirect, DNS, TCP connect (each shown only when non-trivial),
  request‚Üífirst-byte (TTFB), response download, DOM interactive. This attributes
  a slow **document** response (cold/restarting backend, contention) to the right
  sub-phase instead of leaving a blank gap before TTFB.
- `scripts` ‚Äî module eval, plugin load, createRoot.
- `main-thread` ‚Äî Long Tasks (‚â•50ms): the bundle parse/compile/eval +
  plugin-chunk fan-out + first React render that block the thread between
  bytes-arrived and first paint. This fills the historical **0‚Üífirst-span blind
  spot** ‚Äî JS instrumentation can't run while the thread is blocked, so these
  come from the Long Tasks API. Clipped to the boot window (‚â§ FCP / first commit).
- `assets` ‚Äî scripts/stylesheets from Resource Timing, biggest first, as a
  **request** (wait) + **download** (work) split via `WaitWorkRow`. Capped to the
  top-N by transfer size; the remainder rolls up into one summary row (count +
  bytes) so nothing is silently dropped.
- `boot-tasks` ‚Äî boot tasks + the boot-snapshot fetch.
- `resources` ‚Äî each boot-critical resource shipped in the boot snapshot,
  rendered with a **wait** segment (network + queue) and a solid **work**
  segment (server loader/read time) via the custom `WaitWorkRow`.
- `paint` ‚Äî first-paint, first-contentful-paint, first React commit.

## Permalinks (persist & share a captured trace)

`getBootTrace()` returns a fully self-contained, JSON-serializable object and
`BootProfileGantt` is a **pure function of its `trace` prop** (no `performance.*`
or store reads), so a snapshot saved to the DB re-renders byte-for-byte through
the same Gantt with no special-casing.

- **Copy permalink** (live header) persists the current snapshot via
  `POST /api/boot-traces` and copies
  `http://<wt>.localhost:9000/debug/boot-profile/<id>`.
- `/debug/boot-profile/<id>` (`bootProfileDetailPane`) loads the snapshot via
  `GET /api/boot-traces/:id` (404 ‚Üí graceful not-found) and renders it with a
  read-only "Saved snapshot ¬∑ <relative time> ¬∑ <worktree>" banner.
- **Boot Profiles** (Debug sidebar ‚Üí `bootProfileListPane`) lists saved snapshots
  via `GET /api/boot-traces` (metadata only ‚Äî no blob), each row opening the
  detail pane.

Snapshots are stored in the **worktree's own DB** (the permalink keeps the
worktree subdomain) in the `boot_traces` table, derived from the single
`savedBootTraceFields` record (core) via `defineEntity` so column/schema drift is
unrepresentable. The wire `BootTraceSchema` is an explicit zod mirror of the
cross-runtime `BootTrace` (boot-trace/core), pinned to the source type by a
compile-time assertion. A scheduled `defineJob` (`debug.boot-trace-cleanup`,
daily, per-worktree) sweeps snapshots older than 30 days so the table never grows
unbounded.
