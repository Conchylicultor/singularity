# cluster

The **Cluster** tab of the Slow Ops pane. Each worktree writes its `slow_ops`
to its **own forked Postgres DB**, so a cluster-wide contention storm shows up
as ~16 disconnected per-worktree rows. This sub-plugin fans out across every
worktree DB (`listDatabases()` ‚Üí `openShortLivedClient(name)` ‚Üí raw
`SELECT ‚Ä¶ FROM slow_ops`, bounded concurrency via the semaphore primitive) and
merges them into one view.

The fan-out is **pull / user-triggered** (a Refresh button), never live or
polled ‚Äî a 16-worktree query is too heavy and too rarely needed for a live
resource. Per-DB failures (an old-schema fork missing `recent_samples`, a stale
fork) are surfaced as an `ok: false` error row, not swallowed ‚Äî one broken fork
never blanks the whole view.

The server returns the **raw per-worktree rows**; the web layer derives the two
sections client-side (so the merge logic is pure and unit-tested in
`web/internal/aggregate.test.ts`):

1. **Cluster aggregate** ‚Äî rows grouped by `(operationKind, operation)` across
   worktrees: summed count/total, worst max, latest last-seen, and the set of
   affected worktrees ("slow across 12 worktrees").
2. **Contention timeline** ‚Äî every row's `recentSamples` ring flattened across
   all worktrees, newest first, capped at 200. A simultaneous storm appears as a
   dense time cluster sharing a high `load1 / cpu` (warning‚Üídestructive ramp) and
   `pg active` backend count ‚Äî the causal link the per-worktree rows hide.

It contributes its tab into the `SlowOps.View` slot the `pane` sub-plugin owns,
so the pane never imports this component directly.
