# flight-recorder

Slow-event flight recorder. Whenever any server span crosses its slow-op
threshold (the SAME `resolveSlowThreshold` the slow-ops plugin uses ‚Äî "what is
slow" is single-sourced from the `slow-op` config), it persists exactly ONE
coherent-window snapshot to
`~/.singularity/worktrees/<wt>/logs/flight-recorder.jsonl` ‚Äî one JSON line per
snapshot. The snapshot captures the instant of the trip: which ops were
concurrently in flight, which recently completed, how saturated every
concurrency gate was, and the host/DB contention ‚Äî everything blocking-chain
attribution needs, without re-assembling it from aggregates after the fact.

Snapshots are decoupled from the slow-op report/task (no ReportKind, no task
noise); they join to `slow_ops` rows and markers by `(wallTime, kind, label)`.

## Snapshot schema (v1, brief)

```jsonc
{
  "v": 1,
  "atMs": 41234567.8,          // profiler clock
  "wallTime": "2026-07-02T‚Ä¶Z", // wall clock ‚Äî the cross-file join key
  "worktree": "‚Ä¶",
  "trip": {                    // the op that crossed its threshold
    "kind": "loader", "label": "edited-files",
    "durationMs": 4032, "thresholdMs": 2000,
    "parent": { "kind": "flush", "label": "flushNotifies" },
    "waitMs": 3500, "childMs": 0, "selfMs": 532,
    "waits": { "heavy-read-acquire": 3500 }
  },
  "windowStartMs": 41230535.8, // atMs ‚àí max(trip duration, windowMs config)
  "open": [ /* FlightSpan[] ‚Äî entries still in flight at the trip instant */ ],
  "completed": [ /* FlightSpan[] ‚Äî ring spans overlapping the window */ ],
  "gates": { "heavy-read-acquire": { "active": 4, "queued": 11, "max": 4 }, ‚Ä¶ },
  "contention": { /* loadAvg, cpuCount, pg backends, top databases */ }
}
```

## How to read a snapshot (the blocking-chain walk)

1. **`trip.waits` names the layer** the victim queued on ‚Äî e.g.
   `"heavy-read-acquire": 3500` means 3.5 s of the trip's timeline was spent
   waiting on that gate.
2. **`gates[layer]` shows saturation** at the same instant ‚Äî `active 4/4,
   queued 11` confirms the gate was the bottleneck, not the victim's own work.
3. **`open` + `completed` spans whose `waits` include that layer are the
   co-queuers.** The span among them with dominant `selfMs`/`childMs`
   overlapping the window is the holder ‚Äî the op actually occupying the gate
   while everyone else queued.

Gate names use the `chargeWait` layer vocabulary, so `trip.waits`,
`FlightSpan.waits`, and `gates` keys all join directly.

## Timestamp joins

`wallTime` is wall-clock (`Date.now()`), the same idiom as the other JSONL
recorders ‚Äî join a snapshot against `stall-profiles.jsonl` (synchronous
event-loop blocks), `health.jsonl` (event-loop lag / memory around the same
moment), and queue-health reports to distinguish "queued behind a gate" from
"the whole loop was blocked". `atMs`/`t0`/`t1`/`windowStartMs` are on the
profiler clock and only compare to each other.

## Rate limiting

Admission is checked BEFORE any capture work (a storm costs one Map lookup per
slow span):

- **Per-op cooldown** (`cooldownMs`, default 10 s): one snapshot per
  `kind:label` per cooldown window.
- **Global cap** (`maxPerMin`, default 30): hard per-minute ceiling across all
  ops. A cooldown rejection does not consume a minute token.
- `enabled: false` uninstalls the `onSlowSpan` hook entirely.

All knobs live in the `flight-recorder` config (Settings ‚Üí Config), applied
live via `watchConfig` ‚Äî no restart.

## File location + rotation

`~/.singularity/worktrees/<wt>/logs/flight-recorder.jsonl` via
`Log.channel("flight-recorder", { persist: true })`. Bounded without a job:
before each append, if the file exceeds 4 MB it is trimmed to its newest half
(stall-profiler's rotation pattern). Snapshots are tens of KB worst case, so
the cap keeps hundreds.
