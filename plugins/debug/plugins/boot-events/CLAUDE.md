# boot-events

Two durable lines per backend boot, so a **deploy-restart burst is visible as
interval bars** on cross-worktree timelines ‚Äî including a backend that
**wedges mid-boot**, previously invisible exactly during those bursts (see
`research/2026-07-10-global-congestion-observability.md`, Phase A, and
`research/2026-07-11-global-observability-freeze-blind-spots.md`, Stage 7).

Both lines go to the persisted `boot` log channel ‚Üí
`~/.singularity/worktrees/<wt>/logs/boot.jsonl`:

- **`phase: "start"`** ‚Äî written from the plugin's **register phase**, the
  earliest per-process hook, before migrations and any `onReadyBlocking` work.
  (The register phase is documented "no I/O"; this one ~140-byte synchronous
  append is the deliberate exception ‚Äî its entire point is landing before the
  boot work that can wedge.)
- **`phase: "ready"`** ‚Äî written from `onReady` (post-`onReadyBlocking`
  readiness ‚Äî a migrated DB and ready registry), carrying `readyAt`.

```json
{ "sampledAt": <ms>, "worktree": "<wt>", "processStartedAt": <ms>, "phase": "start" }
{ "sampledAt": <ms>, "worktree": "<wt>", "processStartedAt": <ms>, "readyAt": <ms>, "phase": "ready" }
```

A boot is the wall-clock interval `[processStartedAt, readyAt]`
(`processStartedAt` = `Math.round(performance.timeOrigin)`, the pairing key).
Wall-clock ms epoch is the only clock on the line ‚Äî profiler clocks are
incomparable across backends. Pre-cutover lines lack `phase` and parse as
ready lines.

**A log-channel line, not a table**: no migration, survives DB re-forks, and
the main backend can read it off disk while the owning backend is wedged.

**Bounding**: log-channels' `appendEntry` already rotates every channel file at
a 128 MB cap (3 rotated files kept) ‚Äî at two ~140-byte lines per boot that is
effectively unbounded history, so there is deliberately **no custom rotation**
here (the plan's "~256 KB tiny cap" is subsumed by the substrate's rotation).

**Reading**: `readBootEvents(worktree, windowMs)` pairs the lines by
`processStartedAt` (`pair-boot-lines.ts`, pure + tested) into `BootEvent`s
whose interval overlaps the window. A start without a ready line is a boot
that never became ready: `readyAt: null`, bounded by the next attempt's start
(`supersededAtMs`) when one follows ‚Äî a crash-loop must not render as a stack
of eternal open bars ‚Äî and open-ended (the timeline renders it to the window
edge, pulsing) when it is the latest attempt. Mirrors health-monitor's
`read-health-files.ts` ‚Äî a bounded `readChannelEntries` tail, inner-JSON parse,
`safeParse`-drop of malformed lines.

## Plugin reference

- Description: Durable per-boot event line: the onReady hook appends one boot.jsonl log-channel line per backend boot (the wall-clock interval [processStartedAt, readyAt]), and readBootEvents(worktree, windowMs) reads them back ‚Äî so deploy-restart bursts render as interval bars on cross-worktree timelines, with no DB table (survives re-forks, readable while a backend is wedged).
- Server:
  - Uses: `infra/paths.currentWorktreeName`, `primitives/log-channels.Log`, `primitives/log-channels.readChannelEntries`
  - DB schema: `plugins/debug/plugins/boot-events/server/internal/schema.ts`
  - Exports: Types: `BootEvent`; Values: `BootEventSchema`, `readBootEvents`
- Cross-plugin:
  - Imported by: `debug/timeline`
