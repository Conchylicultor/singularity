# boot-events

One durable line per backend boot, so a **deploy-restart burst is visible as
interval bars** on cross-worktree timelines (the boot window is otherwise a
monitoring blind spot — see
`research/2026-07-10-global-congestion-observability.md`, Phase A).

The `onReady` hook (post-`onReadyBlocking` readiness — a migrated DB and ready
registry) appends one line to the persisted `boot` log channel →
`~/.singularity/worktrees/<wt>/logs/boot.jsonl`:

```json
{ "sampledAt": <ms>, "worktree": "<wt>", "processStartedAt": <ms>, "readyAt": <ms> }
```

A boot is the wall-clock interval `[processStartedAt, readyAt]`
(`processStartedAt` = `Math.round(performance.timeOrigin)`). Wall-clock ms epoch
is the only clock on the line — profiler clocks are incomparable across
backends.

**A log-channel line, not a table**: no migration, survives DB re-forks, and
the main backend can read it off disk while the owning backend is wedged.

**Bounding**: log-channels' `appendEntry` already rotates every channel file at
a 128 MB cap (3 rotated files kept) — at one ~130-byte line per boot that is
effectively unbounded history, so there is deliberately **no custom rotation**
here (the plan's "~256 KB tiny cap" is subsumed by the substrate's rotation).

**Reading**: `readBootEvents(worktree, windowMs)` returns the zod-validated
boot events whose interval overlaps the window (filtered on
`readyAt >= now − windowMs`, which is exact overlap since
`processStartedAt <= readyAt`). Mirrors health-monitor's
`read-health-files.ts` — a bounded `readChannelEntries` tail, inner-JSON parse,
`safeParse`-drop of malformed/old-shape lines.
