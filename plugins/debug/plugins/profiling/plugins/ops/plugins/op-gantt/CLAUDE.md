# op-gantt

The reusable Gantt over the ops [`op-log`](../../op-log/CLAUDE.md) records —
`build`, `push`, and `check`, in **one** `group.ops.map`.

## The render model

An op is **one bar** spanning `startMs → startMs + totalMs`, colored by `kind`,
with each `waits[]` entry painted as an overlay segment at its own **true offset
inside that span** (`OpWait.startMs` is relative to the op's own start).

This is deliberately **not** the old `[wait][hold]` head-to-tail split:

- Waits **interleave with real work** — a build reads
  `[build-lock][…migrations/codegen…][duress-valve][host-grant][…heavy work…]`.
- One op may carry **several waits of the same kind** (a build re-queues for the
  host grant across duress requeue cycles).
- Segments **do not tile the bar**: `sum(waits) + holdMs` is generally **less
  than** `totalMs`. The gaps are the op working.

The row's duration column sums each op's own `totalMs` — never `waits + hold`,
which would omit every work gap and under-report every build.

## Three orthogonal color channels

A bar is never ambiguous because kind, blockage, and status each own a separate
channel and never bleed into one another:

| Channel | Answers | Keyed on | Palette |
|---|---|---|---|
| `TYPE_FILL` | *what is this?* — the base bar | `OpKind` | cool: `bg-info` (build), `bg-success` (push), `bg-categorical-5` (check) |
| `WAIT_FILL` | *what is it blocked on?* — the overlays | `WaitKind` | warm ramp: `categorical-3` amber (push-mutex) → `-9` orange (build-lock) → `-4` red (host-grant) → `-8` magenta (duress-valve) |
| `STATUS_TREATMENT` | *how did it go?* | derived status | `animate-pulse` (in flight), `ring-destructive` (failed/interrupted), nothing (ok) |

**The warm/cool split is the gestalt**: any warm patch on a bar means "not
working here", and its hue names *which* resource — self-queued (`push-mutex` /
`build-lock`) vs. fleet-starved (`host-grant`) vs. duress-held
(`duress-valve`). That distinction is the entire diagnostic value of the pane,
which is why the waits get a real palette rather than one undifferentiated
yellow.

Waits draw from the `categorical` token group because that group exists
precisely to supply N mutually-distinguishable, light+dark-tested hues. The
ramp is ordered by severity, and its closest pair (amber/orange) is exactly the
pair that can never co-occur on one bar — a push never takes the build lock, a
build never takes the push mutex — so the weakest contrast is where you never
have to tell the two apart in place.

Status **never** recolors a fill: a running push is green+pulsing (not blue), a
failed build is blue+red-ring (not a different hue). `opStatus()` is **one**
derivation for all three kinds, replacing the separate `pushStatus`/`buildStatus`
that let build and push drift.

The legend lists only the kinds and wait kinds present in the current data, so a
pane with no duress episode never spends a line explaining one.

## Two things that must not regress

- **Interrupted ops** have no known end (`totalMs` 0) → fixed-width
  `INTERRUPTED_MARKER_PX` marker at `startMs`, and **no** wait segments: an
  unbounded span cannot place them honestly.
- **`stopPropagation` on both `pointerdown` and `click`** for every bar and
  segment. Pointerdown: `GanttContainer`'s drag-zoom would `setPointerCapture`
  and retarget the click off the bar. Click: it would otherwise fall through to
  the row's `onWorktreeClick` — a surprising silent redirect into the
  conversation.

`onOpClick(op, worktree)` carries the row's canonical worktree because the
build-profile pane needs it and `OpEntry` does not carry it. Hosts should use
`useOpClick()` from `debug/profiling/ops/web` rather than re-deriving the
kind dispatch.
