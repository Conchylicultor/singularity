# spans

The built-in `spans` trace event class: it captures the runtime profiler's **flight
window** at the trip instant (server) and paints it as a **nested call-tree
waterfall** (web).

## One schema, in `shared/`

`shared/flight-window.ts` is the single zod mirror of the profiler's `FlightWindow`,
imported by both runtimes — the server registers it as the class's section `schema`,
the web parses `snapshot.events.spans` back out with it. It cannot live in
`runtime-profiler/core` (zero-dependency by contract), and it must not be duplicated:
the two hand-mirrors it replaced each hardcoded a 7-kind enum, omitted `cascade`, and
so silently dropped the entire spans section from any trace captured while a cascade
span was in flight. Two rules keep that from recurring:

- `z.enum(SPAN_KINDS)` — the kind list is **derived**, never written out.
- A **bidirectional** compile-time pin. A one-way `FlightWindow = z.infer<Schema>`
  assertion is what let `cascade` through: a schema with a *narrower* `kind` union is
  still assignable. Only the reverse direction catches a missing member.

`parseSpansSection()` returns a discriminated `SpansSection`
(`ok` | `absent` | `legacy` | `invalid`), never `null` — a pre-id trace must read as
"re-capture to see the call tree", not as "no spans in flight".

## The tree is exact, not heuristic

Each `FlightSpan` carries a per-instance `id` and the `parentId` of the enclosing
entry *run*. `web/internal/build-tree.ts` indexes `open ∪ completed` by id (they never
overlap — a span leaves the open-entry registry before it enters the flight ring) and
links by `parentId`, so two concurrent runs of the same `(kind,label)` land under their
own parents. A parent always opens before its child, so `parentId < id` holds by
construction; the builder **refuses to link a back-edge** (`parentId >= id`), which
makes a cycle unrepresentable even from a corrupt payload. Unresolvable parents become
flagged orphan roots (evicted <5 ms parent, or a detached child).
