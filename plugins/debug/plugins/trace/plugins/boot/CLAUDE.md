# boot

Built-in trace event class for **slow server boots**. When a worktree backend's
whole boot (profiler `totalDurationMs`) exceeds `debug/boot-monitor`'s
`totalBootBudgetMs`, the monitor builds a `BootSection` from `getProfilingData()`
(+ the gateway's readiness report when one was POSTed) and fires
`captureTrace({ kind: "boot", label: "server-boot", detail: <BootSection> })`
(and records the `slow_ops` row). This class surfaces that section as a lane on
the resulting trace.

**Trigger-owned, like `stall`.** The other built-in classes read ambient
in-memory state at the trip instant; the boot profile is instead *pre-aggregated
by the producer* — the mint happens on a minute tick long after boot completed,
so there is no live instant to capture. The section arrives via `trigger.detail`
and `captureAtTrip` is a cheap passthrough gated on the `"boot"` kind; every
other trip returns `undefined`.

**Own clock axis.** All section offsets are relative to `wallStartMs` (epoch ms
of process start = `Math.round(performance.timeOrigin)`, the boot-events pairing
key); the gateway fields are raw epoch ms from a *different process*, re-anchored
by the lane as `value − wallStartMs` (spawn-request legitimately lands at a
negative offset). The lane is therefore a **self-contained Gantt card** with its
own `GanttContainer` — the boot never positions on the trace window's axis (the
engine clock-domain rule).

`BootSection` (`core/`) is the single source of truth for the shape, shared by
the `debug/boot-monitor` producer that builds it, this class that validates it,
and the lane that parses it. `BootGatewaySchema` doubles as the body of the
monitor's `POST /api/boot/gateway-report` — every gateway field is optional
because the gateway binary versions independently of the backend (user-gated
recompile), so partial reports from either side of the skew must stay valid.
