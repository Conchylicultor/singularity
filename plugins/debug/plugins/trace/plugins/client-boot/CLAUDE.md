# client-boot

Built-in trace event class for the **browser's own boot decomposition** on a
slow page load. The page-load slow-op beacon (`debug/slow-ops`'
`SlowOpCollector`) attaches `toClientBootSection(getBootTrace())` â€” the
`perfs/boot-trace` capture, trimmed â€” as the optional `clientBoot` field of its
body; the client slow-op endpoint forwards it into the `page-load` trigger's
`detail`, and this class persists it as `snapshot.events["client-boot"]`. One
slow visit = one trace carrying the client evidence *inside* â€” previously the
beacon's trace held only server lanes and the client decomposition was ephemeral
unless a human clicked "Copy permalink" in Debug â†’ Boot Profile.

**Trigger-owned, like `stall`.** The server has no ambient client state to read
at the trip instant â€” the browser pre-builds the section and hands it in via
`trigger.detail`. `captureAtTrip` is a cheap passthrough gated on the
`"page-load"` kind AND payload presence: an older client (or any non-page-load
trip) yields `undefined`, so the section is omitted â€” never faked.

**The trim.** `assets` (Resource Timing rows) is the one unbounded part of a
`BootTrace`; `toClientBootSection(trace, maxAssets = 20)` keeps the top-N by
transferSize and folds the full byte totals + dropped count into `assetRollup`,
keeping the beacon far under the 64KB fetch-keepalive cap without silently
hiding transfer cost (the boot-profile Gantt's own cap+rollup rule, applied at
the wire). `ClientBootSection` (`core/`) is the single source of truth for the
shape, shared by the collector that builds it, this class that validates it,
and the lane that parses it; a compile-time pin keeps it assignable back to the
canonical `BootTrace`.

**The lane reuses `BootProfileGantt`** (`debug/boot-profile`, a pure function of
its `trace` prop): the section minus `assetRollup` reassembles into a
structurally-valid `BootTrace`, so the exact Debug â†’ Boot Profile Gantt renders
inside the trace detail â€” trimmed assets simply paint fewer rows, with the
rollup as a caption. The section's offsets are on the **client's clock**
(`performance.timeOrigin` of the visiting tab), so the lane keeps its own axis,
never the trace window's (engine clock-domain rule).
