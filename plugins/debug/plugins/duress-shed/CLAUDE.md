# duress-shed

The **`duress-shed` report kind**: the post-episode accounting for observability
writes thinned by the duress shed engine
(`plugins/infra/plugins/duress`, Phase C of
`research/2026-07-10-global-congestion-observability.md`).

During a host duress episode, the three observability choke points — trace
capture (`debug/trace/engine`), the slow-op funnel (`debug/slow-ops`), and the
report funnel (`reports`) — shed durable writes past a first-N-per-cascade
grant into bounded in-memory buffers. After the episode clears, each buffer
flushes: shed items replay through their normal durable path, and the buffer's
`onFlushSummary` files ONE report of this kind carrying the full accounting
(`byCascade` shed/dropped counts, `replayed`, `replayErrors`, the episode's
`setAt`).

## Why the kind lives here, not in `infra/duress`

Import direction. `reports` gates its own durable path on the shed engine, so
**`reports` imports `duress`** — duress can therefore never import reports
(cycle), and the shed engine takes an `onFlushSummary` callback instead of
filing reports itself. The kind spec + renderer need `reports`, so they live in
this debug plugin, which imports both freely. Consumers file the summary with a
plain `recordReport({ kind: "duress-shed", data: { ...summary } })` — no import
of this plugin needed; the payload schema carries a compile-time drift guard
against duress's `ShedSummary`.

## `duressExempt` — the accounting must never shed itself

The kind marks itself `duressExempt: true` (a generic `ReportKindSpec` flag —
the reports engine still names no kind). The flush timer only fires when the
latch reads clear, but a flush is not atomic: duress can re-trip while replay
chunks and the trailing summary run. Replayed items just re-enter the gate and
re-buffer (deferred, never lost) — but the summary is the *only* record of what
was shed, and re-shedding it could drop it on buffer overflow. Exempting the
kind closes that race.

## Fingerprint

`duress-shed:<bufferKind>:<episodeSetAt>` — one row per (buffer, episode), so
the traces / slow-ops / reports buffers and successive episodes never dedupe
onto each other. No notification cooldown: each episode is a fresh row.
