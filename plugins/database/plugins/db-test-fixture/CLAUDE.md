# db-test-fixture

The sanctioned primitive for DB-backed test suites: `createTestDb()` provisions
an isolated, throwaway database on the already-running gateway-owned Postgres
cluster (via admin's public barrel — the same primitives `forkDatabase` and the
backup sources use), hands back a drizzle handle plus a raw libpq connection
string, and drops the database on `drop()`.

Use it whenever a suite must run **real SQL** (triggers, `NOTIFY`, `ON CONFLICT`
upserts, ordering predicates) against a real Postgres in isolation, rather than a
fake `db`. Pass a `prefix` so the ephemeral database stays identifiable in
Postgres logs (e.g. `createTestDb({ prefix: "cf_test" })`).

Import it from a `.test.ts` via the server barrel:
`import { createTestDb } from "@plugins/database/plugins/db-test-fixture/server"`.

**Need drizzle-tracked tables in the throwaway DB?** Do NOT hand-write mirror
`CREATE TABLE` DDL in the test (it drifts from the real schema, and the
`imperative-create-table-allowlisted` check rejects any raw `CREATE TABLE`
outside the imperative allowlist). Apply the real migration chain instead — the
runner is db-parametrized:

```ts
import { runMigrations } from "@plugins/database/plugins/migrations/server";
const t = await createTestDb({ prefix: "my_test" });
await runMigrations(t.db); // the full real schema, exactly what production runs
```

Genuinely imperative tables keep their own DDL helpers (e.g. live-state-snapshot's
`ensureSnapshotTable`), interpolating an `IMPERATIVE_PUBLIC_TABLES` constant.
The `page/editor-collab` suite (`server/internal/doc-store.test.ts`) is the
`runMigrations` precedent.
