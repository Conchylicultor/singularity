# fork

Durable, self-healing per-worktree database forks.

`createConversation` (in `conversations`) enqueues the `database.fork` job
instead of calling `forkDatabase` fire-and-forget. The enqueue is a committed
graphile-worker row, so an interrupted fork (e.g. the backend restarting
mid-`pg_restore`) is re-run when the worker reboots rather than permanently
bricking the worktree. The fork itself (`forkDatabase` in `database/admin`) is
idempotent and atomic-publish (populate a `<target>__forking` temp, rename to
the canonical name as the last step), which is the precondition that makes
durable retry safe.

This lives in its own plugin rather than `database/admin` because
`infra/jobs` already imports `connectionString` from `database/admin`; a
`defineJob` consumer back in `admin` would close an import cycle
(`database/admin → infra/jobs → database/admin`). Extracting the jobs here
keeps the graph a DAG: `database/fork → {infra/jobs, database/admin}`.

`database.fork-temp-sweep` is a scheduled (every 15 min, main-runtime-only)
reaper for orphaned `*__forking` temp DBs left by a fork job that exhausted its
retries. It skips any temp with active connections so an in-flight fork's
`pg_restore` is never dropped out from under it.
