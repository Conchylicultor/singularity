# migrations

## Two kinds of migration

Migrations live in `data/` as `<ts>_<sha8>__<slug>.sql`. The runner
(`server/internal/runner.ts`) applies them in timestamp order and tracks applied
state by the **filename `sha8` hash** â€” it never reads drizzle snapshots. The
`sha8` MUST equal the hash of the file's content, or the runner's identity drifts
from what it executes.

- **Schema migration** â€” generated by drizzle from `schema.ts`
  (`./singularity build --migration-name <slug>`). Carries a sibling
  `meta/<tag>_snapshot.json`. The snapshot links into the linear chain enforced by
  the `snapshot-chain-intact` check. Never hand-edit the SQL: it must match the
  snapshot's DDL, and the push-time hand-edit detector aborts if it doesn't.

- **Data / backfill migration** â€” DML only (UPDATE/INSERT/DELETE), created freely
  with `./singularity build --custom-migration --migration-name <slug>`. No human
  approval gate: two automated checks back it up instead â€”
  `data-migration-dml-only` (proves it contains no schema changes) and
  `migration-applies-clean` (a transactional dry-run against main, rolled back,
  that proves it applies before push â€” see **Pre-push verification** below). It has
  **no snapshot**: a backfill changes no schema, so it is deliberately kept out of
  the snapshot chain. Why this matters:
  - It can never Y-fork against a schema migration main adds concurrently, so it is
    **push-safe** regardless of main movement (no "quiet window" needed).
  - drizzle bases the next schema migration on the last *schema* snapshot, which is
    correct since the backfill has zero schema delta.
  - You **hand-edit** the generated empty SQL to add the backfill. It is re-hashed
    on every build (`rehashBranchLocalDataMigrations` in `cli/bin/migrations.ts`)
    so the filename hash always tracks the content â€” editing it re-applies once per
    DB (with a loud drift warning for the old hash), never a silent skip.
  - It is enforced **DML-only** by the `data-migration-dml-only` check, so it cannot
    smuggle schema changes past `schema.ts`. To change the schema, edit `schema.ts`
    and generate a schema migration instead.

## Backfill that must precede a schema change

A data migration that must run **before** a schema change â€” e.g. wiping rows so an
`ADD COLUMN ... NOT NULL` (or any destructive reshape) can apply â€” must land at an
**earlier timestamp** than the schema migration, since the runner applies in
timestamp order. Don't hand-edit the schema migration to inject the DML (the
push-time hand-edit detector aborts). Instead:

1. Create the data migration FIRST, so it gets the earlier timestamp:
   `./singularity build --custom-migration --migration-name <wipe_slug> --no-restart --skip-checks`,
   then hand-edit it to add the `DELETE` / `UPDATE`.
2. Regenerate the schema migration AFTER it:
   `./singularity build --reset-migration --migration-name <schema_slug>`.
   `--reset-migration` drops the branch-local SCHEMA migration (snapshot-carrying,
   absent from `origin/main`) and regenerates it with a later timestamp, while
   **preserving** the snapshot-less data migration from step 1.

The runner then applies wipe â†’ DDL. (`--reset-migration` is also the documented
recovery for a snapshot-chain Y-fork after rebasing onto main.)

## Pre-push verification

The only way a migration "breaks main" is by erroring during boot
(`onReadyBlocking` â†’ `runMigrations`), so the server never comes up. The
`migration-applies-clean` check (`check/index.ts`) catches that automatically
before push:

- **Fast path** â€” if this branch changes no file under `data/` vs `origin/main`
  there are no pending migrations, so it passes instantly with **no DB
  connection** (the ~99% case).
- **Slow path** â€” otherwise it opens a direct connection to the live **main** DB
  and calls `dryRunPendingMigrations` (`server/internal/runner.ts`), which replays
  only the pending delta (every pending migration in **one** transaction, so a
  later one sees an earlier one's DDL, then `rebuildDerivedViews`) and **always
  rolls back** via a sentinel throw. Applying against main's real schema + data
  reproduces the boot failure exactly, while the rollback leaves main untouched. A
  real apply error is surfaced with the offending filename + the pg error.
- `SET LOCAL statement_timeout = '60s'` is the load-bearing bound on how long the
  dry-run can hold locks on live main; `lock_timeout = '1s'` bounds lock
  acquisition so it can never queue behind live traffic. Postgres DDL+DML are both
  transactional and `LISTEN/NOTIFY` only fires on commit, so a rolled-back apply
  is side-effect-free.
- Caveat: a rolled-back `INSERT` still advances serial/identity **sequences**
  (`nextval` is non-transactional), so a dry-run can leave harmless ID gaps. No
  data changes; nothing to fix.

The check runs uniformly in `./singularity build`, `./singularity check`, and the
push checks step â€” which runs after rebase + migration normalization, i.e. when
the branch-local migration set is final and `origin/main` is fresh.

## Schema files must be synchronously loadable

Schema-glob files (`server/**/internal/{tables,tables-*,schema,schema-*}.ts`) must
be leaf modules that are synchronously `require()`-able: drizzle-kit loads each via
a sync `require()`, so an async-only import (top-level await, e.g.
`lexical`/`@lexical/yjs` reached transitively through a plugin barrel) anywhere in
their import graph makes drizzle-kit throw-but-exit-0 and **silently drop the
table** from migration generation. Keep these files importing only synchronous
modules; the `schema-files-loadable` check enforces this invariant.
