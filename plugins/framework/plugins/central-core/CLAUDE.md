# Central Runtime

The **shared central runtime** — a single long-running process that handles cross-worktree concerns (secrets, OAuth, cross-namespace notifications). Unlike the per-worktree `server/`, there is exactly one instance of central running for the entire host.

## RULES

- **NEVER modify anything in `central-core/` unless the user explicitly instructs so.** Changes here affect all worktrees simultaneously and cannot be sandboxed inside a worktree branch. Always get explicit approval before touching this directory.

## File Structure

```
plugins/framework/plugins/central-core/
├── package.json
├── tsconfig.json
├── core/                 # Public API — the framework types and helpers
│   ├── index.ts          # Barrel re-exporting all public API
│   ├── types.ts          # CentralPluginDefinition, WsHandler, HttpHandler
│   └── resources.ts      # defineResource, Resource, notificationsWsHandler
└── bin/                  # Process entry point (gateway runs `bun bin/index.ts`)
    ├── index.ts          # Bun.serve, route tables, lifecycle, orphan watchdog
    ├── topo.ts           # topoSortPlugins (internal)
    ├── plugins.ts        # Re-exports plugins.generated
    └── plugins.generated.ts  # Auto-generated by plugin-registry-gen
```

## How It Works

1. `bin/index.ts` starts `Bun.serve({ unix: process.env.SOCKET_PATH })` on the Unix socket the gateway hands it.
2. Each plugin declares its routes via a `CentralPluginDefinition` (defined in `core/types.ts`).
3. `bin/plugins.ts` is a flat list of plugin imports — structurally identical to the server and web registries.
4. At startup, plugins are topo-sorted by `dependsOn`, registered sequentially, then `onReady` runs concurrently.
