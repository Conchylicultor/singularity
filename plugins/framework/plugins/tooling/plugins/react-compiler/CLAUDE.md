# react-compiler

Enables the **React Compiler** (v1.0, GA Oct 2025) as a Babel plugin across the
*entire* frontend. The compiler auto-inserts the memoization the codebase
maintains by hand today (~654 `useMemo` + ~475 `useCallback` call sites) with
correct deps by construction — structurally retiring the per-subtree manual-memo
treadmill that idle/no-op live-state pushes keep re-opening. It runs in
`compilationMode: "infer"` (the compiler's default — compiles every inferred
component/hook, uniformly across all plugins) and respects existing manual
memoization via the compiler's `preserve-manual-memoization` rule.

> **Do NOT switch to `compilationMode: "all"`.** `"all"` also compiles plain
> top-level helper functions (e.g. `defineCollectedDir`); the compiler injects
> `useMemoCache` into them, and when such a helper runs at module-eval (outside any
> React render, dispatcher null) the app hard-crashes at boot with
> `Cannot read properties of null (reading 'useMemoCache')`. `"infer"` only touches
> functions named like components/hooks, which is exactly the intended coverage.

## The one functional file

`vite/index.ts` is the only functional file. `web-core/vite.config.ts` discovers
every `**/vite/index.ts` generically (it never names this plugin) and feeds each
default export's result to `@vitejs/plugin-react`'s `babel.plugins`. This plugin
has **no runtime barrel** (no `web`/`server`/`core`/`central` `index.ts`) — it is a
pure build-time contribution, exactly like `improve/element-picker`'s `vite/`
source-location transform.

The contribution is kept **fully self-contained**: a single file with only
`node:*` imports (no `@plugins/*`, no `@babel/*`, no sibling `.ts`), because
`vite.config.ts` loads it via a runtime dynamic `import()` of its absolute path
where the `@plugins` alias and extensionless `.ts` resolution are unavailable. It
resolves `babel-plugin-react-compiler` by **absolute path** via `node:module`'s
`createRequire(import.meta.url)` + `require.resolve(...)` (web-core is ESM) rather
than bare-importing it, and hands Babel the resolved path string.

## Structural on/off switch

There is **no env flag**. Presence of this folder == compiler enabled: the
discovery walk finds `vite/index.ts` and the compiler joins the Babel chain.
**Delete this plugin folder** and the walk finds nothing — React runs with no
compiler, byte-identical to before adoption. That makes rollback a folder delete.

## Ordering constraint (critical)

React requires the compiler to run **first** in the Babel plugin list — other
contributors (e.g. element-picker, which stamps `data-source` / `data-ui-owner`
onto JSX) must run *after* the compiler has restructured component bodies. The
contribution returns the ordered wrapper shape `{ order: -100, plugin: [...] }`
that `vite.config.ts` stable-sorts ascending; `-100` ("must run first") guarantees
the compiler precedes the default `order: 0` contributions regardless of
filesystem discovery order. The consumer reads only the generic numeric `order` —
it never names this plugin.

## React 19 native — no runtime polyfill

react/react-dom are `^19.1.0`, which has **native** compiler support. The
contribution passes `target: "19"`, so **no** `react-compiler-runtime` polyfill is
required (the runtime ships inside React itself).

## Future caveat — `@vitejs/plugin-react` v6 drops Babel

We are on `@vitejs/plugin-react@^4.4.1`, which is Babel-based — the seam this
plugin (and element-picker) rides. A future **v6** bump drops Babel in favor of
the oxc/swc path and would require migrating the compiler (and the whole
`findViteContributions` Babel-contribution seam) off Babel. Out of scope here;
flag before any `@vitejs/plugin-react` major bump.
