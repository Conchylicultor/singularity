# entity-projection-safety

Blocking lint rule `no-hand-rolled-entity-projection`.

## What it catches

A live-state resource loader that hand-writes an **identity row projection** over a
drizzle `db.select()` — i.e. `rows.map((r) => ({ id: r.id, name: r.name, … }))`. This is a
footgun: the moment the table grows a column, the hand-written object silently drops it from
the wire (the bug that dropped `recentSamples` from `slow_ops`). Nothing in the type system
catches the omission, because the loader's return type is the hand-written object, not the row.

## The fix

Define the table with **`defineEntity`** (`@plugins/infra/plugins/entities/server`), which
derives the Drizzle `pgTable` **and** the zod wire schema from one `FieldsRecord`. Then
`table.$inferSelect ≡ z.infer<schema>` **by construction**, so the loader returns
`db.select().from(entity.table)` rows **verbatim** — no projection to forget a column in, and a
dropped/renamed column becomes a `tsc` error. See
`research/2026-06-17-global-fields-unified-entities.md`.

## Why it's a conservative nudge (favors false negatives)

Contributed rules run at `error`, so a false positive breaks the build. The matcher is therefore
deliberately narrow — it fires **only** when the receiver is a name-matched `db.select().from(…)`
chain, the callback is an *all-pure* field copy (every value is `p.x`, `p.x.toISOString()`, or
`p.x as T` — anything else, incl. `?? null`, a ternary, or any other call, is treated as a genuine
transform and skipped), and the whole thing sits inside a `defineResource({ loader })`. It is a
NUDGE, not a guarantee: an evasive projection is intentionally allowed to slip through rather than
risk a false positive.

`ignores` carries the one deferred exception: `tasks/auto-start`'s loader is a pure projection but
its table is owned by `defineExtension` (which does not yet derive a wire schema). It is
allowlisted until `defineExtension` grows a `.schema`.
