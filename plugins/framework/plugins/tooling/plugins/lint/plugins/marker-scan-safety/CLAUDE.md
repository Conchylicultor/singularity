# marker-scan-safety

Contributes two repo-wide ESLint rules that close the two shapes of the
marker-scan string/comment-embedding false-positive class:

- **`no-adhoc-marker-scan`** ‚Äî a `maskSource(src, { strings: false })` call
  (masking comments/regex but **keeping string interiors**) is banned everywhere
  except a small, reviewed allowlist of genuine token-in-string scanners. This
  is the *partially-masked* footgun.
- **`no-adhoc-binding-scan`** ‚Äî a **global** regex literal that hand-rolls a
  `const <name> = <call>(` marker-binding scan is banned outright. This is the
  *fully-unmasked* twin: the exact shape of `const X = pgTable("‚Ä¶")` /
  `const X = defineEndpoint(` / `const X = Pane.define({‚Ä¶})` scanners that run a
  regex straight over RAW source, so a call written inside a comment, string, or
  template literal registers a phantom table/route/pane. Route these through
  `markerCallSpans(maskSource(src), "<marker>")` (or `findMarkerCalls`) and read
  the binding name + string value back from the ORIGINAL by offset.

## Why

Several build-time scanners masked with `{ strings: false }` and then located a
`defineX(...)` marker call to read its first string argument (an id, slot id,
route URL). Because string interiors stay visible, a `defineX("id")` written
INSIDE a string or template literal ‚Äî a test fixture, a docs snippet, a codegen
template ‚Äî is still present in the masked text, so the call-locating regex
matches it and its id is extracted as a **real contribution**: a silent false
positive.

The correct, proven contract for a **marker-value** scan is: mask FULLY, locate
the call over the full mask, and read the value back from the *original* source
by offset. Route these through
[`findMarkerCalls(src, "defineX")`](../../../../../../plugin-meta/plugins/parse-utils/CLAUDE.md)
(`@plugins/plugin-meta/plugins/parse-utils/core`) ‚Äî it masks strings fully and
slices `argsText` from the original ‚Äî or `markerCallSpans(maskSource(src), ‚Ä¶)`
when the surrounding member/group context is also needed (pass a **full** mask
and read values from the original at the returned offsets). A string-embedded
call then vanishes from the masked text; a real call's blanked id is recovered
from the original.

## The rules

`no-adhoc-marker-scan` fires on any `maskSource(..., { strings: false })` call (a
`strings` property whose value is the literal `false`), reporting on that
property.

`no-adhoc-binding-scan` fires on any **global** (`/g`) regex literal whose source
hand-rolls a `const <name> = <call>(` binding scan ‚Äî the tell is a literal
`const ‚Ä¶ =` binding reaching a named call's LITERAL `\(` (a dotted identifier is
allowed, e.g. `Pane\.define\s*\(`). A bare capture-group `(`, an object-literal
binding (`const X = { ‚Ä¶ }`), and the `.table` alias form (`const X = Y\.table`,
no `\(`) are deliberately NOT flagged ‚Äî the first two aren't calls, and the alias
form is a construct detector that is safe once run over masked source. The
`isBindingScanSource(pattern)` predicate is exported for unit tests. Dynamic
`new RegExp(...)` construction is out of scope (rare; parse-utils' own primitives
build their regexes there and must not self-flag).

## Sanctioned exceptions (on the `ignores` allowlist)

`no-adhoc-binding-scan` needs no allowlist ‚Äî every marker-binding scanner routes
through `markerCallSpans` / `findMarkerCalls`. The `no-adhoc-marker-scan`
allowlist covers the two shapes where `{ strings: false }` is correct:

`{ strings: false }` is correct in exactly two shapes, both allowlisted here:

1. **Token-in-string scan** ‚Äî the token legitimately lives inside a real string
   literal with **no enclosing marker call** (an `/api/<prefix>` URL a caller
   passes to `fetch(...)`, a MIME string, a hardcoded path). Masking fully would
   erase the very thing being searched for. Prefer
   `grepCode({ maskStrings: false })`; the residual fixture false positive is
   inherent and mitigated by skipping test files. Example: the routes facet's
   `relate()` caller scan.
2. **Careful dual-mask** ‚Äî statement structure is detected on a FULL mask, and
   only the final text is sliced from the strings-kept copy at identical
   offsets. Example: the plugin-boundaries statement splitter.

The remaining allowlist entries are the parse-utils unit tests that exercise the
`{ strings: false }` behavior itself.
