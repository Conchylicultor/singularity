# git-grep-safety

Contributes the repo-wide `no-adhoc-git-grep` ESLint rule: hand-rolling a bare
`git grep` — a `spawn`/`spawnSync` whose argv begins `["git", "grep", …]`, or a
`git grep …` shell command in a string/template — is banned everywhere except
the one sanctioned home, `grep-code.ts`.

## Why

A bare `git grep` only searches **tracked** files, and it is unaware of the
check-runner's scan-tree (`withScanTree`) cache semantics. So a check that
discovers its candidate files with a raw `git grep` silently misses any
not-yet-committed source — precisely the newly-created, still-untracked file an
agent produces when adding a pane/route/endpoint. The check passes, the app
ships, and the invariant only fails at runtime. Worse: under a cached run the
recorded PASS is keyed on a tree hash that *did* include the missed file, so a
green cache entry is recorded against content the check never actually
inspected.

Route candidate discovery through
[`grepCode` / `listCandidateSources`](../../../checks/CLAUDE.md)
(`@plugins/framework/plugins/tooling/plugins/checks/core`). Their shared
`readCandidates` helper is scan-tree-aware (it greps the hashed tree — which
includes untracked files, added via `git add -A` into a throwaway index — and
reads blobs with `git cat-file --batch`) and, in the uncached fallback, greps
the working tree with `--untracked` added. Either way, not-yet-committed files
are seen.

The detection is command-position anchored (argv-array shape for spawns; a
`git grep` at the start of a string or right after a shell separator), so it
stays precise: an unrelated `git add` / `git rev-parse` spawn and prose that
merely mentions the token are not flagged. The sole allowlisted home is
`grep-code.ts`, which owns the `["git", "grep", …]` plumbing every other caller
must route through.
