# table-defs-in-schema-glob

Enforces that every concrete drizzle table definition lives in a file matched by
drizzle-kit's schema glob (`server/**/internal/tables.ts`, `tables-*.ts`,
`schema.ts`, or `schema-*.ts`). drizzle-kit discovers tables by filename glob, so
a `pgTable(...)` — or a table-factory call like `defineLink(...)` — in any other
server file is invisible to migration generation: drizzle treats the table as
dropped and emits a spurious `DROP TABLE` with no error. The glob-matched file
set is read from `drizzle.config.ts` (single source of truth, no duplication),
and the set of table factories lives in the `TABLE_FACTORIES` const in
`check/index.ts` — register a new server-defined factory there (both `name` and
`definedIn`) so the footgun can't be reintroduced silently.

## Imperative-table read handles are exempt

A public table created **imperatively** on boot (`CREATE TABLE IF NOT EXISTS`,
NOT through drizzle — e.g. a derived-tables materialized rollup like
`task_latest_conversation`) must live OUTSIDE the schema glob (drizzle-kit would
otherwise emit a spurious migration for it), yet a loader still wants a typed
`pgTable(...)` READ handle. The footgun this check guards against —
"silently vanishes from migration generation → spurious DROP" — does not apply,
because the table was never drizzle-managed in the first place.

So a `pgTable(<CONST>, ...)` whose **name argument is one of the
`IMPERATIVE_PUBLIC_TABLES` name constants** (read from the same single-source
allowlist the `orphaned-db-tables` check uses,
`plugins/database/plugins/derived-views/core/internal/imperative-tables.ts`) is
exempt. String-literal table names are never exempt — those go through the
schema glob. To add an imperative-table read handle: add its name constant to
`IMPERATIVE_PUBLIC_TABLES`, create the table imperatively, and pass that constant
to `pgTable(...)` in a non-glob file.
