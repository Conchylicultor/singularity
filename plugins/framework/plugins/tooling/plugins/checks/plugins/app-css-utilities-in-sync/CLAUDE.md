# app-css-utilities-in-sync

Fails if
`plugins/primitives/plugins/css/plugins/ui-kit/web/theme/custom-utilities.generated.ts`
drifts from the `/* twmerge: … */` markers in
`plugins/primitives/plugins/css/plugins/ui-kit/web/theme/app.css`.

`app.css` is the **single source of truth** for the custom-`@utility` twMerge
registry: each `@utility` carries a co-located marker declaring how tailwind-merge
must classify it, and the registry (consumed by `cn()` in `lib/utils.ts`) is
**generated** from those markers by `./singularity build` — mirroring the
`token-group-vars-in-sync` codegen pattern. There is no hand-maintained
name-mirroring array anymore, so the old membership-drift bug class is structurally
impossible.

The check re-runs the generator in-memory (`renderCustomUtilities(root)`) and
string-compares it to the committed generated file:

- A thrown **marker-validation** error (an `@utility` with no marker, an `sg-id`
  ref with no matching `/* @twmerge group … */` decl, an unknown built-in group id,
  or an empty `standalone` reason) is reported as a check **failure**, located on
  the offending `@utility`.
- Otherwise, a byte-diff between the committed file and the freshly-rendered output
  fails with the standard "run `./singularity build` and commit" hint.

To add a custom `@utility`: declare it in `app.css` with a co-located marker — one
of `/* twmerge: extend <builtin> */`, `/* twmerge: <sg-id> */` (with a
`/* @twmerge group <sg-id> conflicts: … */` decl in the section header), or
`/* twmerge: standalone -- <reason> */` — then run `./singularity build`. twMerge
wiring is automatic; no other file is touched.
