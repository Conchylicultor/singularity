# no-hardcoded-colors

Bans hardcoded colors in plugin code (`plugins/**/*.{ts,tsx}`). Three forms are caught: raw Tailwind
color-scale classes (`bg-emerald-600`, `dark:text-red-950`), arbitrary color literals in utilities
(`bg-[#abc]`, `bg-[oklch(…)]`), and Tailwind-named CSS vars (`var(--color-emerald-500)`).

There is **no allow-list** — every match must be fixed. Pick the right token:

- **Semantic** — `success` / `warning` / `info` / `destructive` / `muted` / `primary`. Use these
  whenever the color conveys a state (ok/bad/caution/in-progress/neutral/brand-action).
- **Categorical** — for genuinely categorical data-viz where N distinct, meaningless-but-different
  hues are needed (Gantt phases, model tiers, runtime/HTTP-method badges, hash-assigned chips), use
  the themeable categorical palette: `bg-categorical-1` … `bg-categorical-10` / `text-categorical-N`
  (the `ui-tokens-categorical` group). Cycle by index, hash into it, or assign fixed indices — but
  the hue values live in the theme, never in component code.
