# data-migration-dml-only

Enforces that **data/backfill migrations are DML-only**. A data migration is a
`.sql` in `plugins/database/plugins/migrations/data` with **no** sibling
`meta/<tag>_snapshot.json` (created via `./singularity build --custom-migration`).
Such migrations are intentionally decoupled from the drizzle snapshot chain so
they never Y-fork against schema migrations on main — but that also means drizzle
does not validate them. This check is the guardrail: every statement in a
snapshot-less migration must begin with an allowlisted DML keyword
(`UPDATE/INSERT/DELETE/WITH/SELECT`, or `SET LOCAL`). The allowlist is
default-deny, so schema-changing DDL (including `CREATE EXTENSION`, `DO $$…$$`,
`EXECUTE`, `CREATE TABLE AS`) cannot slip through and silently drift the database
from `schema.ts`. To change the schema, edit `schema.ts` and let drizzle generate
a tracked migration instead.

## Statement splitting is trivia-aware

Splitting on `;` is only correct **outside** comments and literals — the same
rule [`checks/CLAUDE.md`](../../CLAUDE.md) states for every source-scanning
check. `maskTrivia` blanks comment bodies, `'…'` literals, `"…"` identifiers and
`$tag$…$tag$` bodies (preserving offsets), and the split reads the mask. So a
backfill may contain a `;`, `--` or `/*` inside a string — e.g.
`regexp_replace(t, '#', '\1;', 'g')`, which a naive `.split(";")` cut in half,
leaving an orphan fragment that failed the allowlist even though the statement
was a plain `UPDATE`.

Masking never weakens the allowlist: it blanks only what is *inside* a delimiter,
so a statement's leading keyword — and any `INTO` in a `SELECT … INTO` — is
always real code. `DO $$ … $$` still leads with `DO` and is still rejected. The
allowlist itself is untouched. `index.test.ts` pins both directions: literals
don't split, and DDL stays visible.
