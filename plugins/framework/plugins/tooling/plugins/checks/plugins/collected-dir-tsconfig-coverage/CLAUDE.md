# collected-dir-tsconfig-coverage

Fails if a collected-dir runtime folder (anything declared via
`defineCollectedDir("X")`) is covered by no `tsconfig*.json` `include` glob. Such
files type-check as orphaned — `type-check` reports them as "belong to no
tsconfig program" — until someone hand-adds `**/plugins/*/X` to the right
tsconfig (this was hit when `composition/` was added).

Every other consumer of the collected-dir set derives from the single source of
truth `discoverCollectedDirs()` (codegen's `standardPluginDirs`, the
plugin-boundaries checker, `plugins-registry-in-sync`). The tsconfig `include`
globs are the one hand-maintained consumer, so they silently drift. This check
re-derives the expected folder set from that same source and asserts each is
covered, making the drift impossible to land silently.

It only asserts that **some** tsconfig covers the folder — it deliberately does
not know each dir's target runtime. `type-check` is the backstop for
*wrong-program* placement (DOM code in a node tsconfig fails to compile); the two
checks compose. Coverage is matched against each tsconfig's **literal** `include`
(via `ts.readConfigFile`, which parses JSONC but does not resolve `extends`), so
the glob must live in the config that actually owns the folder.
