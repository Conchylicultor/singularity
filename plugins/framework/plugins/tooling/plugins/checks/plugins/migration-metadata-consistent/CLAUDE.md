# migration-metadata-consistent

Asserts the three migration-metadata artifact families under
`plugins/database/plugins/migrations/data` cross-reference each other. They are
all keyed by the same `<tag>` (`<YYYYMMDD>_<HHMMSS>_<sha8>__<slug>`):

- `J` = journal tags — `meta/_journal.json` `entries[].tag`
- `S` = `.sql` tags — `data/*.sql` (minus `.sql`)
- `N` = snapshot tags — `data/meta/*_snapshot.json` (minus `_snapshot.json`)

Invariants:

- **`J === S`** — every journal entry has a backing `.sql`, and every `.sql` has
  a journal entry. Reported as two diffs: `orphanSql` (`S \ J`) and
  `orphanJournal` (`J \ S`). Catches a journal entry left behind after its `.sql`
  was deleted (a normal `build` does not regenerate the journal — only
  `--reset-migration`/rename does), and an orphan `.sql` with no journal row.
- **`N ⊆ J`** — every `meta/<tag>_snapshot.json` maps to a real migration.
  Reported as `orphanSnapshot` (`N \ J`; equivalently `N \ S` since `J === S`).
  Catches a ghost snapshot that still sits as a reachable link in the snapshot
  chain.
- **`J ⊆ N` is intentionally NOT asserted.** Data/backfill migrations
  legitimately carry no snapshot (that DML-only contract is owned by
  `data-migration-dml-only`), so a `.sql` without a snapshot must never be
  flagged for the missing snapshot.

The pure verdict logic (`classifyMigrationMetadata(J, S, N) → { orphanSql,
orphanJournal, orphanSnapshot }`) is exported and unit-tested in
`check/index.test.ts`; the impure `readdirSync`/`JSON.parse` stays in `run()`.
It is a pure function of the working tree (no git/DB/drizzle), so it uses the
default tree-hash caching — no `cacheSignature`.

Why each existing migration check misses this gap:

- `migrations-in-sync` only runs `drizzle-kit generate` against `schema.ts`; it
  never reads `meta/*_snapshot.json` and never maps journal → `.sql` (the orphan
  `.sql` slice moved here, so each check owns exactly one invariant).
- `snapshot-chain-intact` walks snapshot `id`/`prevId` links over the snapshot
  id-graph only — it never maps a snapshot **filename** back to a journal entry
  or `.sql`, so a ghost snapshot that is a valid chain link passes.
- `migration-hashes-unique` only groups `.sql` filenames by `sha8`.
- `data-migration-dml-only` only inspects the bodies of snapshot-less `.sql`
  files.
