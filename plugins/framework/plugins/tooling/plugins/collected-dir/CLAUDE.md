# collected-dir

Generic loader for **collected-dir registries** — the build-time counterpart to
the web/server contribution collection.

Every collected runtime (`facet`, `check`, …) marks itself with
`defineCollectedDir("X")` in its `core/`, and codegen emits a `<X>.generated.ts`
exporting a `CollectedEntry[]` (`{ pluginPath, loader, dependsOn }`).
`loadCollectedDir<T>(entries, opts)` turns that entry list into validated, typed
items: it optionally topo-sorts by `dependsOn`, runs every `loader()` via
`Promise.allSettled`, validates each default export (and each item of an array
export) with `opts.isItem`, optionally de-dupes by `opts.dedupeKey`, and warns
(never throws) on a rejected loader.

Used by `loadFacets()` (`facets/core`, `ordered: true`) and `loadAllChecks()`
(`checks/core`, `dedupeKey: c => c.id`).

## Why this is its own leaf plugin

The `CollectedDir` marker (`defineCollectedDir`) lives in `codegen/core`, but the
loader cannot: `codegen → plugin-tree → facets` and `codegen → facets` are
existing import edges, so `facets/core` importing `codegen/core` would close a
cycle. This plugin has **zero cross-plugin imports**, so both `facets/core` and
`checks/core` can depend on it safely.

The caller passes the already-loaded `entries` array, not a module specifier —
some generated registries are imported through a non-literal specifier so the web
bundler cannot statically follow them (see `facets/core/load-facets.ts`), and a
relative specifier inside this leaf would resolve against the wrong module.

`eslint.config.ts` does **not** use this helper: jiti (ESLint's config loader)
cannot resolve `@plugins/*` aliases, so the `lint/` walk reconstructs absolute
paths itself and uses throw-on-failure semantics.
