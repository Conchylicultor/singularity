# reorder-defaults

Review-pane section that lists staged reorder **"default for everyone"** edits and
lets the reviewer **Apply** (write the committed `config/<plugin>/<slot>.jsonc`
override, riding the next `./singularity push`) or **Discard** them.

The staged edits are worktree-global (the staging table is per-worktree, not
per-conversation), so the section ignores `conversationId`/`source` for filtering
and shows every staged slot.

## How it works

- `useResource(stagedReorderDefaultsResource)` (from `@plugins/reorder/plugins/staging/web`)
  is the live list of staged slots. Apply/Discard go through
  `useApplyReorderDefault` / `useDiscardReorderDefault` — both take the `:slotId`
  path param: `mut.mutate({ params: { slotId } })`. On success the live resource
  auto-updates (the server notifies), so there is no manual refetch.
- The before→after diff is rendered with `diffReorderTrees(contributions, before, after)`
  from `@plugins/reorder/web`. The live `contributions` for a slot are read
  generically from `PluginRuntimeContext.bySlot.get(slotId)` (the same registry
  `useConfig`/`<Slot.Render>` use) — never by naming a slot owner.
- **`before`** = the current committed git-layer default, fetched via the
  config_v2 `getConfigRawFile` endpoint (its `gitOverride`, falling back to
  `gitOrigin`) and JSONC-parsed for `items`. **`after`** = the staged row's
  `items` tree. Caveat: an unparseable/missing committed document falls back to
  an empty tree, which `applyTree` renders as the full natural-order catalog
  baseline (nothing is dropped, but every item then reads as "shown").
