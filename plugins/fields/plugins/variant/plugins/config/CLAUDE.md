# variant / config

The **variant** type's config capability: the `variantField` factory + the pure
`validateVariant` helper (core) and the type-dispatched renderer contributed to
`config-v2.fields.renderer` (web).

The factory builds a config_v2 `FieldDef<VariantValue>` carrying the canonical
`fields/` variant token (`@plugins/fields/plugins/variant/core`, id `"variant"`);
dispatch is by id string. Its zod schema is opaque/passthrough
(`z.object({ type: z.string() }).passthrough()`) — only the `type` discriminant
is validated at the config boundary, the payload survives verbatim.

Per-type validation is downstream and web-injected: `variantField({ useVariants })`
carries an optional **web-only** `useVariants(): Map<string, { label, fields }>`
hook (a function type only — the core module pulls no React). The pure
`validateVariant(value, variants)` helper resolves the chosen type's `fields`,
runs `fieldsToZodObject` (from `@plugins/fields/core`) over the payload (value
minus `type`), and fails-soft on an unknown type. The web renderer reads `field.useVariants?.()`: present → a type
selector plus the selected type's sub-fields recursed via `FieldRenderer`;
absent → a read-only display of the current `type`. `FieldRenderer` /
`FieldRendererComponent` / `FieldHeader` come from the slot owner
(`@plugins/config_v2/plugins/fields/web`).
