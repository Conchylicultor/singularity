# server-capabilities-loader

The eager-load trigger for the server-owned field capabilities. Its barrel imports
a generated side-effect manifest (`internal/eager.generated.ts`) of every
`fields/plugins/<type>/plugins/{storage,filter-sql}/server` barrel, so each
`Fields.Storage(...)` / `Fields.FilterSql(...)` self-registers into the
`server-capabilities` eager index at module-eval — before any `defineEntity` body
runs — with NO filesystem scan.

This lives in its own plugin (not in the token-owning `server-capabilities`
library) precisely so that library stays a graph **sink**: importing the capability
barrels there would recreate the `fields ⇄ fields/<type>/{storage,filter-sql}`
import cycle (the barrels import the tokens back). Closure here is
`loader → barrels → server-capabilities` — a clean DAG.

Every eval-time consumer of `resolveFieldStorage` / `resolveFieldFilterSql` imports
this barrel for side-effect (`infra/entities`, `conversations/all-conversations`),
so the eager index is populated before their first resolve call. The manifest is
also the composition-closure edge that pulls the capability barrels into every
release bundle that ships those consumers.

The manifest is generated + committed; the `fields-eager-in-sync` check fails on
drift. Fix by running `./singularity build`.
