# fields

The global unified **fields** primitive: a single `type √ó capability` matrix
shared by every data surface (data-view, Sonata, and config_v2 ‚Äî which now sources
its `FieldType` token directly from `fields/core`).

## Mental model

A **field type** is one canonical identity (`FieldType` token + `FieldIdentity`)
registered once in `fields/core`. Each type's *capabilities* (table cell, filter
control, future: editor, validator) are contributed to **capability slots owned
by the consuming surface** ‚Äî never owned by `fields`. This mirrors the facets
`type √ó capability` pattern: the type dimension is owned here, the capability
dimension is owned by each consumer.

- `fields/core` ‚Äî pure, browser-safe sink. The `FieldType` token, the
  `FieldIdentity` (label, icon, `extends`, `coerce`), and `resolveTypeChain` ‚Äî
  the single source of `extends`-chain truth, reused by every capability
  dispatcher. No `fs`, no server imports; only a type-only `ComponentType`.
- `fields/web` (this umbrella barrel) owns the **identity registry slot**
  `fields.identity`. This is the *type-dimension* registry ‚Äî legitimately owned
  by `fields`. Consumers read it **by string-literal id** (`bySlot.get(
  "fields.identity")`), so no surface imports `fields/web` to read identities.

## Capability resolution & `extends`

A derived type (e.g. `int`) declares `extends: <base type>` and registers only
its identity. When a surface looks up a capability (cell, filter) for `int` and
finds none, it walks `resolveTypeChain` to the base (`number`) and reuses its
capability. Leaf plugins under `plugins/` register one type each.

## Storage ‚Äî the one capability `fields` owns directly

The general rule above is that capability slots are **owned by the consuming
surface, never by `fields`** (data-view owns cell/filter; config_v2 owns the
config renderer). **`storage` is the single carve-out.** It is intrinsic to a
type's identity ‚Äî there is exactly one DB column mapping per type, system-wide ‚Äî
so the `fields` plugin owns it directly, on a **new `fields/server` runtime**
(symmetric to `fields/core` and `fields/web`). It lives on the server runtime
specifically to keep `drizzle-orm/pg-core` out of the browser bundle.

- `fields/server` owns the `fields.storage` server-contribution registry (the
  `Fields.Storage` token) and the generic `resolveFieldStorage(typeId)` resolver.
- Each persisted type contributes its Drizzle column builder from a
  `plugins/<type>/plugins/storage/` server sub-plugin
  (`Fields.Storage({ type, build })`), so adding/removing a type updates the
  matrix with zero consumer edits.
- **Resolution is exact-token ‚Äî no `extends` fallback.** Every persisted type
  declares its own builder; `resolveFieldStorage` does a direct keyed lookup.
  (A future derived type wanting the same column re-declares a one-line builder
  rather than inheriting, keeping storage self-contained on the server with no
  need to mirror the web identity/`extends` map.)

## Adding a field type

1. Create `plugins/fields/plugins/<type>/`.
2. `core/` defines the token + identity via `defineFieldType` /
   `defineFieldIdentity`.
3. `web/index.ts` contributes `Fields.Identity({ identity })` and (optionally)
   the surface capability slots (e.g. `DataViewSlots.Cell`,
   `DataViewSlots.Filter`).
4. Run `./singularity build` ‚Äî the registry is autogenerated from the filesystem.
