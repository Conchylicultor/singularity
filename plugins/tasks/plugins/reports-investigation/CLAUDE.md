# reports-investigation

Bridges the `reports` plugin's investigation flow to the `tasks` domain. When a
user clicks "investigate" on a report, `reports` emits an
`InvestigationTaskRequest` into its `reportInvestigationSink` — this plugin
registers the handler that turns that request into a real task filed under the
**Reports** meta-folder (idempotent: an already-linked live task is reused).

## Why it lives under `tasks/`

The bridge is the piece that legitimately holds the `tasks` dependency
(`createTask` / `getTask` / `ensureMetaTask` / `ContainerTask`). Keeping it in the
`tasks` subtree — rather than under `reports/plugins/` — keeps reports' own server
closure free of `tasks`, so reporting can ship in a self-contained released app.
A composition that omits this plugin simply leaves the sink unregistered: reports'
`investigateReport` then throws loudly (a missing capability, not silent).

The mirror of the Edge-1 `defineReportSink` inversion: the **emitter** (`reports`)
owns the sink; the **capability-holder** (this bridge, via `tasks`) registers into
it. The server registry runs `onReady` for every discovered plugin regardless of
who imports it, so this bridge boots and wires itself even though nothing imports
it.
