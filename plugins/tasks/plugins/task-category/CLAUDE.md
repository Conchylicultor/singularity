# task-category

Per-task **category** — the categorical "what kind of task is this" dimension
(conversations, system, agents, improvements, reports, …), replacing the old
meta-folder parenting. **Registry-driven and system-set only**: there is no user
picker; filing plugins stamp the category via `setTaskCategory` when they create
a task.

## Collection-consumer separation

Each filing plugin contributes its own category via the `TaskCategory`
server-contribution — the list is **never** hardcoded in a consumer:

```ts
// plugins/<owner>/server/index.ts
contributions: [TaskCategory({ id: "reports", label: "Reports", order: 4 })],
```

Consumers read only the generic aggregate:

- **Server:** `GET /api/tasks/categories` returns the registered categories,
  sorted by `order ?? 0` then id (static after boot, cached forever client-side
  via `useTaskCategories()`).
- **Web:** the plugin contributes a `category` enum `FieldDef<TaskListItem>`
  into the task-list's `Tasks.Fields` seam (`CategoryField` closes the field's
  `value` over `useTaskCategoryMap()`), so the tasks DataView can group/filter
  by category with no task-list code naming any category.

## Storage + live state

- `tasks_ext_category` is an entity-extension side-table on `tasks` (one row per
  categorized task, FK CASCADE on task delete). `setTaskCategory(taskId,
  category | null)` upserts / deletes.
- `taskCategoriesResource` is the keyed query-resource over the side-table
  (`{parentId, category}` rows, keyed on `parentId`), declared **bootCritical**
  so the default category-grouped tasks view never flashes "None" on first
  paint.
