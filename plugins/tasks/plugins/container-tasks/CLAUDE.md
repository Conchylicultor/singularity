# container-tasks

Single source of truth for the **system container/meta task ids** that must
never own an attempt (the meta folders: conversations, system, agents,
improvements, reports). A container task is a purely organizational folder — its
status is computed from its children, so attaching an attempt directly to it
would silently flip the folder to `in_progress`.

## Collection-consumer separation

Each meta-owning plugin contributes its own id via the `ContainerTask`
server-contribution — the list is **never** hardcoded in a consumer:

```ts
// plugins/<owner>/server/index.ts
contributions: [ContainerTask({ id: SOME_META_TASK_ID })],
```

Consumers read only the generic aggregate:

- **Server (authoritative):** `assertNotContainerTask(id)` throws `HttpError(400)`.
  `createConversation` calls it before `createAttempt` when a `taskId` is
  explicitly provided, so programmatic misuse fails loudly.
- **Web (UX):** `useIsContainerTask(id)` / `useContainerTaskIds()` read the
  cached `GET /api/tasks/container-ids` endpoint so Launch affordances bound to a
  task's own id hide themselves on container rows (the user never hits the 400).

This plugin is a leaf: it imports only the endpoints primitive and the
server-core contribution token, and is imported *downward* by every consumer —
no cycle.
