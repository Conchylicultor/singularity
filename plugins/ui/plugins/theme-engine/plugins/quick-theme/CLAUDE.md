# quick-theme

The **glance-and-switch** half of theming. `theme-customizer` owns the full pane
(every token row, search, per-app forking); this plugin owns the popover behind
the action bar's palette button, where switching a theme costs no context.

Why a sibling plugin rather than a second component inside `theme-customizer`:
the popover needs its own contribution surface (`QuickTheme.Section`), and its
contributors — today `tweakcn/community-browser` — already contribute to the
pane's `ThemeCustomizer.Section`. Keeping the two slots in separate plugins lets
a contributor pick either surface (or both, with a different component each) and
keeps the popover importable without dragging in the pane.

The panel body is assembled from three generic sources, naming no contributor:

1. `QuickTheme.Section` — contributed quick sections (community themes).
2. `ThemeEngine.VariantGroup`, filtered to `selects: "component"` — the pickers
   whose choice SURVIVES a theme swap (sidebar framing, tab bar, progress bar,
   window titlebar). The `selects: "tokens"` half (palette, shape, density,
   fonts, shadow, …) is deliberately absent: picking a theme above rewrites
   every token group wholesale, so showing them here would offer a control the
   user's next click silently overwrites. The full pane still renders both.
3. A footer that navigates to `themeCustomizerRoute` in the active app.

`selects` is a required field on `VariantGroupContribution` rather than a
default, so a new token-group picker cannot silently inherit `"component"` and
reappear in every compact surface — the filter here names no contributor and
needs no maintenance.

**Scope.** The panel wraps its body in `ThemeScopeProvider` with the same scope
the customizer pane resolves (`app:<id>` once that app's theme is forked, else
base), so a quick switch and a pane edit always write the same layer. Forking
itself stays a pane concern — this surface only follows the choice made there.
