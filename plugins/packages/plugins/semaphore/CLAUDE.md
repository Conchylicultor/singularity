# semaphore

Bounded-concurrency primitive: `createSemaphore(max)` returns `{ run(fn, onWait?) }`
that lets at most `max` `run` bodies execute at once, queueing the rest FIFO. A
counter plus a waiter queue — releases the slot in a `finally` so a rejecting
body never leaks one. `onWait(waitMs)` fires once at acquisition with the time
spent queueing (≈0 when a slot was free), so callers can make the gate
observable — e.g. record a profiler span — without coupling this primitive to a
profiler.

Used at the live-state `wrapLoad` seam
(`plugins/framework/plugins/server-core/core/resources.ts`) to cap concurrent
loader bodies below the per-worktree Postgres pool `max`, so a cascade flush's
~10 loaders queue at the semaphore instead of stampeding `pool.connect()`. The
seam records `onWait` as a `db [loader-acquire]` span (sibling to the pool's own
`[acquire]`) so a saturated cap stays visible in the profiler rather than hiding
queue-wait. See `research/2026-06-15-global-live-state-cascade-contention.md`
(Change 4).
