# composition

The **composition manifest registry** — discovers and loads every named
composition declared in the repo. A composition is a `CompositionManifest`
(`{ name, entryPoints, selectedContributors }`, **owned by and imported from**
[`closure`](../closure/CLAUDE.md) — never redefined here); this plugin is the
*registry* that finds them on disk, the engine is the algorithm that resolves
them. Sibling of `closure` / `plugin-tree` / `facets` under `plugin-meta`;
core-only (no web/server, no `definePlugin`).

## Collected-dir, exactly like `check` and `facet`

`core/collected-dir.ts` declares `defineCollectedDir("composition")`. Codegen
(`./singularity build`) discovers every `<plugin>/composition/index.ts`
default-exporting a `CompositionManifest` (or an array of them) and emits
`core/composition.generated.ts` (`compositionEntries`). `loadCompositions()`
turns that into validated manifests via the shared `loadCollectedDir` helper.
The `plugins-registry-in-sync` check keeps the generated file current; no central
registry is hand-edited.

`loadCompositions()` performs **no de-dupe** — duplicate `name`s must fail the
`composition-closure` check loudly, not be silently dropped. The specifier for
the generated module is held in a variable (mirroring `load-facets.ts`) so the
web bundler cannot statically follow it (Studio will import this loader
client-side later).

## Override is forbidden — by construction

The manifest vocabulary is **additive only** (`entryPoints`,
`selectedContributors`). There is no field that replaces or redirects a plugin's
file, so override is *inexpressible*; resolution is a pure union / hard-closure
with no precedence rules. The `composition-closure` check
(`framework/tooling/checks`) adds validity (ids resolve, names unique, every
selection is a genuine load-bearing soft option), but the no-override invariant
needs no enforcement code.

## Where manifests live

A manifest can sit under any plugin's `composition/` folder; co-locate it with
the app it composes. The seed manifests for the anchor demo live in
`apps/plugins/agent-manager/composition/index.ts` — a full `agent-manager` and an
`agent-manager-lean` that differ by exactly the self-improvement contributors.

## Tests

`core/load-compositions.test.ts` (run AFTER `./singularity build`, which emits the
generated registry) asserts the seed manifests are discovered, resolve with no
redundant selections, and that the full-vs-lean bundle difference is exactly the
self-improvement selection.
