# Plugin view — mini file tree explorer

## Context

The plugin detail view (`pluginViewPane`, studio app) currently shows metadata about a
plugin (source path, runtimes, sub-plugins, health, …) via contributed
`PluginView.Section` slots, but offers no way to look at the plugin's actual files. We
want a **read-only mini file tree** in the detail pane so a user inspecting a plugin can
browse its files, and clicking a file opens that file's content in a **new pane
alongside** the plugin view (no navigation away).

This reuses the existing file-tree UI from `code-explorer` rather than building a new one.

### Decisions (confirmed with user)

- **Interaction:** Tree only. Clicking a file opens it in a *new* pane (`filePeekPane`,
  `mode: "push"`) — the plugin view stays visible as its own Miller column.
- **Scope:** Show every file under the plugin's own directory **except** its nested
  `plugins/` sub-directory (sub-plugins already appear as their own nodes in the plugin
  tree, with their own detail pane + file tree).

## Key facts established during exploration

- `PluginView.Section` slot: each contributed section component receives `{ node: PluginNode }`.
  `node.path` is the path **relative to `plugins/`** (e.g. `plugin-meta/plugins/plugin-view`).
  Mirror the simplest existing contributor: `plugins/plugin-meta/plugins/plugin-view/plugins/source-path/`.
- `code-explorer` already has a self-contained `FileTree` component
  (`plugins/code-explorer/web/components/file-tree.tsx`) — props `{ files: readonly string[];
  selectedPath: string; onSelect: (path) => void }`. It builds the tree from a flat path
  list, has its own search + expand state. **Not currently exported from the web barrel.**
- `getCodeTree` endpoint (`GET /api/code/:worktree/tree` → `{ files: string[] }`) is already
  exported from `@plugins/code-explorer/core`. Returns the full worktree file list.
- `filePeekPane` (routable single-file viewer) is exported from
  `@plugins/conversations/plugins/conversation-view/plugins/code/plugins/file-pane/web`.
  Segment `file/:worktree/:filePath*`; open via `useOpenPane`. Outside a conversation it
  degrades cleanly to `status: "clean"`.
- **Worktree addressing gap:** code-explorer endpoints resolve `:worktree` via
  `resolveWorktreePath(id)` — `"main"` → main checkout, anything else → DB attempt lookup.
  There is **no sentinel for the current running server's own root**. But the plugin view
  shows *this* server's plugins (`PLUGINS_DIR` = `<current-worktree-root>/plugins`). We add a
  `"self"` sentinel resolving to `REPO_ROOT` (the current worktree root, distinct from
  `"main"`). `REPO_ROOT` is exported from `@plugins/infra/plugins/paths/server`.
- Boundaries: `boundary-config.ts` allows `plugin.** -> plugin.**`, so `plugin-meta`
  importing `code-explorer` and `conversations/.../file-pane` is permitted. No cycle is
  introduced (neither imports plugin-view).

## Implementation

### 1. Add a `"self"` worktree sentinel to code-explorer

File: `plugins/code-explorer/server/internal/resolve-worktree-path.ts`

```ts
import { getAttempt } from "@plugins/tasks-core/server";
import { ensureMainWorktreeRoot } from "@plugins/infra/plugins/worktree/server";
import { REPO_ROOT } from "@plugins/infra/plugins/paths/server";

export const MAIN_WORKTREE = "main";
// The current running server's own worktree root (distinct from MAIN_WORKTREE,
// which points at the main checkout even when serving from a worktree).
export const SELF_WORKTREE = "self";

export async function resolveWorktreePath(id: string): Promise<string | null> {
  if (id === MAIN_WORKTREE) return await ensureMainWorktreeRoot();
  if (id === SELF_WORKTREE) return REPO_ROOT;
  const attempt = await getAttempt(id);
  return attempt?.worktreePath ?? null;
}
```

- Verify every code endpoint handler (`tree`, `file`, …) routes through
  `resolveWorktreePath` so `"self"` works for both the tree listing and `filePeekPane`'s
  file-content fetch. (The `getFileContent` handler is the one `filePeekPane` depends on.)
- Update the route param doc comment / `code-explorer/CLAUDE.md` reference block to mention
  the new sentinel (regenerated by `./singularity build`).

`plugins/plugin-meta/plugins/plugin-view/plugins/file-tree/web/components/file-tree-section.tsx`:

```tsx
import { useMemo, useState } from "react";
import { Section, type PluginNode } from "@plugins/plugin-meta/plugins/plugin-view/web";
import { FileTree } from "@plugins/code-explorer/web";
import { getCodeTree } from "@plugins/code-explorer/core";
import { useEndpoint } from "@plugins/infra/plugins/endpoints/web"; // confirm hook import path
import { useOpenPane } from "@plugins/primitives/plugins/pane/web";
import { filePeekPane } from "@plugins/conversations/plugins/conversation-view/plugins/code/plugins/file-pane/web";

const SELF = "self";

export function FileTreeSection({ node }: { node: PluginNode }) {
  const prefix = `plugins/${node.path}/`;
  const nestedPluginsPrefix = `${prefix}plugins/`;
  const { data } = useEndpoint(getCodeTree, { worktree: SELF });
  const openPane = useOpenPane();
  const [selected, setSelected] = useState("");

  // Own files only: under the plugin dir, excluding the nested plugins/ subtree.
  // Strip the prefix so the tree roots at the plugin (web/, server/, core/, …).
  const files = useMemo(() => {
    if (!data) return [];
    return data.files
      .filter((f) => f.startsWith(prefix) && !f.startsWith(nestedPluginsPrefix))
      .map((f) => f.slice(prefix.length));
  }, [data, prefix, nestedPluginsPrefix]);

  return (
    <Section title="Files">
      <div className="max-h-96 overflow-auto rounded-md border">
        <FileTree
          files={files}
          selectedPath={selected}
          onSelect={(rel) => {
            setSelected(rel);
            openPane(filePeekPane, { worktree: SELF, filePath: `${prefix}${rel}` }, { mode: "push" });
          }}
        />
      </div>
    </Section>
  );
}
```

Notes:
- Confirm the exact `useEndpoint` import barrel (`@plugins/infra/plugins/endpoints/web`)
  against an existing caller (e.g. how `health-section.tsx` or code-explorer's
  `file-tree-view.tsx` imports it) and mirror it byte-for-byte.
- `filePeekPane`'s `filePath` is a wildcard (`:filePath*`) that captures slashes, so the
  full repo-relative path (`plugins/<node.path>/<rel>`) is correct.
- The `max-h-96 overflow-auto` wrapper keeps the mini tree compact within the detail pane;
  `FileTree` already ships a sticky search box.
