# Auto-generate `plugins/CLAUDE.md` at build time — v2

Supersedes v1. Changes from v1:

- Body is a **single nested list**, not Markdown headings. Nesting depth is unlimited and visually consistent.
- Per-plugin layout is tightened for scanability — fewer label words, contributions inline.
- Plugins may carry an optional `description` field, surfaced in the doc.

## Context

(Same as v1 — `plugins/CLAUDE.md` drifts because it is hand-maintained; all info is statically declared in code; runtime introspection of the two registries is enough.)

## Format — fully nested list

The body between `<!-- AUTOGENERATED:BEGIN -->` / `<!-- AUTOGENERATED:END -->` is one indented list, rooted at top-level plugins. Nested plugins are list children of their parent. No `###`/`####` headings.

### Rules the generator follows

- **Heading-free nesting.** Each plugin is one `- **\`name\`**` bullet; child plugins are nested bullets under a `- Plugins:` line. Indentation = nesting depth, no limit.
- **Description.** Pulled from `PluginDefinition.description` (frontend) or `ServerPluginDefinition.description` (server-only); rendered as ` — <description>` after the name. Omitted entirely if neither exists.
- **Section order per plugin** (omit any that are empty): description → `Defines` → `Contributes` → `Server` → `Plugins`.
- **`Defines` group** collects everything the plugin exposes for others: `Slots`, `Commands`, and `DB schema` (path to `plugins/<path>/server/schema.ts` when it exists). Rendered as child bullets; omitted entirely if all three are empty.
- **Contribution line format.** ``\`SlotId\``` followed by the most identifying prop only:
  - `Shell.Route` → the `pattern` string (`` `/c/:id` ``).
  - Slots with a `title` or `label` → that string in quotes (`"Logs"`, `"VSCode"`).
  - If `group` is set, append `(group \`name\`)`.
  - If a `component` is set, append `→ ComponentName` (using `displayName ?? name`).
  Goal: one short line per contribution, no JSON dumps.
- **Slots / Commands lines** list IDs only, comma-separated.
- **Plugin order** mirrors `web/src/plugins.ts`; server-only plugins (no frontend definition) appended in `server/src/plugins.ts` order.

## Plugin description field

Tiny framework change:

- `plugin-core/types.ts` — add optional `description?: string` to `PluginDefinition`.
- `server/src/types.ts` — same on `ServerPluginDefinition`.

Plugins opt in:

```ts
const buildPlugin: PluginDefinition = {
  id: "build",
  name: "Build",
  description: "Trigger `./singularity build` from the toolbar.",
  contributions: [...],
};
```

The generator prefers the frontend `description`; falls back to the server's; omits the dash if both are absent. Backfilling descriptions for existing plugins is a one-time cleanup, not blocking — missing descriptions just render as bare names.

## Extraction strategy (unchanged from v1)

Runtime introspection of `web/src/plugins.ts` and `server/src/plugins.ts`. Two one-line framework tweaks: attach `.id` to the factory returned by `defineSlot` (`plugin-core/slots.ts:11`) and `defineCommand` so defined slot/command IDs are enumerable from a plugin's `slots.ts` / `commands.ts` exports.

Nesting on disk drives nesting in the doc: a plugin source dir is `plugins/**/web/index.ts`; segments matching `plugins/<x>/plugins/<y>/...` mark `<y>` as a child of `<x>`. Build a tree from the discovered dirs, then render each registered plugin into its tree node. Plugins registered but not found on disk → fail the build with a clear message.

## Verification

1. `./singularity build` — confirm regenerated file matches expectations; spot-check `conversation` (with `vscode` + `open-app` nested), `logs` (WS route), `db-smoketest` (DB schema line).
2. Hand-edit inside the marker block, rebuild — change is overwritten. Hand-edit outside, rebuild — change survives.
3. `./singularity check --plugins-doc-in-sync` passes clean; revert the file manually → fails. `./singularity push` refuses to ship when out of sync.
4. Add a throwaway nested plugin under `plugins/conversation/plugins/foo/web/`, register it, rebuild — appears as a child of `conversation` at the correct depth. Remove → disappears.
5. Add `description` to one plugin, rebuild — description renders; remove the field, rebuild — bare name renders.
