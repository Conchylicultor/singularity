# Branch Conversation Plugin

## Context

After an agent answers, the user often wants to explore multiple follow-up directions without polluting the current conversation's context. Today two plugins exist:

- **fork-conversation** — creates a *blank* conversation in the same worktree (no session context carried over). Contributes to `Conversation.PromptBar`.
- **fork-session** — forks the Claude session (full context) but with no custom prompt, and navigates away. Contributes to `JsonlViewer.RowAction`.

Neither supports the desired workflow: *fork the session with a custom prompt and stay on the current conversation*. A new **branch** plugin combines both: fork the Claude session + send a prompt + remain on the current view. This enables rapid-fire branching from a single conversation.

`fork-conversation` is replaced and deleted. `fork-session` is unchanged (it serves the complementary "fork and navigate" use case).

## Design

### No server changes needed

The `createConversation()` flow already supports `forkFromConversationId` + `prompt` simultaneously:
- `lifecycle.ts:55-73` — inherits `attemptId`, sets `resumeSessionId` + `forkSession: true`
- `tmux-runtime.ts` — assembles `claude --resume <id> --fork-session -- "$prompt"`

### New plugin: `branch`

**Location:** `plugins/conversations/plugins/conversation-view/plugins/branch/`

**Files:**
```
branch/
  package.json
  web/
    index.ts
    components/
      branch-buttons.tsx
```

**Plugin definition** (`web/index.ts`):
- id: `"conversation-branch"`
- Contributes: `Conversation.PromptBar` → `BranchButtons`, section `"Branch"`, sectionOrder `1`

**Component** (`web/components/branch-buttons.tsx`):
- Reads draft via `useDraft("conversation:prompt", "", { scope: conversation.id })` — same key as `prompt-input`
- Uses `useLaunchConversation` directly (not `LaunchButtons`) with `openAfterLaunch: false`
- `getRequest`: `{ forkFromConversationId: conversation.id, prompt?: plainPrompt }`
- `onLaunched`: clears draft, fires `Shell.Toast({ description: "Branch created", variant: "success" })`
- Returns `null` when `conversation.claudeSessionId` is absent
- Buttons disabled when draft is empty (branching requires an intentional prompt)
- Renders `GitBranch` icon + per-model buttons (same MODELS loop as fork-session)
- Button styling: reuse the `size="icon"` branch of `LaunchButtons` (bare `<button>` with `size-6` and hover styling)

**Key imports:**
- `useLaunchConversation` from `@plugins/primitives/plugins/launch/web`
- `useDraft` from `@plugins/primitives/plugins/persistent-draft/web`
- `draftToPlainText`, `ConversationRecord` from `@plugins/conversations/plugins/conversation-view/web`
- `MODEL_REGISTRY`, `ConversationModel` from `@plugins/conversations/plugins/model-provider/core`
- `ShellCommands` from `@plugins/shell/web`
- `WithTooltip` from `@plugins/primitives/plugins/tooltip/web`
- `GitBranch` from `lucide-react`

### Delete fork-conversation

Remove entire directory: `plugins/conversations/plugins/conversation-view/plugins/fork-conversation/`

No other plugin imports from it — verified by grep. `plugins.generated.ts` is auto-regenerated by the build.

## Implementation steps

1. Delete `plugins/conversations/plugins/conversation-view/plugins/fork-conversation/`
2. Create `plugins/conversations/plugins/conversation-view/plugins/branch/package.json`
3. Create `plugins/conversations/plugins/conversation-view/plugins/branch/web/components/branch-buttons.tsx`
4. Create `plugins/conversations/plugins/conversation-view/plugins/branch/web/index.ts`
5. Run `./singularity build`
6. Verify in browser

## Verification

1. Open a conversation with a `claudeSessionId` → Branch buttons (Sonnet/Opus) appear in the prompt bar
2. Open a conversation without a `claudeSessionId` → no branch buttons
3. Type a prompt, click Branch Sonnet → background conversation created (no navigation), draft cleared, "Branch created" toast shown
4. Empty draft → branch buttons are disabled
5. Create multiple branches in a row from the same conversation → each creates a separate background conversation
6. Verify fork-session row action on last assistant message still works independently
