# ST8 — Slim/retire `tabbed-view` for bespoke-only view hosts

> Sub-task ST8 of [`2026-06-15-global-unified-view-switcher.md`](./2026-06-15-global-unified-view-switcher.md).
> Status: **plan**. Tier: optional / sequenced last.

## Context

The unified-view-switcher roadmap split the app's two "view switcher"
implementations: the **generic** named-instance model now lives in
`data-view` + its `view-core` child (config-driven instances), and the
**presentational chrome** is the shared `view-switcher` primitive (ST1).
`tabbed-view` was the old slot-factory whose generic role is now absorbed.

ST8 asks: now that tasks (ST7) and data-view consumers have migrated off it,
what remains of `tabbed-view`, and should it survive?

**Key finding — ST1 already did most of ST8.** Post-ST1,
`tabbed-view`'s `Host` (`define-tabbed-view.tsx`, 97 lines) is *already*
"active component + shared `ViewSwitcher` chrome + device-local active-id +
props pass-through" — the heavy bordered box is gone. Only **two** consumers
remain, both irreducibly bespoke and identical in shape:

- `conversations/conversations-view` — 3 opaque views (queue / grouped /
  history), nav-callback `ViewProps`, `LaunchControl` header.
- `debug/slow-ops/pane` — 2 opaque views (Local / Cluster), empty `ViewProps`,
  no header.

So ST8 is a **decision + one small structural cleanup**, not a rewrite.

### Decisions (confirmed with user, 2026-06-29)

1. **`tabbed-view` survives in its already-slim form, keeping its name.** It is
   the bespoke-only opaque-view host. Retiring it (inlining slot+switcher+active-id
   into each consumer) is rejected — it would duplicate ~30 lines of identical host
   logic across both consumers, re-introducing exactly what a primitive prevents.
   A rename to `bespoke-view-host` is rejected as pure churn (regenerates the whole
   doc cascade, touches both consumers) for no behavior change.
2. **Dedup the active-id hook.** `tabbed-view`'s inline active-id localStorage
   logic (`define-tabbed-view.tsx:40-59`) is byte-identical to
   `view-core/web/internal/use-active-view.ts` (`useActiveViewId`) — same
   `${storageKey}:active-view` key, same DOMException guard. ST6 forked the second
   copy; ST8 reunifies them onto one canonical hook.

### Why `view-switcher` is the shared hook's home

The hook is *the switcher's device-local selection persistence* (its key suffix is
literally `:active-view`), so it belongs with the view-switching primitive, not as
a generic localStorage hook (`persistent-draft` already is that, and its default
**TTL auto-expiry** makes it a wrong fit — the active selection must persist
indefinitely). Co-locating the chrome + its persistence helper in one primitive is
the most discoverable home and avoids a near-empty new plugin. The `<ViewSwitcher>`
**component stays pure-presentational** (selection passed by caller); the hook is a
separate opt-in sibling export.

## Plan

### 1. `view-switcher` — own the canonical `useActiveViewId`

- Add `web/internal/use-active-view.ts` (move the impl verbatim from
  `view-core`'s copy): `useActiveViewId(storageKey): { activeViewId, setActiveView }`,
  reading/writing `${storageKey}:active-view`, DOMException-guarded. Export
  `ActiveViewState` type too.
- Export both from `plugins/primitives/plugins/view-switcher/web/index.ts`.
- The `<ViewSwitcher>` component (`web/components/view-switcher.tsx`) is unchanged
  and stays pure.
- Update `view-switcher/CLAUDE.md` + the plugin `description`: it now offers the
  presentational chrome **and** the opt-in device-local active-id helper
  (component stays pure-presentational).

Files: `plugins/primitives/plugins/view-switcher/web/{index.ts,internal/use-active-view.ts}`,
`plugins/primitives/plugins/view-switcher/CLAUDE.md`.

### 2. `tabbed-view` — consume the shared hook, reframe docs

- In `define-tabbed-view.tsx`, replace the inline `useState` + `selectView` +
  `localStorage` block (lines ~25, 40-59) with
  `const { activeViewId, setActiveView } = useActiveViewId(id);` imported from
  `@plugins/primitives/plugins/view-switcher/web` (already imported for
  `ViewSwitcher` — zero new cross-plugin edge). Drop the local `storageKey` const.
- `onSelect={setActiveView}` on the `ViewSwitcher`.
- Reframe `tabbed-view/CLAUDE.md` to document its post-roadmap role: bespoke-only
  opaque-view host; the generic named-instance role lives in `data-view`/`view-core`;
  chrome + active-id come from `view-switcher`. (The autogen reference block is
  regenerated by build; only the hand-written prose section is edited.)

Files: `plugins/primitives/plugins/tabbed-view/web/internal/define-tabbed-view.tsx`,
`plugins/primitives/plugins/tabbed-view/CLAUDE.md`.

### 3. `view-core` — drop its fork, import from `view-switcher`

- Delete `view-core/web/internal/use-active-view.ts`.
- In `view-core/web/internal/use-view-model.ts`, import `useActiveViewId` from
  `@plugins/primitives/plugins/view-switcher/web` instead of `./use-active-view`.
- Remove `useActiveViewId` + `ActiveViewState` from
  `view-core/web/index.ts`. **Do not re-export them from view-switcher through
  view-core's barrel** — that violates the no-cross-plugin-re-export rule. Verified
  no external plugin imports them from view-core (only `use-view-model.ts` internal
  + the barrel line). This adds a legal `view-core → view-switcher` DAG edge (no
  cycle: view-switcher imports only `toggle-chip`).
- Update `view-core/CLAUDE.md` barrel-API list (remove the two symbols).

Files: `plugins/primitives/plugins/data-view/plugins/view-core/web/{index.ts,internal/use-view-model.ts}`,
delete `.../internal/use-active-view.ts`, `view-core/CLAUDE.md`.

### Out of scope (explicitly)

- No changes to the two consumers' code (`conversation-list.tsx`, slow-ops
  `slots.ts`/`panes.tsx`) — they keep using `defineTabbedView` unchanged.
- No fold of bespoke views into the generic `data-view`/`view-core` model
  (confirmed product decision #1: bespoke stays on a separate primitive).
- No rename of `tabbed-view`.
