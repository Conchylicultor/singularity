# Plugin Graph Visualization

## Context

The repo has ~90 plugins with multiple kinds of relationships between them — imports (cross-plugin barrel imports, enforced as a DAG by `--plugin-boundaries`), slot contributions, server-endpoint calls, event subscriptions, and umbrella parent/child containment. Today this graph is visible only as static markdown (`docs/plugins-details.md`, `docs/plugins-compact.md`) and as per-plugin `CLAUDE.md` reverse-index blocks generated by `cli/src/docgen.ts`. There is no in-app, interactive way to:

- Get an architectural overview ("what does this app actually look like?").
- Answer "if I touch X, what breaks?" — surface the blast radius of a change.
- See how a slot is used / who calls a route / who's downstream of a primitive.
- Spot near-cycles, orphaned plugins, or unexpected coupling.

The intended outcome is a new sidebar entry that opens a sidepane with a global, interactive graph, plus a focused side-pane (Miller column) for any selected plugin showing its full reverse index and deep-links into source. Splitting both the data layer and every visualization feature into independent plugins keeps the surface composable: a future "marketplace publish diff", "boundary-violation heatmap", or "what-if simulator" can reuse the same parsed graph and contribute new lenses/rails without touching the core.

## Decisions

| Question | Decision | Why |
|---|---|---|
| Data plugin location | `plugins/infra/plugins/plugin-inspector/` | Cross-cutting parse-project-state primitive, alongside `git-watcher`/`paths`. Reusable by `publish`, `plugin-graph`, future CLI unification. |
| Visualization plugin | `plugins/plugin-graph/` (umbrella) | Sub-plugins (lenses, rails, filters, toolbar items) live under `plugins/plugin-graph/plugins/`. |
| Render library | `@xyflow/react` + `dagre` | Already in root `package.json` and used by `plugins/tasks/plugins/task-graph/`. Zero new deps; byte-for-byte precedent. |
| Layout | dagre `rankdir: "TB"` for hierarchy lens; `"LR"` for default import lens | Mirrors `task-graph`'s pattern; predictable, stable, scrollable. |
| Umbrella parent/child | Visual nesting via dagre compound graph (`g.setParent`), NOT regular edges | Containment ≠ data flow; would double edge density and mislead. |
| Data freshness | One-shot `fetch` on mount + manual Reload button (mirrors `publish/`) | Graph is filesystem-derived, doesn't change without a build. Live-state via `git-watcher.refAdvanced` is a v-late enhancement. |
| Lens computation boundary | Client-side, pure over `GraphNode[]` | Graph payload is small; keeps lenses zero-server. |
| Sidebar group | `"System"` | Joins `Publish`, `Settings`, `Accounts`. |
| CLI parser unification | Out of scope; mark duplicated parser block with `// DUPLICATED from cli/src/docgen.ts — unify in v-late` | CLI cannot import plugins today; structural unification is its own task. |

## Architecture

### Two top-level plugins

```
plugins/infra/plugins/plugin-inspector/      # data layer (parser + types + endpoint)
plugins/plugin-graph/                        # umbrella for visualization
  └── plugins/<lens|rail|filter|toolbar>/    # one sub-plugin per feature
```

### `plugin-inspector` — data layer

- `shared/index.ts` — exports `GraphPayload`, `GraphNode`, `GraphEdge`, `EdgeKind` (Zod schemas + types).
- `server/index.ts` — registers `GET /api/plugin-inspector/graph`.
- `server/internal/parser.ts` — duplicated subset of `cli/src/docgen.ts`: `findAllPluginDirs`, `collectPlugin`, `collectAllPlugins`, `computeReverseIndexes`, `buildPluginTree`, plus the `stripTypes`/`readIfExists`/regex helpers they depend on. Marked with the `// DUPLICATED …` comment block.
- Imports nothing from `cli/`. Bun's transpiler is available in plugin server runtime.

#### Wire-format (canonical schema; sub-plugins consume these only)

```ts
type EdgeKind = "import" | "slot" | "endpoint" | "event" | "umbrella";
type Runtime = "web" | "server" | "central" | "shared";

interface GraphNode {
  id: string;                         // canonical hierarchy id, e.g. "active-data.conv"
  name: string;                       // last segment
  description?: string;
  loadBearing: boolean;
  parentId?: string;                  // umbrella parent id (containment)
  runtimes: Record<Runtime, boolean>;
  exports: Record<Runtime, number>;   // counts; full lists kept server-side for now
  definedSlots: { id: string; group: string }[];
  httpRoutes: string[];
  resources: { key: string; mode: string }[];
  // Reverse-index sources for client-side lenses:
  apiUses: string[];                  // ids this node imports from
  importedBy: string[];               // ids that import this node
  slotContributors: string[];         // ids that contribute to this node's slots
  endpointCallers: string[];          // ids that call this node's endpoints
}

interface GraphEdge {
  from: string;                       // node id
  to: string;                         // node id
  kind: EdgeKind;
  detail?: string;                    // e.g. slot id, route, runtime
}

interface GraphPayload {
  nodes: GraphNode[];
  // No precomputed edges in v1; lenses derive edges from node fields client-side.
  // Once lens-events lands, server emits explicit eventEdges to avoid leaking trigger internals.
}
```

### `plugin-graph` — umbrella

- `web/index.ts` — registers `pluginGraphPane` + `pluginGraphNodePane` + `Shell.Sidebar` (group `"System"`); re-exports `PluginGraph` slots.
- `web/slots.ts` — defines four slots:
  - `PluginGraph.EdgeLens` — `{ id; label; color; getEdges(nodes: GraphNode[]): GraphEdge[] }`
  - `PluginGraph.NodeFilter` — `{ id; label; predicate(node: GraphNode): boolean }`
  - `PluginGraph.Toolbar` — `{ id; component }`
  - `PluginGraph.Rail` — `{ id; order?: number; title; component }` (mirrors `TaskDetail.Section`)
- `web/panes.tsx`:
  - `pluginGraphPane` — `path: "/plugin-graph"`, content = global graph.
  - `pluginGraphNodePane` — `parent: pluginGraphPane`, `path: "node/:nodeId"`, `chrome: { history: false }`, content = focused rail. Opens as Miller child column on node click. Reads selection via `pluginGraphNodePane.useParams()`.
- `web/components/global-graph.tsx`:
  - Fetches `/api/plugin-inspector/graph` once (mirrors `publish-view.tsx`).
  - Renders `PluginGraph.Toolbar` items in pane chrome via `pluginGraphPane.Actions`.
  - Reads `PluginGraph.EdgeLens.useContributions()` and `PluginGraph.NodeFilter.useContributions()`; toggles live in an internal store (Zustand or React Context — defer to v1 sub-task).
  - Filters nodes through enabled filters → applies enabled lenses → dagre layout → ReactFlow render. Mirrors `plugins/tasks/plugins/task-graph/web/components/task-graph.tsx`.
  - Custom node component shows: name, load-bearing badge (ringed border), runtime badges, click-to-open child pane.
- `web/components/focused-view.tsx`:
  - Reads selected node id from `pluginGraphNodePane.useParams()`.
  - Renders `PluginGraph.Rail.useContributions()` sorted by `order ?? 100` as stacked sections (`stats-panel.tsx` shape).

### Visual encoding (set in v1, refined later)

- Load-bearing: ringed/bolder border on the node card.
- Umbrella vs leaf: subtle background differentiation; v2 lens-hierarchy upgrades this to actual nested clusters.
- Runtime tint: small color dots per runtime contributing exports (web=blue, server=green, central=violet, shared=gray).
- Edge color: per-lens (each lens declares its own color; legend rendered in toolbar).

## Critical files / precedents to reuse

- `plugins/tasks/plugins/task-graph/web/components/task-graph.tsx` — **primary template** for `@xyflow/react` + `dagre` integration in this repo.
- `plugins/publish/server/internal/tree-handler.ts` + `plugins/publish/web/components/publish-view.tsx` — fs walk + one-shot fetch.
- `plugins/stats/web/{index.ts,slots.ts,panes.tsx,components/stats-panel.tsx}` — umbrella + slot + stacked-section render.
- `plugins/conversations/plugins/conversation-view/plugins/tasks-panel/` — toolbar button → child Miller column with toggle.
- `plugins/tasks/plugins/task-detail/` — `Rail`-style section slot pattern (and `order` precedent).
- `cli/src/docgen.ts` — parser to clone in v1 (specifically: `findAllPluginDirs`, `collectPlugin`, `collectAllPlugins`, `computeReverseIndexes`, `buildPluginTree`, `stripTypes`, `parseImports`, `parseBarrelExports`, `matchBracket`, `parseStringField`, `parseBoolField`).
- `plugin-core/types.ts` — `PluginDefinition` shape; the parser already encodes statically computable fields.

## Verification

- After v1: run `./singularity build` (regenerates migrations even though we don't add any; rebuilds frontend; restarts server) → open `http://<worktree>.localhost:9000` → click Plugin Graph in the sidebar → verify ~90 nodes laid out with import edges → click a load-bearing plugin (e.g. `tasks-core`) → child column shows correct importer/contributor counts → URL is `/plugin-graph/node/tasks-core` → reload preserves selection.
- After v2: cycle through edge-lens toggles → graph re-renders with the right edges; hierarchy lens shows umbrella clusters around `infra`/`active-data`/`stats`/etc.
- After v3: every reverse-index entry is clickable and navigates focus.
- After v4: filtering "server only" hides web-only plugins; search jumps to the matched node.
- After v5: event-lens edges connect `git-watcher` ↔ subscribers, etc.
- After v7: `bunx eslint .` (via `./singularity check`) and `plugins-doc-in-sync` both pass; both consumers (CLI docgen, plugin-inspector endpoint) emit identical reverse indexes for at least one spot-checked plugin.

## Out of scope

- Editing the graph from the UI (read-only).
- Persisting per-user filter/lens preferences (will fall out naturally if the umbrella wires its store through the existing `config` plugin later).
- Boundary-violation visualization beyond what `plugin-boundaries.ts` already enforces — separate plan if pursued.
- Migrating `publish/` to consume `plugin-inspector` — natural follow-up but independent.
