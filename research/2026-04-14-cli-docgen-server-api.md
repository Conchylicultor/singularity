# Docgen: document server api.ts exports and consumers

## Context

`cli/src/docgen.ts` generates `plugins/CLAUDE.md` during `./singularity build`. Today it documents plugin descriptions, defined slots/commands, frontend contributions, HTTP/WS routes, and `schema.ts` presence — but is blind to `plugins/*/server/api.ts`, which is the documented inter-plugin contract layer on the backend (e.g. `logs/server/api.ts` exports `Log`, consumed by `build` via `@plugins/logs/server/api`).

Result: the dependency graph between server plugins is invisible in generated docs. This plan adds two pieces of information:

1. **Exports** of each `server/api.ts` (names only).
2. **Forward consumer links**: under each plugin, which other plugins' server APIs it imports.

## Scope (confirmed with user)

- Detail level: **names only** (no kinds, no signatures).
- Consumer rendering: **forward only** — each consumer plugin gets a `Uses:` line. No reverse index under the definer.
- Coverage: **server only** (`plugins/*/server/api.ts`). `web/api.ts` deferred.

## Design

### 1. Discover & parse `server/api.ts`

Extend `findAllPluginDirs` / the per-plugin collection pass in `cli/src/docgen.ts` to also read `<plugin>/server/api.ts` when present.

Extract top-level exported symbol names with a regex pass (consistent with existing style — Bun transpiler strips types first, then regex). Capture:

- `export const X`, `export function X`, `export class X`
- `export type X`, `export interface X`
- `export { A, B as C }` re-export lists
- `export type { ... }` lists

Store as `apiExports: string[]` on the plugin record. Dedupe and sort.

### 2. Trace cross-plugin consumers

Walk every `.ts`/`.tsx` under each plugin's `server/` directory and scan import statements for:

```
from "@plugins/<name>/server/api"
```

(The path alias form is the documented pattern — see `plugins/build/server/internal/handle-build.ts`.) Also accept a trailing `/index` just in case.

For each match where `<name>` ≠ the current plugin, record an edge `consumer -> definer`. Aggregate per plugin into `apiUses: string[]` (sorted list of definer plugin names).

Reuse the existing `parseImports` helper's regex shape if practical, but a dedicated narrow regex over raw file text is sufficient and avoids coupling.

### 3. Render in output

Add two lines to the per-plugin block in the renderer (around `cli/src/docgen.ts:362–399`), placed inside the existing `Server:` section for locality:

```
  - Server:
    - API: `Log`, `LogChannel`, `LogStream`
    - Uses: `logs`, `conversations`
    - `GET /api/logs/channels`
    - `WS /ws/logs`
```

Rules:

- Omit `API:` line if `apiExports` empty.
- Omit `Uses:` line if `apiUses` empty.
- If the plugin has no routes and no api.ts and no uses, do not emit a `Server:` section (current behavior preserved).

### Critical files

- `cli/src/docgen.ts` — add parser for `server/api.ts` exports, add consumer scan, extend render.
- `plugins/CLAUDE.md` — regenerated output (autogenerated block).
- Reference examples while implementing:
  - `plugins/logs/server/api.ts` (definer)
  - `plugins/build/server/internal/handle-build.ts` (consumer: `import { Log } from "@plugins/logs/server/api"`)

### Non-goals

- No signatures, JSDoc extraction, or type rendering.
- No reverse index ("consumed by") under definers.
- No web/api.ts coverage.
- No new AST library — stay with regex + Bun transpiler, matching existing style.

## Verification

1. Run `./singularity build` from the worktree.
2. Inspect `plugins/CLAUDE.md` between the `AUTOGENERATED` markers:
   - `logs` plugin shows `API: Log, LogChannel, LogStream`.
   - `build` plugin shows `Uses: logs` under its `Server:` section.
   - Plugins with no `server/api.ts` show no `API:` line.
   - Plugins with no cross-plugin server imports show no `Uses:` line.
3. Spot-check at least one plugin that has server routes but no api.ts — its `Server:` section should still render routes unchanged.
4. Confirm deterministic output: run `./singularity build` twice, diff should be empty.
