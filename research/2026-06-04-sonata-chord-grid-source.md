# Sonata: Chord-Grid Input Source (authored chords -> derived notes)

## Context

Sonata is a compiler-pattern music app: many input **Sources** compile into one
canonical `Score` IR (`score/core`), and **Displays** render it. The `Score` has
two layers — `notes` (literal pitches) and `annotations` (typed meaning, e.g.
chord symbols) — and the load-bearing idea is that *either layer can be authored
or derived*.

Today the only Source is **MIDI**, which authors `notes` and lets the
chord-analyzer *derive* chord annotations. This exercises only one direction of
the two-layer model.

We are adding a **chord-grid Source** that exercises the *opposite* direction:
chord symbols are authored truth (`annotations`, `source:"authored"`) and notes
are *generated* from them via a voicing strategy. It also exercises **multiple
sources merging into one Score** via the existing `mergeScores` (load MIDI +
chord-grid simultaneously). The intended outcome: type a grid like
`| C G | Am F |`, see chord labels on the timeline and voiced notes on the piano
roll, and have those compose with a loaded MIDI file.

## Verified facts (from code, not assumed)

- `score/core/types.ts` ALREADY exports `ChordData = {symbol, root, quality}`,
  `VoicingData`, `VoicingAnnotation`, `ChordAnnotation`. No new IR types needed.
- `mergeScores(scores)` (helpers.ts): namespaces ids `L{layer}:{id}`; meta /
  tempoMap / timeSigMap = **first non-empty wins, by array order**. `mergeAnnotations`
  appends derived after authored. `scaleTempo`, `beatToSeconds`, `scoreEndBeat`,
  `bars`, `emptyScore` all present.
- `shell/web/context.tsx`: SINGLE active source. State `raw: unknown`,
  `activeSourceId`. Pipeline memo compiles the one active source, runs analyzers
  on it, `mergeAnnotations`. `setRaw(raw)` writes the single `raw`. Cursor resets
  on `[activeSourceId, raw]`.
- `shell/web/slots.ts`: `Sonata.Source = { id, label, icon?, LoaderComponent:
  ({onRaw}) => …, compile: (raw) => Score }`. No voicing slot exists.
- MIDI source: `loader.tsx` (dropzone -> ArrayBuffer -> onRaw), `compile.ts`
  (pure, type-guards ArrayBuffer, emits `annotations: []`), `index.ts` barrel
  (`Sonata.Source({...})`, `satisfies PluginDefinition`).
- Chord taxonomy lives ONLY in `chord-analyzer/web/components/chord-detect.ts`
  (`TEMPLATES`: quality+symbol+intervals; `PC_NAMES`). There is **no**
  string->ChordData parser anywhere.
- `chord-overlay` renders EVERY `chord` annotation regardless of `source`
  (overlay-host filters on `a.type === o.annotationType` only). So authored
  chords render with zero overlay changes — AND duplicate derived chords would
  double-render. The redundancy concern is real and visible.
- **Plugin discovery is filesystem-based.** `web.generated.ts` is regenerated by
  `./singularity build` from each plugin's `<runtime>/index.ts`; `dependsOn` is
  derived from cross-plugin barrel imports. **No manual registry edit** — but a
  build is required, and `plugins-registry-in-sync` fails on drift.
- Sonata sub-plugins are plain `package.json` (name `@singularity/plugin-apps-
  sonata-…`, `private`, `version`, `description` at root, singularity metadata
  under `"singularity"` key). The `sources` umbrella has `"singularity":
  {"collapsed": true}`. `score/package.json` proves a `core`-only plugin needs no
  special fields.

## Decisions (LOCKED — confirmed with user 2026-06-04)

1. **Voicing** → in-source pure registry *now*, but designed so it can be
   promoted to an **extensible slot scoped inside the chord-grid plugin**
   (`ChordGrid.Voicing`, NOT a global `Sonata.*` slot) later. See Decision 2.
2. **Theory home** → new `theory/core` plugin + refactor chord-analyzer to
   consume it. See Decision 1.
3. **Multi-source UX** → accumulate all sources that have input; merge via
   `mergeScores`. See Decision 3.

## Decisions (detail)

### 1. Chord vocabulary home: NEW `theory/core` plugin (not `score/core`)

Recommend a new pure leaf plugin
`plugins/apps/plugins/sonata/plugins/theory/core/` owning the chord vocabulary:
the quality->intervals->symbol table plus `parseChordSymbol(str): ChordData |
null` (forward, new), `qualityToIntervals(quality): number[]`,
`formatChordSymbol(data): string`, and `PC_NAMES`.

Reasoning:
- `score/core` is explicitly documented as "the narrow waist… depends on
  nothing… the DAG leaf" — a minimal IR. A parser + interval tables + a voicing
  algorithm are *theory operations on* `ChordData`, not the IR itself. Keeping
  them out preserves score/core's minimality. `theory/core` may import
  `score/core` (for the `ChordData` type) and stays a near-leaf.
- It DRYs the taxonomy currently hard-coded in `chord-detect.ts`. **Refactor
  chord-analyzer** to consume `theory/core` (replace its private `TEMPLATES` /
  `PC_NAMES` with imports) so the vocabulary has exactly one home — this is the
  "fix the structural issue" win and is the main reason to prefer a shared plugin
  over burying the parser inside chord-grid.
- DAG check: `theory/core -> score/core` (leaf). `chord-grid/web ->
  {theory/core, score/core, shell/web}`. `chord-analyzer/web -> {theory/core,
  score/core, shell/web}`. No cycles. Shell does NOT need theory/core.

Tradeoff acknowledged: it adds one plugin. Worth it because two consumers
(analyzer + grid) share the table and score/core stays a pure IR leaf.

### 2. Voicing strategies: pure in-source registry (NOT a Sonata.Voicing slot)

Keep voicing as a **pure registry** array
`{ id; label; voice: (events: ChordEvent[], opts) => Note[] }` living in
`chord-grid/web/voicings.ts`. The loader imports it to populate the dropdown;
`compile.ts` imports it to apply the chosen strategy.

Why not a `Sonata.Voicing` shell slot + a generation pass symmetric to the
analyzer pass (the alternative the prompt asks me to weigh)?
- `Source.compile` is typed pure `(raw) => Score` and **slot-blind** — it cannot
  read contributions. A slot-based voicing pass would have to move voicing OUT of
  compile into the shell pipeline, making chord-grid's compiled Score *incomplete*
  (authored chords but no notes) until the shell fills them. That forces a new
  ordering contract (voicing must run before analyzers; analyzers then re-derive
  from freshly-voiced notes) and a way to mark "these notes are voiced from
  annotation X" — strictly more machinery.
- The extension axis this task proves is **Sources** (and multi-source merge),
  not voicing. Voicing is an implementation detail of *this* source. Shipping 2-3
  strategies in a local registry demonstrates extensibility without inventing a
  cross-plugin slot.
- A pure registry keeps `compile` self-contained and deterministic, matching the
  MIDI source's shape.

**Future path (per user): make voicing extensible later — as a slot scoped
*inside* the chord-grid plugin, not a global `Sonata.*` slot.** Design for that
now WITHOUT building it:
- The `Voicing` interface (`{ id; label; voice: (events: ChordEvent[], opts) =>
  Note[] }`) is the **stable contract**. A future `ChordGrid.Voicing` slot would
  carry exactly this shape, so promotion is mechanical.
- The only thing blocking a slot today is that `compile` is pure/slot-blind. The
  clean promotion path (when wanted): move voicing application out of `compile`
  and into the chord-grid **loader** (a React component that CAN read its own
  plugin's slot via `useContributions()`), having the loader pre-voice and pass
  `notes` through `raw`; OR keep a thin built-in registry as the fallback and let
  the slot extend it. Either way `ChordEvent` + `Voicing` stay unchanged.
- For THIS iteration: ship the registry as `export const VOICINGS: Voicing[]` in
  `chord-grid/web/voicings.ts`; the loader reads it for the dropdown and `compile`
  reads it to apply the chosen strategy. No slot yet.

Ship strategies: `block-triad` (root-position close voicing of the chord tones,
whole-bar block), `root-position-7th` (adds the 7th when quality has one), and
`arpeggio-up` (chord tones spread across the bar's subdivisions). All build
intervals from `theory/core.qualityToIntervals`.

### 3. Multi-source UX: accumulate-all, ordered by contribution order

Upgrade the shell from single-active to **merge-all-sources-that-have-input**:
- Replace `raw: unknown` with `rawById: Record<string, unknown>`. `setRaw(raw)`
  writes `rawById[activeSourceId]`. `activeSourceId` still only chooses which
  Loader is shown (and is the picker's job).
- Pipeline: compile every source whose `rawById[id] !== undefined`, in
  **`sources` contribution order**, `mergeScores(compiledList)`, then run
  analyzers on the merged Score once, then `mergeAnnotations`.
- Cursor reset dep changes from `[activeSourceId, raw]` to a stable key over the
  *inputs that affect the Score* — i.e. the set of source ids that have raw and
  their raw identities — NOT `activeSourceId` (switching the visible loader must
  not reset the playhead). Use a memoized key like the sorted list of ids with
  input; reset when that membership changes. (Switching source alone no longer
  changes the Score, so it must not reset.)

**Tempo/timeSig ordering policy.** `mergeScores` takes the first non-empty
tempoMap/timeSigMap by array order. chord-grid emits a default 120bpm / 4/4 map;
MIDI emits the real one. To avoid grid's placeholder tempo winning over MIDI's
real tempo, **order the compiled list so note-authoring sources come first**:
sort compiled scores by `notes.length > 0` descending (stable within that). MIDI
(real tempo + notes) then precedes a grid whose tempo is a placeholder. Document
this in `context.tsx`. (Simpler alternative considered — rely on contribution
order with MIDI registered first — is fragile to registration order; prefer the
explicit notes-first sort.)

### 4. Analyzer redundancy: make chord-analyzer authored-aware

After merge, the analyzer sees the merged Score *including* the grid's authored
chord annotations, and would re-derive chords from the voiced notes — producing
duplicate `derived` chords that double-render in the overlay. The pipeline
contract is "an Analyzer never clobbers authored truth," so guard inside
`analyze`:
- Before emitting, gather authored chord coverage:
  `const authored = score.annotations.filter(a => a.type === "chord" && a.source
  === "authored")`.
- Skip any window whose `[start,end)` overlaps an authored chord span (same
  overlap test already used by `isSounding`: `w.start < a.end && w.end >
  a.start`). This drops detection inside authored regions entirely (the simplest,
  cleanest rule), leaving authored truth as the sole chord annotation there while
  still analyzing MIDI-only regions with no authored coverage.
- This is purely additive to `analyze.ts` (one filter + one guard in the window
  loop) and needs no slot/pipeline change.

### NEW plugin: `sources/plugins/chord-grid`

- `chord-grid/package.json` — `name @singularity/plugin-apps-sonata-sources-
  chord-grid`, `description`, `private`, `version`. No npm deps.
- `chord-grid/web/index.ts` — barrel:
  `Sonata.Source({ id:"chord-grid", label:"Chord Grid", icon: MdGridOn (or
  MdViewModule), LoaderComponent: ChordGridLoader, compile })` ->
  `satisfies PluginDefinition`. Imports `Sonata` from shell/web. Default export
  only (registry rule).
- `chord-grid/web/loader.tsx` — `({onRaw}) =>` a `<textarea>` for grid text
  (placeholder `| C G | Am F |`) + a voicing `<select>` populated from the
  voicings registry (`VOICINGS.map(v => …)`) + optional octave input. On change,
  `onRaw({ text, voicingId, octave })`. Debounce or fire on change/blur. Surface
  parse errors visibly (per repo "never swallow" style).
- `chord-grid/web/compile.ts` — pure `compile(raw): Score`:
  1. Type-guard `raw` is `{text: string, voicingId: string, octave?: number}`.
  2. Parse `text`: split on `|` -> bars (trim, drop empties); each bar's tokens
     split on whitespace -> chord symbols; bar length default 4 qn (4/4); each
     chord gets equal subdivision `barLen / chordsInBar`. Accumulate beat
     offsets across bars to build `ChordEvent[] = {symbol, data, start, end}`
     using `parseChordSymbol` from theory/core (skip/flag unparseable tokens).
  3. Emit one `Annotation<"chord",ChordData>` per event, `source:"authored"`.
  4. Look up the voicing strategy from the local `VOICINGS` registry by
     `voicingId`; `voice(events, {octave})` -> `Note[]` on a synthetic track
     `cg0`. Note ids `cg-n{j}`.
  5. Return self-contained Score: `meta:{}`, `tracks:[{id:"cg0",
     name:"Chord Grid"}]`, `tempoMap:[{beat:0,bpm:120}]`,
     `timeSigMap:[{beat:0,numerator:4,denominator:4}]`, voiced `notes`, authored
     chord `annotations`.
- `chord-grid/web/voicings.ts` — pure registry: `interface ChordEvent`,
  `interface Voicing {id,label,voice}`, `export const VOICINGS: Voicing[]` with
  `block-triad`, `root-position-7th`, `arpeggio-up`. Build pitches from
  `qualityToIntervals(data.quality)` + `data.root` + octave base (e.g. C4=60 ->
  `60 + 12*(octave-4) + root + interval`). Velocity ~80.
- `chord-grid/CLAUDE.md` — autogenerated by build.

## Data flow + raw shape (chord-grid)

```
ChordGridLoader textarea+select
  -> onRaw({ text: "| C G | Am F |", voicingId: "block-triad", octave: 4 })
  -> shell setRaw writes rawById["chord-grid"]
  -> compile(raw):
       parse text -> ChordEvent[]  (parseChordSymbol per token, equal bar split)
       events -> authored chord Annotations (source:"authored")
       VOICINGS[voicingId].voice(events) -> Note[] on track "cg0"
       => Score{ default 120/4-4 maps, notes, authored annotations }
  -> shell mergeScores([...notes-first sorted compiled scores])
  -> analyzers.analyze(merged)  (chord-analyzer skips authored-covered windows)
  -> mergeAnnotations -> scaleTempo -> score
  -> piano-roll draws voiced notes; chord-overlay labels authored chords.
```

`raw` shape: `{ text: string; voicingId: string; octave?: number }`.

## Pitfalls

- **Barrel purity**: `index.ts` / `core/index.ts` may ONLY import + re-export own
  files and carry the single default export (chord-grid). theory/core barrel must
  not re-export from other plugins.
- **Default-export imports only in registry files**: only `chord-grid/web/
  index.ts` carries the `satisfies PluginDefinition` default export.
- **DAG**: theory/core -> score/core; chord-grid + chord-analyzer -> theory/core.
  No cycle, shell stays theory-free. Verify with the build (it'll fail on a
  cycle).
- **mergeScores namespacing**: voiced note ids become `L{layer}:cg-n{j}` and
  their `track` becomes `L{layer}:cg0`; authored annotation `target.noteIds`
  remap automatically. Fine — nothing downstream assumes raw ids.
- **Tempo ordering**: grid's placeholder 120bpm must not override MIDI's real
  tempo — hence the notes-first sort before `mergeScores`.
- **Analyzer double-render**: without the authored-aware guard, derived chords
  duplicate authored ones in the overlay. Guard is required, not optional.
- **Cursor reset**: must key on input-membership, not `activeSourceId`, or
  toggling the visible loader resets the playhead mid-play.
- **Unparseable tokens**: `parseChordSymbol` returns null -> skip token + surface
  a visible loader error; never throw inside compile for user typos? (compile is
  pure and may throw on a hard type-guard failure like the MIDI source does, but
  for per-token typos prefer skipping + a count, shown in the loader.)
- **Build required**: a new plugin is invisible until `./singularity build`
  regenerates the registry.

## Implementation refinements (discovered while building)

Two changes to the plan above, made during implementation + verification:

1. **Controlled loaders instead of self-emitting ones.** Each source's
   `LoaderComponent` now receives `raw?` (the source's persisted input) in
   addition to `onRaw`, and the shell exposes `activeRaw`. The chord-grid loader
   is fully *controlled* by `rawById` (no local `useState`), so switching the
   visible source and back never loses the typed grid. The slot contract gained
   one optional prop; the MIDI loader is unaffected (it ignores `raw`).

2. **Chord grid declares no tempo/time-sig (empty maps), no notes-first sort.**
   The planned "order note-authoring sources first" heuristic was wrong: voicing
   *generates* notes, so the grid also has `notes.length > 0` and the sort
   couldn't distinguish it from MIDI. Instead the grid emits `tempoMap: []` /
   `timeSigMap: []` (no authored opinion); `mergeScores`' first-non-empty rule
   then lets a real source (MIDI) own the timeline, and the `score/core` helpers
   fall back to 120 bpm / 4-4 when the grid is alone — same musical result,
   simpler shell (plain `mergeScores(compiled)` in contribution order).
   *Verified:* grid-only advances ≈120 bpm; grid+MIDI advances ≈90 bpm (MIDI's
   authored tempo).
