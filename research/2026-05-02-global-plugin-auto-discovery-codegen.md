# Plugin auto-discovery via build-time codegen

## Context

The lazy `Registration[]` lifecycle landed in commit `de05f47d` — registry writes (MCP tools, runtimes, jobs, events, durable hooks) now flow through topo-sorted phases instead of side-effect-on-import. With ordering no longer load-bearing, the only remaining cost of adding a plugin is editing one of the three hand-maintained registry files:

- `web/src/plugins.ts` — 103 imports, ~208 lines
- `server/src/plugins.ts` — 50 imports, ~102 lines
- `central/src/plugins.ts` — 4 imports, ~19 lines

Every new plugin requires touching one of these. With nesting (`plugins/conversations/plugins/conversation-view/plugins/code/plugins/file-pane/plugins/diff/web`), the entries are noisy and the file is purely mechanical — nothing in it expresses intent that isn't already on disk.

Goal: replace the hand-written list with codegen output derived from the existing `findAllPluginDirs` walker in `cli/src/docgen.ts`. Adding a plugin becomes: create the dir, run `./singularity build`. Background and full design rationale (codegen vs runtime glob, R5 implications, naming) is in [`research/2026-04-29-global-plugin-auto-discovery.md`](./2026-04-29-global-plugin-auto-discovery.md). This plan covers only the auto-discovery half.

## Design

Three pieces:

1. **Generator** — pure function that walks plugins, returns the rendered text for one runtime's `plugins.generated.ts`.
2. **Build hook** — call the generator from `./singularity build` between docs gen and tsc.
3. **Sync check** — `plugins-registry-in-sync` mirrors `plugins-doc-in-sync`: re-renders, string-compares, fails with a `./singularity build` hint.

Generated files are committed to git (drizzle-migration precedent). Hand-written `plugins.ts` files become 1-line re-exports so consumption sites (`web/src/App.tsx`, `server/src/index.ts`, `central/src/index.ts`) don't move.

### Generator

New file: `cli/src/plugin-registry-gen.ts`. Reuses `findAllPluginDirs` from `cli/src/docgen.ts` (currently internal — export it).

Per runtime (`web` | `server` | `central`):

1. Walk `<root>/plugins`; filter to dirs where `<dir>/<runtime>/index.ts` exists.
2. Compute relative path from `<root>/plugins` (e.g. `conversations/plugins/conversation-view/plugins/code/plugins/file-pane/plugins/diff`).
3. Sort by relative path (`localeCompare`) — deterministic, machine-independent.
4. For each, derive an import name: split path on `/`, drop the literal `plugins` connector segments, camelCase the rest, suffix `Plugin`.
   - Example: `conversations/plugins/conversation-view/plugins/code/plugins/file-pane/plugins/diff` → segments `["conversations", "conversation-view", "code", "file-pane", "diff"]` → `conversationsConversationViewCodeFilePaneDiffPlugin`.
   - Names will be longer than today's hand-curated ones (which drop redundant prefixes). That's fine — only the generator reads this file.
5. Emit:

```ts
// DO NOT EDIT — autogenerated by ./singularity build (cli/src/plugin-registry-gen.ts).
// Run ./singularity build to regenerate; the plugins-registry-in-sync check fails on drift.
import type { ServerPluginDefinition } from "./types";
import activeDataPlugin from "@plugins/active-data/server";
// …
import worktreeCleanupPlugin from "@plugins/debug/plugins/worktree-cleanup/server";

export const plugins: ServerPluginDefinition[] = [
  activeDataPlugin,
  // …
  worktreeCleanupPlugin,
];
```

The type annotation differs per runtime (`PluginDefinition` from `@core` for web; `ServerPluginDefinition` / `CentralPluginDefinition` from `./types` for server/central). Otherwise identical shape.

Public API:

```ts
export function renderPluginRegistry(opts: { root: string; runtime: "web" | "server" | "central" }): string;
export function pluginRegistryPath(root: string, runtime: "web" | "server" | "central"): string;
export function generatePluginRegistry(opts: { root: string }): Promise<void>;  // writes all 3
```

### Build hook

`cli/src/commands/build.ts` line 340 area — insert after `generatePluginDocs({ root })` and before the server tsc step (line 347). The generator must run before tsc so the generated file is on disk when type-checking starts.

```ts
console.log("Generating plugin registry...");
await generatePluginRegistry({ root });
```

### Sync check

New file: `cli/src/checks/plugins-registry-in-sync.ts`. Mirrors `cli/src/checks/plugins-doc-in-sync.ts` exactly (read on-disk file, compare to `renderPluginRegistry(...)`, fail with `./singularity build` hint). Three runtimes → three comparisons in sequence; first mismatch returns. Missing-file branch is identical.

Register in `cli/src/checks/index.ts` `CHECKS` array, immediately after `pluginsDocInSync`.

### Plugin-boundaries update

`cli/src/checks/plugin-boundaries.ts:14-20` — extend `FRAMEWORK_FILES`:

```ts
const FRAMEWORK_FILES: ReadonlySet<string> = new Set([
  "web/src/plugins.ts",
  "web/src/plugins.generated.ts",
  "server/src/plugins.ts",
  "server/src/plugins.generated.ts",
  "server/src/index.ts",
  "central/src/plugins.ts",
  "central/src/plugins.generated.ts",
  "central/src/index.ts",
]);
```

Without this, R5 (default-imports-only-in-registry-files) fires on every line of the generated files.

### Hand-written files become re-exports

After codegen lands and once: replace each hand-written `plugins.ts` body with a single line:

```ts
// web/src/plugins.ts
export { plugins } from "./plugins.generated";
```

Same pattern for `server/src/plugins.ts` and `central/src/plugins.ts`. Comments in `server/src/plugins.ts:45-55` (auth-on-central explanation, "order is no longer load-bearing") are dropped — vestigial after this change; the relevant context lives in the auth plugin's CLAUDE.md and the [Registration[] research doc](./2026-04-29-global-plugin-auto-discovery.md).

## Why this is safe

- **Type checking equivalence**: generated file has explicit `import xPlugin from "@plugins/.../runtime"` statements with a `PluginDefinition[]` annotation; tsc verifies each barrel against the type at the array site. Same as today.
- **Order is not load-bearing**: `topoSortPlugins` runs in `server/src/index.ts:14`, `central/src/index.ts:9`, and `plugin-core/context.tsx:18` before any `Registration.register()` invocation. Alphabetical-by-path is purely a tiebreaker for slot rendering order — and "alphabetical by import path" is just as defensible as "the order Claude happened to add them."
- **Slot rendering order changes will be visible** once: anywhere two plugins contribute to the same slot, their relative order may flip. Checked via screenshots during verification.
- **Sync enforced two ways**: the build always regenerates (so post-build the file is fresh); the check fails CI/`push` if a hand-edited or stale file is committed.

## Implementation order

1. **Generator + check, no consumers yet.** Implement `plugin-registry-gen.ts` and the new check. Wire the build to call it. Run `./singularity build` once — it produces `*.generated.ts` files alongside the existing hand-written `plugins.ts` files. Generated files are present on disk but unused; nothing breaks. Commit.
2. **Allowlist generated files in plugin-boundaries** so R5 doesn't fire on them.
3. **Convert hand-written files to re-exports.** One-line re-export, drop vestigial comments. `./singularity build` succeeds; gateway responds; spot-check that slot rendering hasn't broken (toolbar, sidebar, conversation view buttons). Commit.
4. **Verify and push.**

Each step is independently testable and revertible.

## Verification

- `./singularity build` succeeds end-to-end. `*.generated.ts` files match the committed versions (check passes).
- `./singularity check --plugins-registry-in-sync` passes on a clean tree; fails after manually editing a generated file with a clear "out of sync" message and the `./singularity build` hint.
- `./singularity check --plugin-boundaries` passes (no R5 violations on the generated files).
- Add a trivial new plugin under `plugins/scratch/web/index.ts` with a `definePlugin` default export → run `./singularity build` → confirm the new plugin appears in `web/src/plugins.generated.ts` and is reachable in the running app. Remove it; rebuild; confirm the entry is gone.
- Open `http://<worktree>.localhost:9000`, click through: conversation view (toolbar buttons render), sidebar (entries grouped correctly), debug pane, settings pane. No regressions from slot ordering changes.
- `./singularity check --typescript` passes (catches barrel type errors, confirming type checking is intact).
- Smoke-test the full lifecycle: launch a Sonnet conversation (proves runtimes registered), trigger a build (proves jobs/events registered), open MCP tools (proves MCP tools registered) — all of these should work because the lifecycle is unchanged.
