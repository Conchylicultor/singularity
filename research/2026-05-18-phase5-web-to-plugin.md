# Phase 5: `web/` → `plugins/framework/plugins/web-core/`

Implementation plan for the DAG migration phase that moves the Vite SPA entry point into the plugin tree. Part of [`2026-05-12-global-plugin-dag-migration.md`](./2026-05-12-global-plugin-dag-migration.md).

## Context

`web/` is the Vite SPA entry point. It has no cross-plugin consumers — nothing imports from it — making it the simplest phase in the migration.

Two structural changes accompany the move:

1. **Rename `src/` → `web/`** — the source folder takes the plugin zone name (`web/`) instead of the conventional `src/`. This is consistent with how every other plugin in the tree names its frontend zone.
2. **Vite `root` → `"./web"`** — since `index.html` must live alongside the source files rather than at the plugin package root, Vite's `root` is configured to `"./web"`. `public/` also moves inside `web/` so Vite's default `publicDir` (`<root>/public`) continues to work without any extra config.

The path alias depth from `plugins/framework/plugins/web-core/` to `plugins/` is `../../../` — three levels, identical to the server plugin.

## Pre-existing stale alias (fix in this phase)

`web/vite.config.ts` still has `"@server": path.resolve(__dirname, "../server/src")`, pointing to a directory deleted in Phase 4. (`tsconfig.app.json` was already cleaned up.) No file in `web/src/` imports from `@server`, so the alias is dead. **Drop it entirely** in the new config rather than retargeting it.

Mirror the server plugin's structure exactly. From `plugins/framework/plugins/web-core/`, `plugins/` is three levels up (`../../../`).

```diff
-      "@/*": ["./src/*"],

+      "@/*": ["./web/*"],
+      "@plugins/*": ["../../../*"]
     }
```

**exclude** — update depth:
```diff
   "exclude": [
-    "../plugins/plugin-meta/plugins/plugin-tree/**",
-    "../plugins/plugin-meta/plugins/barrel-import/**"
+    "../../../plugin-meta/plugins/plugin-tree/**",
+    "../../../plugin-meta/plugins/barrel-import/**"
   ],
```

**include** — rename `"src"` → `"web"`, retarget globs from new depth, drop `lint` and explicit `web-sdk` entry:
```diff
   "include": [
-    "src",
-    "../plugins/framework/plugins/web-sdk",
-    "../**/plugins/*/web",
-    "../**/plugins/*/core",
-    "../**/plugins/*/shared",
-    "../**/plugins/*/lint"
+    "web",
+    "../../../../**/plugins/*/web",
+    "../../../../**/plugins/*/core",
+    "../../../../**/plugins/*/shared"
   ]
```

Going up 4 levels reaches the repo root. From there `**/plugins/*/web` covers all depths: `**` = zero segments → `plugins/shell/web` (depth-1); `**` = `plugins/tasks` → `plugins/tasks/plugins/task-detail/web` (depth-2); and so on.

`lint` is omitted — lint rule files are ESLint infrastructure, not web runtime code. The explicit `web-sdk` entry is subsumed by the `**/plugins/*/web` glob.

## Execution order

1. Create `plugins/framework/plugins/web-core/web/` directory
2. Move `web/src/*` → `plugins/framework/plugins/web-core/web/`
3. Move `web/public/` → `plugins/framework/plugins/web-core/web/public/`
4. Move `web/index.html` → `plugins/framework/plugins/web-core/web/index.html` (update script tag)
5. Move config files (package.json, tsconfig*.json, components.json, vite.config.ts, vitest.config.ts, CLAUDE.md) → `plugins/framework/plugins/web-core/` (apply all content edits)
6. Update external files (boundary.config.ts, root tsconfig.json, package.json, eslint.config.ts, cli/build.ts, tooling/...)
7. Delete `web/` from repo root
8. Run `./singularity build`
9. Run `./singularity check`

---

## Done when

- `./singularity build` succeeds and SPA loads in the browser at `http://<worktree>.localhost:9000`
- `./singularity check` passes (plugin-boundaries, typescript, eslint, all checks)
- `web/` directory is deleted from repo root
- Worktree spec JSON `web` field points to `plugins/framework/plugins/web-core/dist`
