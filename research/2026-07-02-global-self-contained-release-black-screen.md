# Self-contained release black screen — fix plan

**Date:** 2026-07-02
**Category:** global (composition config + live-state primitive)
**Status:** planned

## Context

A self-contained release (`./singularity release --composition sonata --target web --dev`,
and by extension the Tauri `.app`) boots its backend but renders a **black screen**:
`#root` mounts only the toast host (~2 nodes), the whole app surface is missing.

### Root cause (verified)

The React root (`framework/web-core/web/App.tsx`, `RootRenderer`) renders **every**
`Core.Root` contributor. The one that paints the actual app surface is
`apps-core/plugins/layout` — it contributes `Core.Root({ component: AppsLayout })`,
and `AppsLayout` assembles the tab bar + rail + tab surface.

That plugin is **excluded from the sonata composition bundle** because of how the
closure engine (`plugin-meta/plugins/closure`) works:

- The bundle = the **hard-import closure** of the entry points, plus explicitly
  **selected** soft contributors. Soft contributors are never auto-included.
- `apps-core/plugins/layout` has **zero inbound hard edges** (nothing imports it —
  it only exports a `Core.Root` contribution), **owns no slot**, and its only edge is
  the outbound soft edge *contributor → `Core.Root` owner*. So it is a graph
  dead-end with no path in — it enters a bundle only if the manifest names it.
- `loadBearing: true` is **documentation-only**; the closure engine never reads it.

So sonata's bundle drops the surface renderer and `Core.Root` has nothing to paint but
the toast host. (Once `layout` *is* in the bundle, `tab-bar` / `app-rail` /
`tab-surface` / `tabs` come for free — `AppsLayout` hard-imports them,
`apps-layout.tsx:18–21`.)

### Secondary issue in scope

`/ws/central-notifications` **404** on every self-contained release. `NotificationsClient`
opens the central WS channel **unconditionally** in its constructor, even for an app
with zero central-origin resources. A release ships no central runtime, so that socket
404s and reconnects forever — pinning the overall connection status at `reconnecting`
and tripping `release-boot-verify`'s console-error / request-storm checks.

### Explicitly out of scope (by decision)

- **Guard check** — not adding a composition-closure assertion that app bundles must
  contain the surface. Data fix only.
- **git-watcher "Could not determine main worktree root"** — left as-is. It is a
  backend log line already caught by `startGitWatcher` (does not block boot, does not
  fail the browser-only `release-boot-verify`). Silencing it risks masking a real error;
  a cleaner fix can be designed later if it proves to matter.
- **Making the surface chrome (tab bar / rail / desktop) opt-out** — filed as a
  separate follow-up task (`task-1782988653814-vdgifj`).

## Fix 1 — seed the app surface into `served-baseline` (the black screen)

Every app composition already `extends: ["served-baseline"]` (the `app()` helper in
`plugins/plugin-meta/plugins/composition/core/config.ts:195–212`, plus both
agent-manager profiles). `served-baseline` already **force-seeds `shell.toast`** — another
`Core.Root` host — as an **entry point**, with a comment explaining it must be forced in
or its contributions "silently vanish." `apps-core/plugins/layout` is the same category.

**Change:** add `"apps-core.layout"` as an entry point of the `served-baseline` subsystem.

- File: `plugins/plugin-meta/plugins/composition/core/config.ts`
- Location: the `subsystem("served-baseline", "aN5", [ … ])` array (lines ~158–169).
- Add `"apps-core.layout"` to the entry list (put it first; extend the adjacent comment
  to note it is the mandatory `Core.Root` app surface — without it a filtered app boots
  to a black screen, the same rationale already stated for the toast host).

One line, fixes **all** app/profile compositions at once (all extend `served-baseline`).
Because it is force-seeded as an entry, it must **not** also appear in any app's
`selectedContributors` (the closure check's "no redundant selection" rule would reject
that) — none currently do.

Derived artifacts (committed `config/plugin-meta/composition/compositions.origin.jsonc`,
etc.) are regenerated by `./singularity build`; do not hand-edit them.

## Fix 2 — open the central WS channel lazily (the 404)

Only open `/ws/central-notifications` when a resource whose `origin === "central"` is
actually observed. Sonata has none (`auth` is excluded), so the socket never opens →
no 404, no reconnect storm, overall status stays healthy. This is a structural
improvement for every app (no wasted central socket), not a release-only hack.

- File: `plugins/primitives/plugins/live-state/web/notifications-client.ts`
- Keep the **worktree** channel eager (always used — lowest risk, byte-identical
  behavior). Create the **central** channel lazily on first central-origin `observe()`.
- Concrete shape:
  - Make `this.channels` a partial map; add a private `channelFor(kind): SocketChannel`
    that creates + opens + caches a channel on first use. Call `channelFor("worktree")`
    in the constructor to preserve eager worktree open.
  - `observe()` routes through `channelFor(kind)` (this is what lazily creates central).
  - Read-only paths that assume a channel exists (`etagFor`, `noteHttpEtag`, `unobserve`)
    are safe: you can only reach them for a `(key, origin)` you already `observe()`d, which
    created the channel. Add a defensive early-return if the channel is absent.
  - Methods that iterate all channels (`probeMissedUpdates`, `debugSnapshot`) already use
    `Object.values/entries(this.channels)` — with a partial map they naturally skip an
    unopened central. Guard `leaderInfo("central")` in `debugSnapshot` for absence.
  - `getStatus()` reads the `channelStatuses` map, which only ever holds **opened**
    sockets — so a never-opened central is already excluded from the overall status
    (this is the core of the fix). Keep `getChannelStatuses()` tolerant of an absent
    central (benign default for the debug pane; keep the `ChannelStatuses` type stable).

No server change: the route only exists on the central runtime (`central-core`), which a
release never ships. When central *is* present (e.g. the agent-manager composition, which
includes `auth`), the first auth-resource observe opens the channel exactly as before.

## Verification

1. `./singularity build` — regenerates the composition origin config + registries; must
   succeed.
2. `./singularity check composition-closure` — must pass. Confirms sonata's
   `excludes: ["agent-runtime", "auth"]` still holds after adding `layout` to the baseline
   (layout's hard closure is app-neutral chrome — apps-core / app-rail / tab-bar /
   tab-surface / tabs — and must stay disjoint from agent-runtime/auth). **If this fails,
   it is a real finding**: the surface chrome transitively reaches agent/auth infra.
3. Dev app sanity: screenshot `http://<worktree>.localhost:9000` — app still renders
   normally; central-backed apps (agent-manager) still connect auth (central channel
   opens lazily on first auth observe).
4. Release render:
   - `./singularity release --composition sonata --target web --dev`
   - launch the staged `<dir>/launch` from a fresh `SINGULARITY_DIR`
   - `bun e2e/release-boot-verify.mjs --url http://localhost:<port>/ --settle 15000`
   - Expect **PASS**: `#root` > 10 nodes after settle, zero console errors, no `ws`
     request-storm (no `/ws/central-notifications` requests at all).
