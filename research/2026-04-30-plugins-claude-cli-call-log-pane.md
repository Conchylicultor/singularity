# Claude CLI single-shot call log + debug pane

## Context

The codebase makes short-lived "single-shot" LLM calls (Haiku today, but the helper supports Sonnet/Opus too) via `runClaudePrint` in `plugins/infra/plugins/claude-cli/server/internal/run-claude-print.ts`. These power task-title generation, conversation-category classification, and turn summaries. Today the helper is fire-and-forget: no logging, no DB row, no telemetry. When a Haiku call produces a wrong category or a poor summary, there is no way to inspect the prompt or output after the fact.

We want a **Debug pane** that lists every single-shot call with its prompt, output, timestamp, and a "what launched this" label (e.g. *Task title for task abc…*, *Turn summary for conv X turn 3*). The user will use this to debug Haiku quality in the wild.

## Approach

Two changes:

1. **Instrument `runClaudePrint` itself** so every call writes a row, regardless of caller. Adding telemetry per call site is brittle; wrapping the helper guarantees coverage.
2. **Two-plugin split** that follows the codebase's conventions:
   - **Storage + recording** stays inside `plugins/infra/plugins/claude-cli/`. The plugin already owns the primitive; it should own the log of calls to itself. Adds a `_claudeCliCalls` table, a push live-state resource, and a record-on-call wrapper.
   - **UI pane** lives at `plugins/debug/plugins/claude-cli-calls/` under the existing Debug umbrella, alongside `logs`, `queue`, `db-backup`, `worktree-cleanup`. UI-only — no DB, no endpoints. It consumes the resource exported by `claude-cli`.

This avoids a circular dependency (debug → infra is fine; infra → debug would be wrong) and keeps the infra plugin responsible for its own observability.

We use the **push live-state pattern** (`defineResource({ mode: "push" })` + `notify()`) — same as `plugins/agents/server/internal/resources.ts`. The pane updates in real time as new calls land. The Debug umbrella siblings poll, but push is the modern pattern and the live-update UX is the whole point of this pane.

## DB schema

`_claudeCliCalls` (in `plugins/infra/plugins/claude-cli/server/internal/tables.ts`):

```ts
export const _claudeCliCalls = pgTable(
  "claude_cli_calls",
  {
    id: uuid("id").primaryKey().defaultRandom(),
    createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
    model: text("model").notNull(),                           // "haiku" | "sonnet" | "opus"
    sourceName: text("source_name").notNull(),                // e.g. "task-title"
    sourceContext: jsonb("source_context").$type<Record<string, unknown>>(),
    prompt: text("prompt").notNull(),
    system: text("system"),                                   // null when omitted
    output: text("output"),                                   // null on error
    error: text("error"),                                     // null on success
    durationMs: integer("duration_ms").notNull(),
  },
  (t) => [index("claude_cli_calls_created_at_idx").on(t.createdAt.desc())],
);
```

Migration is generated by `./singularity build` — never run drizzle-kit manually.

## API change to `runClaudePrint`

```ts
export interface RunClaudePrintInput {
  model: ClaudePrintModel;
  prompt: string;
  system?: string;
  timeoutMs?: number;
  source: {                                       // NEW, required
    name: string;                                 // stable label, e.g. "task-title"
    context?: Record<string, unknown>;            // taskId / conversationId / etc.
  };
}
```

Implementation wraps the existing spawn:

```ts
const startedAt = performance.now();
let output: string | undefined;
let error: Error | undefined;
try {
  output = await /* existing spawn logic */;
  return output;
} catch (e) {
  error = e as Error;
  throw e;
} finally {
  const durationMs = Math.round(performance.now() - startedAt);
  void recordClaudeCliCall({
    model: input.model,
    sourceName: input.source.name,
    sourceContext: input.source.context ?? null,
    prompt: input.prompt,
    system: input.system ?? null,
    output: output ?? null,
    error: error ? error.message : null,
    durationMs,
  });
}
```

`recordClaudeCliCall` is `void`-returning and swallows its own errors — recording must never affect the calling path. After insert it calls `claudeCliCallsResource.notify()` and runs trim.

## Retention

After every insert, prune rows beyond the most recent 1000:

```ts
DELETE FROM claude_cli_calls
WHERE id NOT IN (
  SELECT id FROM claude_cli_calls ORDER BY created_at DESC LIMIT 1000
);
```

Hard cap of 1000 — no time-based component. Cheap to run on every insert (one indexed scan); avoids a separate scheduled job. If volume grows, we can move this to a graphile-worker job later.

## Resource + endpoint

Push resource only; no separate REST endpoint. The `defineResource` machinery provides:
- WS push via `/ws/notifications` whenever `notify()` is called
- HTTP fallback on the standard live-state path for cold loads

The loader returns the latest 1000 rows ordered by `createdAt desc`.

Shared schema (`shared/resources.ts`):

```ts
export const ClaudeCliCallSchema = z.object({
  id: z.string().uuid(),
  createdAt: z.string(),       // ISO
  model: z.enum(["haiku", "sonnet", "opus"]),
  sourceName: z.string(),
  sourceContext: z.record(z.unknown()).nullable(),
  prompt: z.string(),
  system: z.string().nullable(),
  output: z.string().nullable(),
  error: z.string().nullable(),
  durationMs: z.number().int(),
});
export type ClaudeCliCall = z.infer<typeof ClaudeCliCallSchema>;

export const claudeCliCallsResource = resourceDescriptor<ClaudeCliCall[]>(
  "claude-cli-calls",
  z.array(ClaudeCliCallSchema),
);
```

## UI structure

Pane title: **Claude CLI Calls**. Sidebar entry under Debug, icon `MdAutoAwesome` (or similar).

Layout:

```
┌────────────────────────────────────────────────────────────────┐
│ filters: [all models ▾] [all sources ▾]      total: 247 calls  │
├────────────────────────────────────────────────────────────────┤
│ ▸ [haiku] [task-title]   2m ago · 412ms                        │
│   "Add per-conversation category chip…"  → "Add category chip" │
│ ▸ [haiku] [conversation-category]  5m ago · 387ms              │
│   "<6 turns…>"  → "Coding"                                     │
│ ▼ [haiku] [turn-summary]  7m ago · 902ms                       │
│   ─ Source context: { conversationId: "abc", turnNumber: 3 }   │
│   ─ Prompt: <full>                                             │
│   ─ System: <full>                                             │
│   ─ Output: <full markdown>                                    │
│ ▸ [haiku] [task-title]  18m ago · ERROR (timeout)              │
└────────────────────────────────────────────────────────────────┘
```

- Newest first.
- Collapsed row: model chip, source chip, relative time (`<RelativeTime>`), duration in ms, single-line output preview (or red error pill).
- Expanded row: source context JSON, full prompt, full system, full output OR error message. Code-block formatting with copy buttons.
- Filter chips on top: model (`all` / `haiku` / `sonnet` / `opus`), source name (populated from observed labels in the data).
- No pagination — capped at 1000 rows by retention.

## Verification

After implementation:

1. `./singularity build` — must succeed, run migration cleanly.
2. Open `http://<worktree>.localhost:9000`, open Debug → **Claude CLI Calls** pane. Pane should render empty.
3. Trigger each source:
   - **task-title**: create a new task with a description → row appears with `sourceName: "task-title"`, `context.taskId` set, output is the generated title.
   - **conversation-category**: launch a conversation, send a turn → after the classify job runs, row appears with `sourceName: "conversation-category"`, output is the category.
   - **turn-summary**: same conversation, send a turn → row appears with `sourceName: "turn-summary"`, context includes `conversationId` and `turnNumber`, output is the markdown summary.
   - **improve-title**: open the Improve form, submit a card → row appears with `sourceName: "task-title"` (improve goes through `scheduleTaskTitleUpdate`), `context.taskId` set.
4. Expand a row — prompt/system/output show in full.
5. Force an error (set `SINGULARITY_CLAUDE_BIN` to `/bin/false` temporarily, or kill claude during a call) → confirm row is recorded with `error` populated and `output: null`.
6. Generate >1000 calls (loop a small task creation) — confirm retention prunes oldest rows.
7. WS push: open the pane in two browser tabs, trigger a call, both update without refresh.

## Out of scope

- Aggregate stats (cost, p50/p99 latency, error rate) — easy follow-up once the log exists.
- Replay / re-run a call from the pane — interesting but separate.
- Filtering by free-text search — start with model + source filters, add later if needed.
- Persisting calls beyond 1000 rows or exporting to JSONL — follow-up if the log proves useful for offline analysis.
