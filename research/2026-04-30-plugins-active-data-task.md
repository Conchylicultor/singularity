# Active-data `task` widget

## Context

We want assistant-generated text to be able to propose new tasks inline. Today, an agent has no way to surface a "here's a task you might want to spin up" suggestion that the user can immediately act on without leaving the transcript. The `improve` plugin gives the user a way to author a card, but there's no equivalent for the *agent* to author one.

The `active-data` plugin already exists for exactly this kind of use case: sub-plugins claim an XML-like tag that the model emits, and the rendered widget gets injected into the assistant text. Adding a `<task>prompt</task>` widget lets any model output a draggable, editable card with **Create** and **Launch** actions — same affordances as `improve`'s box, but generated by the model and inlined where the suggestion was made.

Outcome: when an assistant emits

```
<task>refactor the X module into Y</task>
```

it renders as an editable card. The user can tweak the prompt and click **Create** (just inserts a child task under the host conversation's task) or one of the **Launch** buttons (Sonnet / Opus — pre-creates the task, then immediately spins up a conversation against it). After action, the card collapses to a chip linking to the result.

## Design decisions (confirmed)

- **Parent task**: child of the host conversation's task — same nesting semantics as the `new-child-task` toolbar.
- **Editor**: Lexical `PromptEditor` from `@plugins/primitives/plugins/paste-images/web` (visual parity with `improve`, paste-image support).
- **Post-action UI**: card collapses to a chip — same idiom as `ConvChip`. Created task → task chip linking to its detail. Launched conversation → reuse the existing `ConvChip` for consistency.
- **Tag mode**, not pattern mode: the prompt content is the children, not an attribute, but we want unambiguous boundaries (so the model isn't constrained to a regex shape).

4. **`plugins/active-data/plugins/task/web/components/task-card.tsx`** — the widget.

   Key shape (sketch):
   ```tsx
   import { useId, useState } from "react";
   import { conversationPane } from "@plugins/conversations/plugins/conversation-view/web";
   import { PromptEditor } from "@plugins/primitives/plugins/paste-images/web";
   import { LaunchButtons } from "@plugins/primitives/plugins/launch/web";
   import { ConvChip } from "@plugins/active-data/plugins/conv/web/components/conv-chip"; // see note
   import { Button } from "@/components/ui/button";

   export function TaskCard({ children }: { attrs: Record<string, string>; children: string }) {
     const initial = children.trim();
     const editorNs = useId();
     const { conversation } = conversationPane.useData();
     const hostTaskId = conversation.taskId;

     const [prompt, setPrompt] = useState(initial);
     const [creating, setCreating] = useState(false);
     const [createdTaskId, setCreatedTaskId] = useState<string | null>(null);
     const [createdConvId, setCreatedConvId] = useState<string | null>(null);

     // Collapsed states
     if (createdConvId) {
       return <ConvChip attrs={{}}>{createdConvId}</ConvChip>;
     }
     if (createdTaskId) {
       return <CreatedTaskChip taskId={createdTaskId} />; // small chip → /tasks pane
     }

     const onCreate = async () => { /* POST /api/tasks → setCreatedTaskId */ };

     return (
       <div className="border bg-background rounded-md p-2 my-2 flex flex-col gap-2">
         <PromptEditor
           value={prompt}
           onChange={setPrompt}
           namespace={`active-data-task-${editorNs}`}
           placeholder="What should be done?"
           minRows={3}
           maxHeight="20rem"
         />
         <div className="flex justify-end items-center gap-2">
           <Button variant="ghost" size="sm" onClick={onCreate} disabled={creating || !prompt.trim()}>
             Create
           </Button>
           <LaunchButtons
             size="icon"
             disabled={creating || !prompt.trim()}
             getRequest={async () => {
               const task = await createTaskRequest({
                 parentId: hostTaskId,
                 description: prompt,
               });
               setCreatedTaskId(task.id);
               return { taskId: task.id, prompt };
             }}
             onLaunched={(conv) => setCreatedConvId(conv.id)}
           />
         </div>
       </div>
     );
   }
   ```

   - **Create flow**: `POST /api/tasks` with `{ parentId: hostTaskId, description: prompt }`. On 200, parse with `TaskSchema` (from `@plugins/tasks-core/shared`) and store the id.
   - **Launch flow**: `getRequest` is async — pre-creates the task with the host as parent, then returns `{ taskId, prompt }` to `LaunchButtons`, which posts `/api/conversations`. `onLaunched` collapses to a `ConvChip`.
   - **Empty / whitespace-only `<task>` content**: render nothing (defensive — same as `ConvChip` does for blank ids).

   *Note on the conv-chip import path*: `ConvChip` is currently a deep import that crosses plugin boundaries (`@plugins/active-data/plugins/conv/web/components/conv-chip`), which the boundary checker rejects. **We have two clean options** — pick one during impl:
   - (a) Have `conv` re-export `ConvChip` from its barrel, then import via `@plugins/active-data/plugins/conv/web`. Lightweight; keeps things self-contained.
   - (b) Inline a tiny chip in `task-card.tsx` that opens `conversationPane` directly (no `ConvChip` reuse). Avoids cross-sub-plugin dependency entirely.
   - Default: **(a)** — `ConvChip` is the established post-launch chip idiom and re-exporting it is two lines.

## Constraints to respect

- **Plugin boundaries**: only import `@plugins/<name>/{web,server,shared}`. No deep paths (`/internal/`, `/components/`) across plugins. `ConvChip` reuse must go through `conv`'s barrel.
- **Barrel purity**: `web/index.ts` is import / re-export / type alias / single default-export only.
- **Lexical namespace uniqueness**: `useId()` per instance — multiple `<task>` tags in one transcript must not collide.
- **Tag rules**: lowercase, allowed in raw HTML inside markdown source. `task` is fine.
- **Active-data is rendered inside markdown `<p>`** — using a card with `border`/`rounded-md` and `my-2` matches what `improve-card` does and renders fine inline.

5. **Create**: click → card collapses to a task chip pointing to `/tasks?selected=<id>` (or whichever the task pane URL form is). Verify task exists in the task tree under the host conversation's task.
6. **Launch (Sonnet)**: click Sonnet play → card collapses to a `ConvChip` showing the new conversation, status spinner; clicking opens it in the side pane. Verify the new conversation's task is parented under the host task and its initial prompt is the (possibly edited) text.
7. **Launch (Opus)**: same as above with Opus.
8. **Disabled state**: clear prompt → both Create and Launch are disabled.
9. **Multi-instance**: render two `<task>` tags in one assistant message → both have independent state and don't collide on Lexical namespace.
10. **Reload**: refresh the page → cards reset to idle (acceptable for MVP; the created tasks/convs persist independently).
