# Create child task from a conversation

## Context

Agents working in a conversation routinely spot follow-up work — new bugs,
TODOs, refactors — that should land as a subtask of whatever task the
conversation was launched for. Today the only place to add tasks is the
`/tasks` view, which means leaving the conversation, navigating back to the
task tree, and finding the right parent. That friction makes it tempting to
scribble TODOs somewhere else and lose them.

This change adds an inline affordance: the conversation title (top-left of
`/c/:id`) becomes a clickable button that opens a small popover with a
multi-line textarea. Submitting creates a task whose `parentId` is the task
the conversation was launched for (via its attempt). The existing task tree
(`plugins/tasks/web/components/tasks-list.tsx`) already renders nested tasks
and will pick up the new child automatically through the `tasks` resource
subscription.

## Plan

### 1. Expose `taskId` on the conversations view (backend, 1 line of SQL)

The conversation's parent task is reachable via `attemptId → _attempts.taskId`,
but no current frontend hook exposes it. Extend the existing
`conversations_v` view — which already joins `_attempts` — to select
`taskId` alongside `worktreePath`.

**File:** `plugins/conversations/server/schema.ts`

- Line 20–29: add `taskId: _attempts.taskId` to the view's select list.
- Line 36–38: add `taskId: z.string()` to the extended `ConversationSchema`
  so the client receives it and types pick it up.

`./singularity build` will regenerate the migration; commit the generated
file. Everything downstream (`useConversation`, `ConversationEntry`) is
already typed from `ConversationSchema`, so it flows through automatically.

### 2. Add a Popover UI primitive (frontend, new file)

The repo uses Base UI via the shadcn registry but has no Popover wrapper
yet. The `@base-ui/react/popover` primitive is already available — the
existing `dropdown-menu.tsx` (`web/src/components/ui/dropdown-menu.tsx:2`)
shows the wrapping pattern to copy.

**New file:** `web/src/components/ui/popover.tsx`

Thin shadcn-style wrapper exporting `Popover`, `PopoverTrigger`,
`PopoverContent`. Keep it to the bare minimum — no Arrow, no Close, no
Portal customization — to match current repo style (same as `sheet.tsx`).

### 3. Add the `CreateChildTaskForm` popover content (frontend, new file)

**New file:**
`plugins/conversations/plugins/conversation-view/web/components/create-child-task-form.tsx`

Props: `{ parentTaskId: string; onClose: () => void }`.

Behavior:
- Local `value` state for the textarea (4–6 rows, auto-focused on open).
- `Create` button disabled when `value.trim()` is empty.
- Submit: `POST /api/tasks` with `{ parentId: parentTaskId, title: value.trim() }`.
  This matches the existing handler at
  `plugins/tasks/server/internal/handle-create.ts:5–21` — no backend
  changes needed.
- On successful response, call `onClose()` and reset the state. The
  `tasks` resource push notification at `handle-create.ts:19` already
  refreshes the tree.
- Keyboard: `Cmd/Ctrl+Enter` submits, `Esc` closes (handled by Popover).
- Use plain `fetch` directly (matches `tasks-list.tsx:54–58`); no shared
  helper needed.

### 4. Wire the title as the popover trigger

**File:**
`plugins/conversations/plugins/conversation-view/web/components/conversation-view.tsx`

At line 54–56, wrap the title in `<Popover>` + `<PopoverTrigger>` (rendered
as a button with ghost styling so it looks like the current title but has a
hover state). Its `<PopoverContent>` renders `<CreateChildTaskForm
parentTaskId={conversation.taskId} onClose={…} />`.

Guard: only render the trigger when `conversation?.taskId` is set. In
practice every conversation has an attempt, so this is defensive but
inexpensive.

## Not changing

- `handleCreate` already accepts `{ parentId, title }` — no API change.
- The task tree rerenders automatically via the `tasks` resource.
- No new plugin; this is a UI affordance inside `conversation-view`.
- Task description stays empty on creation; user fills it in via
  `/tasks/:id` if they want (matches the "Single textarea" UX choice).

## Verification

1. `./singularity build` — builds, runs generated migration, deploys to
   `http://claude-1776361590.localhost:9000`.
2. In a browser: open any conversation, click the title → popover opens
   with focused textarea.
3. Type text, press `Cmd+Enter` (or click Create) → popover closes, new
   child appears under the correct parent in the left sidebar's Tasks list
   (or `/tasks` page).
4. Verify parent linkage: the new task's parent in the tree is the task
   this conversation was launched in (the one whose attempt owns this
   worktree), not a sibling or a root task.
5. Edge cases to click through:
   - Empty textarea → Create button disabled.
   - `Esc` closes popover, state reset (reopen → empty textarea).
   - Open another conversation → popover appears for that one's parent.
