# Read tool renderer for the JSONL viewer

## Context

The JSONL viewer's per-tool renderer system (design doc: `research/2026-04-27-conversations-jsonl-per-tool-renderers.md`) has renderers for Bash, Edit, Write, and add_task. Read — the most frequent tool call — still falls through to `GenericToolView`, which dumps raw JSON for both input and result. The reader can't see the file path at a glance, can't tell the requested range, and the file content isn't syntax-highlighted.

This plan adds a dedicated Read renderer with: clickable file path, line range badge, syntax-highlighted code with line-number gutter, image thumbnail for image files, and proper error rendering.

## Data shape

**Input:** `{ file_path: string; offset?: number; limit?: number }`

**Result content:** `cat -n` formatted string — `"1\tline one\n2\tline two\n..."`. Each line is `<line_number>\t<content>\n`. When `offset` is provided, line numbers start from `offset + 1`. Images do NOT embed base64 data in the result.

## Components

### 1. `read-tool-view.tsx` — main renderer

Receives `{ event: ToolCallEvent }`. Casts `event.input` as `ReadInput`.

**Summary line:** `<ToolFilePath>` + a `<LineRangeBadge>` showing `L42–84` when offset/limit are present.

```
LineRangeBadge({ offset, limit }):
  if both null → null
  start = (offset ?? 0) + 1
  end = limit != null ? start + limit - 1 : null
  label = end ? "L{start}–{end}" : "L{start}+"
  render: <span className="shrink-0 rounded bg-muted px-1.5 py-0.5 font-mono text-[10px] text-muted-foreground">
```

**Body dispatch:**
1. Error (`event.result?.isError`) → red text
2. Image file (extension match) → `<ReadImageView>`
3. Otherwise → `<CodeWithLineNumbers>`

`defaultOpen={false}` — Read is high-volume, collapsed by default.

**Image detection:** Local `IMAGE_EXTS` set (`png, jpg, jpeg, gif, webp, svg, ico, bmp, avif`). Cannot import from `file-pane/plugins/image/web/internal/` — boundary rules forbid internal cross-plugin imports. Duplication is minimal.

### 2. `code-with-line-numbers.tsx` — code + gutter

Uses the shiki transformer approach from `raw-view.tsx` (`file-pane/plugins/raw/web/components/raw-view.tsx`). Does NOT use `<HighlightedCode>` — that component doesn't support transformers.

**`parseCatN(content: string): { startLine: number; lines: string[] }`**
- Split on `\n`, drop trailing empty
- For each line: split on first `\t`, parse left side as line number, right side as content
- First valid number becomes `startLine`
- Malformed lines (no tab) kept as-is

**Line number transformer factory:**
```ts
function makeLineNumberTransformer(startLine: number): ShikiTransformer {
  line(node, lineIdx) {
    // lineIdx is 1-based per shiki
    const displayNum = startLine + lineIdx - 1;
    node.children.unshift(<span class="ln">{displayNum}</span>);
  }
}
```

**Rendering:** `useEffect` calling `getHighlighter(lang).then(hl => hl.codeToHtml(code, { lang, theme, transformers: [lineNumberTransformer] }))`, stored in state, rendered via `dangerouslySetInnerHTML`. Same pattern as `raw-view.tsx`.

**Gutter styling:** Reuse the exact Tailwind classes from `raw-view.tsx`:
```
[&_.ln]:mr-4 [&_.ln]:inline-block [&_.ln]:w-7 [&_.ln]:select-none
[&_.ln]:text-right [&_.ln]:text-muted-foreground/50 [&_.ln]:tabular-nums
```

**Height cap:** `max-h-[280px] overflow-auto` — matches Write.

**Empty content:** Show `(empty result)` in muted italic.

**Imports:** `getHighlighter`, `languageForPath`, `SHIKI_LANGS`, `themeForMode`, `useDarkMode` from `@plugins/primitives/plugins/syntax-highlight/web`.

### 3. `read-image-view.tsx` — image thumbnail

Fetches from `/api/code/${worktree}/image?path=${path}` (same API as `file-pane/plugins/image`). Gets `worktree` passed as a prop from the parent (which reads `conversation.attemptId` from `conversationPane.useData()`).

Click toggles between thumbnail (`max-h-32 max-w-xs`) and expanded (`max-h-[80vh]`). Uses the pixelated→auto rendering heuristic from `ImageView`.

## Integration

```ts
JsonlViewerTool.Renderer({ name: "Read", component: ReadToolView })
```
