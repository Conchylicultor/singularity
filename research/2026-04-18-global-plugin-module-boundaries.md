# Plugin Module Boundaries & Self-Healing Infrastructure

## Context

Duplication is accumulating across plugins: `descriptor<T>()` copy-pasted 5×, `isDescendant` 4×, `buildTree`/`computeDrop` duplicated, `push-and-exit` prompt constants byte-identical in two files, etc. More worrying than the specific cases: *there's no mechanism to prevent this from recurring*. Agents can't discover existing utilities, so they reinvent them; module boundaries are implicit, so they import whatever compiles; and "just put it in `plugin-core/shared`" would relocate the debt, not fix it.

We need infrastructure that makes the plugin model self-healing:

- **Discoverable** — agents know what already exists before writing new code.
- **Bounded** — module boundaries are explicit and enforced, not trust-based.
- **Structured** — there's one clear place for shared code, with a clear owner.
- **Ergonomic** — creating a new plugin is mechanical, not error-prone.

Good news: most of the foundation already exists. The `internal/` folder convention is used in `plugins/agents/server/internal/`, `plugins/push-and-exit/server/internal/`, etc. Several plugins expose a server public API via `api.ts` (`plugins/tasks/server/api.ts`, `plugins/agents/server/api.ts`). `docs/plugins.md` is auto-generated by `cli/src/docgen.ts` and verified by the existing `plugins-doc-in-sync` check. The check framework (`cli/src/checks/`) with a simple `{ id, description, run() }` interface already gates `./singularity push`. This plan formalizes the conventions, extends what's there, and closes the enforcement loop — it does not introduce a new subsystem.

## The Module Taxonomy

Three categories, each with a distinct role:

### 1. Framework primitives — `plugin-core/`
Only what makes a plugin *be* a plugin: the slot registry, `descriptor<T>()`, `PluginDefinition` types, lifecycle glue. Grows only when the plugin protocol itself grows — a slow, deliberate rate.

### 2. Feature plugins — `plugins/<name>/`
Contribute to slots, routes, DB schemas, UI. User-visible features. The usual case.

### 3. Library plugins — `plugins/<name>/` (same structure, zero slot contributions)
Export only via imports: types, functions, components. Other plugins import them. Already exists — `plugins/launch` is the canonical example (its `index.ts` declares `contributions: []`), and `plugins/tasks`/`plugins/conversations` expose library APIs alongside their feature contributions. Formalize as a first-class pattern so new shared code (e.g. tree primitives) has a natural owned home instead of being dumped in core.

**Library plugins share the exact shape of feature plugins.** They still have `index.ts` with a default-exported `PluginDefinition` (empty `contributions: []`); the difference is that `api.ts` carries the load and no slot contributions are declared. Same scaffolder, same check rules, same registry import path — zero-contribution registration is a cheap no-op. `plugins.md` still lists them; they simply show "Exports:" without a "Contributes:" section. No bifurcation in the plugin system.

**Rule of thumb:** feature plugins are *nouns in the product* (agents, tasks, screenshot). Library plugins are *mechanisms* (tree, rank, descriptor). If a user could "turn it off," it's a feature. If not, it's a library. Both live in `plugins/`, share the same folder shape, and are discoverable via the same `plugins.md`.

## Structure Conventions

Every plugin has this skeleton:

```
plugins/<name>/
├── package.json              # @singularity/plugin-<name>
├── shared/
│   ├── api.ts                # library surface (no plugin registration here)
│   └── internal/             # private
├── web/
│   ├── index.ts              # default export = PluginDefinition (registration entry)
│   ├── api.ts                # (optional) library surface — named exports for other plugins
│   ├── components/           # private unless re-exported via api.ts
│   └── internal/             # private
└── server/
    ├── index.ts              # default export = ServerPluginDefinition (registration entry)
    ├── api.ts                # (optional) library surface — named exports, including schema re-exports
    └── internal/
        └── schema.ts         # (optional) Drizzle tables — private; drizzle-kit reads it via config path
```

**Two barrel roles:**
- **`index.ts` — the plugin registration entry.** Default export is the `PluginDefinition` / `ServerPluginDefinition`. Consumed by `web/src/plugins.ts` and `server/src/plugins.ts` (the explicit registries). Exists in `web/` and `server/` only.
- **`api.ts` — the one public surface.** Named exports consumed by *other plugins*. Import path `@plugins/<name>/<runtime>/api` communicates "I'm using this plugin's library," distinct from `@plugins/<name>/<runtime>` which means "I'm registering this plugin." Exists wherever the plugin exposes public symbols; `shared/` only has `api.ts` (no registration happens there). DB schema surface flows through here too — see **DB schemas** below.

This matches the existing convention in `plugins/tasks/server/api.ts` and `plugins/agents/server/api.ts`; it formalizes the rule and extends it to `web/` and `shared/`.

**Other rules:**
- **Private** = anything under `internal/`, or not re-exported from `api.ts` / `index.ts`.
- **One domain per plugin**, split internally by runtime (`web`/`server`/`shared`), not by technology.
- **UI and logic for the same domain stay in one plugin.** A `<TreeView>` and its tree algorithms live together in `plugins/tree/` — `shared/api.ts` for the pure functions, `web/api.ts` for the React component. Split plugins only when domains genuinely differ.

## DB Schemas

DB tables are *owned* by exactly one plugin. Drizzle table definitions live at `plugins/<name>/server/internal/schema.ts` — **inside `internal/`**, private by the boundary rule. Drizzle migration tooling finds it via its config path (a one-line change to `drizzle.config.ts`'s `schema` glob).

**Whatever the owner wants public, `api.ts` re-exports explicitly:**
```typescript
// plugins/tasks/server/api.ts
export type { TaskRow, NewTaskRow } from "./internal/schema";
export { tasks } from "./internal/schema";  // for cross-plugin FK use
export { nextRankUnder, tasksResource } from "./internal/...";
```

No special path, no `schema` whitelist entry in the check, no "third barrel." `api.ts` remains *the* public surface; schemas are just one kind of thing that might flow through it.

**Consequences of this model:**

- **Explicit surface.** If `TaskRow` isn't in `api.ts`, it's not part of the contract. Owner decides intentionally. Contrast with "schema is public": then every inferred type is implicitly exposed.
- **Cross-plugin foreign keys require opt-in.** To write `references(() => tasks.id)` in another plugin's schema, the `tasks` owner must `export { tasks }` from `api.ts`. One line per FK-able table — a mild cost that makes "other plugins may depend on this table's identity" an explicit decision.
- **Narrowing is free.** If the owner wants to expose `Omit<TaskRow, 'secret_field'>`, or a wrapped read-only view, they do it in `api.ts` without consumers noticing.
- **Cross-plugin writes are still forbidden by convention** (`db.update(otherTable)` from a consumer plugin). The check can't fully prevent this — a plugin could still construct queries against a re-exported table — but the re-export signals "FK use expected," and reviewers can flag writes. Future stricter rule: expose FK'd tables as branded read-only types.

**Migrations stay global.** `drizzle-kit generate` (run by `./singularity build`) produces one set of migrations covering all plugin schemas. The only config change needed is updating the schema glob to `plugins/*/server/internal/schema.ts`.

## Enforcement: `plugin-boundaries` Check

New check at `cli/src/checks/plugin-boundaries.ts`, registered in `cli/src/checks/index.ts`. Runs automatically before `./singularity push`.

**Verifies:**

1. Every `plugins/**/` directory has a `package.json` with name `@singularity/plugin-<name>`.
2. Every cross-plugin import matches an allowed public path. Allowed suffixes:
   - `@plugins/<name>/web` → `web/index.ts` (plugin-registration entry; used only by the plugin registries)
   - `@plugins/<name>/web/api` → `web/api.ts` (library surface)
   - `@plugins/<name>/server` → `server/index.ts` (plugin-registration entry; used only by the plugin registries)
   - `@plugins/<name>/server/api` → `server/api.ts` (library surface; re-exports schema types/tables as needed)
   - `@plugins/<name>/shared/api` → `shared/api.ts` (library surface; no registration entry in shared)
   - Nested plugins recurse with the same grammar: `@plugins/<name>/plugins/<sub>/<suffix>`.
   - Plugin-registration entries (`web`, `server`) should only be imported by `web/src/plugins.ts` and `server/src/plugins.ts`; importing one plugin's registration from another is a separate warning.
3. No import references a path inside an `internal/` folder from outside the owning plugin. Intra-plugin deep imports are fine.

**Implementation:** walk `.ts`/`.tsx` files under `plugins/`, `web/src/`, `server/src/`, regex-match `from ["']@plugins/...["']`, compare the suffix against the allowed grammar. Emit the first ~10 violations with a hint pointing to the right barrel (e.g. "move this export to `plugins/tree/web/api.ts` and import from `@plugins/primitives/plugins/tree/web/api`"). Mirrors the shape of existing checks like `plugins-doc-in-sync.ts`.

**Why a check, not `package.json` `exports`:** the root `tsconfig.json` has a `paths` alias (`@plugins/*` → `./plugins/*`) that resolves imports directly to folder paths, bypassing any `exports` field. Adding `exports` would require removing the alias and switching to package-name imports — a large mechanical churn for marginal gain over a dedicated check. Defer until the check proves insufficient.

## Discoverability: Extend `plugins.md` Generator

`cli/src/docgen.ts` already walks plugins and emits slot/command/route contributions. Extend it to also emit **exported library APIs** so "what already exists?" has a single canonical answer.

**For each plugin, add a new section:**

```markdown
- Exports:
  - `web/api` — TreeView, buildTree, computeDrop, isDescendant
  - `shared` — TreeNode, TreeOps
```

**Source:** parse each barrel file (`index.ts`, `api.ts` at each runtime-folder root) and list its top-level `export` declarations (excluding `export default` plugin definitions, which are the feature-plugin side). Same AST/regex infrastructure the generator already uses for slot extraction.

**Why this is the self-healing piece:** the existing `plugins-doc-in-sync` check fails if `docs/plugins.md` diverges from generator output. Once library exports are part of generation, they become part of the verified contract — you cannot add a library export without `plugins.md` updating, and you cannot ship `plugins.md` drift. Agents read `plugins.md` before writing new code; it now surfaces library APIs as well as contributions.

Add one line to `CLAUDE.md`: *"Before writing a helper, search `docs/plugins.md` for it."*

## Scaffolding: `./singularity plugin new <name>`

New command at `cli/src/commands/plugin.ts`, registered in `cli/src/index.ts` alongside `registerBuild/Check/Push`. Generates the minimum required to pass `plugin-boundaries` and appear in `plugins.md`.

**Command:**

```
./singularity plugin new <name> [--web-only] [--server-only] [--parent <path>]
```

**Generates:**

- `plugins/<name>/package.json` — name `@singularity/plugin-<name>`, standard workspace deps (`@singularity/plugin-core`, react via workspace).
- `plugins/<name>/web/index.ts` — stub with a `// TODO: define plugin contributions` comment and a minimal `PluginDefinition` default export.
- `plugins/<name>/server/index.ts` — stub (unless `--web-only`).
- No `api.ts` or `shared/` files generated by default — create them only when the plugin actually exposes library symbols. The scaffolder keeps the starting surface minimal; `plugins.md` will simply show no exports until they exist.

**Updates:**

- Appends import + array entry to `web/src/plugins.ts`.
- Appends import + array entry to `server/src/plugins.ts` (unless `--web-only`). Preserve the existing ordering comment on line 18–22 (server runtime-loading order).

**`--parent <path>`** (e.g. `--parent conversations/plugins/conversation-view`) creates a nested sub-plugin. `server/tsconfig.json` currently enumerates three levels of nesting (`plugins/*/plugins/*/plugins/*/`); reject deeper parents until tsconfig is updated.

**Non-goals:** no sample slot contributions, no demo components, no mock DB schemas. Agents fill in real code by imitating nearby plugins — scaffolding exists to eliminate registration-boilerplate errors and make "create a plugin" a single command, not an 8-step recipe. A bloated scaffold is worse than none — it generates cruft agents have to delete and can't tell apart from the task.

**Migration (prerequisite for the check to go green):** the first run of `plugin-boundaries` will surface existing violations. Fix iteratively:

- `plugins/launch/web/index.ts:12` — move the `LaunchButtons` / `LaunchButtonsProps` / `LaunchRequest` named exports into a new `plugins/launch/web/api.ts`, leaving `index.ts` with only the `PluginDefinition` default export. Update consumers to import from `@plugins/primitives/plugins/launch/web/api`.

- Do not land the check in `CHECKS` until a clean run is achieved (guard with a commented-out registration during migration if needed).

## Non-Scope (Deferred)

- **`package.json` `exports` field enforcement.** Bypassed by the `@plugins/*` tsconfig alias. Revisit only if the check proves insufficient.
- **Concrete dedup refactors** (`descriptor`, `buildTree`, `isDescendant`, `formatDate`, etc.). Follow-up work enabled by this plan. Natural first extraction: a `plugins/tree/` library plugin for `isDescendant` + `buildTree` + `computeDrop` + `<TreeView>`.
- **ESLint `no-restricted-imports` rules.** Redundant with the check; skip unless editor-time feedback becomes a priority.
- **Library plugin naming prefix** (`lib-tree/` vs `tree/`). Use flat naming; `plugins.md` already shows (zero) contributions — no folder-level split needed.
- **Auto-discovery of plugins** (avoiding the manual edit to `web/src/plugins.ts` and `server/src/plugins.ts`). The scaffolder handles the common case; full auto-discovery would need dependency-ordering logic (server plugins currently have a hand-maintained load order).

## Verification

1. Run `./singularity check --plugin-boundaries` on the current tree. Expect violations; fix them iteratively until clean.
2. Run `./singularity build`. Confirm `docs/plugins.md` now includes an "Exports:" section per plugin with library symbols.
3. Run `./singularity plugin new tree`. Inspect the skeleton; it must pass `plugin-boundaries` immediately without edits, and its entries must appear in `web/src/plugins.ts` and `server/src/plugins.ts`.
4. Add a deliberately forbidden deep import — e.g. `import { TreeView } from "@plugins/primitives/plugins/tree/web/components/tree-view"` — in a test file. Run the check; verify it fails with a hint pointing at `@plugins/primitives/plugins/tree/web/api`. Remove the test import.
5. Run `./singularity check` (no flags). Confirm `plugin-boundaries` runs alongside existing checks.
6. On a throwaway branch, run `./singularity push` and confirm the check runs as the first step before the merge flow.
