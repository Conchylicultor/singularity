# Mail: attachment indicator without a full body fetch

## Context

Mail's on-demand sync mirrors messages as **envelopes only** (Gmail
`format=metadata`), which carries no MIME parts. So attachment presence is
unknown until a message is *hydrated* (opened): `mail_attachments` rows are
written only on `format=full` hydration, and the thread rollup
`mail_threads.hasAttachments` (computed by `recomputeThread` from those rows)
therefore stays `false` for every unopened message — even when the real Gmail
mail has attachments. Result: no paperclip indicator on unread/unopened mail,
which users expect at a glance. This is the exact gap flagged in
`plugins/apps/plugins/mail/plugins/sync/CLAUDE.md` (Limitations →
"`hasAttachments` / attachment metadata lag hydration").

## Key insight

Gmail's **`has:attachment` search operator** is the authoritative, metadata-only
signal for "real attachment" — the *same* classification behind Gmail's own
paperclip (it excludes inline/`cid:` images). `messages.list?q=… has:attachment`
returns just the id refs (cheap, no per-message GET, no body fetch). Nothing
else in the metadata surface is reliable (`sizeEstimate` is noisy;
`multipart/mixed` is a soft heuristic; `messages.list` has no per-row flag).

## Approach

Introduce a **message-level** `hasAttachments` boolean (the paperclip source of
truth), populate it from a cheap `has:attachment` list pass across all three
ingestion paths, and derive the thread rollup from it.

### 1. Schema — message-level flag

`plugins/apps/plugins/mail/plugins/mail-core/core/internal/fields.ts`
- Add `hasAttachments: boolField()` to `mailMessageFields` (after `isSent`).

`plugins/apps/plugins/mail/plugins/mail-core/server/internal/tables.ts`
- Add `hasAttachments: { default: false }` to the `mail_messages` `columns` meta.

`MailMessage` (derived via `fieldsToZodObject`) then carries `hasAttachments`
automatically → it flows to the search endpoint + web with zero extra wiring.
Migration is auto-generated by `./singularity build`.

### 2. Store layer — `store.ts`

- `writeMessage`: on the **full** (hydration) path only, set
  `hasAttachments: parsed.attachments.some((a) => !a.inline)` (non-inline =
  paperclip semantics). Spread it like `body` into both the insert values and
  the `onConflictDoUpdate` set. The **metadata** path never touches the column
  (insert → DB default `false`; update → preserved), so a delta re-fetch never
  clobbers a scan-set or hydration-set flag.
- New `markMessagesWithAttachments(accountId, ids)`: `UPDATE _mailMessages SET
  hasAttachments = true … WHERE accountId = … AND id = ANY(ids) AND
  hasAttachments = false RETURNING threadId`; recompute each distinct affected
  thread. Positive-only + idempotent (Gmail message content is immutable, so a
  message never *loses* an attachment; hydration still corrects a false-positive
  down to the exact value).
- `recomputeThread`: derive `hasAttachments: messages.some((m) =>
  m.hasAttachments)` and **drop** the now-unused `mail_attachments` count query.
  This unifies the rollup on the message flag (populated by scan OR hydration)
  and fixes the root bug.

### 3. Attachment-scan job — new `attachment-scan.ts`

- `scanAttachmentFlags(token, accountId, q)`: paginate
  `listMessages(token, { q: `${q} has:attachment`, maxResults: 500 })`, collect
  ids up to a page cap (`MAX_ATTACHMENT_SCAN_PAGES`, log if hit), then one
  `markMessagesWithAttachments`.
- `attachmentScanJob` (`mail.attachment-scan`, `defineJob`): input
  `{ accountId, windowDays }`, `dedup` keyed by `accountId` (overlapping scans
  coalesce), `event: z.never()`. Resolves a fresh token and runs
  `scanAttachmentFlags` for `newer_than:${windowDays}d`. Registered in the sync
  server barrel alongside the other jobs.

### 4. Wire the three ingestion paths

- **Backfill completion** (`backfill.ts`): after flipping `status → delta`,
  `attachmentScanJob.enqueue({ accountId, windowDays: BACKFILL_WINDOW_DAYS })` —
  one full-window scan once the window is mirrored. Decoupled job so a scan
  failure never dead-letters the backfill.
- **Delta** (`delta.ts` + `history-sync.ts`): change `applyHistorySince` to
  return `{ historyId, addedIds }` (ids from `messagesAdded`). Backfill's
  `renewWatermark` unwraps `.historyId`. Delta, on a successful pass with
  `addedIds.length > 0`, enqueues `attachmentScanJob` with a short window
  (`ATTACHMENT_SCAN_DELTA_WINDOW_DAYS = 2`, new mail is recent) — no wasted scan
  when nothing arrived.
- **Remote-search** (`remote-search.ts`): on the first page (`!pageToken`), one
  extra `listMessages({ q: `${query} has:attachment`, maxResults: 100 })`,
  intersect with the folded page ids, `markMessagesWithAttachments`, *then* read
  the rows back so results carry the flag. (The `has:attachment` first-page
  spans a far wider recency range than the 25-result query page — attachment
  bearers in the page are reliably covered. Deep-page marking is skipped;
  pagination isn't wired yet.)

New config constant `ATTACHMENT_SCAN_DELTA_WINDOW_DAYS` and
`MAX_ATTACHMENT_SCAN_PAGES` in `sync/core/config.ts`.

### 5. UI — the paperclip

`plugins/apps/plugins/mail/plugins/search/web/components/mail-search-row.tsx`
- Render `<MdAttachFile className="text-muted-foreground shrink-0" />` (muted,
  secondary) in the sender/time `Line`, before the star, when
  `message.hasAttachments`. Remove the stale "attachment unknown until hydrated"
  follow-up comment. (Mirrors the existing raw-icon precedent — `MdStar` with a
  direct `className`.)

The search-result row is the only visible list surface today (no inbox list view
exists yet). The message-level flag + thread rollup are now correct for any
future thread-list surface with zero extra work.

### 6. Docs

Update `sync/CLAUDE.md` (Limitations + jobs list: new `mail.attachment-scan`,
the `has:attachment` mechanism), `search/CLAUDE.md` (attachment chip follow-up →
done), and `mail-core/CLAUDE.md` (message `hasAttachments` column).

## Out of scope (file as follow-ups)

- **Reader attachment chips include inline images.** `mail-message-reader.tsx`
  lists every `mail_attachments` row (incl. inline `cid:` logos) as a chip,
  which diverges from the paperclip's non-inline semantics. Should filter/label
  inline.
- **Older-than-window, non-searched mail** never gets a scan. The backfill-window
  scan + delta scan cover the active window; a broader background reconcile is a
  later-phase follow-up (already tracked for the mirror generally).
- **Deep search pages** don't mark attachments (has:attachment first-page only);
  lands with real search pagination.
