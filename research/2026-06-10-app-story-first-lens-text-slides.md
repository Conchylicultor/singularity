# Story Builder — First End-to-End Lens (text content + Slides renderer)

> Implements **T5** of [`2026-06-03-app-story-builder-plan-v2.md`](./2026-06-03-app-story-builder-plan-v2.md)
> ("Renderers" + "Content renderers" sections).

## Context

The Story app can already **author** a story (a page-as-block-tree) and switch
its view, but it cannot **render** to any format yet. The render substrate is in
place and inert:

- `story-core` produces the renderer-agnostic `StoryNode[]` IR
  (`buildStoryTree`, `role: "content" | "break"`).
- `render` owns the two dispatch slots `Story.Renderer` and `Story.Content`,
  the `<StoryRender pageId rendererId/>` surface, and the **visible** fallbacks
  (`NoRenderer`, `UnsupportedContent`). It has **zero contributors** today —
  switching to a renderer in the shell shows "No renderer available".

This task lands the **first end-to-end lens** by adding two pure-additive
contributor plugins — **no core changes**, proving the renderer architecture:

1. `content/text` → contributes `Story.Content` for the `text` block payload.
2. `renderers/slides` → contributes `Story.Renderer`; turns the top-level block
   tree into a deck (split at top-level dividers, else one slide per top block).

Intended outcome: in `/story`, author a story, switch to **Slides**, and see a
real, live-updating deck. Adding these as sub-plugins with no edit to `render`,
`story-core`, or `shell` is the live proof that extension is zero-core-change.

## Scope

**In:** `content/text`, `renderers/slides`, and their two collapsed umbrella
namespaces (`content/`, `renderers/`).
**Out (later tasks):** blog renderer (T6), image/code content (T7),
pages-integration (T8). Those are independent additive follow-ups.

## The contracts (already built — consume, don't touch)

- `Story.Content` (`render/web/slots.ts:24`) —
  `defineDispatchSlot<{ node: StoryNode }, string>`, key = `node.type`, fallback
  `UnsupportedContent`. Contributor shape: `Story.Content({ match, component })`;
  `component` receives `{ node: StoryNode }`.
- `Story.Renderer` (`render/web/slots.ts:11`) —
  `defineDispatchSlot<{ story: StoryNode[]; activeRendererId: string }, string,
  { id; label; icon? }>`, key = `activeRendererId`, fallback `NoRenderer`.
  Contributor shape:
  `Story.Renderer({ match, id, label, icon?, component })`; **`match` must equal
  `id`** (sonata convention, `piano-roll/web/index.ts:14`). `component` receives
  `{ story, activeRendererId }`.
- `StoryNode` (`story-core/core/types.ts`) — `{ id, type, data, role, depth,
  index, children }`. `role: "break"` marks a divider; renderers interpret it
  structurally and **never name "divider"**.
- `textBlock` (`@plugins/page/plugins/text/core`) — `textBlock.parse(data) →
  { text: string }` (zod `{ text }`).
- The shell view-switcher (`story-view-switcher.tsx:28`) already enumerates
  `Story.Renderer.useContributions()` — the Slides segment appears automatically
  with **no shell change**.

## `content/text`

`web/components/text-content.tsx`:
```tsx
import { Text } from "@plugins/primitives/plugins/text/web";
import { textBlock } from "@plugins/page/plugins/text/core";
import type { StoryNode } from "@plugins/apps/plugins/story/plugins/story-core/core";

export function TextContent({ node }: { node: StoryNode }) {
  // Defensive parse: data may be transient/empty mid-edit (plan "Gotchas").
  // The slot-render item error boundary would contain a throw to this one leaf,
  // but a safe fallback avoids a visible boundary flicker while typing.
  const result = textBlock.schema.safeParse(node.data);
  const text = result.success ? result.data.text : "";
  if (!text) return null; // empty paragraph → render nothing
  return <Text as="p" variant="body">{text}</Text>;
}
```
> Confirm `textBlock.schema` is exported from `page/text/core`; it re-exports
> `textBlock` whose `.schema` is the zod object (`define-block.ts`). If `.schema`
> isn't on the public handle, use `textBlock.parse` wrapped in a try/catch
> (specific catch, not bare) instead.

`web/index.ts`:
```ts
import type { PluginDefinition } from "@plugins/framework/plugins/web-sdk/core";
import { Story } from "@plugins/apps/plugins/story/plugins/render/web";
import { TextContent } from "./components/text-content";

export default {
  description: "Renders the text block payload as a Story.Content widget.",

```

## `renderers/slides`

**Slicing logic** (`buildSlides(story: StoryNode[]): StoryNode[][]`):
- If any **top-level** node has `role === "break"` → split the top-level array on
  those break nodes; each run between breaks is one slide (break nodes dropped).
  Drop empty runs (leading/trailing/consecutive dividers).
- Else → each top-level node is its own slide (`story.map(n => [n])`).
- A slide is a `StoryNode[]` (its top-level nodes); each node renders its own
  content **and its descendants** via the recursive content tree below.

**Recursive content** (`story-content-tree.tsx`) — keeps renderers type-agnostic:
```tsx
export function StoryContentTree({ nodes }: { nodes: StoryNode[] }) {
  return (
    <>
      {nodes.map((node) =>
        node.role === "break" ? (
          // a nested divider inside a slide → structural rule, not content.
          <hr key={node.id} className="my-3 border-border" />
        ) : (
          <div key={node.id} style={{ marginLeft: node.depth * 16 }}>
            <Story.Content.Dispatch node={node} />
            {node.children.length > 0 && <StoryContentTree nodes={node.children} />}
          </div>
        ),
      )}
    </>
  );
}
```
> Indent by `depth` so nested bullets read as a hierarchy. `Story.Content` is
> imported from `@plugins/apps/plugins/story/plugins/render/web`.

**Slides component** (`slides-renderer.tsx`): local `useState` current index,
prev/next buttons + slide counter, 16:9 framed surface. Receives
`{ story }` (ignores `activeRendererId`). Clamp index when slide count changes
(`Math.min(index, slides.length - 1)`); empty story → a muted "Empty story"
placeholder.

`web/index.ts`:

## Notes / risks

- **Two contributions per block type going forward** (`Editor.Block` to edit,
  `Story.Content` to view) — inherent to edit/view separation; only `text` here.
- **Defensive parse** in `TextContent` avoids a visible error boundary while the
  user is mid-typing (data transiently `{}`/empty).
- **`match === id`** for the renderer must stay in lockstep (single literal here).
- `useResource` (inside `StoryRender`) needs `NotificationsProvider` above it —
  app-global; confirm on first `/story` load (substrate already relies on it).
