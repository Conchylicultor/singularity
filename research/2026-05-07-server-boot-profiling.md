# Server Boot Profiling — Gantt Debug Pane

## Context

Server boot time is opaque — there's no way to see which plugin is slow or which phase dominates. The server has ~57 plugins, and the boot has 3 main phases (register, infra, onReady) but no timing instrumentation. This adds a profiler module that captures per-plugin spans during boot and a Gantt-chart debug pane to visualize them.

## Design

### Data model

```ts
type PhaseId = "register" | "awaitPgReady" | "runMigrations" | "routePopulation" | "socketBind" | "onReady";

interface Span {
  id: string;        // "register:tasks-core", "onReady:jobs", "awaitPgReady"
  phase: PhaseId;
  plugin?: string;   // undefined for standalone phases (awaitPgReady, etc.)
  label: string;
  startMs: number;   // relative to boot t=0
  durationMs: number;
}
```

### Profiler module — `server/src/profiler.ts` (NEW)

A module-level `bootStart = performance.now()` captured at import time. Collects spans in a plain array. API:

- `profilerStart(id, phase, label, plugin?) → () => void` — returns a closer that records the span
- `getProfilingData() → { spans, totalDurationMs }` — read-only accessor for the endpoint

Zero-cost: just array pushes, no serialization until the API is called.

The closure pattern (start returns end callback) matches the existing `performance.now()` before/after idiom in `run-claude-print.ts` and avoids needing a mutable active-spans map.

### Instrumentation — `server/src/index.ts` (MODIFY)

Surgical wraps around existing code, no structural changes:

1. **Register loop** (lines 15-24): wrap each plugin's `register()` calls in `profilerStart/end` with `finally` for safety
2. **awaitPgReady** (line 26): single span
3. **runMigrations** (line 27): single span
4. **Route population** (lines 90-100): single span wrapping the sync loop
5. **Socket bind** (line 115): single span wrapping `Bun.serve()`
6. **onReady** (lines 161-170): per-plugin span with `finally(() => end())` in the Promise chain; skip plugins without `onReady`

### Server plugin — `plugins/debug/plugins/profiling/server/` (NEW)

- `index.ts` — `ServerPluginDefinition` with `GET /api/debug/profiling`
- `internal/handle-profiling.ts` — calls `getProfilingData()` from the profiler module, returns JSON

Response shape:
```json
{
  "spans": [{ "id": "register:config", "phase": "register", ... }],
  "totalDurationMs": 2341
}
```

### Web plugin — `plugins/debug/plugins/profiling/web/` (NEW)

- `index.ts` — `PluginDefinition` with `Pane.Register` + `Debug.Item` (icon: `MdSpeed`)
- `panes.tsx` — `Pane.define({ id: "debug-profiling", segment: "debug/profiling", after: [null] })`
- `components/gantt-view.tsx` — Gantt chart component

**Gantt rendering** (pure React + CSS, no charting library):

- **TimeAxis** — horizontal tick marks across total boot duration (6 evenly spaced)
- **PhaseGroup** — grouped sections for each phase (register, infra phases, onReady), each with:
  - Phase header: color dot, label, total phase duration, start offset
  - Span rows: label (w-48) | bar track (flex-1, bar positioned via `left%` / `width%`) | duration (w-16)
- **SpanRow** — horizontal bar using absolute positioning: `left = startMs/total * 100%`, `width = durationMs/total * 100%`. Min width clamped to 0.3% for visibility. Color-coded by phase.
- **SpanDetail** — sticky footer showing hovered span's full details
- Parallel spans (onReady) naturally overlap on the time axis, making parallelism visible

Phase colors (Tailwind): register=blue, awaitPgReady=amber, runMigrations=orange, routePopulation=emerald, socketBind=purple, onReady=sky.

One-shot fetch on mount + Refresh button (boot data never changes after startup).
