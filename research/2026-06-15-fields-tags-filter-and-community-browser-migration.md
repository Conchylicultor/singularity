# Plan B — Multi-value `tags` filter field + community-browser migration

> Part 2 of 2. **Depends on Plan A**
> (`2026-06-15-data-view-embedded-mode.md`), which adds `mode="embedded"` to
> `DataView`. This plan consumes that; it does not design it.

## Context

The tweakcn **community-browser** is a theme gallery that hand-rolls its own grid +
search + tag chips instead of reusing `DataView`. Migrating it cleanly requires
`DataView` to own the **whole toolbar** (search + tag filter) so the section keeps
no bespoke chrome. The blocker: a theme's `tags: string[]` is a **multi-valued
row** and the data-view filter model is single-valued. The enum filter is already
multi-*select* (toggles `selected: string[]`), but its predicate tests one scalar's
membership — it can't express "the row holds a set of tags; match if it intersects
the selected set."

So we add a generic, reusable **multi-value `tags` field type** to the `fields`
primitive, then migrate community-browser onto `DataView` consuming it (plus
`mode="embedded"` from Plan A and the gallery `renderCard` for the swatch card).

## Design decisions

**A. A new standalone `tags` field type** (not `extends enum`, not widening `enum`).
Additive — leaves `enum`'s scalar cell/filter/config paths frozen. It must **not**
`extends enum`: `extends` would resolve enum's *scalar* predicate via
`resolveTypeChain`, which is the wrong semantics. `tags` stands alone and ships its
own filter contribution; it reuses the multi-select `ToggleChip` Control *pattern*
(by copy, not by chain).

**B. Add a parallel `values?: (row) => string[]` to `FieldDef`** (do **not** widen
`FieldValue`). Strictly additive; `FieldValue` stays scalar so every existing
cell/sort/search path and cell contribution is untouched. A `tags` field leaves
`value` undefined (so sort/scalar-search skip it) and carries `values`. The filter
predicate call site reads `values` when present.

**C. Dynamic options via the consumer.** Theme tags are derived from data, not a
fixed list. Rather than new machinery, the consumer passes a memoized
`field.options` computed from rows (exactly the existing `sortedTags` memo). The
`tags` filter Control reads `field.options` like `EnumFilter` does.

**D. Presentation — tags behind the funnel (accepted), with a deferred opt-out.**
The data-view filter lives behind the `MdFilterList` funnel toggle (`FilterBar`).
Today the community browser shows tag chips *always-visible* above the grid; after
migration the user clicks the funnel to reveal the same multi-select chips. This is
one extra click but fully generic and needs zero chrome changes. **Open question
for the user** (see end): if always-visible chips are a hard requirement, add an
optional `alwaysOpenFilters?: boolean` to `DataViewProps` that force-opens
`FilterBar` and hides the funnel — a small, separable data-view chrome change.

## Part 1 — Primitive edits (data-view)

### `plugins/primitives/plugins/data-view/core/internal/types.ts`
- `FieldDef<TRow>`: add
  ```ts
  /** Multi-value projection for tags-style fields. Folded into search and passed
   *  to the array-aware filter predicate. Mutually exclusive with `value`. */
  values?: (row: TRow) => string[];
  ```
- `FilterContribution.predicate`: widen 2nd param to
  `(filterValue: unknown, fieldValue: FieldValue | readonly string[]) => boolean`.
  This is a widening — existing scalar predicates (enum/text/number/bool/date)
  still compile (they narrow with `String(...)`/`Number(...)` internally and never
  receive an array, since only a `tags` field defines `values`).

### `plugins/primitives/plugins/data-view/web/internal/use-flat-rows.ts`
- `isSearchable` (L9-14): add `|| type === "tags"` to the default-true branch.
- Search accessor (L38-45): drop the `&& f.value` requirement; map each field to
  `f.values ? f.values(row).join(" ") : String(f.value?.(row) ?? "")`.
- Filter loop (L51-60): compute
  `const rowValue = field.values ? field.values(row) : field.value ? field.value(row) : undefined;`
  and pass `rowValue` to `contribution.predicate`.
- Sort path (L63-77): unchanged — already guards on `field.value`, so a
  `values`-only field is skipped (tags is non-sortable).

`FilterBar` / `hasFilters` need no change — they auto-include any field whose type
resolves a filter contribution, so `tags` appears in the funnel automatically.

## Part 2 — New `tags` field-type plugin tree

Mirror the `enum` shape (identity barrel + a `filter` sub-plugin). No `config`
sub-plugin (not config_v2-editable here) and no `table` cell initially (the gallery
uses a custom card; add a `Badge`-row cell later if a table view of themes is wanted).

```
plugins/fields/plugins/tags/
  package.json            # @singularity/plugin-fields-tags, { singularity: { collapsed: true } }
  CLAUDE.md               # with AUTOGENERATED block (filled by build)
  core/
    index.ts              # re-export own internal token + identity
    internal/tags.ts      # defineFieldType<string[]>("tags") + defineFieldIdentity (label "Tags", MdLabel; NO extends, NO coerce)
  web/
    index.ts              # Fields.Identity({ identity: tagsIdentity })
  plugins/filter/
    package.json
    CLAUDE.md
    web/
      index.ts            # DataViewSlots.Filter({ match: "tags", Control: TagsFilter, predicate, isActive })
      components/tags-filter.tsx       # ToggleChips from field.options (EnumFilter shape, TagsFilterValue)
      internal/tags-filter-logic.ts    # isActive + array-aware predicate
```

**`tags-filter-logic.ts`** (the load-bearing predicate — OR / non-empty intersection):
```ts
export interface TagsFilterValue { selected?: string[]; }
export function isActive(fv: unknown): boolean {
  return ((fv as TagsFilterValue)?.selected?.length ?? 0) > 0;
}
export function predicate(fv: unknown, fieldValue: FieldValue | readonly string[]): boolean {
  const selected = (fv as TagsFilterValue)?.selected;
  if (!selected || selected.length === 0) return true;
  const rowValues = Array.isArray(fieldValue) ? (fieldValue as readonly string[]) : [];
  return selected.some((s) => rowValues.includes(s));
}
```
(`FieldValue` imported from `@plugins/primitives/plugins/data-view/web`. `.some` =
match-any; swap to `.every` only if match-all is later requested.)

**`tags-filter.tsx`** — identical structure to `enum-filter.tsx`: one `ToggleChip`
per `field.options`, toggling membership of `selected: string[]`, calling
`onChange({ selected: next })`. Reuse `ToggleChip` from
`@plugins/primitives/plugins/toggle-chip/web`.

Barrels stay pure; imports go through runtime barrels (`@plugins/fields/core`,
`@plugins/fields/web`, `@plugins/primitives/plugins/data-view/web`,
`@plugins/primitives/plugins/toggle-chip/web`). `./singularity build` discovers the
new plugins from the filesystem — no manual registration.

## Part 3 — Community-browser migration

`plugins/ui/plugins/tweakcn/plugins/community-browser/web/components/community-browser-section.tsx`

**Delete:** the local `themeQuery` `SearchInput` (DataView's search replaces it),
the `activeTag` `FilterChip` row (the `tags` filter replaces it), the hand-rolled
`grid grid-cols-2` of cards, the "N themes" count, and the `matchesQuery`/`visible`
in-grid filtering.
**Keep:** `sortedTags` memo (repurposed into `field.options`), the `applyMutation`
/ `applyingId` / `handleApply` / `handleCardClick` apply flow verbatim, the catalog
`useEndpoint`, and `<ImportByUrl>` as the footer.
**Unchanged:** `CommunityThemeCard` — it already owns its own `onClick → onApply`
and `isPending`, matching the gallery custom-card contract (rendered as-is, not
wrapped in `DataCard`, `onRowActivate` not wired).

**Field schema:**
```ts
const tagOptions = useMemo(() => sortedTags.map((t) => ({ value: t, label: t })), [sortedTags]);
const fields: FieldDef<CatalogTheme>[] = useMemo(() => [
  { id: "name", label: "Name", type: "text", primary: true, value: (t) => t.name },
  { id: "tags", label: "Tags", type: "tags", values: (t) => t.tags, options: tagOptions, sortable: false },
], [tagOptions]);
```

**The call:**
```tsx
<DataView<CatalogTheme>
  mode="embedded"                       // Plan A dependency
  storageKey="tweakcn.community-browser"
  rows={hostFilteredThemes}             // see host-search note below
  fields={fields}
  rowKey={(t) => t.id}
  views={["gallery"]}
  defaultView="gallery"
  loading={isLoading}
  searchAccessor={(t) => `${t.name} ${t.tags.join(" ")}`}
  emptyState={<Text as="p" variant="body" tone="muted">No themes match your search.</Text>}
  viewOptions={{ gallery: { minCardWidth: 220, renderCard: (t) => (
    <CommunityThemeCard theme={t} isPending={applyingId === t.id} onApply={() => handleCardClick(t.id)} />
  ) } }}
/>
```
Render shape becomes `<div className="flex flex-col gap-md"><DataView …/><ImportByUrl …/></div>`.
The standalone `<Loading variant="cards" />` is dropped (forwarded as `loading`).

**Host `search` prop (from the ThemeCustomizer pane) — two scopes:**
1. *Section visibility:* keep the `sectionMatchesSearch` gate, simplified to
   `if (!sectionMatchesSearch) return null;` (DataView's `emptyState` now covers the
   zero-rows case; `ImportByUrl` still matches on its own keywords). Slight behavior
   change: the section no longer auto-hides purely because zero rows match the host
   query.
2. *Row narrowing:* pre-filter `rows` by the host `q` (name/tags substring) in a
   `useMemo` (`hostFilteredThemes`) so typing in the settings-wide search still
   narrows the grid; DataView's own search box layers on top. Keeps the two search
   scopes cleanly separated.

**Imports:** drop `FilterChip` / `SearchInput` / `Loading`; add `DataView` +
`FieldDef` from `@plugins/primitives/plugins/data-view/web`. Do **not** import the
gallery view child — only `DataView` + types + the `viewOptions={{ gallery: {…} }}`
literal (collection-consumer separation).

## Open question for the user

Decision **D**: accept the tag filter living **behind the funnel toggle** (generic,
zero chrome change, one extra click), or add an `alwaysOpenFilters` option to
`DataView` to preserve today's **always-visible** tag chips above the grid? Default
recommendation: behind the funnel; revisit if the always-visible affordance matters.
