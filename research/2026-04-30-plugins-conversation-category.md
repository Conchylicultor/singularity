# Conversation Category Chips

## Context

We want every conversation in Singularity to surface, at a glance, what *kind* of work it is:

- General question
- Small feature
- Load bearing infra
- Bug
- Other

The list must be user-configurable from the Settings pane, and Haiku auto-classifies a conversation after its first assistant turn (with a manual re-classify affordance for the user). The chip should appear both on the sidebar conversation rows and in the conversation toolbar (next to model/status).

This is a useful triage signal — the agent manager app currently has no fast filter for "show me all the bugs" or "show me load-bearing changes" — and is also a stepping stone toward Haiku-driven metadata for conversations more generally (later: priority, area, complexity, …).

## High-level design

A new sub-plugin under the `conversations` umbrella, `conversation-category`, owns:

- A configurable `string-list` of category labels (Settings → Conversation Categories).
- A plugin-owned table `conversation_categories(conversation_id PK, category, classified_at, model, source)`.
- A durable job `classify-conversation` that calls `runClaudePrint({ model: "haiku", … })` with a tight prompt and validates output against the configured list.
- A trigger on the existing `afterTurn` event that enqueues the job when the row has no category yet.
- A push `categoriesResource` so the web layer gets live updates.
- A UI chip rendered in **two** places:
  1. The conversation **toolbar** via `conversationPane.Actions` (matches `model`/`status`).
  2. The conversation **sidebar row** via a new `ConversationItem.Chips` slot defined in the `item` plugin (no equivalent slot exists today; adding one cleanly enables future per-row chip plugins).

Clicking the chip opens a small popover that lets the user (a) re-classify via Haiku, or (b) override the category manually.

We deliberately do **not** create a new `haiku` infra plugin — `claude-cli` already exposes the haiku model and `runClaudePrint`. A wrapper plugin would be ceremony with no payoff at this scale.

## Data model

### New table — `plugins/conversations/plugins/conversation-category/server/internal/tables.ts`

```ts
export const _conversationCategories = pgTable("conversation_categories", {
  conversationId: text("conversation_id")
    .primaryKey()
    .references(() => _conversations.id, { onDelete: "cascade" }),
  category: text("category").notNull(),
  source: text("source").notNull(), // "haiku" | "manual"
  model: text("model"),             // null for manual; "haiku" for auto
  classifiedAt: timestamp("classified_at", { withTimezone: true })
    .defaultNow()
    .notNull(),
});
```

`onDelete: "cascade"` keeps category cleanup automatic when the parent conversation is deleted.

Migration is generated by `./singularity build`.

### Resource

```ts
export const categoriesResource = resourceDescriptor<
  { id: string }, // (none — list-wide)
  Array<{ conversationId: string; category: string }>
>({ name: "conversation-categories", strategy: "push" });
```

Notify on insert/update/delete in the job handler and the manual-override route.


(Exact contributions shape follows the existing plugin barrel grammar — see `plugins/conversations/plugins/summary/server/index.ts` for the closest reference.)

### Job — `internal/job-classify.ts`

```ts
export const classifyConversationJob = defineJob({
  name: "classify-conversation",
  input: z.object({ conversationId: z.string(), force: z.boolean().optional() }),
  handler: async ({ conversationId, force }) => {
    if (!force) {
      const existing = await db
        .select()
        .from(_conversationCategories)
        .where(eq(_conversationCategories.conversationId, conversationId))
        .limit(1);
      if (existing.length) return; // already classified
    }

    const { categories } = await readConfig(conversationCategoryConfig);
    const conversation = await getConversation(conversationId); // tasks-core
    if (!conversation) return;

    const turns = await readConversationTurns(conversationId); // conversations
    const transcript = turns
      .slice(0, 6) // cheap; first few turns capture intent
      .map(turn => `### ${turn.role}\n${turn.content}`)
      .join("\n\n");

    const system = `You categorize software-engineering chat conversations. Reply with EXACTLY ONE of these labels, verbatim, on a single line, with no surrounding quotes, prose, or punctuation:
${categories.map(c => `- ${c}`).join("\n")}`;

    const raw = await runClaudePrint({
      model: "haiku",
      system,
      prompt: transcript,
      timeoutMs: 15_000,
    });

    const picked = pickCategory(raw, categories); // case-insensitive trim, fallback to last entry ("Other" by convention)

    await db
      .insert(_conversationCategories)
      .values({ conversationId, category: picked, source: "haiku", model: "haiku" })
      .onConflictDoUpdate({
        target: _conversationCategories.conversationId,
        set: { category: picked, source: "haiku", model: "haiku", classifiedAt: new Date() },
      });

    categoriesResource.notify();
  },
});
```

`pickCategory(raw, list)` is a pure helper: trim, drop trailing punctuation, case-insensitive match against `list`. If no match → return the last entry in the configured list (by convention this is "Other"; documented in the config description). Tested with a small unit test alongside the file.

### Trigger

Bind to the existing `afterTurn` event from `conversations` (see `plugins/conversations/server/index.ts` and `plugins/conversations/plugins/summary/server/handle-generate.ts` for binding examples):

```ts
trigger({
  event: afterTurn,
  job: classifyConversationJob,
  input: ({ conversationId }) => ({ conversationId }),
});
```

The job is idempotent (the existing-row check skips when a category is already set), so we don't need to filter the trigger to "first turn only" — `afterTurn` firing repeatedly is harmless and the second-onwards calls short-circuit cheaply.

### Routes — `internal/routes.ts`

- `POST /api/conversation-category/:conversationId/classify` → enqueue `classifyConversationJob` with `force: true`.
- `POST /api/conversation-category/:conversationId` `{ category }` → manual override; validates `category` against current configured list, upserts with `source: "manual"`, notifies the resource.

### Components

- `CategoryChipBase` — shared visual: rounded-full pill, deterministic color hashed from the label (so categories stay color-stable as the list edits). Used by both the toolbar and row.
- `CategoryChipToolbar` — reads `conversationPane.useData()`; reads `categoriesResource` for the conversation; on click opens a popover with: configured categories list (radio-select for manual override), divider, "Re-classify with Haiku" button.
- `CategoryChipRow` — same as toolbar but smaller (`text-[10px]`), no popover (clicks pass through to the row's normal navigate behavior). If no category yet, render nothing (no placeholder).

### Hook — `useCategoryFor(conversationId)`

```ts
export function useCategoryFor(conversationId: string): string | null {
  const all = useResource(categoriesResource, {});
  return all?.find(r => r.conversationId === conversationId)?.category ?? null;
}
```

## Verification

1. `./singularity build` — confirms migrations generate cleanly, frontend + server build, gateway re-registers.
2. Open `http://<worktree>.localhost:9000`. Create a fresh conversation with a clear bug-report prompt ("the homepage 500s on /foo"). After the first assistant turn, the toolbar chip should appear (≈ within ~5 s). The same chip should appear on the sidebar row.
3. Click the toolbar chip. Popover lists the five default categories. Pick a different one — chip updates immediately on both surfaces (manual override path).
4. Click "Re-classify with Haiku" — chip changes back to Haiku's pick.
5. Open Settings → Conversation Categories. Edit the list (add "Refactor", remove "Other"). Confirm the popover lists the updated set on next render. Existing conversations keep their (now possibly-removed) category until re-classified — that's intentional and matches `quick-prompts`' behavior.
6. Verify `./singularity check --plugin-boundaries` passes (new plugin's barrel is pure; cross-plugin imports go through `@plugins/<name>/{web,server,shared}` only).
7. Manual unit test: run the unit test for `pickCategory` (case differences, trailing periods, garbage output → fallback).
