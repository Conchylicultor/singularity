# Facets v3 Phase 3 — `commands` facet render sub-plugins

## Context

The facets v3 migration (`research/2026-06-02-global-facets-rendering-separation-v3.md`,
Phase 3) splits each facet's three **browser** renderers — diff (PR review), detail
(Forge plugin pane), catalog (Forge aggregated tables) — into `web/` sub-plugins so that
adding a facet surface means adding a folder subtree and touching no consumer.

Seven of nine facets already have their render trio: `contributions`, `cross-refs`,
`db-schema`, `exports`, `resources`, `routes`, `slots`. The **`commands`** facet still has
none — it is currently rendered nowhere in detail or catalog, and has no diff renderer.
The design doc notes commands "need new `toComparable`s" (no legacy `commandStrings()`
existed in `compute-plugin-diff.ts`).

This plan adds the three browser renderers for `commands`, following the `db-schema`
precedent byte-for-byte.

## Facet data shape

`commands/facet/index.ts` extracts `defineCommand()` calls from each plugin's
`web/commands.ts`. Data is **`CommandDef[]`** (a bare array, unlike `db-schema`/`resources`
which wrap in an object):

```ts
interface CommandDef { memberName: string; commandId: string; groupName: string; }
```

e.g. `Shell.Toast: defineCommand<ToastArgs, void>("shell.toast")` →
`{ memberName: "Toast", commandId: "shell.toast", groupName: "Shell" }`.
`renderDoc` already renders these as `` `Shell.Toast` `` (`${groupName}.${memberName}`).

## Registration is automatic

`plugins/framework/plugins/web-sdk/core/web.generated.ts` is codegen'd from the filesystem
by `./singularity build`; `dependsOn` is auto-derived from imports. Creating the three
`plugins/render-*/web/index.ts` barrels (+ `package.json`) is all that's needed — no
manual registry edits.

## Changes

### 1. `core/` — add `toComparable` (pure, runtime-agnostic)

- **NEW** `plugins/plugin-meta/plugins/facets/plugins/commands/core/to-comparable.ts`
  ```ts
  import type { CommandDef } from "./types";

  /** Diff projection: one `Group.Member` string per command, mirroring renderDoc
   *  (facet/index.ts). No legacy commandStrings() existed; this defines the diff. */
  export function commandsToComparable(data: CommandDef[]): string[] {
    return data.map((c) => `${c.groupName}.${c.memberName}`);
  }
  ```
- **EDIT** `commands/core/index.ts` — add `export { commandsToComparable } from "./to-comparable";`

### 2. `plugins/render-diff/web` — `DiffRenderer`

Mirror `db-schema/plugins/render-diff`. Files:
- `package.json` — name `@singularity/plugin-plugin-meta-facets-commands-render-diff`
- `web/index.ts`:
  ```ts
  PluginChangesSlots.DiffRenderer({
    facetId: "commands",
    label: "Commands",
    toComparable: (data) => commandsToComparable(data as CommandDef[]),
  })
  ```
- `CLAUDE.md` — prose only (build inserts the AUTOGENERATED reference block).

### 3. `plugins/render-detail/web` — `PluginView.Section`

Mirror `resources/plugins/render-detail` (flat list, no cross-refs). Files:
- `package.json` — `…-commands-render-detail`
- `web/components/commands-detail-section.tsx` — reads `node.facets["commands"]` as
  `CommandDef[]`; returns `null` if absent/empty. Renders one row per command:
  `<code>{groupName}.{memberName}</code>` left, `commandId` muted/right. Uses `Section`
  from `@plugins/plugin-meta/plugins/plugin-view/web` (type-only `CommandDef` import from
  the facet core is erased — browser-safe).
- `web/index.ts` — `PluginViewSlots.Section({ id: "commands", label: "Commands", component: CommandsDetailSection })`
- `CLAUDE.md`.

### 4. `plugins/render-catalog/web` — `Catalog.FacetTable`

Mirror `db-schema/plugins/render-catalog`. Files:
- `package.json` — `…-commands-render-catalog`
- `web/commands-facet-table.tsx` — `defineFacetTable<CommandRow>` with `facetId: "commands"`,
  `label: "Commands"`, `icon: MdKeyboardCommandKey` (verified present in `react-icons/md`).
  Columns: **Command** (`${groupName}.${memberName}`, mono), **Command ID** (`commandId`,
  muted mono), **Plugin** (`<PluginChip hierarchyId=… />`). `rows(entries)` flattens
  `entry.data as CommandDef[]` across plugins; `rowKey: (r) => \`${hierarchyId}:${commandId}\``.
- `web/index.ts` — `Catalog.FacetTable(commandsFacetTable)`
- `CLAUDE.md`.

## Critical files (reference precedent)

| File | Role |
|---|---|
| `…/facets/plugins/db-schema/plugins/render-{diff,detail,catalog}/**` | exact structural precedent |
| `…/facets/plugins/resources/plugins/render-detail/web/components/resources-detail-section.tsx` | flat-list detail precedent (commands has no cross-refs) |

   `commands/plugins/render-*` entries; `commands/CLAUDE.md` + the three new `CLAUDE.md`
   get their AUTOGENERATED reference blocks.

3. Forge detail pane for a plugin with commands (e.g. `shell`) shows a **Commands** section
   (`Shell.Toast` + `shell.toast`). Forge catalog shows a **Commands** tab aggregating all
   `defineCommand` across plugins. Verify via `e2e/screenshot.mjs` against
   `http://<worktree>.localhost:9000`.
   - Note: per design doc, render sub-plugins stay dormant until the Phase 4 `skipBarrelImport`
     gate flips — `node.facets` may be empty in the served tree today. Commands is a
     **static** facet (parsed from source, no barrel import), so it should populate once
     `buildPluginTree` runs extract unconditionally; if the served tree still gates facets,
     detail/catalog will render empty until Phase 4. The diff renderer is exercised by the
     PR review pane. This matches the dormancy caveat recorded for the other Phase 3 facets.
