# Auto-generate `plugins/CLAUDE.md` at build time

## Context

`plugins/CLAUDE.md` is the index agents consult to understand what each plugin defines and contributes. Today it is hand-maintained and already drifts from reality (e.g. `welcome` and `db-smoketest` missing before recent edits; `Shell.Main` not listed as a defined slot). Every new plugin, slot, or route is a doc-update risk.

All the information in that file is statically declared in code:

- Frontend plugins export a `PluginDefinition` with a `contributions: Contribution[]` array, where each contribution is `{ _slotId, ...props }` (see `plugin-core/slots.ts:12`).
- Server plugins export a `ServerPluginDefinition` whose `httpRoutes` / `wsRoutes` keys are literal strings like `"GET /api/build"` or `"/ws/logs"` (see `server/src/types.ts`).
- Plugin registries are static lists: `web/src/plugins.ts`, `server/src/plugins.ts`.
- Nested plugins live on disk under `plugins/<parent>/plugins/<child>/…` — nesting is recoverable from import paths.

Goal: generate the plugin index during `./singularity build` so it always matches the source.

## Proposed format

Keep a small **static header** (architecture pointers) and regenerate everything below a marker. The header lives in `plugins/CLAUDE.md.header` (or an inline `<!-- header -->…<!-- /header -->` block); the generator replaces the body between `<!-- AUTOGENERATED:BEGIN -->` and `<!-- AUTOGENERATED:END -->`.

Body layout — one section per **top-level** plugin; nested plugins become subsections. Each plugin shows only the axes it actually uses (empty axes are omitted):

```markdown
# Plugins

Architecture: [plugin-core/CLAUDE.md](../plugin-core/CLAUDE.md) · [server/CLAUDE.md](../server/CLAUDE.md)

<!-- AUTOGENERATED:BEGIN — do not edit; regenerated by `./singularity build` -->

### `shell`
- Defines slots: `Shell.Sidebar`, `Shell.Toolbar`, `Shell.Route`, `Shell.Main`
- Defines commands: `Shell.OpenPane`, `Shell.Toast`
- Frontend contributions:
  - `Core.Root` → `ShellLayout`

### `welcome`
- Frontend contributions:
  - `Shell.Route` pattern=`/`

### `conversation`
- Defines slots: `Conversation.Toolbar`
- Frontend contributions:
  - `Shell.Route` pattern=`/c/:id`
- Nested plugins:
  #### `conversation/vscode`
  - Frontend contributions:
    - `Conversation.Toolbar` label="VSCode"
  #### `conversation/open-app`
  - Frontend contributions:
    - `Conversation.Toolbar` label="Open"

### `build`
- Frontend contributions:
  - `Shell.Toolbar` group=`actions` → `BuildButton`
- Server routes:
  - `POST /api/build`

### `logs`
- Frontend contributions:
  - `Shell.Sidebar` title="Logs"
- Server routes:
  - `GET /api/logs/channels`
  - `WS /ws/logs`

### `db-smoketest`
- Server routes:
  - `GET /api/smoketest`
  - `POST /api/smoketest`
- DB schema: `plugins/db-smoketest/server/schema.ts`

<!-- AUTOGENERATED:END -->
```

Formatting rules the generator follows:

- Heading depth = nesting depth (top-level `###`, first-level nested `####`, etc.).
- Per-contribution line: `` `SlotId` `` followed by the contribution's identifying props in insertion order — short, scannable. Identifying props are the serializable ones: `title`, `label`, `pattern`, `group`, `variant`. The `component` (if present) renders as `→ ComponentName` using `component.displayName ?? component.name`. Unknown/unserializable props are skipped.
- Plugin order mirrors `web/src/plugins.ts` (frontend-registered order), with server-only plugins appended in `server/src/plugins.ts` order.
- Empty axes are omitted — no `- Frontend contributions: (none)` lines.
- DB schema line is included only when `plugins/<name>/server/schema.ts` exists on disk.

## Extraction strategy — runtime introspection, not AST

Because plugin registries are plain Bun-compatible TS modules with static imports and literal data, a tiny Bun script can `import` them and read the in-memory objects. This avoids a TypeScript AST walker and stays correct-by-construction with the framework.

Two small framework additions make this clean:

1. **Expose the slot id on the slot factory.** `plugin-core/slots.ts:11` — change `defineSlot` so the returned factory has a non-enumerable `id` field (`slot.id = id`). Cost: one line. Benefit: `Object.values(Shell).map(s => s.id)` yields the defined slot IDs for a plugin without parsing source.
2. **Same for `defineCommand`** in `plugin-core/commands.ts` — attach `.id`. Enables listing commands per plugin.

With those, the generator (`cli/src/docgen.ts`) does:

```ts
import { plugins as webPlugins } from "../../web/src/plugins";
import { plugins as serverPlugins } from "../../server/src/plugins";
// For each plugin, find its source dir by matching `plugin.id` to folder
// names under plugins/** (recursive). Read its `slots.ts` / `commands.ts`
// if present and enumerate exported Slot/Command objects to get defined IDs.
```

Source-dir discovery: glob `plugins/**/web/index.ts` and `plugins/**/server/index.ts`, dynamic-`import` each, match default-export `.id` to the registry plugin. That also yields the nesting path (`plugins/conversation/plugins/vscode/web/index.ts` → parent `conversation`, child `vscode`).

### Alternative considered (rejected)

AST parsing via `ts-morph` / Babel. More code, more fragile to stylistic changes (e.g. spread, constant extraction), and buys nothing — everything we need is already materialized at runtime.

## Files to add / modify

- **New** `cli/src/docgen.ts` — the generator. Exports `generatePluginDocs({ root }): Promise<void>`. Reads both registries, discovers source dirs, renders Markdown, writes `plugins/CLAUDE.md` between the markers.
- **New** `cli/src/checks/plugins-doc-in-sync.ts` — mirrors `migrations-in-sync`: regenerate in memory, diff against on-disk `plugins/CLAUDE.md`, fail if different. Register in `cli/src/checks/index.ts`. This is what protects against "agent committed without `./singularity build`".
- **Modify** `cli/src/commands/build.ts:73` — add step between frontend build and registry write: `await generatePluginDocs({ root });`. Cheap, so no flag needed.
- **Modify** `plugin-core/slots.ts:11` — attach `.id` to the slot factory.
- **Modify** `plugin-core/commands.ts` — attach `.id` to the command function (same pattern).
- **Modify** `plugins/CLAUDE.md` — replace hand-written body with the `<!-- AUTOGENERATED:BEGIN/END -->` block; keep the short header.

## Verification

1. `./singularity build` in the worktree; confirm `plugins/CLAUDE.md` is regenerated and diff matches what a human would write for the current plugins (spot-check `conversation/vscode`, `logs` WS route, `db-smoketest` schema line).
2. Hand-edit inside the autogen block, re-run `./singularity build` — change should be overwritten.
3. Hand-edit outside the block (header) — change should survive.
4. `./singularity check --plugins-doc-in-sync` passes on a clean tree; fails after manually reverting the file. `./singularity push` refuses to ship when out of sync.
5. Add a throwaway `plugins/foo/web/index.ts` exporting a `PluginDefinition` registered in `web/src/plugins.ts`, rebuild, confirm it appears; remove it and confirm it disappears.
