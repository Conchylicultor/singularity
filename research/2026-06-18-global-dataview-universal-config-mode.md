# Universalize DataView config mode (reorder-parity)

**Date:** 2026-06-18
**Category:** global (touches `primitives/data-view`, `framework/tooling/codegen`, and 8 consumer plugins)

## Context

`<DataView>` has two runtime modes, chosen per mount by whether the consumer registered
`viewsDescriptor(storageKey)` via `ConfigV2.{Register,WebRegister}`:

- **Default mode** (8 of 9 consumers): one synthesized view-instance per view-type,
  no instance actions, sort/filter in **localStorage** (device-local, not git-promotable).
- **Config mode** (only `sonata:library`): N config-authored named views, full
  add/rename/duplicate/delete/reorder + per-view saved sort/filter persisted to a
  **config row** (git-promotable), editable view-switcher.

There is no architectural reason for the split — every DataView already passes a required
`storageKey`, and "favorites / recents / saved filters" are legitimate on every surface
(a launcher wants a "recently used" view; a nav tree wants a "conflicts only" view). The
only blocker is that `config_v2` registration is **static, barrel-time** while `storageKey`
is a runtime prop. `sonata` worked around this by hand-registering in its own barrel.

**Goal:** make **every** DataView config mode and **delete default mode**, mirroring the
`reorder` primitive — where *every* `defineRenderSlot` is unconditionally config-backed
with **zero per-consumer registration boilerplate**, via build-time codegen + a central
registration in the primitive's own barrel.

**Decisions (confirmed with user):**
- Registration discovery: a **`defineDataView("id")` marker** scraped by codegen
  (type-safe branded `DataViewId`; mirrors how reorder scrapes `defineRenderSlot`).
- Scope: **force on all, no opt-out** — every DataView gets the editable switcher.

## How reorder does it (the model we copy)

- `plugins/reorder/shared/reorderable-slots.generated.ts` — build-time list of every slot,
  generated by `plugins/framework/plugins/tooling/plugins/codegen/core/reorderable-slots-gen.ts`.
- `plugins/reorder/shared/directive.ts` — `reorderDirectiveDescriptor(slotId)` →
  `defineConfig({ name: slotId, source: "reorder", ... })`: **slotId IS the descriptor name**,
  giving a distinct store path per slot.
- `plugins/reorder/{web,server}/internal/config-registrations.ts` — map over the generated
  list, emit one `ConfigV2.{WebRegister,Register}` per slot, **all under the reorder pluginId**,
  spread into the reorder barrel's `contributions`.
- `plugins/reorder/web/internal/descriptors.ts` — `Map<slotId, descriptor>`, reference-stable.
- Runtime: `useConfig(reorderDescriptors.get(slotId))` — no scopeId, all render sites share one row.

**Settled constraint:** a single registered descriptor cannot serve many ids via `scopeId`
(an unregistered descriptor throws on read; scopeId only multiplexes rows *under* a registered
descriptor). So each id needs a registered descriptor — centralized under the data-view pluginId,
distinguished by `descriptor.name = id`.

## Implementation

### 1. `defineDataView` marker + branded type
New `plugins/primitives/plugins/data-view/core/internal/define-data-view.ts`:
```ts
export type DataViewId = string & { readonly __dataViewId: unique symbol };
export function defineDataView(id: string): DataViewId {
  if (!/^[a-zA-Z0-9._-]+$/.test(id)) throw new Error(`defineDataView: id "${id}" must match [a-zA-Z0-9._-]`);
  return id as DataViewId;
}
```
- Export `defineDataView` + `DataViewId` from `core/index.ts`; re-export from `web/index.ts`.
- The grammar assertion **bans `:`** so `descriptor.name` is filename-safe at
  `config/<hier>/<name>.jsonc` (dots are already used by existing config names; colon is the
  only novel char). Convention: `defineDataView` calls live under a plugin's `web/`.

### 2. Codegen pass (riskiest part)
New `plugins/framework/plugins/tooling/plugins/codegen/core/data-views-gen.ts`, modeled on
`reorderable-slots-gen.ts`:
- `collectDataViews(root)`: reuse the cached `buildEnrichedTree(root)`; for each node, scan its
  `web/**` source with `findMarkerCalls(maskSource(src, { strings: false }), "defineDataView")`
  (the blessed scanner — `plugins/plugin-meta/plugins/parse-utils/core/find-marker-calls.ts`;
  do **not** hand-roll a regex). Extract the first string-literal arg (reuse the id regex from
  `parseSlotCalls`). Dedupe + sort. **Only the set of ids is needed** (no pluginId — all register
  under data-view), which is simpler than the reorder manifest.
- `renderDataViewsManifest(root)`, `dataViewsManifestPath(root)`, `generateDataViews({ root })` —
  same trio shape as reorder. Export from the codegen `core/index.ts`.
- New `plugins/primitives/plugins/data-view/shared/data-views.generated.ts` (commit an empty-array
  **stub first** so `descriptors.ts` imports resolve before the first build):
  ```ts
  export interface DataViewEntry { id: string }
  export const dataViews: DataViewEntry[] = [ /* { id: "agents-list" }, ... */ ];
  ```
- Wire into `plugins/framework/plugins/cli/bin/commands/build.ts`: call `generateDataViews({ root })`
  immediately after `generateReorderableSlots` and **before** `generateConfigOrigins` (the new
  descriptors must be registered when origins regenerate).
- New check plugin `plugins/framework/plugins/tooling/plugins/checks/plugins/data-views-in-sync/`
  (copy `reorderable-slots-in-sync/` verbatim) — fails on manifest drift.

### 3. Descriptor: name = id
`plugins/primitives/plugins/data-view/shared/views-config.ts` — change
`defineConfig({ name: "views", ... })` → `name: id` (keep the module-cache `Map<id, descriptor>`
for reference identity). Store path becomes `config/primitives/data-view/<id>.jsonc`.

### 4. Central registration in the data-view barrels
- New `web/internal/descriptors.ts`:
  ```ts
  export const dataViewDescriptors = new Map(dataViews.map(v => [v.id, viewsDescriptor(v.id)]));
  export const dataViewDescriptorEntries = dataViews.map(v => ({ id: v.id, descriptor: dataViewDescriptors.get(v.id)! }));
  ```
- New `web/internal/config-registrations.ts`: map `dataViewDescriptorEntries` →
  `ConfigV2.WebRegister({ descriptor, pluginId: asPluginId("primitives.data-view") })`.
- New `server/internal/config-registrations.ts`: same over `ConfigV2.Register` (server builds its
  own descriptor map — server identity is independent of web, per the existing views-config doc).
- `web/index.ts`: spread `...dataViewConfigContributions` into `contributions`; add
  `export { defineDataView }` + `export type { DataViewId }`.
- `server/index.ts`: set `contributions: dataViewConfigRegistrations`.
- `web/internal/use-views-config.ts`: read the descriptor via `dataViewDescriptors.get(storageKey)`
  (guarantees reference identity against the registration).

### 5. Collapse to one path / deletions
- `web/components/data-view.tsx`: delete the `isRegistered` / `useConfigRegistrations` /
  `DefaultDataView` vs `ConfigDataView` branch. `DataView` always builds `useConfigViewModel`
  and renders `DataViewInner`. In `DataViewInner` the switcher branch collapses to always render
  `<EditableViewSwitcher>`. Remove the stale "mode once per mount" doc comment.
- `web/internal/use-view-model.ts`: delete `useDefaultViewModel`; make `ViewModel.actions`
  non-nullable (`actions: ViewActions`).
- `web/internal/use-view-state.ts`: delete `setLocalSort` / `setLocalFilter` and the `sort`/`filter`
  fields (default-mode-only). **Keep** localStorage for `active-view`, `query`, `expanded`
  (device-local; both modes always used these). Reader stays tolerant of legacy blobs.
- `web/components/view-switcher.tsx` (the data-view-local read-only wrapper): now dead — **delete**
  it and its import. ⚠️ Do **not** touch the separate `primitives/view-switcher` primitive that
  `EditableViewSwitcher` composes.
- `core/internal/types.ts`: `storageKey: string` → `storageKey: DataViewId`.

`storageKey="<id>"` → `storageKey={X}`. **No `web/index.ts` / `server/index.ts` registration is
added** — that's the whole point (central). Web-only consumer plugins thus avoid needing a new
server barrel.

## UX consequence (accepted: force on all)
Every surface — including the **Home launcher** and **Settings config-nav** — now shows the
editable view-switcher with a `+ View` affordance. This is the intended outcome (favorites /
recents / game-views as real saved views). Screenshot-review the two chrome surfaces (launcher,
settings-nav) and the `embedded` tweakcn browser for polish; nothing blocks them functionally.
