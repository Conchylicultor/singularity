# Auto-resolve generated-file conflicts during `./singularity push`

## Context

Agents working in parallel worktrees keep hitting merge conflicts in `./singularity push` because every build commits regenerated artefacts that two agents inevitably touch. The two recurring offenders are:

1. **Docs**: `docs/plugins-compact.md`, `docs/plugins-details.md`, and the `<!-- AUTOGENERATED:BEGIN -->` blocks inside every `plugins/**/CLAUDE.md`. Generated by `cli/src/docgen.ts::generatePluginDocs` (writes at lines 1076, 1077, 1084).
2. **Drizzle migrations**: `server/src/db/migrations/*.sql`, `meta/*_snapshot.json`, `meta/_journal.json`. Generated by `cli/src/migrations.ts::generateMigration`.

`push.ts` line 101 stages everything with `git add -A`, then line 157 rebases the worktree branch onto `main`. When two agents both modified the same generated content, the rebase aborts (line 174) — the agent then has to manually resolve files that are 100% deterministic functions of source code already merged a few commits earlier.

The fix: teach git to auto-resolve these specific paths via custom merge drivers, then re-derive the canonical content from the rebased source tree in a post-rebase normalize step. The agent only ever sees a real conflict when the *source* (e.g. `schema.ts`, plugin `index.ts`, hand-written CLAUDE.md prose) actually conflicted — never for downstream artefacts.

## Why a hybrid design (drivers + post-rebase normalize)

A merge driver alone is not sufficient because:

- Drivers only fire on **modify-modify** conflicts. When agent B's commit *adds* a brand-new `YYYYMMDD_HHMMSS_<hash>__newcol.sql` file, the rebase replays it as a clean add — no driver invocation, no chance to clean it up — even though the file is now stale (its hash and SQL were derived from a pre-rebase snapshot).
- Even when drivers fire, the cleanest output is "regenerate from source" — but at driver-invocation time the source tree may still be partially merged, and the driver can't easily reach across files. Doing the regen *after* the full rebase succeeds gives us a consistent source tree to derive from.

So drivers do the minimum needed to keep `git rebase` running (don't abort on cosmetic conflicts), and a post-rebase normalize step handles the canonicalization.

## Safety: hand-edited migrations

Agents sometimes hand-edit a freshly-generated `.sql` migration before pushing (e.g. to add a manual data backfill or fix a column type drizzle got wrong). The naïve post-rebase normalize would silently delete those edits, because `resetBranchLocalMigrations` (`cli/src/migrations.ts:151`) deletes any branch-local SQL file unconditionally.

**Detection**: each migration filename embeds the SHA-256 prefix of the SQL content (`<timestamp>_<sha8>__<slug>.sql`, see `cli/src/migrations.ts` `renameMigrations`). For each branch-local SQL file (those absent from `origin/main`), recompute SHA-256 of the on-disk content and compare against the embedded `sha8`. If any file mismatches, the agent edited it.

**Behavior on mismatch**: abort the auto-resolve before any deletion. Surface an error like:

```
hand-edited migration detected:
  20260502_140000_3a7f9b21__add_col.sql
  filename hash:  3a7f9b21
  content  hash:  8c2d4e10
auto-rebase would discard your edits. Resolve the rebase manually.
```

`process.exit(1)` from the normalize step. Push.ts surfaces the failure the same way as a real conflict — agent falls back to the existing manual-rebase flow, strictly no worse than today.

Note: hand-editing migrations *already on `main`* is a pre-existing footgun unrelated to this plan — the runtime parses the applied-hash from the filename (`server/src/db/migrate.ts:19`), so an edit to a merged migration is silently ignored on running databases. Out of scope here.

### 2. Merge driver scripts (NEW, `cli/git-merge-drivers/`)

Each script receives `%O %A %B %P` (ancestor, current/temp, other, working-tree path). Git uses the final content of `%A` as the merged result.

**`regen-docs.sh`** — for fully-generated docs. The current/`%A` side is the upstream version (during rebase). We keep it; the post-rebase normalize step will overwrite with canonical content derived from the rebased plugin sources.

```sh
#!/bin/sh
# args: %O %A %B %P
# Take the upstream/current side. Post-rebase normalize regenerates canonical content.
exit 0
```

**`regen-claudemd.sh`** — per-plugin `CLAUDE.md` mixes hand-written prose with an autogen block. Run a 3-way merge for the prose, then re-run docgen so the autogen block is canonical regardless of how it conflicted. If real prose conflict markers remain, exit 1 so the agent sees the conflict.

```sh
#!/bin/sh
# args: %O %A %B %P
# 1. 3-way merge into a temp file (handles hand-written prose).
git merge-file -p "$2" "$1" "$3" > "$2.tmp" || true
mv "$2.tmp" "$2"
# 2. Stage the merged content into the real path so docgen reads it.
cp "$2" "$4"
# 3. Regenerate the autogen block in place (writeIfChanged keeps prose intact).
bun cli/src/index.ts regen-docs >/dev/null 2>&1 || exit 1
# 4. Copy the regen'd file back to %A so git uses it.
cp "$4" "$2"
# 5. If conflict markers survived (i.e. real prose conflict), surface it.
if grep -qE '^(<{7}|={7}|>{7}) ' "$2"; then exit 1; fi
exit 0
```

**`regen-migrations.sh`** — same shape as `regen-docs.sh`. The actual reset+regenerate happens in the post-rebase normalize step (because pure-add SQL files never trigger a driver).

```sh
#!/bin/sh
# args: %O %A %B %P
# Take the upstream side. Post-rebase normalize will reset branch-local migrations
# and regenerate fresh ones from the rebased schema.
exit 0
```

All three need `chmod +x`.

### 3. `cli/src/git/register-merge-drivers.ts` (NEW)

Idempotent install — reads current config, only writes if changed. Mirrors the shape of `ensureHooksPath`.

```ts
import { execFileSync } from "node:child_process";
import { join } from "node:path";

interface Driver { name: string; script: string; }

const DRIVERS: Driver[] = [
  { name: "regen-docs",       script: "cli/git-merge-drivers/regen-docs.sh" },
  { name: "regen-claudemd",   script: "cli/git-merge-drivers/regen-claudemd.sh" },
  { name: "regen-migrations", script: "cli/git-merge-drivers/regen-migrations.sh" },
];

export async function registerMergeDrivers({ root }: { root: string }): Promise<void> {
  for (const d of DRIVERS) {
    const key = `merge.${d.name}.driver`;
    const want = `${join(root, d.script)} %O %A %B %P`;
    let current = "";
    try { current = execFileSync("git", ["config", "--local", "--get", key], { cwd: root, encoding: "utf8" }).trim(); } catch {}
    if (current !== want) {
      execFileSync("git", ["config", "--local", key, want], { cwd: root });
    }
    // Optional: set merge.<name>.name for `git config --list` clarity.
  }
}
```

`git config --local` from inside any worktree updates the shared `.git/config` (worktrees don't enable `extensions.worktreeConfig` here), so one install covers all worktrees.

**`cli/src/commands/regen-migrations.ts`**:

```ts
import { createHash } from "node:crypto";
import { readdirSync, readFileSync } from "node:fs";
import { join } from "node:path";
import { generateMigration, resetBranchLocalMigrations } from "../migrations.ts";

export async function runRegenMigrations({ root, migrationName }: { root: string; migrationName?: string }): Promise<void> {
  await assertNoHandEditedBranchLocalMigrations({ root });
  await resetBranchLocalMigrations({ root });
  await generateMigration({ root, migrationName: migrationName ?? deriveName(root) });
}

// Each branch-local migration filename embeds the sha256 prefix of its SQL.
// If the on-disk content no longer matches, the agent hand-edited it and
// auto-regen would silently discard those edits — abort instead.
async function assertNoHandEditedBranchLocalMigrations({ root }: { root: string }): Promise<void> {
  const migrationsDir = join(root, "server/src/db/migrations");
  const tracked = await listMigrationsOnRef(root, "origin/main", "main");
  const offenders: { file: string; expected: string; actual: string }[] = [];
  for (const f of readdirSync(migrationsDir)) {
    if (!f.endsWith(".sql")) continue;
    if (tracked.has(f)) continue; // already on main; not in scope here
    const m = f.match(/^\d{8}_\d{6}_([0-9a-f]{8})__/);
    if (!m) continue; // unfamiliar shape — leave alone
    const expected = m[1];
    const actual = createHash("sha256").update(readFileSync(join(migrationsDir, f))).digest("hex").slice(0, 8);
    if (expected !== actual) offenders.push({ file: f, expected, actual });
  }
  if (offenders.length) {
    console.error("hand-edited migration detected:");
    for (const o of offenders) console.error(`  ${o.file}\n    filename hash: ${o.expected}\n    content  hash: ${o.actual}`);
    console.error("auto-rebase would discard your edits. Resolve the rebase manually.");
    process.exit(1);
  }
}

function deriveName(root: string): string {
  // Default name for auto-resolved rebases. Parsing the deleted SQL filename's
  // slug to preserve agent intent could be a v2 nicety.
  return `merged_${new Date().toISOString().slice(0, 10).replaceAll("-", "")}`;
}
```

(`listMigrationsOnRef` mirrors the existing `listTrackedMigrationBasenames` helper at `cli/src/migrations.ts:191`; either reuse it via export or duplicate the few lines. Verify the actual exported names of `resetBranchLocalMigrations` and `generateMigration` against `cli/src/migrations.ts` during implementation — `resetBranchLocalMigrations` is currently file-local and will need to be exported.)

Register as `singularity regen-migrations`.

```ts
// Post-rebase normalize: regenerate canonical content from the rebased source
// tree, in case merge drivers took the upstream side for generated artefacts.
await run(["bun", "cli/src/index.ts", "regen-docs"]);
await run(["bun", "cli/src/index.ts", "regen-migrations"]);

const { stdout: dirty } = await run(["git", "status", "--porcelain"]);
if (dirty.trim()) {
  await run(["git", "add", "-A"]);
  await run([

    "--amend", "--no-edit",

}
```

The same block should be added in the `--from-main` branch of `push.ts` (lines ~121–128) for completeness, since builds from main can also leave generated dirt. Apply at the same logical position in that flow.

## Verification

End-to-end: simulate the conflict scenario.

1. **Setup**: `git config --local --get merge.regen-docs.driver` → confirms registration after `./singularity build`.
2. **Doc conflict reproduction**:
   - From `main`, branch A: add a new plugin under `plugins/foo/`. `./singularity build`. `./singularity push -m "foo"`.
   - From `main` (pre-A), branch B: add a new plugin under `plugins/bar/`. `./singularity build`. `./singularity push -m "bar"`.
   - Expected: B's push rebases over A. `docs/plugins-compact.md` would have conflicted; `regen-docs` driver eats it, post-rebase normalize regenerates with both `foo` *and* `bar`. Final commit on main contains the canonical doc with both plugins. No agent intervention.
3. **Migration conflict reproduction**:
   - From `main`, branch A: add a column to a table in `schema.ts`. `./singularity build` (generates migration). `./singularity push -m "col-a"`.
   - From `main` (pre-A), branch B: add a different column to the same table. `./singularity build`. `./singularity push -m "col-b"`.
   - Expected: B rebases over A. B's stale migration files are removed by `regen-migrations`. Drizzle-kit diffs the rebased `schema.ts` (which has both columns after a clean rebase of the schema source itself) against A's tip snapshot, emits one new migration containing only B's column. Final main has both migrations in correct timestamp order.
4. **Real prose conflict surfaces**: edit the same line of a plugin's `CLAUDE.md` prose section in two branches; confirm `regen-claudemd` exits 1 and the rebase pauses with conflict markers in the prose section only (autogen block is clean).
5. **Real schema conflict surfaces**: have both branches add a column with the same name; confirm rebase aborts with conflict in `schema.ts` (no driver applies there) — the agent must resolve manually.
5b. **Hand-edited migration is detected**: branch B adds a column, runs `./singularity build`, then hand-edits the generated `.sql` (e.g. adds a `-- backfill` comment). After branch A pushes, B pushes — confirm the post-rebase normalize aborts with the "hand-edited migration detected" error instead of silently overwriting B's edit.
6. **Idempotence check**: run `./singularity build` twice in a row; second run should not modify any tracked file (`git status` clean both times). Confirms `writeIfChanged` + `register-merge-drivers` are idempotent.
7. **Fresh worktree**: cut a brand-new worktree, run `./singularity build` once, confirm `git config --get merge.regen-docs.driver` is set.

## Out of scope

- Optimizing for the case where `regen-docs` is invoked many times during a single rebase (once per conflicted CLAUDE.md). Each invocation is idempotent and only ~1–3s; acceptable for v1.
- Preserving B's original migration slug when auto-regenerating (we use `merged_YYYYMMDD`). Could be added later by parsing the deleted SQL filename in `resetBranchLocalMigrations`.
- The drift-warning case where B already applied its old migration to its local DB before the rebase — re-fork the worktree DB resolves it. Already documented in CLAUDE.md.
