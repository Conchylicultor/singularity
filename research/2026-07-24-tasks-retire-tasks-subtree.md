# Retire `TasksSubtree` and the `tasks-subtree` DataView surface

## Context

`TasksSubtree` (`plugins/tasks/plugins/task-list/web/components/tasks-subtree-view.tsx`)
renders the task tree scoped two mutually-exclusive ways — by `rootTaskId` (a folder
subtree) or by `members` (an arbitrary id set as a creation forest). It owns its own
`<DataView>` under the `tasks-subtree` storage key, with a matching config surface at
`config/tasks/task-list/tasks-subtree.jsonc`.

Both of its modes have lost their consumers, in two steps:

- **`7947a9f8f`** (*drop the inline task tree view*) removed the `rootTaskId` consumer
  (`task-detail`'s `task-tree-detail.tsx`) when the task-detail pane collapsed to a
  single detail-only mode.
- **`7d5a6a204`** (*multi-source DataView*) removed the last `members` consumer.
  `task-deps-tree`'s "Created" view could not reuse `<TasksSubtree>` — a merged-surface
  source's `render(bundle)` *is* the inner DataView, so nesting a component that owns
  its own DataView is structurally wrong. It instead composes task-list's exported
  building blocks directly (`plugins/tasks/plugins/task-deps-tree/web/internal/deps-sources.tsx:105-129`).

Today `TasksSubtree` has **zero runtime consumers**. A repo-wide search finds it only in
its own definition, the barrel export, autogenerated docs, and a comment in
`deps-sources.tsx` explaining why it is *not* used. The multi-source design doc
(`research/2026-07-23-global-multi-source-data-view.md:466`) filed this as an explicit
follow-up: *"Delete `TasksSubtree` + the `tasks-subtree` surface/config if it stays
consumer-less."* It stayed consumer-less.

**Outcome:** delete it. The reusable public surface is the *building blocks*, not the
component.

## Decision: delete, not keep

The "keep it as a public reusable surface" case is weak on the merits:

- **The building blocks already are that surface, and strictly dominate it.**
  `task-list/web/index.ts` exports `taskFields`, `clusterTaskHierarchy`, and
  `buildTreeOptions` (plus `Tasks.TaskActions`). A consumer composing those gets
  everything `TasksSubtree` offered *and* can be used as a merged-surface source.
  `TasksSubtree` cannot — its owned `<DataView>` is exactly what disqualifies it. It is a
  dead-end shape in the architecture the repo moved to.
- **Nothing is lost.** The one piece of authored value in `tasks-subtree.jsonc` — the
  `hide-completed` filter + preset — was already copied into
  `config/tasks/task-deps-tree/task-deps-tree.jsonc` under the `created` view.
- **Carrying cost is real, not notional.** Dead code here drags a config file pair, a
  live `config_v2` descriptor, an entry in `data-views.generated.ts`, and doc entries in
  three autogenerated surfaces — all of which must keep compiling as the DataView API
  evolves, with no consumer exercising them.
- **Repo precedent is explicit.** `7947a9f8f`'s own message: when a collapse makes a
  chain of things dead, they are *"all removed rather than left dangling."*

## Changes

### 1. Delete the component and its barrel export

- **Delete** `plugins/tasks/plugins/task-list/web/components/tasks-subtree-view.tsx`.
- **Edit** `plugins/tasks/plugins/task-list/web/index.ts:10` — drop
  `export { TasksSubtree } from "./components/tasks-subtree-view";`.

The three building-block exports on lines 11-15 (`taskFields`, `clusterTaskHierarchy`,
`buildTreeOptions`) **stay** — they are the supported reuse path and `task-deps-tree`
depends on them.

### 2. Prune what the deletion makes dead in `web/internal/tasks-data-view.tsx`

- `readOnlyTaskHierarchy` (line 78) was consumed only by `TasksSubtree`. Its remaining
  role is as the derivation base for `clusterTaskHierarchy` — **drop the `export`
  keyword**, keep the const and its comment.
- `buildTreeOptions`'s `rootTaskId` parameter (lines 93-97) loses its last caller. Both
  survivors pass no root: `tasks-list-view.tsx:36` (`buildTreeOptions({})`) and
  `deps-sources.tsx:109` (`{ readOnly: true, defaultExpanded: true }`). **Remove the
  param**, and with it:
  - `rootId: rootTaskId` from the returned `TreeViewOptions`;
  - `addLabel: readOnly || rootTaskId ? null : "Add"` → `addLabel: readOnly ? null : "Add"`.

  The generic `rootId` capability stays where it belongs — in the tree primitive
  (`plugins/primitives/plugins/data-view/plugins/tree/web/internal/types.ts:50`) — so
  subtree scoping remains expressible if a consumer ever needs it again.

  `readOnly` and `defaultExpanded` both stay: `deps-sources.tsx` uses them.

`taskHierarchy` stays exported — `tasks-list-view.tsx:35` uses it.

### 3. Let the build handle the config, manifest, and docs

Deleting the `defineDataView("tasks-subtree")` marker is the only manual step;
`./singularity build` propagates it. **Do not hand-delete these** — the build's
`pruneOrphanedConfigFiles` (`codegen/core/config-origin-gen.ts:373-403`, invoked from
`generateConfigOrigins:405-434`) removes them, and hand-editing inconsistently is what
trips `config-origins-in-sync`:

- `config/tasks/task-list/tasks-subtree.jsonc` + `tasks-subtree.origin.jsonc` — pruned
  automatically (this is why `7d5a6a204`'s diff shows the retired queue configs as plain
  `D` lines: the agent ran the build and committed the result).
- `plugins/primitives/plugins/data-view/shared/data-views.generated.ts:47` — regenerated
  by `data-views-gen.ts:73-152`, which text-scans `web/**` for marker calls. It is in the
  pre-barrel manifest set, so the same build that drops the id also prunes its configs.
- `plugins/tasks/plugins/task-list/CLAUDE.md` autogen block, `docs/plugins-details.md`,
  `docs/plugins-compact.md` — regenerated by `generatePluginDocs`.

**Known, accepted leftover:** the propagated user-layer copy under
`~/.singularity/config/<worktree>/tasks/task-list/tasks-subtree.*` is unversioned and
deliberately never auto-deleted (`config_v2/server/internal/orphan-audit.ts:39-41` — user
files are classified, never removed). It surfaces as a `noise`-class row in
Debug → Config Orphans and has no runtime effect.
