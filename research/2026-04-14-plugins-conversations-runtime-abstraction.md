# Conversations: runtime abstraction + Claude session id

## Context

The `conversations` plugin today assumes every conversation is backed by a **tmux** session running the `claude` CLI. The poller (`poller.ts`), lifecycle ops (`createConversation`, `deleteConversation` in `tmux.ts`), and the SSE `tmux` event are all hard-wired to that backend.

We want to add a second backend in the future: **API mode**, where Claude runs in-process via the Anthropic Agent SDK (no tmux, no pane_title). Both modes must coexist per-conversation — a user can have some conversations driven by tmux and others by the SDK simultaneously.

Secondary goal: capture and persist the **Claude session id** (from `~/.claude/sessions/<pid>.json`) so the UI can deep-link to transcripts and so future features (resume, transcript tail, status badge) have a stable handle on the underlying Claude session. Confirmed: that file exists on this machine and contains `{pid, sessionId, cwd, startedAt, kind, entrypoint}`.

## Design

### 1. `ConversationRuntime` interface

New file `plugins/conversations/server/internal/runtime.ts`:

```ts
export type RuntimeKind = "tmux" | "api";

export interface RuntimeInfo {
  task: string;               // human-visible activity label ("" when idle)
  idle: boolean;              // true = waiting for user input
  claudeSessionId: string | null; // from ~/.claude/sessions/<pid>.json, null until resolved
}

export interface ConversationRuntime {
  readonly kind: RuntimeKind;

  /** Start a new conversation: set up environment + spawn Claude. Returns when spawned. */
  create(id: string, worktreePath: string): Promise<void>;

  /** Terminate a running conversation. Idempotent. */
  delete(id: string): Promise<void>;

  /** Snapshot all live conversations handled by this runtime, keyed by conversation id. */
  list(): Promise<Map<string, RuntimeInfo>>;
}
```

### 2. Tmux runtime

Move the tmux-specific code from `tmux.ts` into `runtimes/tmux-runtime.ts` implementing `ConversationRuntime`:

- `list()` = current `listTmuxSessions()` **extended** to resolve the Claude session id per pane:
  1. `tmux list-panes -t <id> -F '#{pane_pid}'` → shell pid.
  2. `pgrep -P <shell_pid>` → `claude` child pid.
  3. Read `~/.claude/sessions/<pid>.json` → `sessionId`.
  4. Cache by `(id, claude_pid)` to avoid disk I/O every tick; invalidate when pid changes or file vanishes.
- `create()` = current `createConversation()` body (worktree + DB row insert stays in the orchestration layer, see §4; only the spawn lives here).
- `delete()` = current `deleteConversation()`.

Preserve `cleanPaneTitle`, `getMainWorktreeRoot`, `worktreePathFor` as utilities in `tmux.ts` (not runtime-specific — worktree paths are shared).

### 3. API runtime (stub for now)

`runtimes/api-runtime.ts` with `kind: "api"`, all methods throwing `NotImplemented`. Registered but inert. This proves the abstraction compiles without two real implementations and gives a concrete target to fill in later.

### 4. Runtime registry + orchestration

`internal/runtimes/index.ts`:

```ts
const runtimes: Record<RuntimeKind, ConversationRuntime> = {
  tmux: tmuxRuntime,
  api: apiRuntime,
};
export function getRuntime(kind: RuntimeKind): ConversationRuntime { ... }
```

High-level lifecycle moves to `internal/lifecycle.ts` (new):

```ts
export async function createConversation(kind: RuntimeKind = "tmux"): Promise<Conversation> {
  const id = `claude-${Math.floor(Date.now() / 1000)}`;
  const wtPath = await worktreePathFor(id);
  // git worktree + forkDatabase (shared, not runtime-specific)
  await setupWorktree(id, wtPath);
  const [row] = await db.insert(conversations).values({ id, worktreePath: wtPath, runtime: kind }).returning();
  await getRuntime(kind).create(id, wtPath);
  return row!;
}

export async function deleteConversation(id: string): Promise<void> {
  const row = await db.select().from(conversations).where(eq(conversations.id, id));
  await getRuntime(row.runtime).delete(id);
}
```

This is where the HTTP handlers (`handle-create.ts`, `handle-delete.ts`) import from — they become runtime-agnostic.

### 5. Poller becomes runtime-agnostic

`poller.ts` iterates all registered runtimes and **unions** their `list()` maps. On collision (same id in two runtimes — shouldn't happen), last-write-wins with a warning. The rest of the poller (orphan adoption, title sync, broadcast) stays the same shape, but:

- It now also syncs `claudeSessionId` to the DB when it changes.
- It emits one new event type in addition to the existing `tmux` one (or renames — see §7).

### 6. DB schema changes

Add two columns to `conversations` (`plugins/conversations/server/schema.ts`):

```ts
runtime: text("runtime").$type<RuntimeKind>().notNull().default("tmux"),
claudeSessionId: text("claude_session_id"),
```

A new migration file is generated by `./singularity build`. Default `"tmux"` covers all existing rows.

### 7. Shared protocol

Rename the `tmux` event to `live` in `plugins/conversations/shared/protocol.ts`, carrying the runtime kind and optional session id:

```ts
export type ConversationEvent =
  | { type: "created"; conversation: Conversation }
  | { type: "deleted"; id: string }
  | { type: "title"; id: string; title: string | null }
  | { type: "live"; id: string; runtime: RuntimeKind; task: string; idle: boolean; claudeSessionId: string | null }
  | { type: "live"; id: string; gone: true };
```

Update the one consumer (`web/stream/client.ts` + `use-conversation-stream.ts`) and the web `TmuxLive` type → `RuntimeLive` in `shared/types.ts`. Local app, single-process wire — no back-compat concerns.

## Verification

1. `./singularity build` — regenerates migration, restarts server.
2. `http://<this-worktree>.localhost:9000` — sidebar still lists existing tmux conversations, titles still update live.
3. SQL: `SELECT id, runtime, claude_session_id FROM conversations;` — `runtime` = `"tmux"` for all rows; `claude_session_id` populated within a few seconds of the poller running for any active tmux session.
4. Create a new conversation from the UI → new row has `runtime="tmux"`, worktree and tmux session spawn as before.
5. Delete from the UI → tmux session killed, DB row deleted (unchanged behavior).
6. `api-runtime.ts` is imported but never invoked — server boots cleanly.
7. No regression in the status toolbar (pulled from `status` plugin) — title and idle still drive status transitions.
