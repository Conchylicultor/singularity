# Stage 2 — Desktop multi-window arrangement (execution plan)

**Date:** 2026-06-14
**Category:** global (apps + primitives reuse)
**Status:** Plan — awaiting approval
**Supersedes (for execution):** the Stage 2 section of
[`research/2026-06-13-global-desktop-multi-window.md`](./2026-06-13-global-desktop-multi-window.md).
Stage 1 (scope-by-context refactor) has landed (commit `efd386167`); this doc
re-validates the design against the current code and fixes the details that
shifted.

## Context

Stage 1 made every per-surface read resolve from the surface's own `PaneStore` +
`appId` instead of global singletons, so several app surfaces can now render
concurrently without their rail highlight / theme scope / column-maximize
clashing. Today all open tabs are already mounted at once (`AppTabsBody` keep-alive
with `display:none` on the unfocused ones — `apps-layout.tsx:65-87`); exactly one
is *visible* and *live* (mirrors to the URL).

Stage 2 adds a **switchable surface arrangement**: a new "desktop" mode that lays
the same `Tab[]` out as free-floating, draggable, resizable windows on one screen,
versus today's "tabs" mode (one fullscreen tab). It is a pure spatial
re-arrangement of the same `Tab` / `PaneStore` lifecycle — no change to tabs,
stores, routing, or app shells.

### Decisions (confirmed with user)

- **URL model:** focused window owns the URL (Stage-1 `activate` already does
  this). One `live` store; background windows hold their route in memory.
- **Window style:** free-floating — drag / resize / overlap, z-order on focus,
  maximize, minimize. Built on raw pointer events (the `ResizeHandle` idiom), no
  new dependency.
- **Switchable arrangement** modeled on the `app-rail-framing` variant region
  (tabs vs desktop), picker in the theme customizer.
- **Rail click → replace focused window** (today's `replaceTabApp`, unchanged).
  The rail therefore never needs to read the active arrangement, so **no import
  cycle** and the region can live entirely in a child plugin — a faithful mirror
  of `app-rail-framing`. New windows come from the existing tab-creation paths
  (Home launcher, the tab bar `+`) and a `+` affordance on the desktop itself.
- **Minimize → collapse to a titlebar strip** on the desktop; the top
  `AppTabBar` stays visible as a taskbar / window list in desktop mode too.

## Validated current-code facts (post Stage 1)

- `apps-layout.tsx:256-269` `FramedSurface` reads `Apps.RailFraming.useContributions()[0]`
  and passes `body = <AppTabsBody />` as `RailFramingProps`. Falls back to
  `DefaultRailFraming` when no framing plugin is loaded.
- `apps-layout.tsx:56-89` `AppTabsBody` maps `useTabs().tabs` → per-tab
  `<div absolute inset-0 display:focused?block:none>` →
  `<PaneSurfaceProvider store basePath={appPathFor(tab.appId, apps)} appId={tab.appId}>`
  → `<TabTitleReporter/>` + `renderIsolated(Apps.App.id, app)`. The outer container
  is `relative ... flex-1 transform-gpu` (containing-block trick for fixed sidebars).
- `Tab = { tabId, appId, store }` (`tabs-store.ts:11-15`). `basePath` is **not**
  on the tab — it's computed via `appPathFor(appId, apps)` (`tabs-store.ts:92`).
- `useTabs(): TabsApi` returns `{ tabs, focusedTabId, titles, setTabTitle,
  openTab, replaceTabApp, navigate, focusTab, closeTab, moveTab }`
  (`use-tabs.tsx`). `focusTab` calls the internal `activate` → flips liveness +
  mirrors the focused store's route to the URL (Stage-1 behavior).
- `Apps` slots live in `apps/web/slots.ts`: `App` (render slot) + `RailFraming`
  (`defineSlot<RailFramingContribution>`). `RailFramingProps` lives in
  `apps/core/types.ts`, re-exported from `apps/core`.
- Variant-region factory (`plugins/ui/plugins/variant-region/{core,web,server}`):
  - `defineVariantRegion<Props>({ id, label, defaultVariant, scope? })` → core
    (owns the ConfigV2 descriptor). No `scope` ⇒ global (base config).
  - `defineVariantRegionWeb(core)` → `{ Variant, Region, Picker, contributions }`.
    `contributions` already bundles `ConfigV2.WebRegister`, `DynamicEnum.Options`,
    and `ThemeEngine.VariantGroup` (the customizer picker). The `Region` host
    forwards its `Props` verbatim to the active variant's `component` via
    `renderIsolated`.
  - `variantRegionServerContribution(core)` (server) registers the descriptor.
- **Plugins are auto-discovered** by codegen (`web.generated.ts`,
  `DO NOT EDIT — autogenerated by ./singularity build`). Creating
  `<runtime>/index.ts` files is enough; no manual registry edits.
- Reuse idioms confirmed: pointer-drag `resize-handle.tsx:9` (onPointerDown →
  `window` pointermove/pointerup, emit deltas); per-key external store
  `use-column-widths.ts` (module `Map` + `Set` + `useSyncExternalStore` +
  storage); per-tab key seed `getTabId()` (`@plugins/primitives/plugins/tab-id/web`).
- Lint: inline `style={{ zIndex }}` is **not** subject to `no-adhoc-zindex`
  (that rule only bans `z-<n>`/`z-[…]` className tokens) — so per-window z-order
  uses numeric inline zIndex. Window chrome uses named `rounded-*` / `gap-*` /
  `p-*` (or `<Stack>`/`<Inset>`) to satisfy `no-adhoc-radius`/`no-adhoc-spacing`.

## Plan

### A. `apps` host changes (extract the shared per-tab core + add the slot)

The per-tab render core is identical for both arrangements; extract it so both
variants reuse it (DRY, keeps `PaneSurfaceProvider` mounting byte-identical).

1. **`plugins/apps/web/components/tab-surface.tsx`** (new) — move the per-tab
   body + the title-reporter machinery (`TabTitleReporter`, `IndexTitleReporter`,
   `LeafTitleReporter`, `TitleClear`) out of `apps-layout.tsx`. Export:
   ```tsx
   export function TabSurface({ tab }: { tab: Tab }) {
     const apps = Apps.App.useContributions();
     const app = apps.find((a) => a.id === tab.appId);
     if (!app) return null;
     return (
       <PaneSurfaceProvider store={tab.store}
         basePath={appPathFor(tab.appId, apps)} appId={tab.appId}>
         <TabTitleReporter tabId={tab.tabId} />
         {renderIsolated(Apps.App.id, app as unknown as Contribution)}
       </PaneSurfaceProvider>
     );
   }
   ```
2. **`apps-layout.tsx`** — `AppTabsBody` becomes the thin keep-alive container
   over `TabSurface` (same outer `relative ... flex-1 transform-gpu` + per-tab
   `absolute inset-0` + `display:none`). Keep `AppTabsBody` here as the
   **fallback** body. `DocumentTitleSync` stays. `FramedSurface` changes to read
   the new slot:
   ```tsx
   function FramedSurface() {
     const framing = Apps.RailFraming.useContributions()[0];
     const arrangement = Apps.SurfaceArrangement.useContributions()[0];
     const body = arrangement
       ? renderIsolated(Apps.SurfaceArrangement.id, arrangement as unknown as Contribution, {})
       : <AppTabsBody />;
     // …existing rail-framing wrap unchanged…
   }
   ```
3. **`apps/core/types.ts`** — add `export type SurfaceArrangementProps = Record<string, never>;`
   (both variants pull everything from `useTabs()`, so props are empty). Re-export
   from `apps/core/index.ts`.
4. **`apps/web/slots.ts`** — add to `Apps`:
   ```ts
   export interface SurfaceArrangementContribution {
     component: ComponentType<SurfaceArrangementProps>;
   }
   SurfaceArrangement: defineSlot<SurfaceArrangementContribution>(
     "apps.surface-arrangement", { docLabel: () => "Surface arrangement" }),
   ```
5. **`apps/web/index.ts`** — additionally export `TabSurface`, `AppTabsBody`,
   the `Tab` type, and `SurfaceArrangementContribution` so the child variants can
   consume them.

> `apps` never imports the `surface-arrangement` plugin — the slot indirection
> (same as `RailFraming`) keeps the graph acyclic.

`core/region.ts` imports `SurfaceArrangementProps` from `@plugins/apps/core` and
`defineVariantRegion` from `@plugins/ui/plugins/variant-region/core`. The picker

### C. `desktop` variant — the windows

```
plugins/apps/plugins/surface-arrangement/plugins/desktop/web/
├── index.ts                       SurfaceArrangement.Variant({ id:"desktop",
│                                     label:"Desktop", match:"desktop",
│                                     component: AppWindowsBody })
├── components/app-windows-body.tsx
├── components/window-frame.tsx
├── components/window-resize-handles.tsx
└── hooks/use-window-geometry.ts
```

- **`AppWindowsBody`** — `const { tabs, focusedTabId, focusTab, closeTab, titles }
  = useTabs()`. Render a `relative h-full w-full overflow-hidden transform-gpu`
  desktop backdrop; map each `tab` → `<WindowFrame tab focused title onFocus
  onClose />`. **All** windows mounted concurrently (keep-alive, like tabs).
  A small `+` button (or double-click empty desktop) calls `openTab(focused app)`
  / opens the launcher — the desktop's own new-window affordance.
- **`WindowFrame`** —
  - `const [geo, setGeo, bringToFront] = useWindowGeometry(tab.tabId)`.
  - Outer `<div className="absolute rounded-lg border bg-background shadow-lg overflow-hidden"
    style={{ left: geo.x, top: geo.y, width: geo.w, height: geo.h, zIndex: geo.z }}>`.
    Inline numeric `zIndex` (no `z-*` className → no `no-adhoc-zindex` violation);
    the set `zIndex` makes each window its own stacking context so inner app
    `z-nav`/`z-float` chrome can't bleed across windows.
  - `onPointerDownCapture` anywhere → `focusTab(tab.tabId)` + `bringToFront()`.
  - **Titlebar** (drag): pointer idiom from `resize-handle.tsx` — `onPointerDown`
    → `window` pointermove emits `{dx,dy}` → `setGeo(g => ({…g, x:g.x+dx, y:g.y+dy}))`.
    Shows app icon + `titles[tab.tabId] ?? app.tooltip`; buttons: minimize /
    maximize / close (`onClose`). Buttons `stopPropagation` on pointerdown so they
    don't start a drag.
  - **Resize** — `WindowResizeHandles`: 8 edge/corner handles, each the same
    pointer idiom, applying `{dx,dy}` per edge (right: `w+=dx`; left: `x+=dx;
    w-=dx`; etc.), clamped to a min size. Hidden while minimized/maximized.
  - **Maximize**: store current geo as `restore`, set geo to fill the desktop
    (`x:0,y:0,w:100%,h:100%`); toggle restores. **Minimize**: collapse to a
    titlebar strip (content wrapper `display:none` but **kept mounted** to
    preserve React state); toggle restores. (References: `use-column-maximize.ts`
    / `use-column-collapse.ts` toggle shapes.)
  - **Content**: `<div className="relative min-h-0 flex-1 transform-gpu">`
    (per-window containing block) → `<TabSurface tab={tab} />`. Unchanged app
    shell fills the box (`h-full`).
- **`useWindowGeometry(tabId)`** — module-global `Map<tabId, Geometry>` + `Set`
  subscribers + `useSyncExternalStore`, mirroring `use-column-widths.ts`.
  `Geometry = { x, y, w, h, z, minimized: boolean, maximized: boolean, restore?:
  {x,y,w,h} }`. Persist the whole map to **sessionStorage** under
  `"app-windows:" + getTabId()` (matches the per-browser-tab session model of
  `PersistedTabs`; keeps tabs-mode persistence untouched). First access for a new
  `tabId` ⇒ cascade default `{ x: 40 + n*28, y: 40 + n*28, w: 880, h: 600 }` and
  `z = ++nextZ`. `bringToFront` = `z = ++nextZ`. On mount, prune map entries whose
  `tabId` is no longer in `useTabs().tabs` (closed windows).

### D. No change needed

- `AppTabBar` already focuses a tab on click → serves as the desktop taskbar /
  window list unchanged; minimized windows still appear there.
- `app-rail.tsx` unchanged (rail click = `replaceTabApp` in both modes).
- App shells, `PaneStore`, routing, Stage-1 `activate` — all unchanged.
