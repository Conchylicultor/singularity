# Dash the typography sub-scale token vars (`--font-size2xs` → `--font-size-2xs`)

## Context

The typography token group declares four sub-scale keys — `fontSize2xs`,
`fontSize3xs`, `lineHeight2xs`, `lineHeight3xs` — in camelCase. `defineTokenGroup`
derives each CSS var via `camelToKebab`, which only inserts a dash **before
uppercase letters**, never before a digit. So those four resolve to
`--font-size2xs` / `--font-size3xs` / `--line-height2xs` / `--line-height3xs`
(no dash before the digit), while every other var in the group is properly
dashed (`--font-size-body`, `--font-size-caption`, …).

This inconsistency previously masked a latent bug: the `@theme inline` bridges in
`app.css` referenced the *dashed* `--font-size-2xs` form that no descriptor
supplied — the only thing that supplied it was the now-deleted static `:root`
fallback. The bridges were realigned to the descriptor's actual undashed names as
a stopgap, but the var names themselves are still ugly and inconsistent with the
rest of the token system.

**Goal:** make the four vars consistently dashed (`--font-size-2xs`,
`--font-size-3xs`, `--line-height-2xs`, `--line-height-3xs`) and realign every
consumer, so the naming matches the rest of the token system.

## Approach: kebab-literal schema keys (mirrors existing precedent)

The codebase already solved this exact problem. The **chart** and **categorical**
token groups need digit-suffixed vars (`--chart-1`, `--categorical-10`) and get
them by writing the schema keys as **kebab-literal strings**, not camelCase:

```ts
// plugins/ui/plugins/tokens/plugins/chart/shared/group.ts
export const chartGroup = defineTokenGroup("chart", {
  "chart-1": { default: "…", label: "Chart 1" },   // → --chart-1
  ...
});
```

`camelToKebab("chart-1")` is a no-op (no uppercase), so the var comes out exactly
`--chart-1`. We apply the same, established pattern to the four typography keys.
This is the precedent-matching fix — **not** a new inconsistency, and **not** a
change to `camelToKebab` (which would be a wider blast radius for no benefit: the
generated manifest confirms these four are the *only* letter-then-digit vars in
the entire token system, so a global transform change would buy nothing the local
key rename doesn't).

1. **`plugins/ui/plugins/tokens/plugins/typography/shared/group.ts`** (lines 14–17)
   Rename the four keys to kebab-literal strings:
   ```ts
   "font-size-2xs":   { default: "0.6875rem", label: "Font size 2xs" },
   "font-size-3xs":   { default: "0.625rem",  label: "Font size 3xs" },
   "line-height-2xs": { default: "1rem",      label: "Line height 2xs" },
   "line-height-3xs": { default: "0.875rem",  label: "Line height 3xs" },
   ```

   `TypographyTokenValues` (the mapped type over `schema` keys, lines 36–38) and
   `typographyGroup.vars` adapt automatically.

2. **`plugins/ui/plugins/tokens/plugins/typography/web/presets.ts`** (lines 22–25)
   Update the four keys in `defaultPreset` to the same kebab-literal strings:
   ```ts

   "font-size-3xs": "0.625rem",
   "line-height-2xs": "1rem",
   "line-height-3xs": "0.875rem",
   ```
   `TypographyTokenValues` is exhaustive, so TypeScript flags any key left stale.

3. **`plugins/framework/plugins/web-core/web/theme/app.css`** (lines 83–86)
   Realign the four `@theme inline` bridge references to the dashed names:
   ```css
   --text-2xs: var(--font-size-2xs);
   --text-2xs--line-height: var(--line-height-2xs);
   --text-3xs: var(--font-size-3xs);
   --text-3xs--line-height: var(--line-height-3xs);
   ```

   (The static `@theme` block at lines 10–30 hardcodes literal fallbacks like
   `--text-2xs: 0.6875rem` and does **not** reference these vars — leave it.)

## Risk

Minimal. Three small hand-edits, all type-checked or check-enforced; the rest is
codegen. Worst case (a missed consumer) fails `css-vars-supplied` loudly at build
rather than silently. Pre-existing stored config overrides keyed by the old
camelCase names (none committed today) would simply be ignored — the values are
empty strings.
