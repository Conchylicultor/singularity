# Phase 4b: `central/` → `plugins/framework/plugins/central-core/`

Implementation plan for the DAG migration phase that moves `central/` into the plugin tree. Part of [`2026-05-12-global-plugin-dag-migration.md`](./2026-05-12-global-plugin-dag-migration.md).

## Context

`central/` is structurally identical to `server/` (pre-Phase 4): a top-level directory that sits outside the plugin DAG, with its own zone in `boundary.config.ts`. Plugins import from it via `@central/*`, but it cannot import from plugins cleanly. Phase 4 solved this for `server/`; this phase applies the same treatment to `central/`.

Central plays two roles:

| Role | Files | Consumers |
|------|-------|-----------|
| **Framework API** — types, resources (live-state pub/sub engine) | `types.ts` (7 imports), `resources.ts` (1 import) | 8 import statements across 4 plugin `central/index.ts` barrels, all via `@central/*` alias |
| **Boot harness** — Bun.serve on Unix socket, plugin topo-sort, lifecycle (register → onReady → shutdown), orphan watchdog, HTTP/WS routing | `index.ts`, `topo.ts`, `plugins.ts`, `plugins.generated.ts` | Zero external consumers — only the gateway spawns this |

These two roles separate cleanly into `core/` and `bin/`, mirroring Phase 4's server split.

### Why `central-core/`

Phase 4 landed `server/` → `plugins/framework/plugins/server/`, then immediately renamed to `server-core/` ([`2026-05-18-framework-rename-server-to-server-core.md`](./2026-05-18-framework-rename-server-to-server-core.md)) to avoid name collision with per-plugin `server/` runtime directories. The same collision exists for `central/` — plugins have `central/` runtime directories (e.g. `plugins/auth/central/`, `plugins/infra/plugins/secrets/central/`). Landing directly as `central-core/` avoids the intermediate rename step.

## Target directory structure

```
plugins/framework/plugins/central-core/
  package.json          ← @singularity/plugin-framework-central-core
  tsconfig.json         ← path aliases retargeted
  CLAUDE.md
  core/                 ← public barrel — the framework API
    index.ts            ← re-exports from types + resources
    types.ts              CentralPluginDefinition, WsHandler, WsData, HttpHandler, ResourceLike, Registration
    resources.ts          defineResource, Resource, notificationsWsHandler, handleResourceHttp
  bin/                  ← process entry point (gateway runs `bun bin/index.ts` from this directory)
    index.ts            ← Bun.serve, route tables, lifecycle, orphan watchdog
    topo.ts             ← topoSortPlugins (only used by index.ts)
    plugins.ts          ← re-exports plugins.generated
    plugins.generated.ts ← auto-generated by plugin-registry-gen
```

### Why `core/` + `bin/`

Same reasoning as Phase 4. The boot harness is an executable entry point, not library code. `bin/` is the standard convention for executables. This plugin doesn't contribute routes/registrations/onReady — it IS the central runtime. It has `core/` (public API) and `bin/` (process entry), but no `server/` or `web/` barrel in the plugin system sense.

## Gateway — zero code changes

The gateway spawn is fully data-driven:

1. `cli/build.ts` writes `~/.singularity/worktrees/central.json` with `{ "server": "<abs-path>" }`
2. Gateway reads this spec, stores it in `Spec.Server`
3. `startBackend()` runs `exec.Command("bun", "bin/index.ts")` with `cmd.Dir = spec.Server`

The entry point command (`bin/index.ts`) was already updated in Phase 4 (same code path for all backends). The `spec.Server` value changes because `cli/build.ts` now writes the new path. The gateway's `w.Name == "central"` never-sweep check and eager-spawn logic use the worktree name, not the directory path — no change.

Central-routes manifest: `central-routes.json` is generated by scanning plugin `central/index.ts` barrels for `httpRoutes`/`wsRoutes`. The scan walks plugin directories, not the framework's `central-core/` — no path change needed in the scanner itself.

## Module identity — singleton safety

`resources.ts` holds module-level singletons (`registry: Map<string, RegistryEntry>`, notification batching state, WebSocket state). All consumer imports use `@central/resources`. After the alias retarget, every import resolves to the same physical file (`plugins/framework/plugins/central-core/core/resources.ts`). Bun's module cache is path-keyed — same file = same module instance = singleton preserved.

Unlike `server/`, there is no `no-relative-central-imports` check. With only 8 consumer imports (all already using `@central/*`), the risk of module identity splits is low. Creating an equivalent check is optional — defer unless a regression appears.

## Implementation steps

Steps 1–5 must be sequential. Steps 6–11 are independent (parallelizable).

### Step 1: Create directory structure

```bash
mkdir -p plugins/framework/plugins/central-core/{core,bin}
```

### Step 2: Move files via `git mv`

```bash
# Framework API → core/
git mv central/bin/types.ts      plugins/framework/plugins/central-core/core/
git mv central/bin/resources.ts  plugins/framework/plugins/central-core/core/

# Boot harness → bin/
git mv central/bin/index.ts              plugins/framework/plugins/central-core/bin/

git mv central/bin/plugins.ts            plugins/framework/plugins/central-core/bin/

git mv central/package.json    plugins/framework/plugins/central-core/
git mv central/tsconfig.json   plugins/framework/plugins/central-core/
git mv central/CLAUDE.md       plugins/framework/plugins/central-core/
```

### Step 3: Create `core/index.ts` barrel

```typescript
export * from "./types";
export * from "./resources";
```

### Step 4: Rewrite `bin/index.ts` internal imports

Replace relative imports to files that moved to `core/` with alias imports:

```
./types      → @central/types
```

**Verification gate:** `bunx tsc --noEmit` from `plugins/framework/plugins/central-core/`.

### Step 6: Update `plugin-registry-gen.ts`

### Step 7: Update `boundary.config.ts`

- Remove `zone("central", ...)` definition
- Remove `central` from runtime layers

### Step 8: Update `resolve.ts`

Zone → `plugin.framework.central-core`, runtime → `core`.

### Step 9: Update `cli/build.ts`

Two `resolve(...)` calls: `centralDir` and `worktreeCentralDir`.

### Step 10: Update tooling checks

Six locations across 4 files (see "Other tooling updates" section).

### Step 11: Update root `package.json`

### Step 12: Update CLAUDE.md

## Risk register

| Risk | Severity | Mitigation |
|------|----------|------------|
| **Module identity split** — relative imports bypass alias, duplicating `resources.ts` singleton registry | Low | Only 8 consumers, all already use `@central/*`. No `no-relative-central-imports` check exists, but consumer count is small enough to verify manually. |
| **tsconfig depth errors** — `@plugins/*` → `../../../*` resolves incorrectly | Medium | Verification gate at step 5: `bunx tsc --noEmit`. Mirror proven server-core/tsconfig.json pattern. |
| **Stale spec JSON** — running central has `server: <old-path>` in `central.json` | Low | `./singularity build` rewrites the spec. Gateway fsnotify picks up the change. Already-running central is unaffected (path only used at spawn). |
| **Generated file `typeImport`** — `./types` breaks when types.ts moves to `core/` | Medium | Fixed by step 6 (change to `@central/types`). Build failure is immediate. |
| **Central-core always runs main's code** — worktree builds write `centralDir = resolve(mainRoot, ...)`, so the running process uses main's files. After merge to main, the new path is active. | None | This is by design. Worktree tsc validates the worktree's copy. |
| **`plugin-boundaries.ts` runtime handling** — `central` runtime token in `VALID_RUNTIMES`, barrel checks, cycle detection | Low | These reference `"central"` as a per-plugin runtime name (the directory convention), not the framework plugin path. The runtime name stays `central` — no change needed. |
