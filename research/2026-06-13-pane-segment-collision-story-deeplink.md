# Story `/story/s/:pageId` deep-link "Not Found" — pane segment collision

## Context

Opening the Story editor by URL — `/story/s/<pageId>` — fails with the pane-guard
"This link refers to a resource that no longer exists." on any URL-driven load
(fresh deep-link, cross-app `navigate()` from the Pages "Open in Story Builder"
link, and reload). Clicking a card in the `/story` gallery opens the same pane
correctly.

### Root cause: two panes share one URL-matching pattern

Two panes in **different apps** declare segments that match the *same* URLs:

- `plugins/apps/plugins/story/plugins/shell/web/panes.tsx:29` — `storyDetailPane`, `segment: "s/:pageId"`
- `plugins/apps/plugins/deploy/plugins/servers/web/panes.tsx:39` — `serverDetailPane`, `segment: "s/:serverId"`

They aren't string-equal, but both match `s/<anything>` (literal `s` + one param).

`parseUrl` (`plugins/primitives/plugins/pane/web/pane.ts:281`) matches a URL
against the **global** registry of every registered pane across all apps — the
app `basePath` (`/story`, `/deploy`) is stripped *before* matching, so both
collapse into the same `/s/:id` space. For `/s/<id>` both panes return
`consumed: 2`; the tie-break keeps the first the registry iterated
(`result.consumed > bestMatch.consumed`, strict `>` — pane.ts:299), i.e. plugin
registration order. So `/story/s/<id>` non-deterministically resolves to
whichever of the two panes registered first.

When `serverDetailPane` wins, the route becomes `[deploy-server-detail{serverId:<pageId>}]`;
its `useResolveServer` looks up a server by that id, finds none, and renders the
guard's "Not Found". (The reporter attributed the failing resolve to
`useStoryDetailResolve`; the actual failing resolve is whichever colliding pane
wins the tie — the guard text is identical either way.)

**Why the gallery card works:** `openPane(storyDetailPane, { pageId }, { mode: "root" })`
(story-gallery.tsx:38) passes the `paneId` + params explicitly — no URL parse, no
ambiguity. Reload restores from `history.state.route` (pane.ts:523), which also
carries the explicit `paneId`. Only paths that go through `parseUrl` —
fresh deep-link (`syncRouteFromUrl` → `parseUrl`) and cross-app `navigate()`
(use-tabs.tsx:296 calls `parseUrl` directly) — hit the collision.

### Why nothing caught it

`pane/CLAUDE.md:78` states "Segments must be globally unique across all
registered panes", but **nothing enforces it**. `useSyncPaneRegistry`
(pane.ts:1212) de-dupes only by exact pane `id`; two distinct ids with
colliding segment *patterns* register silently and mis-route at runtime.

## Fix

### 1. Disambiguate the collision (correctness)

Rename the deploy segment so the two no longer overlap. Story's
`/story/s/:pageId` is the documented, bug-reported deep-link and stays; deploy's
server-detail URL is internal with no hardcoded references (every use goes
through the `serverDetailPane` object — verified: `useParams`, `useRouteEntry`,
`openPane(serverDetailPane, …)`).

- `plugins/apps/plugins/deploy/plugins/servers/web/panes.tsx:39`
  `segment: "s/:serverId"` → `segment: "server/:serverId"`

No other deploy edits needed (the segment string is the only definition of the
URL form; persisted routes key on `paneId`+params, not the segment, so existing
state is unaffected). Update the pane's CLAUDE.md autogen block via the normal
doc-sync (regenerated by `./singularity build`).

### 2. Enforce uniqueness at the registry boundary (structural — fail loudly)

Make the documented invariant real where panes enter the matcher, so a future
colliding segment crashes loudly on app boot instead of silently mis-routing.

In `useSyncPaneRegistry` (`plugins/primitives/plugins/pane/web/pane.ts:1212`),
while populating `registry`, also build a `Map<normalizedPattern, paneId>`. On
collision, `throw` an Error naming both panes and their segments.

- **Normalize** a segment to its match-pattern: split on `/`, map each part —
  `:name*` → `:*`, `:name` → `:`, literal → itself — and re-join. This makes
  `s/:pageId` and `s/:serverId` both normalize to `s/:`, so they collide;
  `page/:pageId` (`page/:`) and `s/:pageId` (`s/:`) do not.
- **Skip empty segments** (`""` / `"/"`): index panes don't participate in URL
  matching (they resolve via `appPath` in `useIndexMatch`), so multiple empty
  segments are legal — only normalize/compare non-empty segments.
- Throw rather than `console.warn` (the existing duplicate-`id` branch at
  pane.ts:1217 only warns; a pattern collision is a genuine routing corruption,
  not a benign re-register). `useSyncPaneRegistry` reads *all* contributions
  globally on the first render of any layout renderer, so the throw surfaces on
  app boot regardless of which app is opened — effectively as early as a build
  check, with no new infra.

This closes the entire class: any two panes whose segments match the same URLs
fail immediately and point at both culprits.

## Files

- `plugins/apps/plugins/deploy/plugins/servers/web/panes.tsx` — rename segment (fix 1).
- `plugins/primitives/plugins/pane/web/pane.ts` — collision detection + throw in `useSyncPaneRegistry` (fix 2).
- `plugins/apps/plugins/deploy/plugins/servers/CLAUDE.md` — autogen block refresh via build (no manual edit).
