# Category Avatars for Conversation List

## Context

Conversation categories are currently shown only as neutral text chips in
the sidebar. The conversation list item's `Item.Avatar` slot renders a blank
disc (just a status dot) for all non-agent conversations, because no plugin
contributes an avatar for them.

The goal is to make categories visible as avatars in the list — an icon on a
colored disc, auto-colored by category label, with per-category customisation
via the existing `AvatarPicker` primitive. Users configure the icon/color in
Settings.

The existing color override table (`conversation_category_colors`) already
tracks per-category color keys. The `avatar` primitive already exports
`Avatar` + `AvatarPicker` with the same 10-color key space. The
`Item.Avatar` slot already has first-match semantics.

---

## Key observations

- `colorClassFor()` / `useCategoryColors()` is called in `CategoryChipRow`
  but the chip is always `bg-muted text-muted-foreground` — color overrides
  are **invisible in the list today**. The color system only affects the
  Settings swatch preview.
- `AVATAR_COLORS` in the avatar primitive uses the same key names (`sky`,
  `emerald`, …) as `COLOR_PALETTE` in the category plugin — fully compatible.
- `Avatar` takes `fallbackKey` to derive a deterministic auto-color when
  `color` is null — using the category label gives consistent per-category
  coloring without manual setup.

## Changes

### 1. DB schema — `tables-colors.ts`

```
plugins/conversations/plugins/conversation-category/server/internal/tables-colors.ts
```

- Make `colorKey` nullable (remove `.notNull()`).
- Add `iconKey: text("icon_key")` (nullable, no default).

A row now represents any manual avatar override for a category; both fields
can independently be null. The migration is generated by `./singularity build`.

### 2. Server resource — `colors-resource.ts`


```ts
type CategoryAvatarOverride = { colorKey: string | null; iconKey: string | null };
```

Update the `loader` to map `{ colorKey: r.colorKey, iconKey: r.iconKey }` per

### 3. Server routes — `colors-routes.ts`

```
plugins/conversations/plugins/conversation-category/server/internal/colors-routes.ts
```

  `{ category: string, colorKey?: string | null, iconKey?: string | null }`.
  Upsert with both fields; at least one must be non-null (otherwise just
  delete the row). Call `categoryColorsResource.notify()`.
- Keep **`handleDeleteColor`** unchanged (deletes the whole row, resetting
  both fields).
- Update the route registration in `server/index.ts`:
  `POST /api/conversation-category/colors` still works (same path, new body
  shape accepted).

### 4. Client hook — `use-category-colors.ts`

### 5. Settings UI — `category-color-settings.tsx`

```
plugins/conversations/plugins/conversation-category/web/components/category-color-settings.tsx
```

Replace the chip preview + color swatch grid with an `<Avatar>` wrapped in
`<AvatarPicker>` per category row.

- Import `Avatar`, `AvatarPicker`, `AvatarSpec` from

- For each category, derive `value: AvatarSpec = { icon: override?.iconKey ?? null, color: override?.colorKey ?? null }`.
- `onChange` sends `POST /api/conversation-category/colors` with

- Show an "auto" badge next to categories with no override (same as before).
- Update the section `title` to `"Category avatars"` and `description`

### 6. Chip row simplification — `category-chip-row.tsx`

```
plugins/conversations/plugins/conversation-category/web/components/category-chip-row.tsx
```

- Remove the `useCategoryColors()` call and `colorClassFor` import — the

- Hardcode `className="bg-muted text-muted-foreground"` (the value

### 7. New component — `category-avatar-row.tsx` (new file)

```
plugins/conversations/plugins/conversation-category/web/components/category-avatar-row.tsx
```

```tsx
import { Avatar } from "@plugins/primitives/plugins/avatar/web";
import { CONV_STATUS_DOT } from "@plugins/conversations/plugins/conversation-ui/plugins/item/web";
import type { ConversationItemConv } from "@plugins/conversations/plugins/conversation-ui/plugins/item/web";
import { useCategoryFor } from "../internal/use-category";
import { useCategoryColors } from "../internal/use-category-colors";

export function CategoryAvatarRow({ conv }: { conv: ConversationItemConv }) {
  const category = useCategoryFor(conv.id);
  const overrides = useCategoryColors();
  const override = category ? overrides[category] : undefined;
  return (
    <Avatar
      icon={override?.iconKey ?? null}
      color={override?.colorKey ?? null}
      size="sm"
      statusDot={CONV_STATUS_DOT[conv.status]}
      fallbackKey={category ?? conv.id}
    />
  );
}
```

- `fallbackKey={category ?? conv.id}` — deterministic auto-color per category
  label; falls back to per-conversation color when unclassified.
- No manual override → auto-colored disc (no icon). With override → icon +
  color from `AvatarPicker`.

### 8. Wire up — `web/index.ts`

```
plugins/conversations/plugins/conversation-category/web/index.ts
```

Add `Item.Avatar` contribution (import `Item` from
`@plugins/conversations/plugins/conversation-ui/plugins/item/web`):

```ts
Item.Avatar({
  match: (conv) => conv.kind !== "agent",
  component: CategoryAvatarRow,
}),
```

Agent conversations are already handled by the `agents` plugin's
first-match contribution; this covers all others.

## Verification

1. `./singularity build` — generates migration for the nullable `colorKey` +
   new `iconKey` column, builds and deploys.
2. Open the app → Settings → "Category avatars" — each category shows an
   `<Avatar>` disc. Clicking opens the `AvatarPicker` popover; picking icon +
   color saves and live-updates.
3. Open the conversation sidebar — non-agent conversations now show a colored
   disc with the category's avatar (or auto-color by category label).
4. Agent conversations are unaffected (still show their agent avatar).
5. Clear the avatar via the popover — disc reverts to auto-color.
6. Verify the category chip (sidebar) still renders in neutral muted color
   (unchanged behaviour).
