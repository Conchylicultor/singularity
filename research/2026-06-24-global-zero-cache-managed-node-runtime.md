# Managed Node-24 runtime for zero-cache + plugin-contributed provisioning

## Context

The per-worktree zero-cache sidecar (Stage 2) can only start when a Node binary
whose major equals `ZERO_NODE_MAJOR` (24) is reachable. `@rocicorp/zero-sqlite3`'s
native addon is built for exactly one ABI (Node 24, NODE_MODULE_VERSION 137);
loading it under a different major throws `ERR_DLOPEN_FAILED`. The dev host runs
Node 25, so at runtime `resolveNode()` in
`plugins/database/plugins/zero/plugins/cache-service/scripts/start.ts` finds no
compatible Node, throws "no compatible Node runtime found", and the process exits
before ready. The gateway's `waitZeroReady` sees the early exit and returns an
**opaque 502** on `/zero/*`. Stage-2 verification only worked because the gateway
was started with `SINGULARITY_ZERO_NODE` pointing at a homebrew `node@24`; a normal
`./singularity start` leaves the `SINGULARITY_ZERO_CACHE` opt-in non-functional.

Two structural problems:

1. **The Node-24 runtime is an unmanaged host requirement** — no install/resolution
   path. (The *addon* is already managed: `ensure-zero-sqlite3.ts` builds it for the
   Node-24 ABI regardless of host Node. Only the runtime binary is missing.)
2. **The root `package.json` postinstall hardcodes zero's plugin paths** —
   `bun e2e/ensure-chromium.mjs && bun plugins/database/plugins/zero/.../ensure-zero-sqlite3.ts`.
   The app core should not name a feature plugin; provisioning a plugin needs is the
   plugin's concern, injected through a discovery convention (like checks / lint),
   not wired by hand in the root.

Intended outcome: zero-cache works on any host with no manual Node install; a
download/extract failure fails the **build** loudly and early instead of surfacing
as a runtime 502; and the root postinstall references only framework plumbing +
its own e2e infra, never a feature plugin.

## Approach

Three independent parts.

### Part 1 — Generic plugin-contributed `provision` convention (framework)

Mirror the existing `defineCollectedDir("check")` / lint discovery so any plugin can
contribute an install-time provisioning step that is discovered automatically — the
root never names it.

New plugin **`plugins/framework/plugins/tooling/plugins/provision/`**:

- `core/index.ts` — barrel (`export default definePlugin(...)`).
- `core/collected-dir.ts` — `export const provisionCollectedDir = defineCollectedDir("provision")`
  (imports the canonical marker from `…/collected-dir/core`; this file is only ever
  read by codegen's regex scan, never imported in the alias-unfriendly postinstall
  context, so the alias is safe here — unlike lint which must inline its own copy).
- `core/provision.generated.ts` — committed registry, regenerated by `./singularity build`
  codegen (`discoverCollectedDirs` already finds the marker and walks every
  `plugins/<name>/provision/index.ts` with a default export). Seed it with the
  cache-service entry so the first postinstall works.
- `scripts/run-provisions.ts` — the runner, invoked by the root postinstall.
  **Must be alias-free** (`@plugins/*` does not resolve during `bun install`
  postinstall — see the note in `node-runtime.ts`). It:
  - imports `provisionEntries` from `../core/provision.generated` (relative),
  - for each entry, dynamic-imports
    `pathToFileURL(join(root, "plugins", e.pluginPath, "provision", "index.ts"))`
    — the **absolute-path** form, exactly like
    `lint/core/build-lint-config.ts:loadContributions` (which ignores the alias
    `loader` and reconstructs the path from `pluginPath` for the same reason),
  - awaits each module's `default` (an `async () => void`), collecting failures and
    **throwing if any fails** (fail-loud, like the lint loader — not the warn-and-
    continue of `loadCollectedDir`).

Convention: a plugin contributes provisioning via
`plugins/<name>/provision/index.ts` default-exporting `async function provision(): Promise<void>`,
**alias-free** (node builtins + relative imports only, because it runs in the
postinstall context).

Root `package.json` postinstall becomes:

```json
"postinstall": "bun e2e/ensure-chromium.mjs && bun plugins/framework/plugins/tooling/plugins/provision/scripts/run-provisions.ts"
```

The root now references only its own e2e infra (chromium is repo-level e2e tooling,
owned by no plugin) and the framework runner — never zero. New provisioning needs
are added by dropping a `provision/index.ts`; this line never changes again.

> Bootstrap/lag note: the runner consumes the **committed** `provision.generated.ts`
> (postinstall runs at build step 5, before codegen at step 6). A newly-added
> provision contribution therefore first runs on the *second* build — the same lag
> all collected-dir registries have, guarded by `plugins-registry-in-sync`.

### Part 2 — Managed Node-24 runtime (zero / cache-service)

Provision the Node-24 runtime the same way the addon is provisioned. Download the
official Node tarball from nodejs.org, checksum-verify, cache under `~/.singularity`,
and have `resolveNode()` prefer it.

- **New `plugins/database/plugins/zero/plugins/cache-service/scripts/ensure-zero-node.ts`**
  — exports `async function ensureZeroNode()`. Alias-free; imports
  `ZERO_NODE_BUILD_TARGET` + a new `zeroNodeCacheDir()` from `../shared/internal/node-runtime`.
  Logic (model it on `ensure-zero-sqlite3.ts`: stat-first idempotent, single
  subprocess, dependency-free):
  1. If `<cacheDir>/bin/node --version` reports `v24.*` → return (noop).
  2. If host `node` on PATH is already `v24.*` → return (host satisfies; no download).
  3. Else download `node-v<TARGET>-<plat>-<arch>.tar.gz` from
     `https://nodejs.org/dist/v<TARGET>/`, download `SHASUMS256.txt`, verify the
     tarball's sha256 against the matching line (`node:crypto`), extract to a
     `<cacheDir>.tmp` then atomically rename to `<cacheDir>` (temp+rename, like the
     pid sidecars). Platform/arch map: `process.platform` ∈ {darwin, linux},
     `process.arch` ∈ {arm64→arm64, x64→x64}; `.tar.gz` (host `tar -xzf`, no `xz`
     dep). glibc-linux only (same constraint as `@embedded-postgres`).
  - Fail loud on network/checksum/extract error with an actionable message naming
    `SINGULARITY_ZERO_NODE` as the offline/proxy escape hatch.
  - Cache dir: `~/.singularity/node/v<TARGET>/` (under `SINGULARITY_DIR`, so release
    isolation carries it).

- **`shared/internal/node-runtime.ts`** — add
  `export function zeroNodeCacheDir(singularityDir: string)` returning
  `join(singularityDir, "node", ZERO_NODE_BUILD_TARGET)` so `ensure-zero-node.ts` and
  `start.ts` share one path derivation and can never drift. Stays alias-free.

- **`scripts/ensure-zero-sqlite3.ts`** — refactor the top-level body into an exported
  `async function ensureZeroSqlite3()` (keep behavior byte-for-byte).

- **New `plugins/database/plugins/zero/plugins/cache-service/provision/index.ts`** —
  `export default async function provision()` → `await ensureZeroSqlite3(); await ensureZeroNode();`.
  Alias-free. This is the discovered contribution from Part 1.

- **`scripts/start.ts` `resolveNode()`** — insert the managed cache as candidate 2:
  order `SINGULARITY_ZERO_NODE` → `join(zeroNodeCacheDir(SINGULARITY_DIR), "bin", "node")`
  → `node` on PATH (compute `SINGULARITY_DIR` inline as
  `process.env.SINGULARITY_DIR ?? join(homedir(), ".singularity")`; do not import the
  paths plugin — keep alias-free). Update the error message to name the managed cache.

> Gating: provisioning runs at every `bun install`, but step 2 above **skips the
> ~35 MB download when a compatible Node is already on PATH or cached** — only hosts
> lacking Node 24 pay it, once. This matches the existing posture (zero-sqlite3 is
> already built for everyone unconditionally; chromium ~200 MB downloads for
> everyone). `SINGULARITY_ZERO_CACHE` can't gate this — it isn't set during install.

### Part 3 — Loud gateway signal (stderr tail in the 502)

Make the lazy failure path legible instead of "exited before ready".

- **`gateway/logring.go`** — add
  `func (r *logRing) Tail(n int, streamPrefix string) []logEntry` returning the last
  `n` entries whose `Stream` starts with `streamPrefix` (locked snapshot, like
  `Subscribe`).
- **`gateway/worktree.go` `EnsureZeroCache`** — in the `readyErr != nil` branch, pull
  `w.logBuf.Tail(20, "zero-")` and fold those lines into both the `slog.Warn` and the
  returned `wrapped` error. Since `proxy.go` returns `err.Error()` as the 502 body
  (proxy.go:204/222), the real cause (e.g. the `resolveNode` message) now appears in
  the response and the gateway log — no more opaque 502.

## Verification

1. **Clean managed path (the bug):** ensure no Node 24 on PATH (host is Node 25),
   remove `~/.singularity/node/`, unset `SINGULARITY_ZERO_NODE`. Run `./singularity build`
   with `SINGULARITY_ZERO_CACHE=1`. Expect: postinstall downloads + verifies + extracts
   Node 24 to `~/.singularity/node/v24.17.0/bin/node` (loud, at build time).
2. **Steady state noop:** re-run `./singularity build` — `ensure-zero-node` is a single
   stat + `--version`, no download.
3. **Runtime:** load `http://<worktree>.localhost:9000/zero/...` (a `/zero/*` route).
   `waitZeroReady` succeeds; gateway logs `zero-cache ready`. Confirm in
   `~/.singularity/worktrees/<wt>/logs/` the line `zero-cache: using Node v24.x (<cacheDir>/bin/node)`.
4. **Loud failure path:** temporarily point `SINGULARITY_ZERO_NODE` at a Node-25 binary
   and corrupt/remove the cache so resolution fails; hit `/zero/*`. Expect the 502 body
   and `gateway.log` to contain the actual `zero-cache: no compatible Node runtime found …`
   tail, not a bare "exited before ready".
5. **Boundary / discovery:** `git grep "zero" package.json` returns nothing; the root
   postinstall names only `e2e/ensure-chromium.mjs` + the framework runner.
   `./singularity check` passes (`plugins-registry-in-sync` confirms `provision.generated.ts`
   matches the filesystem; `plugin-boundaries` passes for the new plugin).
6. **`./singularity check`** clean (type-check, boundaries, registry-in-sync).
