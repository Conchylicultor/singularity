# Launch agent from task view

## Context

Tasks today are just text (title/description/status) and have a `task_attempts` table that is never populated â€” no API writes to it. Conversations have a `taskAttemptId` column but nothing sets it. The user wants a button in the task view that, given a task's description, spawns a Claude agent in a fresh worktree, with the task description as the initial prompt, and the resulting conversation linked to a newly created attempt for that task.

Outcome: clicking **Launch agent** on a task creates `taskAttempts` row â†’ creates a conversation row with `taskAttemptId` set â†’ tmux runtime spawns `claude` preloaded with the description as the prompt â†’ UI navigates to the conversation pane.

## Approach

Extend the existing `POST /api/conversations` flow to accept `{ taskId?, prompt? }`, rather than creating a parallel endpoint. This keeps a single conversation-creation path and lets tasks plug in.

### Backend

1. **`plugins/conversations/server/internal/lifecycle.ts`** â€” extend `createConversation`:
   - Accept `opts?: { runtimeId?; taskAttemptId?; prompt? }`.
   - Persist `taskAttemptId` in the insert.
   - Pass `prompt` through to `runtime.create(id, wtPath, { prompt })`.

2. **`plugins/conversations/server/api.ts`** â€” widen `ConversationRuntime.create` signature to accept an optional `{ prompt?: string }` 3rd arg. Update both runtimes:
   - **`runtime-tmux/server/internal/tmux-runtime.ts`** â€” when a prompt is given, append it as a single-quoted positional arg to the `claude` invocation (shell-escape single quotes with `'"'"'`). The existing command already single-quotes the shell string; we need to build the inner command safely.
   - **`runtime-api`** â€” stub runtime, just accept the extra arg; no-op.

3. **`plugins/conversations/server/internal/handle-create.ts`** â€” parse JSON body if present (`{ taskId?, prompt? }`). If `taskId` provided:
   - Insert a `taskAttempts` row (`id = ` short id, `taskId`) using a generated id (mirror existing id scheme in tasks plugin).
   - Call `createConversation({ taskAttemptId: attempt.id, prompt })`.
   - Otherwise behave as before.
   - Import `taskAttempts` from `@plugins/tasks/server/schema`. Cross-plugin server import is already the pattern (see plugins doc: conversations exports `Runtime`, tasks depends on conversations, etc.).

4. (Optional sanity) **`plugins/conversations/server/schema.ts`** â€” no change needed; `taskAttemptId` already exists.

### Frontend

5. **`plugins/tasks/web/components/task-detail.tsx`** â€” add a **Launch agent** button in the header row (next to "Savingâ€¦"). On click:
   - Flush any pending debounced save first (clear timers and `await save(...)` with current values) so the agent sees the latest description.
   - `POST /api/conversations` with JSON body `{ taskId, prompt: description }`.
   - Parse with `ConversationSchema`, then call `Shell.OpenPane(conversationPane({ session_id: id }))` â€” same pattern as `conversation-list.tsx:52-57`.
   - Disable the button when description is empty or while launching.

6. **`docs/plugins.md`** â€” regenerated by build; no manual edit.

### Migrations

`./singularity build` will regenerate a migration only if the schema changes. We aren't changing columns, so expect no new migration.

3. Click **Launch agent**. Expect: pane switches to the new conversation; worktree created; tmux session running `claude` with the prompt preloaded.

   - `select id, task_attempt_id from conversations order by created_at desc limit 1;` â†’ new row with `task_attempt_id` matching the attempt above.
5. `tmux list-sessions` shows a `claude-*` session; attaching shows the description as the first user message.
6. Empty-description case: button disabled.
7. Existing **New conversation** button in the sidebar still works (regression: POST with no body).
