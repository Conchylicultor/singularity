# Event-driven queue pin — remove the `queueRanks → conversationsLive` catch-all edge

## Context

`queueRanksResource` (`plugins/conversations/plugins/conversations-view/plugins/queue/server/internal/resource.ts`)
declares `dependsOn: [conversationsLiveResource]`, forcing a FULL recompute on
every poller status tick. A prior contention analysis
(`research/2026-06-15-global-live-state-cascade-contention.md`, Change 3) flagged
this edge as "pure waste," on the premise that the loader reads only ranks + pin
validation and **never reads conversation status**.

**That premise is wrong.** The loader's `validatePin()` (`resource.ts:21`) reads
conversation status and *writes* the pin (`setPinnedId`). A pin is valid only
while the pinned conversation is `waiting`, unblocked, group-selected, and has no
`working` sibling — all status-derived. So the edge is **not** dead: it is the
load-bearing **catch-all** that re-validates the pin whenever any conversation
changes status. Dropping it as-is ships a stale-pin bug.

**The real gap.** Verification (traced in this doc's investigation) shows:

- Every queue *mutation* (reorder/promote/demote/step-down, seed-rank, validate-pin,
  advance-pin, task-status-pin, repair-blocked-order) already validates/sets the
  pin **transactionally** and calls `queueRanksResource.notify()` directly. The
  loader's `validatePin()` is redundant for all of these.
- The four explicit triggers (`conversationCreated`, `conversationTurnCompleted`,
  `userTurnSent`, `taskStatusChanged`) do **not** cover a conversation going
  `gone`. `gone` emits no turn/userTurn event, and `taskStatusChanged` fires only
  when the parent task's *derived* status flips — which it doesn't when the task
  still has another live conversation. So a **pinned conversation that crashes/exits
  in a multi-conversation (group) task leaves the pin stuck pointing at a `gone`
  conversation** unless the conversations cascade re-runs `validatePin`.

**Root smell.** Two anti-patterns are entangled: (1) a live-state *loader* performs
a *write* side-effect (`validatePin → setPinnedId`); (2) pin correctness depends on
a grab-bag of partial triggers plus an incidental resource cascade as the
backstop, with a real gap. There is no single authoritative "a conversation's
status changed" signal, even though one chokepoint already exists.

**Intended outcome.** Pin validity becomes driven by an explicit, complete
conversation-status event. The loader becomes a pure read. The
`conversationsLiveResource` edge is removed for the *right* reason — its job is
done by a proper primitive, not deleted-and-hoped. The `gone`-in-a-group gap is
closed (a correctness improvement over today). Cascade cost is a wash-to-slightly
positive (the per-flush queueRanks recompute is replaced by a singleton-deduped
job).

## Design

### 1. New trigger event `conversation.statusChanged` (mirror `taskStatusChanged`)

**File:** `plugins/tasks/plugins/tasks-core/server/internal/tables-events.ts`

tasks-core owns the `conversations` table and every status write, and already
defines `taskStatusChanged` here — the exact precedent to mirror.

```ts
export interface ConversationStatusChangedPayload {
  conversationId: string;
  taskId: string | null;
  status: ConversationStatus;
  previousStatus: ConversationStatus;
  [key: string]: unknown;
}

export const { event: conversationStatusChanged, table: _conversationStatusChangedTriggers } =
  defineTriggerEvent<ConversationStatusChangedPayload>({
    name: "conversation.statusChanged",
    filters: {}, // global delivery; pin revalidation is global + idempotent
  });
```

Export `conversationStatusChanged` (and the table) from the tasks-core **server
barrel** (`plugins/tasks/plugins/tasks-core/server/index.ts`), exactly as
`taskStatusChanged` / `_taskStatusChangedTriggers` are. The backing trigger table
is DB schema → the migration is generated by `./singularity build` (never by hand).

### 2. Emit at the status-write chokepoint

**File:** `plugins/tasks/plugins/tasks-core/server/internal/mutations/conversations.ts`

All conversation-status writes funnel through this one file and already snapshot
task status `before` the write for `emitStatusChangeIfChanged`. Add the symmetric
conversation-level emit. The functions already know the *new* status; capture the
*old* one cheaply (fold it into the existing `taskIdForConversation` lookup, which
becomes "taskId + current status" in one query; for `markConversationGone` the old
status is whatever active value the row held).

Add a small helper next to `emitStatusChangeIfChanged` (in `status-emit.ts`),
e.g. `emitConversationStatusChange(conversationId, taskId, previousStatus, nextStatus)`
that emits only when `previousStatus !== nextStatus`. Wire it into:

- `updateConversation` — when `patch.status` is set and differs (poller working↔waiting).
- `markConversationGone` — on the `→ gone` transition (**closes the gap**).
- `markConversationClosed` — on `→ done`.
- `insertConversation` / `insertConversationOnConflictDoNothing` — new conv
  (`previousStatus` = the inserted status; harmless, keeps the signal complete).

Emit **after** the write commits, matching `emitStatusChangeIfChanged`'s ordering,
so subscribers re-querying see the new state.

### 3. New `pinRevalidateJob` (singleton-deduped) in the queue plugin

**File (new):** `plugins/conversations/plugins/conversations-view/plugins/queue/server/internal/pin-revalidate-job.ts`

```ts
export const pinRevalidateJob = defineJob({
  name: "queue.pin-revalidate",
  input: z.object({}).passthrough(),
  event: z.object({}).passthrough(),
  dedup: "singleton",      // coalesce a burst of status flips into one revalidate
  maxAttempts: 2,
  run: async () => {
    await validatePin();
    queueRanksResource.notify();
  },
});
```

`dedup: "singleton"` (supported per `jobs/.../registry.ts:85`) collapses a tick's
worth of conversation status flips into a single `validatePin()` — making the
downstream cost flat regardless of how many conversations changed. `validatePin`
reads live DB state, so coalescing is safe.

Register + bind in the queue **server index**
(`.../queue/server/index.ts`):

```ts
Trigger({ on: conversationStatusChanged, do: pinRevalidateJob, with: {}, oneShot: false }),
// register: [...existing, pinRevalidateJob]
```

### 4. Make `queueRanksResource` a pure read

**File:** `.../queue/server/internal/resource.ts`

- Replace `const pinnedConversationId = await validatePin();` with
  `const pinnedConversationId = await getPinnedId();` (read-only; `getPinnedId`
  already exists in `pinned.ts`).
- Remove `dependsOn: [{ resource: conversationsLiveResource }]` and the
  `conversationsLiveResource` import.

The loader no longer writes. The pin is authoritative in `queue_state`, written by
(a) the transactional mutation handlers and (b) the event-driven `pinRevalidateJob`.

### 5. Retire the now-fully-redundant `validatePinJob`

**Files:** delete `.../queue/server/internal/validate-pin-job.ts`; remove its
`import`, its `Trigger({ on: conversationTurnCompleted, ... })`, and its `register`
entry from `.../queue/server/index.ts`.

`validatePinJob` did nothing but `validatePin()` + `notify()` on
`conversationTurnCompleted`. The equivalent transition (working→waiting) now fires
`conversation.statusChanged` → `pinRevalidateJob`, which is **more** correct: it
fires on the actual DB status flip, whereas `conversationTurnCompleted` fires from
the JSONL watcher possibly *before* the DB says `waiting` (so the old
`validatePin` could see the conv still `working` and skip pinning it).

**Keep** the other two — they cover signals the conversation-status event does
*not*:

- `advancePinJob` (on `userTurnSent`): low-latency fast-path. `sendTurn`
  (`runtime.ts:99`) does not write status; the flip to `working` lands on the next
  poller tick (≤1s). Keeping `advancePinJob` advances the pin off the just-sent
  conversation immediately instead of up to a second later.
- `taskStatusPinJob` (on `taskStatusChanged`): a task becoming blocked/unblocked
  changes `notBlocked` in `validatePin` **without** any conversation status change,
  so it is not covered by `conversation.statusChanged`. Keep it (rerank logic +
  `validatePin`) unchanged.

`seedRankJob`'s trailing `validatePin()` is left as-is (redundant but cheap, and
keeps the pin set the instant a conversation is created).

## Verification

1. **Build/migrate:** `./singularity build` (generates the
   `_conversation_status_changed_triggers` migration; never run drizzle-kit by
   hand). Then `./singularity check` (`migrations-in-sync`, `type-check`,
   `plugin-boundaries`, `plugins-doc-in-sync`).
2. **The closed gap (primary):** create a task with **two** live conversations
   (e.g. fork/`+Sonnet` in the same worktree so they share a task group). Pin the
   first. Kill/exit it so it goes `gone` while the sibling stays live. Confirm the
   pin **advances off the gone conversation** (today, without the cascade, it would
   stick). Drive via `e2e/screenshot.mjs` against
   `http://<worktree>.localhost:9000` (Queue view in the conversations sidebar).
3. **No regressions:** normal working→waiting pin set still lands promptly; queue
   reorder/promote/demote/step-down still set the pin (transactional, unchanged);
   `userTurnSent` still advances the pin **immediately** (advancePinJob kept);
   blocked/unblocked still re-pins (taskStatusPinJob kept).
4. **Cost:** `mcp__singularity__get_runtime_profile` `kind:"loader"` — confirm
   `queue-ranks` loader no longer fires on every conversations flush; `kind:"db"`
   `[acquire]` unchanged or improved. Watch that `conversation.statusChanged` emit
   volume + the singleton job do not introduce a new hot spot (`kind:"db"` /
   `query_db` on the trigger/jobs tables during a status storm). Singleton dedup
   should keep `validatePin` invocations ≈ one per burst.
5. **Pin still globally correct after a burst:** trigger a mass status change
   (advance several conversations at once) → exactly one `validatePin` run, pin
   resolves to the correct top.

## Notes / non-goals

- This is primarily a **correctness + architecture** fix (loader purity; close the
  `gone`-in-a-group gap; replace the catch-all edge with a real primitive). The
  cascade-contention relief is secondary — `conversationsLiveResource` already
  carries `debounceMs: 250`, which was the dominant lever.
- `conversation.statusChanged` is a generally useful primitive; other consumers
  that currently lean on the live resource for status (e.g. progress/op-status)
  could adopt it later. Out of scope here.
- Boundary: the queue plugin already imports tasks-core; subscribing to a
  tasks-core event keeps the dependency direction (queue → tasks-core) intact.
  tasks-core must **not** import the queue job — the event/trigger indirection is
  exactly the decoupling seam.
