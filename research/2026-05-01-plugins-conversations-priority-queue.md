# Conversations Priority Queue (v1)

## Context

When many conversations are open it's easy to lose track of which one needs attention next. Some never get resolved because they fall off-screen. The Grouped sidebar view is good for organisation but gives no signal about *what to do next*.

This adds a sibling sidebar view — **Queue** — that shows only conversations currently `status === 'waiting'`, ordered by a single global stack rank the user controls via drag. The queue is modelled as an **Anki-style deck**: the top card is what to do next, drawing a card removes it from the deck, and finishing a card returns it to the deck *after* the new top, so the user's next-action target stays stable.

**Decisions:**
- Eligibility: `status === 'waiting'` only.
- Ordering: pure stack rank (drag-to-reorder). One global ordered list — **no per-attempt scope**.
- Storage: lexical strings via the existing `fractional-indexing` helper.
- All conversations get a rank from creation. The queue display doesn't filter on `rank IS NOT NULL`; the assignment hooks guarantee it's never null for active rows.
- Cycling rule: when a conversation transitions `working → waiting` (a turn finished), it is reinserted **after the current first item** (between top-waiting and second-waiting ranks). The top stays where it is.
- New-conversation rule: `insertConversation` assigns the rank at the **end** of the deck. New tasks land at the bottom; the user drags up if they want to prioritise.
- Drag-to-reorder lets the user override at any time. Manual ranks are only reshuffled when the conversation re-enters waiting.

## Schema (tasks-core)

Add **one nullable column** to `_conversations` and **one index**.

`plugins/tasks-core/server/internal/tables.ts`:

| Column | Type     | Notes                                                       |
|--------|----------|-------------------------------------------------------------|
| `rank` | `rankText` (existing custom type) | Nullable in the DB; the application guarantees non-null on insert and during the queue plugin's onReady backfill. Old rows that pre-date this feature get backfilled at next boot. |

Index:

```sql
CREATE INDEX conversations_status_rank_idx
  ON _conversations (status, rank);
```

This is the canonical query: "all waiting conversations, sorted by rank".

Migration is generated by `./singularity build` (drizzle-kit). Per CLAUDE.md, never run `drizzle-kit generate` manually.

**Why nullable:** keeps the schema additive — no NOT NULL constraint forcing a backfill before drizzle accepts the column. The application layer fills it in at insert time and the queue plugin's `onReady` backfills any existing rows that lack one.

## Server: rank assignment in tasks-core

Two chokepoints in `plugins/tasks-core/server/internal/mutations/conversations.ts`:

### `insertConversation` / `insertConversationOnConflictDoNothing`

On insert, compute `endRank()` and write it. Every new conversation lands at the bottom of the global deck, regardless of status.

### `updateConversation`

When `patch.status === 'waiting'` AND `prev.status !== 'waiting'` AND `patch.rank === undefined`:

```
patch.rank = positionTwoRank()
```

This applies on every transition into waiting (`starting → waiting`, `working → waiting`, `gone → waiting`). The conversation's prior rank is overwritten. Cycling reasserts the position-2 rule.

Manual drag (the reorder endpoint) sets `patch.rank` explicitly, which short-circuits the auto-assignment.

### Helpers

```ts
// End of deck: rank greater than every currently-waiting rank.
async function endRank(): Promise<string> {
  const [last] = await db.select({ rank: _conversations.rank })
    .from(_conversations)
    .where(and(eq(_conversations.status, 'waiting'), isNotNull(_conversations.rank)))
    .orderBy(desc(_conversations.rank))
    .limit(1);
  return generateKeyBetween(last?.rank ?? null, null);
}

// Position 2: between the top waiting rank and the second waiting rank.
// 0 waiting → just any rank (becomes the only item).
// 1 waiting (the existing top) → after it.
async function positionTwoRank(): Promise<string> {
  const top2 = await db.select({ rank: _conversations.rank })
    .from(_conversations)
    .where(and(eq(_conversations.status, 'waiting'), isNotNull(_conversations.rank)))
    .orderBy(asc(_conversations.rank))
    .limit(2);
  const top = top2[0]?.rank ?? null;
  const second = top2[1]?.rank ?? null;
  return generateKeyBetween(top, second);
}
```

The current row is excluded from these queries because at the time of SELECT its status is still the pre-update value (`working`), so `WHERE status='waiting'` filters it out.

## Server: queue plugin (`plugins/.../queue/server`)

Owns two things: the reorder HTTP route and an `onReady` backfill for legacy rows.

### `POST /api/conversations-queue/reorder`

```
body: { conversationId: string, rank: string }
```

Handler:
1. Load conversation, confirm it exists.
2. Validate `rank` is a non-empty string ≤ 256 chars.
3. Call `updateConversation(conversationId, { rank })` — which already calls `recentConversationsResource.notify()` for live-state.

The client computes the rank string via `generateKeyBetween` (matching the tasks precedent at `tree.ts:40`); the wire format is the precomputed string, not `{ beforeId, afterId }`.

### `onReady` backfill

```
SELECT id FROM _conversations WHERE rank IS NULL AND status <> 'gone' ORDER BY created_at ASC
```

Walk the result and assign sequential ranks via `generateKeyBetween` (each starting from the previous one). Idempotent — re-runs are no-ops because the WHERE clause filters out already-ranked rows. Lives in the queue plugin (not tasks-core) because the rank concept is queue-specific and tasks-core stays neutral.

## Client: queue view component

`queue-view.tsx`:

1. **Data:** call `useConversations()`. Filter `active` to `status === 'waiting'` (no `rank !== null` check — the application guarantees rank presence).
2. **Sort flat by rank ascending.** No per-attempt sectioning.
3. **Position chip:** small "1", "2", … pill on the left edge of each row, reinforcing "do the top one next".
4. **Top item highlight:** subtle left-border accent on the top row.
5. **Empty state:** `"All clear — no conversations are waiting on you."`

### Drag-to-reorder

Use `@dnd-kit/core` directly, mirroring the `grouped` plugin's pattern:

- Each row is a `useDraggable` source with id `queue-conv-<id>`.
- Each row exposes two `useDroppable` zones: `before-<id>` and `after-<id>`.
- A single `DndContext` wraps the entire flat list.
- On drop, compute rank client-side:
  - `before-<targetId>`: `generateKeyBetween(prev?.rank ?? null, target.rank)`
  - `after-<targetId>`:  `generateKeyBetween(target.rank, next?.rank ?? null)`
- POST `/api/conversations-queue/reorder` with `{ conversationId, rank }`.
- No optimistic update — wait for the live-state push.

### Visuals

- Top item: subtle left-border accent.
- Drag affordance: drag works on the row body; row otherwise identical to grouped's row.

## Verification

1. **Schema migration.** `./singularity build` regenerates the migration: drops the old `(attempt_id, status, rank)` index, creates the new `(status, rank)` index. Inspect generated SQL before commit.
2. **Insert.** Create a fresh conversation. DB row has `rank` populated immediately (end of deck).
3. **First waiting transition.** Send a prompt; agent finishes a turn. Confirm `rank` is reassigned to position 2 (between top-waiting and second-waiting). Even if there is no other waiting conversation, the rank still resolves cleanly.
4. **Cycling.** Drag a conversation to the top, prompt it (working), wait for waiting. Confirm it returns to position 2 (one slot below the new top) — drag-set rank is *not* honoured across cycles.
5. **No grouping by attempt.** Open the Queue view with conversations from multiple attempts. Confirm flat rendering with global rank order; no section headers.
6. **Drag-to-reorder.** Drag the third item to the top. Confirm `POST /api/conversations-queue/reorder` is sent with a precomputed rank lower than the previous head. Live-state push lands and UI re-renders.
7. **Backfill.** Inspect the DB on first boot after upgrading: confirm any `status <> 'gone'` rows previously lacking a rank now have one. Re-running the boot should be a no-op.
8. **Empty state.** Close all waiting conversations: empty-state message renders.
9. **e2e snapshot.** `bun e2e/screenshot.mjs --url http://<worktree>.localhost:9000/ --click "Queue" --out /tmp/queue` to capture the switcher click.
