# Fields config-migration — batch 2 (enum, dynamic-enum, color, avatar)

## Context

We are unifying all config field-types onto the canonical `fields/` matrix. Batch 1
(commit `acb36b611`) relocated the primitive config field-types (text, bool, int, float,
multiline-text) from `config_v2/plugins/fields/plugins/<type>` into
`plugins/fields/plugins/<type>/plugins/config`, binding each factory + renderer to the
canonical `fields/` token and deleting the legacy `config_v2` plugins.

Batch 2 does the same for **enum, dynamic-enum, color, avatar**. These four are in
inconsistent states today (some have a `fields/` identity, some don't; enum's renderer is
already relocated but its factory isn't). The goal is to bring all four to the uniform
batch-1 end-state: the `fields/` identity owns the token, `plugins/config` owns the
factory + renderer, and the legacy `config_v2` field plugin is deleted. Behavior-preserving:
slot dispatch, schemas, defaults, and the options/picker semantics (including dynamic-enum's
slot-resolved options) are unchanged.

**Scope decisions (confirmed with user):** identity + config capability only — no new
table/filter cells for avatar/dynamic-enum. Rename dynamic-enum's options slot id from
`config-v2.fields.dynamic-enum.options` → `fields.dynamic-enum.options`.

## The batch-1 pattern (target shape per type)

```
plugins/fields/plugins/<x>/
├── core/internal/<x>.ts        defineFieldType("<x>") + defineFieldIdentity(...)   ← identity
├── core/index.ts               re-exports <x>FieldType, <x>Identity
├── web/index.ts                Fields.Identity({ identity: <x>Identity })
└── plugins/config/
    ├── core/internal/<x>.ts    <x>Field() factory + <X>FieldDef, binds to fields/ token
    ├── core/index.ts           re-exports factory + types
    ├── web/components/<x>-renderer.tsx   FieldRendererComponent<T> with .type = <x>FieldType
    ├── web/index.ts            contributions: [Fields.Renderer(<X>Renderer)]
    ├── package.json            "@singularity/plugin-fields-<x>-config", singularity.collapsed
    └── CLAUDE.md               prose only (build codegen inserts the reference block)
```

- Renderers import `Fields`, `FieldHeader`, `FieldRendererComponent`, `useLocalValue` from
  `@plugins/config_v2/plugins/fields/web` — **config_v2 remains the slot owner** of
  `config-v2.fields.renderer`. Field plugins only *contribute*.
- Dispatch is by `field.type.id` string; the renderer's `.type` static prop must be the
  `fields/` token. `pickMeta` comes from `@plugins/config_v2/core`.
- `web.generated.ts` is **autogenerated** by `./singularity build` — never hand-edit. Loader
  entries regenerate from the filesystem after source importers are updated.

## Cross-cutting notes

- **`config_v2/core` stays the home** of `pickMeta`, `getFieldResolver`/`registerFieldResolver`,
  `FieldDef`, `FieldMeta`, and the `defineFieldType` shim — unchanged. Field tokens now come
  from `@plugins/fields/core`.
- **Plugin boundary rules:** only runtime barrels may be imported cross-plugin. All new imports
  target `.../core` or `.../web` barrels (e.g. `@plugins/fields/plugins/enum/plugins/config/core`).
  No authored `id:` in barrels (loader-derived); barrels contain only imports + re-exports +
  single `definePlugin` default.
- **CLAUDE.md files:** write prose only — `./singularity build` codegen inserts the
  `## Plugin reference` AUTOGENERATED block. Update parent `plugins/fields/plugins/{avatar,dynamic-enum}/`
  CLAUDE.md (new) and the `enum`/`color` parent CLAUDE.md sub-plugin lists if present.
- **Docs:** `docs/plugins-compact.md` / `plugins-details.md` regenerate on build; the
  `plugins-doc-in-sync` check enforces it.
