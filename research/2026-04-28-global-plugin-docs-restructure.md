# Plugin docs restructure

## Context

`docs/plugins.md` is auto-generated from plugin source by `cli/src/docgen.ts` and currently inlined into the root `CLAUDE.md` via `@docs/plugins.md`. As the plugin tree has grown (~40 plugins, deeply nested umbrellas) the file has become large enough to dominate every agent's context window — defeating its purpose.

But the plugin tree itself is load-bearing context: an agent that doesn't see it will reinvent helpers, miss public exports, or contribute to the wrong slot. The fix is not to remove the doc but to split it into a slim always-loaded index and a fuller on-demand reference, and to push the heavy per-plugin detail into per-plugin `CLAUDE.md` files that are read only when an agent is actually working on that plugin.

While restructuring, we add what's currently missing: **reverse-index sections** (who imports my exports, who contributes to my slots, who calls my endpoints) so a plugin author can see their consumers without grepping the whole repo.

## Goals

- Slim `docs/plugins-compact.md` (always-loaded): tree + name + one-line description for every plugin, plus full detail for plugins flagged `loadBearing`.
- Fat `docs/plugins-details.md` (read on demand): everything currently in `docs/plugins.md` + reverse-index sections.
- Per-plugin `CLAUDE.md` at `plugins/<…>/CLAUDE.md`: human-written prose above an `<!-- AUTOGENERATED -->` fence; the autogen block contains that plugin's section from details.md (including reverse indexes).
- New `loadBearing?: boolean` field on `PluginDefinition` for explicit opt-in to the compact doc's full detail.
- Replace the existing `plugins-doc-in-sync` check with one that validates all three artifacts (compact, details, every per-plugin CLAUDE.md). Add a new `plugins-have-claudemd` check.
- Update root `CLAUDE.md` to load `@docs/plugins-compact.md` and point the "before writing a helper" rule at `docs/plugins-details.md`.

## Non-goals

- No automatic derivation of `loadBearing` (e.g. by import count). Explicit flag only — usage limited to critical core infra (e.g. `plugin-core`, `tasks-core`, `conversations`, `infra/secrets`, `infra/jobs`, `infra/events`, `infra/attachments`, `auth`, `config`, `shell`, `primitives/pane`, `primitives/live-state`).
- No MCP doc tool (raised as a future option; out of scope for now).
- No deletion of existing per-plugin prose; the migration preserves what's in the four existing `CLAUDE.md` files.

## Output layout

```
docs/
  plugins-compact.md     # always-loaded; tree + 1-liner + full detail for loadBearing plugins
  plugins-details.md     # read on demand; full detail for every plugin + reverse indexes
plugins/
  <…>/CLAUDE.md          # one per plugin; hand-written prose + AUTOGEN fence
```

`docs/plugins.md` is removed.

### `plugins-compact.md` shape

For non-load-bearing plugins, one indented bullet per plugin with name + description:

```
- **`screenshot`** — Capture the current page and edit it…
- **`shell`** [load-bearing] — Foundational app layout; defines slots most plugins extend.
  - Defines slots: `Shell.Sidebar`, `Shell.Toolbar`
  - Exports (web): Shell, ShellCommands
  - …full detail block as in details.md…
```

A `[load-bearing]` marker after the name signals that the full detail block follows. Children of an umbrella are nested as today.

### `plugins-details.md` shape

Same content as today's `plugins.md` for every plugin, plus three new reverse-index lines per plugin:

```
- **`auth`** — Shared authentication infrastructure…
  - Exports (web): …
  - Contributes: …
  - Server: …
  - Imported by: `auth/google`, `auth/notion`, …          ← NEW
  - Slot contributors: `auth/google`, `auth/notion`        ← NEW (who contributes to MY slots)
  - Endpoint callers: `shell`, `conversation-view`, …      ← NEW (best-effort)
```

"Endpoint callers" is best-effort: docgen greps `web/` and `server/` for `fetch("/api/<my-route>...")` patterns. False negatives are acceptable; treat as a hint, not authority. If implementation cost grows, ship without it and add later.

### Per-plugin `CLAUDE.md` shape

```markdown
# <Plugin name>

<hand-written prose: topology, gotchas, examples, deferrals>

<!-- AUTOGENERATED:BEGIN — do not edit; regenerated by `./singularity build` -->

## Plugin reference

- Description: …
- Exports (web): …
- Contributes: …
- Server: …
- Imported by: …
- Slot contributors: …
- Endpoint callers: …

<!-- AUTOGENERATED:END -->
```

The hand-written prefix is preserved across regenerations. The fence text matches the existing convention in `cli/src/docgen.ts:4-5`.

For plugins with no hand-written prose (the new majority), the file starts with a one-line title and goes straight into the AUTOGEN block.

## Mechanism

### Single pass, three outputs

`docgen.ts` already collects all plugins via `collectPlugin()` into `byDir: Map<dir, PluginInfo>` (line 700). After that map is built, build the reverse indexes once:

- **`importedBy`**: invert each `PluginInfo.apiUses` — if A's apiUses contains `B.foo`, push A onto `B.importedBy`.
- **`slotContributors`**: invert `contributions` — if A contributes to slot `B.Foo`, push A onto `B.slotContributors`.
- **`endpointCallers`**: scan every plugin's source for fetch/url-string occurrences of `/api/<prefix>/`; map prefix → owning plugin via the existing `httpRoutes` registry.

Then render three outputs:

1. `renderCompactDoc(byDir)` → `docs/plugins-compact.md`
2. `renderDetailsDoc(byDir)` → `docs/plugins-details.md`
3. For each plugin, `renderPluginClaudeMd(info)` → `plugins/<…>/CLAUDE.md` (preserve content above the AUTOGEN fence; rewrite content inside)

`renderPlugin()` (the existing per-plugin renderer) is the basis for both `renderDetailsDoc` and the AUTOGEN body of `renderPluginClaudeMd` — share the same function and pass a flag for whether to include reverse-indexes (always yes now).

### `loadBearing` flag

Add to `plugin-core/types.ts`:

```ts
export interface PluginDefinition {
  id: PluginId;
  name: string;
  description: string;
  loadBearing?: boolean;
  contributions?: Contribution[];
}
```

Add to `PluginInfo` in `cli/src/docgen.ts:29` and read it from the parsed barrel default-export object alongside `name` and `description` (existing parse path in `collectPlugin`).

Confirmed flag-set:

- `tasks-core`
- `conversations`
- `auth`
- `config`
- `shell`
- `infra/secrets`, `infra/jobs`, `infra/events`, `infra/attachments`, `infra/mcp`
- `primitives/pane`, `primitives/live-state`, `primitives/networking`

(`plugin-core` is the framework, not a plugin in `plugins/`, so it is not in the set.)

### Preserving hand-written prose in `CLAUDE.md`

Existing per-plugin docs:

- `plugins/auth/CLAUDE.md`
- `plugins/config/CLAUDE.md`
- `plugins/infra/plugins/secrets/CLAUDE.md`
- `plugins/primitives/plugins/pane/web/CLAUDE.md` ← misplaced; move to `plugins/primitives/plugins/pane/CLAUDE.md` as part of this work

Algorithm for `renderPluginClaudeMd(info)`:

1. Read existing file. If absent, treat prose-prefix as `# <name>\n\n`.
2. Find `<!-- AUTOGENERATED:BEGIN` line. Everything before it is the prose-prefix; preserve verbatim.
3. Replace from `BEGIN` line through `END` line (inclusive) with the freshly-rendered AUTOGEN block.
4. Write only if content changed.

If the file exists but has no AUTOGEN fence (the four pre-existing files), append the AUTOGEN block at the end on the first run.

## Implementation steps

1. **Plumb `loadBearing` through types.** Edit `plugin-core/types.ts` to add the field. Edit `cli/src/docgen.ts` `PluginInfo` interface (line 29) and `collectPlugin()` (line 446) to extract it from the barrel's `definePlugin` default export.

2. **Add reverse-index computation.** In `renderPluginDocs()` (line 696), after `byDir` is built, compute the three reverse maps (`importedBy`, `slotContributors`, `endpointCallers`) and attach to each `PluginInfo`.

3. **Refactor renderers.**
   - Rename existing `renderPlugin()` to `renderPluginDetail()`; extend to emit reverse-index lines.
   - Add `renderPluginCompact(info)`: name + description + `[load-bearing]` marker + (if load-bearing) the full detail body.
   - Add `renderPluginClaudeMd(info, existingFileContent | null)`: prose-preserving merge described above.
   - Add three top-level functions: `renderCompactDoc`, `renderDetailsDoc`, `generateAllDocs` — the last orchestrates writing all artifacts.

4. **Replace `generatePluginDocs()`.** Have it write `plugins-compact.md`, `plugins-details.md`, and every per-plugin `CLAUDE.md` instead of `plugins.md`. Delete `plugins.md` from disk on first run (one-shot cleanup, separate commit).

5. **Update the in-sync check.** `cli/src/checks/plugins-doc-in-sync.ts` now validates all three artifact families — compact, details, and that every per-plugin AUTOGEN block matches expected content. Keep the same `id` for now (no migration friction). Add a new check `plugins-have-claudemd` that fails when any plugin lacks a `CLAUDE.md`. Register both in `cli/src/checks/index.ts`.

6. **Move misplaced doc.** Move `plugins/primitives/plugins/pane/web/CLAUDE.md` → `plugins/primitives/plugins/pane/CLAUDE.md`.

7. **Set `loadBearing` flags** on the confirmed set (see list above).

8. **Update root `CLAUDE.md`.**
   - Replace `@docs/plugins.md` with `@docs/plugins-compact.md`.
   - Update the rule "Before writing a helper, search `docs/plugins.md` for it" to point at `docs/plugins-details.md`.
   - Add a one-line note that per-plugin `CLAUDE.md` carries detailed plugin docs and is auto-loaded when working in that plugin's tree.

9. **Run `./singularity build`** to regenerate everything; commit the generated artifacts and the source changes together.
