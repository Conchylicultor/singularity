# Decouple conversation "done" from tmux process state

## Context

Today `ConversationStatus` has 4 values: `starting | working | waiting | gone`. `gone` is the only terminal status and serves double duty — it means both "the tmux process died" and "the conversation is finished". The poller (1s interval) automatically sets `gone` whenever the tmux pane disappears, regardless of whether the user intended to close the conversation.

User exit actions (exit, drop-and-exit, hold-and-exit, push-and-exit) kill the tmux session, then the poller detects the dead pane and writes `gone` — indistinguishable from a crash.

**Goal:** separate "process not running" from "user explicitly closed". This enables future resource savings — old tmux sessions could be killed without changing UX, since the conversation would still show as active based on user intent, not process state.

## Design

Add a 5th status: `done`. After the change:

| Status | Meaning | Active? | Has live process? |
|---|---|---|---|
| `starting` | Spawning tmux / warming worktree | yes | pending |
| `working` | Claude is computing | yes | yes |
| `waiting` | Claude paused for user input | yes | yes |
| `gone` | Process dead, NOT user-completed | yes | no |
| `done` | User explicitly closed | no | no |

- **`gone`** = process is dead, but conversation is conceptually active. Appears in active list. Can be resumed.
- **`done`** = user explicitly marked as finished. Terminal. Appears in history/done list.

## Implementation

### 1. Type/schema changes

**`plugins/conversations/server/status.ts`** — Add `"done"` to enum, update `isActiveStatus`, add `hasLiveProcess` helper:

```ts
export const ConversationStatusSchema = z.enum([
  "starting", "working", "waiting", "gone", "done",
]);

export function isActiveStatus(status: ConversationStatus): boolean {
  return status !== "done";
}

export function hasLiveProcess(status: ConversationStatus): boolean {
  return status === "working" || status === "waiting" || status === "starting";
}
```

**`plugins/tasks-core/server/internal/tables.ts`** — Update local `ConversationStatus` type to include `"done"`.

**`plugins/tasks-core/server/internal/schema.ts`** — Three changes:
- Local `ConversationStatusSchema`: add `"done"`
- `conversations_v` view: `active` becomes `status <> 'done'` (was `status <> 'gone'`)
- `attempts_v` view: `hasLiveConv` becomes `status NOT IN ('gone', 'done')` (was `status <> 'gone'`) — only truly running processes count as "live" for attempt status

**`plugins/tasks-core/server/internal/mutations/conversations.ts`** — Add `"done"` to all type unions (`InsertConversationInput.status`, `UpdateConversationPatch.status`, `insertConversationOnConflictDoNothing`). Change `markConversationClosed` to write `status: "done"` instead of `"gone"`.

**`plugins/tasks-core/server/internal/mutations/cross-table.ts`** — Add `"done"` to `AdoptOrphanInput.status` type.

### 2. Server behavior

**`plugins/conversations/server/internal/poller.ts`** — Two changes:
- Skip `"done"` rows alongside `"gone"` in the dead-pane check: `if (dbRow.status === "gone" || dbRow.status === "done") continue`
- Skip `"done"` rows in the missing-from-runtime sweep: `if (dbRow.status === "gone" || dbRow.status === "done") continue`
- Resurrection logic stays `gone`-only — `done` rows are never auto-resurrected

**`plugins/conversations/server/internal/lifecycle.ts`** — `resumeConversation` guard: allow both `"gone"` and `"done"` (user can resume a conversation they previously closed):
```ts
if (row.status !== "gone" && row.status !== "done") {
  throw new Error(`Cannot resume conversation ${id} (status: ${row.status})`);
}
```

**`plugins/conversations/server/internal/handle-close.ts`** — Add `markConversationClosed` call before `deleteConversation`. This ensures the DB row is set to `"done"` before the poller sees the dead pane.

**Exit action servers** — Each should call `markConversationClosed(id)` before `deleteConversation(id)`:
- `plugins/conversations/plugins/conversation-view/plugins/exit/server/index.ts`
- `plugins/conversations/plugins/conversation-view/plugins/drop-and-exit/server/index.ts`
- `plugins/conversations/plugins/conversation-view/plugins/hold-and-exit/server/index.ts`

**`plugins/conversations/plugins/conversation-view/plugins/push-and-exit/server/internal/exit-clean-finalize-job.ts`** — Add `markConversationClosed(conversationId)` before `deleteConversation(conversationId)`.

### 3. DB migration

The migration (generated by `./singularity build` from schema changes) will rebuild `conversations_v`, `attempts_v`, and `tasks_v` views. Additionally, include a data migration:

```sql
UPDATE conversations SET status = 'done' WHERE status = 'gone';
```

This migrates all historical `gone` rows to `done` for a clean slate.

### 4. UI changes

**`plugins/conversations/plugins/conversation-ui/plugins/item/web/components/conversation-item.tsx`** — Add `"done"` to `CONV_STATUS_DOT` (same muted color as current `gone`). Update `muted` check: `conv.status === "gone" || conv.status === "done"`.

**`plugins/conversations/plugins/conversation-view/plugins/status/web/components/status-badge.tsx`** — Add `"done"` to `STATUS_CLASSES`. Display `"gone"` as "disconnected" for better UX.

**`plugins/conversations/plugins/conversations-view/plugins/queue/web/components/queue-view.tsx`** — `gone` conversations appear mixed into active sections (dimmed, like blocked). Rename "Recently gone" section to "Done".

**`plugins/conversations/plugins/conversation-view/plugins/resume/web/components/resume-button.tsx`** — Allow resume for both `"gone"` and `"done"`.

**Exit buttons** (exit, drop-and-exit, hold-and-exit, push-and-exit web components) — Replace `live.status === "gone"` checks with `!hasLiveProcess(live.status)` for semantic clarity.

**`plugins/conversations/plugins/conversation-view/plugins/prompt-input/web/components/prompt-input.tsx`** — Update `disabled` and placeholder for both `gone` and `done`.

**`plugins/conversations/plugins/conversation-view/plugins/jsonl-viewer/web/components/jsonl-pane.tsx`** — Fade transcript for both `gone` and `done`.

**`plugins/conversations/plugins/conversation-view/plugins/terminal-pane/web/components/terminal-pane-body.tsx`** — Track transitions from both `gone` and `done` back to active for terminal reattach.

### 5. Key invariants

- The poller NEVER writes `"done"` — it only writes `"gone"`, `"working"`, `"waiting"`
- Only explicit user actions write `"done"` (via `markConversationClosed`)
- `gone` conversations are `active = true` in the DB view
- `done` conversations are `active = false` in the DB view
- Attempt `hasLiveConv` requires an actually running process (`status NOT IN ('gone', 'done')`)
- Task status computation is unaffected — it reads `hasLiveConv` transitively

2. Open the app, create a conversation, verify it shows as `working`/`waiting`
3. Kill the tmux session manually (`tmux kill-session -t conv-XXX`) — verify the conversation shows as `gone`/`disconnected` in the active list, NOT in the done section

5. Use Drop & Exit — same, conversation goes to Done

8. Resume a `done` conversation — should also work
9. Check that task statuses compute correctly: a task with only `gone` conversations = `attempted`, with only `done` + push = `done`
