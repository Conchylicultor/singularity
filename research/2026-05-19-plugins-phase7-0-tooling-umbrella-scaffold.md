# Phase 7.0: Tooling Umbrella Scaffold

## Context

Part of the tooling → plugin migration ([Phase 7 plan](./2026-05-19-phase7-tooling-to-plugin.md)). The `tooling/` package needs to become an umbrella plugin at `plugins/framework/plugins/tooling/`. This first sub-task creates the scaffold: `package.json`, `tsconfig.json`, and a `core/index.ts` barrel exporting the shared `Check`/`CheckResult` types.

These types are the shared currency that breaks the cycle between the future `boundaries` and `checks` sub-plugins — both need the types, but neither should depend on the other for them.

The old `tooling/` remains fully functional. Nothing is deleted or re-pointed.

---

## Files to create

### 1. `plugins/framework/plugins/tooling/package.json`

Follows the umbrella convention from `plugins/framework/package.json` and `plugins/infra/package.json`.

```json
{
  "name": "@singularity/plugin-framework-tooling",
  "description": "Umbrella for build-time tooling: boundary checker, lint rules, checks, guards, codegen",
  "private": true,
  "version": "0.0.1"
}
```

No `"type": "module"` — umbrella has no runnable code of its own (sub-plugins will add it as needed).

### 2. `plugins/framework/plugins/tooling/tsconfig.json`

Single compilation unit for all sub-plugin `core/` and `bin/` dirs. Modeled after `server-core/tsconfig.json` but scoped to tooling's own sub-plugins only.

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "skipLibCheck": true,
    "noEmit": true,
    "isolatedModules": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "jsx": "react-jsx",
    "types": ["@types/bun"],
    "paths": {
      "@plugins/*": ["../../../*"]
    }
  },
  "exclude": [
    "../../../framework/plugins/web-sdk/**",
    "**/node_modules"
  ],
  "include": [
    "core",
    "plugins/*/core",
    "plugins/*/bin"
  ]
}
```

Key decisions:
- `@plugins/*` → `"../../../*"` — correct depth: `plugins/framework/plugins/tooling/` → back 3 levels to `plugins/`
- `include` sweeps umbrella `core/` + future sub-plugin `core/` and `bin/` dirs. TSC silently ignores globs matching no files, so this is forward-safe.
- Excludes `web-sdk/**` (same as `server-core`) to avoid pulling React/JSX compilation into the tooling unit.
- Does NOT include the broad `../../../../**/plugins/*/server` globs — tooling is build-time-only and doesn't type-check server/web plugin code.

### 3. `plugins/framework/plugins/tooling/core/index.ts`

Types lifted from `tooling/src/checks/types.ts`:

```typescript
export interface Check {
  id: string;
  description: string;
  run(): Promise<CheckResult>;
}

export type CheckResult =
  | { ok: true }
  | { ok: false; message: string; hint?: string };
```

Flat single-file definition (no separate `types.ts`). Can be split later if the umbrella core grows.

### 4. Root `package.json` workspaces — no change

Current: `["plugins/**", "cli", "tooling"]`

The new umbrella is already covered by `"plugins/**"`. The old `tooling/` still needs its explicit entry — removing it risks breaking `bun install` dep hoisting for the still-active old tooling. Both packages have different names (`@singularity/tooling` vs `@singularity/plugin-framework-tooling`), so no workspace conflict. Remove `"tooling"` in sub-task 7.6 when the old directory is deleted.

## Verification

1. `bun install` — new workspace package resolves without errors
2. `bunx tsc -p plugins/framework/plugins/tooling/tsconfig.json --noEmit` — passes (core/index.ts has no imports)
3. `./singularity build` — succeeds, generates `CLAUDE.md` for the new umbrella, updates `docs/plugins-compact.md` and `docs/plugins-details.md`
4. `./singularity check` — all checks pass, including `plugins-have-claudemd` (generated by build) and `plugins-doc-in-sync`
5. Old tooling unaffected: `./singularity check --list` still works via old `@tooling/*` paths
6. `plugins.generated.ts` files unchanged — tooling has no `web/`, `server/`, or `central/` barrel, so registry codegen skips it
