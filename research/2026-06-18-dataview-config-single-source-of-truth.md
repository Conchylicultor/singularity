# DataView config: single source of truth + plugin-tree placement (reorder-parity, fixes)

**Date:** 2026-06-18
**Category:** global (touches `primitives/data-view`, `primitives/data-view/view-core`, `framework/tooling/codegen`, a new data-view check, and 12 consumer config files)
**Follows:** [`2026-06-18-global-dataview-universal-config-mode.md`](./2026-06-18-global-dataview-universal-config-mode.md) (the universal-config-mode change that already landed)

## Context

The universal-config-mode change made every `<DataView>` config-backed and shipped on commit `9a20f2b5d`. Reviewing the result against the reorder precedent it was meant to mirror surfaced three gaps:

1. **Configs live in the wrong place.** Every DataView descriptor registers under the `primitives.data-view` plugin id, so all configs land in `config/primitives/data-view/<id>.{jsonc,origin.jsonc}`. They should live in the **consuming plugin's** tree (e.g. `config/apps/sonata/library/…`, `config/conversations/agents/…`), exactly as reorder plants each slot's directive under its *defining* plugin via the `pluginId` override.

2. **Two sources of truth; views are empty.** Today the code synthesizes one default view-instance per view-type at runtime (`seedRows()` / the `null`-mirror fallback in `use-views-config.ts`), and the origin files are `{ "views": [] }`. So a view's definition lives both in code (synthesized defaults) *and* in config. The **config must be the only source** — the views (with their sort/filter) are defined explicitly in committed config files, and a reorder-style **`configs-authored` check fails by default** until an agent authors them. This forces agents to compose views in config rather than relying on a code fallback.

3. **Sonata config was deleted.** The prior change orphaned/deleted `config/apps/sonata/library/views*.jsonc`. The sonata library's named views (Cards / All / Longest / Composed, including their sort/filter) must be restored as an explicit authored config at the new plugin-tree path.

**Intended outcome:** every DataView's views are defined *only* in a committed `config/<consuming-plugin>/<id>.jsonc`; the runtime reads config and never synthesizes; a build/push check fails until each DataView's config is authored; sonata's views are restored.

**Confirmed design decisions (with user):**
- **Terse authored rows.** Author just `{ name, view }` per row; the resolver derives `id` (slug of name, else index) and `rank` (array order) on read. Matches the desired authored shape; easiest to hand-write.
- **Visible placeholder on empty.** When a DataView has no views in config at runtime, render an honest "No views configured — author `config/<plugin>/<id>.jsonc`" placeholder. The build-time check is the real forcing function; runtime degrades visibly without crashing the pane.

## The model we copy: reorder

- `reorderable-slots.generated.ts` carries **`{ slotId, pluginId }`** — `pluginId` is the *defining* plugin's tree path.
- `reorder/{web,server}/internal/config-registrations.ts` passes `pluginId: asPluginId(s.pluginId)` to `ConfigV2.{WebRegister,Register}`, so config_v2 derives the path as `config/<asPath(pluginId)>/<name>.jsonc` (path = override `pluginId` ?? registering `_pluginId`, then dot→slash; see `config_v2/server/internal/registry.ts:275,282,284`).
- `reorder:configs-authored` check (`plugins/reorder/check/index.ts`) fails the build/push if any reorderable slot lacks a hand-authored `config/<pluginId>/<slotId>.jsonc` (via `git ls-files --cached --others`). `alwaysRun: true`, `cacheSignature: () => null`. This is the "fail by default" forcing function.

## Implementation

### 1. Codegen: capture the defining pluginId per DataView

`plugins/framework/plugins/tooling/plugins/codegen/core/data-views-gen.ts`:
- Change `collectDataViews` to return `{ id, pluginId }[]` instead of `string[]`. When a `defineDataView("<id>")` marker is found in a node's `web/**`, record that node's plugin id (the enriched-tree node id — confirm the field name on `tree.byDir` nodes; reorder's static scanner returns `{ slotId, pluginId }`, use the same node identity). Dedupe by id (first definer wins), sort by id.
- Update `DataViewEntry` to `{ id: string; pluginId: string }` and `renderManifest` to emit both fields.
- Update the header comment: configs now land under `config/<pluginId>/<id>.jsonc`, **not** `config/primitives/data-view/`.
- `plugins/primitives/plugins/data-view/shared/data-views.generated.ts` will regenerate with the new shape on build. (It's autogenerated; no manual stub edit needed, but the interface widening must compile against its consumers — update those in step 2 first or in the same pass.)

### 2. Register each config under its consuming plugin

The view-core registration helpers are generic and currently take a single `pluginId`. Make them per-entry:
- `view-core/web/internal/build-descriptors.ts` — `buildViewConfigContributions(entries, pluginId)` → accept `entries: Array<{ descriptor; pluginId }>` and register each under its own `pluginId`. (`buildViewDescriptors(ids)` stays: the descriptor is keyed by id and is pluginId-independent.)
- `view-core/server/internal/config-registrations.ts` — `buildViewConfigRegistrations(ids, pluginId)` → `buildViewConfigRegistrations(entries: Array<{ id; pluginId }>)`.
- `data-view/web/internal/descriptors.ts` — build `dataViewDescriptorEntries` from the manifest as `{ id, descriptor, pluginId }` (descriptor still `viewsDescriptor(v.id)`; `dataViewDescriptors` Map<id, descriptor> unchanged for `useViewsConfig`'s reference-identity lookup).
- `data-view/web/internal/config-registrations.ts` + `data-view/server/internal/config-registrations.ts` — pass the per-entry pluginId; drop the central `DATA_VIEW_PLUGIN_ID` constant.

Descriptor `name` stays `= id` (e.g. `sonata.library`), so the file is `config/apps/sonata/library/sonata.library.jsonc`. The id grammar already bans `:`, keeping names filename-safe.

### 3. Config = single source of truth (delete code synthesis)

`view-core/web/internal/use-views-config.ts`:
- **Delete `seedRows()`** and the synthesized-defaults path. Drop the `defaults: ResolvedViewInstance<T>[]` parameter.
- **Normalize-on-read** (new helper) to support terse rows: map each raw config row to `{ id, rank, name, view }`, deriving `id` = explicit `id` ?? slug(name) ?? `view-${index}`, and `rank` = explicit `rank` ?? a generated `Rank.between` sequence following array order. Apply when reading `config.views` and in the reconcile effect.
- `mirror` initial state: `normalize(configRows)` (no longer `null` for empty — empty config means **no views**).
- Reconcile effect: incoming empty → `[]` (remove the "stay un-materialized / display defaults" branch).
- `displayRows = mirror ?? []`; `applyMutation` base = `prev ?? []`. First UI edit thus materializes from the authored rows (or empty), never from code.
- Keep `buildInstanceFromRow` + `contributions` (needed to resolve a row's `type` → renderer and to power the add-view menu).

`view-core/web/internal/use-view-model.ts` (and `data-view`'s `use-data-view-model.ts` host wrapper):
- Stop passing synthesized `defaults` into `useViewsConfig`. Keep `useResolvedInstances`/contributions **only** for the addable-types menu (`available`) — that is "what view-types exist", not "default instances".
- When the resolved instance list is empty, surface it so the host renders the placeholder (return an `isEmpty`/empty `instances` the `<DataView>` body can branch on).

`data-view/web/components/data-view.tsx`:
- When `viewModel.instances.length === 0`, render a `<Placeholder>` ("No views configured — author `config/<plugin>/<id>.jsonc`") instead of the (empty) view body. Use `primitives/css/placeholder.Placeholder`.

### 4. New `data-view:configs-authored` check

New `plugins/primitives/plugins/data-view/check/index.ts` (copy `plugins/reorder/check/index.ts` structure):
- Read `dataViews` (`{ id, pluginId }`) from the generated manifest.
- `overridePathFor(pluginId, id)` = `config/${asPath(asPluginId(pluginId))}/${id}.jsonc` (identical helper to reorder's).
- Fail if any DataView lacks an authored override in `git ls-files --cached --others --exclude-standard -- config/`.
- `id: "data-view:configs-authored"`, `alwaysRun: true`, `cacheSignature: () => null`.
- **No grandfathering** — all 12 configs are authored in step 6, so the check passes clean. (If staging in parts, add a `grandfathered-slots.ts` twin; not needed if all authored at once.)
- Message/hint: tell the agent to author `config/<plugin>/<id>.jsonc` with a `views` array of `{ name, view }` rows (terse), pointing at the sonata file as the worked example.

### 5. Move/clean origin files

- The build's `generateConfigOrigins` regenerates origins at the new `config/<pluginId>/<id>.origin.jsonc` paths (origin stays `{ "views": [] }` — descriptor default; stable hash, independent of view-types, so authored overrides never go stale when a view-type is added).
- **Delete the stale** `config/primitives/data-view/*.origin.jsonc` (12 files) — they no longer correspond to any registration. `config-origins-in-sync` would otherwise flag them.

## Risks / notes

- **Manifest interface widening** (`DataViewEntry` gains `pluginId`) is a breaking change to every reader — update `descriptors.ts` (web+server) in the same pass so the build compiles.
- **Node pluginId field**: confirm the exact field on enriched-tree nodes used to attribute a marker to its plugin (mirror reorder's static scanner identity), so sub-plugin call sites (e.g. `tasks-recent` under `tasks.task-list.recent`) get the precise path.
- **sort/filter value shapes** inside the `view` variant blob must match what the host reads (`viewFor`/`updateView` merge `sort`/`filter` as host-injected keys) — verify against `SortState`/`FilterGroup` before authoring the 12 configs, especially the sonata `filter`.
- **No materialized-origin preparer** (unlike reorder): data-view origins stay `{ "views": [] }` with a stable hash, because each authored view is self-contained (names its own `type`) — adding a view-type never invalidates committed configs. Simpler than reorder's catalog materialization.
