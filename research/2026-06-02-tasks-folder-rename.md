# Rename tasks parent/child → "folder" (decouple display hierarchy from the dependency DAG)

## Context

Agents repeatedly confuse two distinct task relationships:

1. **The dependency DAG** (`task_dependencies`: `taskId depends_on dependsOnTaskId`) — the only execution-relevant relationship. This is what matters.
2. **The parent/child hierarchy** (`_tasks.parent_id`) — purely a display tree for human organization.

Why agents confuse them:

- **The data conflates them at creation.** `add_task` silently writes `parentId = currentTaskId` alongside the dependency wiring, so in practice every dependency edge is *also* a parent edge. The hierarchy looks load-bearing because it's minted next to a real dependency.
- **The hierarchy leaks back as semantic prompt context.** When `includeParentTask` is set, the server injects `<parent-task id=…>title + description</parent-task>` into the new task's opening prompt — training the agent to treat its "parent" as the thing it belongs to / depends on. The injected task is actually the **folder** (`task.parentId` → a *sibling* relationship via `targetForSibling`), i.e. pure organization, not the dependency target.
- **The name itself.** "parent/child" is a structural name that implies containment/inheritance, which reads as dependency.

**Goal:** make the hierarchy obviously display-only so agents stop reasoning about it for execution. The chosen model (decided with the user):

- **Orthogonal "folder" model.** The hierarchy becomes a `folder` concept: *any task can act as a folder* for tasks filed under it. `folder_id` still references `_tasks.id`. It is display/organization only and never agent-facing as execution context.
- **Keep the silent default.** `add_task` continues to file new tasks under the current task — that's provenance ("spawned while working on X"), purely for the human tree. Not up for debate.
- **Close the leak.** Remove the `<parent-task>` injection path entirely (confirmed: remove, not rename).
- **Full relabel.** Rename data, types, agent-facing strings, *and* human UI copy to "folder" vocabulary.
- **`groupId` untouched.** Leave the separate `group_id` column alone, but note: its stated intent ("a meta-task containing sub-tasks") overlaps with "folder" — flag for a future reconciliation so the two container-ish axes don't drift further.

### Key design decision: rename scope = tasks domain only

`parentId` is also the vocabulary of the **generic `tree` primitive** (`TreeItem`, `buildTree`, `computeDrop`, `isDescendant`, `TreeList`) at `plugins/primitives/plugins/tree/`. There, "parent" is correct, domain-neutral tree vocabulary used by other consumers — **do not rename it.** The `folder` rename lives entirely in the **tasks domain**. At the tree-rendering boundary we map `folderId ↔ parentId`:

- Feeding tasks into `buildTree`/`TreeList`: map `{ ...task, parentId: task.folderId }`.
- DnD `onMove` returns `{ parentId, rank }` → tasks layer maps to `{ folderId: parentId, rank }` when calling `patchTask`.

Also **do not rename** `tasks_ext_auto_start.parent_id` — that is the entity-extensions PK FK to `tasks.id`, an unrelated convention column.

## Implementation

### 1. Schema + migration (`plugins/tasks-core`)

- `server/internal/tables.ts`:
  - `parentId: text("parent_id")` → `folderId: text("folder_id")` (keep `.references(() => _tasks.id, { onDelete: "cascade" })`).
  - Index `tasks_parent_rank_idx` on `(parentId, rank)` → `tasks_folder_rank_idx` on `(folderId, rank)`.
- Migration is **generated by `./singularity build`** (never hand-run drizzle-kit). The generated SQL should be a column + index rename:
  - `ALTER TABLE "tasks" RENAME COLUMN "parent_id" TO "folder_id";`
  - drop `tasks_parent_rank_idx`, create `tasks_folder_rank_idx` on `("folder_id","rank")`.
  - Verify the generated migration is a rename (not drop+add, which would lose data). If drizzle emits drop/add, author the rename manually in the generated file before committing.
- `server/internal/schema.ts` — no change needed: `getTableColumns(_tasks)` and `createSelectSchema(_tasks)` auto-pick up the rename, so `TaskSchema.folderId` flows everywhere automatically.
- `server/internal/tables-events.ts` — `TaskStatusChangedPayload.parentId` → `folderId`.
- `server/internal/status-emit.ts` — `ParentSnapshot.parentId`, the `.select({ parentId: … })`, and `parent?.parentId ?? null` → folder naming.

### 2. Repository / mutations / queries (`plugins/tasks-core`)

- `server/internal/mutations/tasks.ts`:
  - `CreateTaskInput.parentId` / `UpdateTaskPatch.parentId` → `folderId`.
  - All local `parentId` vars, `dbPatch.parentId`, the reparent guards (self-folder + descendant checks), `.where(eq(_tasks.parentId, id))` (children-exist delete guard), `.set({ parentId: metaTaskId })` in `backfillMetaParent`, `isNull(_tasks.parentId)` → folder.
  - Rename `backfillMetaParent` → consider `backfillMetaFolder` for consistency (optional; check callers).
- `server/internal/queries/tasks.ts`:
  - `findNextRankUnder(parentId, …)` → `findNextRankInFolder(folderId, …)` (or keep name, rename param); passes `_tasks.folderId` as scope column. Update all 3 callers.
  - server-side `isDescendant` walk over `parentId` → `folderId`.
  - The raw SQL `JOIN tasks_ext_auto_start tas ON tas.parent_id = td.task_id` — **leave `tas.parent_id`** (extension table PK, not renamed).
- `server/internal/mutations/cross-table.ts` — `parentId: CONVERSATIONS_META_TASK_ID` → `folderId`.

### 3. Endpoints + types (`plugins/tasks/core`, `plugins/tasks/web`)

- `core/endpoints.ts` — `CreateTaskBodySchema.parentId`, `UpdateTaskBodySchema.parentId`, `InsertBetweenBodySchema.targetParentId` → `folderId` / `targetFolderId`.
- `core/task-chain-types.ts` — discriminated union `kind: "child"` / `parentTaskId` → `kind: "folder"` / `folderTaskId`. **Remove `includeParentTask`** from `TaskChainCardSchema`.
- `web/client.ts` — `TaskPatch.parentId` → `folderId`.

### 4. Server handlers (`plugins/tasks/server`)

- `handle-create.ts` — `body.parentId` → `folderId`.
- `handle-insert-between.ts` — `targetParentId` → `targetFolderId`, `parentId:` in `createTask` → `folderId`.
- `handle-create-chain.ts`:
  - `parentId` var (from `target` union) → `folderId`; comment update.
  - **Remove** the `parentTaskRef` path: the `isHead && card.includeParentTask && kind==="child"` branch, the `parentTaskRef` param of `renderTaskDescription`, and the entire `<parent-task>` block (lines ~182–191). The `parentTask = await getTask(folderId)` existence check can stay only if still needed for validation; otherwise drop it.
- `mcp-tools.ts` — `parentId: currentTaskId` → `folderId: currentTaskId` (behavior unchanged: still files under the current task). `groupId` line stays.

### 5. Tree boundary mapping + web components (`plugins/tasks/plugins/*`)

- `task-list/plugins/tree/web/tasks-list.tsx`:
  - Map `folderId → parentId` when building rows for `TreeList`; map back in `onMove` → `patchTask(id, { folderId: dest.parentId, rank: dest.rank })`.
  - `createTaskRow` args `parentId` → `folderId`; `isInSubtree` walk over `folderId`.
- `task-list/plugins/.../child-count-action.tsx` — `t.parentId === taskId` → `t.folderId === taskId`.
- `task-dependencies/web/components/task-dependencies.tsx`:
  - `targetForSibling` reads `task.parentId` → `task.folderId`; emits `{ kind: "folder", folderTaskId: task.folderId }`.
  - `parentCandidate` / `addParentAsDep` logic → `folderId`.
  - **Remove** both `captures={["parentTask"]}` props.
  - Relabel button **"Add parent as dep"** → **"Add folder as dep"** (full relabel).
- `task-graph/web/components/task-graph.tsx` + `insertable-edge.tsx` — `t.parentId` / `targetParentId` → folder naming.

### 6. Draft form — remove the capture (`plugins/tasks/plugins/task-draft-form`)

- `web/components/task-draft-form.tsx` — remove `includeParentTask` from `CardDraft` + `makeCard`; remove `"parentTask"` from `CaptureKind`; remove `supportsParentTask` and the toggle UI (~lines 237–244).
- `web/internal/submit.ts` — remove the `includeParentTask: …` line from the card payload.
- `CLAUDE.md` — drop the `parentTask` bullet from the `captures` doc.

### 7. Docs + descriptions (full relabel)

- `plugins/tasks/plugins/task-dependencies/CLAUDE.md` + plugin description — "quick-add button for the parent task" → "folder".
- `plugins/tasks/plugins/task-list/*` descriptions — keep "tree view" (fine), but any "parent/child" prose → "folder".
