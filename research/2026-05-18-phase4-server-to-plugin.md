# Phase 4: `server/` → `plugins/framework/plugins/server/`

Implementation plan for the DAG migration phase that moves `server/` into the plugin tree. Part of [`2026-05-12-global-plugin-dag-migration.md`](./2026-05-12-global-plugin-dag-migration.md).

## Context

`server/` plays two roles today:

| Role | Files | Consumers |
|------|-------|-----------|
| **Framework API** — types, resources, contributions, error reporting, profiling | `types.ts` (111 imports), `resources.ts` (76), `contributions.ts` (4), `error-reporter.ts` (4), `profiler.ts` (1) | 196 import statements across 163 plugin files, all via `@server/*` alias |
| **Boot harness** — Bun.serve, route-table construction, plugin topo-sort, lifecycle (register → onReady → shutdown), orphan detection | `index.ts`, `topo.ts`, `paths.ts`, `plugins.ts`, `plugins.generated.ts` | Zero external consumers — only the gateway spawns this |

These two roles separate cleanly into `core/` and `bin/` directories.

## Target directory structure

```
plugins/framework/plugins/server/
  package.json          ← @singularity/server (name unchanged)
  tsconfig.json         ← path aliases retargeted
  CLAUDE.md
  core/                 ← public barrel — the framework API
    index.ts            ← re-exports from all 5 modules below
    types.ts              ServerPluginDefinition, WsHandler, WsData, HttpHandler, Registration, ResourceLike, ConfigDescriptorLike
    resources.ts          defineResource, Resource, withNotifyBatch, notificationsWsHandler, handleResourceHttp
    contributions.ts      defineServerContribution, collectContributions, ServerContributionToken
    error-reporter.ts     reportServerError, setErrorReporter
    profiler.ts           profilerStart, getProfilingData
  bin/                  ← process entry point (gateway runs `bun bin/index.ts` from this directory)
    index.ts            ← Bun.serve, route tables, lifecycle
    topo.ts             ← topoSortPlugins (only used by index.ts)
    paths.ts            ← GIT binary path (only used internally)
    plugins.ts          ← re-exports plugins.generated
    plugins.generated.ts ← auto-generated by plugin-registry-gen
```

### Why `bin/`

The boot harness is an executable entry point, not library code. `bin/` is the standard convention for executables (Node.js `package.json` `bin` field, Unix `/usr/bin`, Go `cmd/`). `bin/index.ts` reads clearly as "the thing you run."

This plugin doesn't contribute routes/registrations/onReady — it IS the server. It has `core/` (public API) and `bin/` (process entry), but no `server/` or `web/` barrel in the plugin system sense.

### Gateway change: `src/index.ts` → `bin/index.ts`

The gateway currently hardcodes `exec.Command("bun", "src/index.ts")` with `cmd.Dir = spec.Server` (`gateway/worktree.go:527`). This changes to `exec.Command("bun", "bin/index.ts")`. The same one-line change applies to `central/`, which uses the identical spawn path. No other backends exist.

## Plugin tsconfig.json

The moved tsconfig needs deeper relative paths to reach `plugins/` from `plugins/framework/plugins/server/`:

```jsonc
{
  "compilerOptions": {
    "paths": {
      "@server/*": ["./core/*"],           // was: ["./src/*"]
      "@plugins/*": ["../../../*"]          // was: ["../plugins/*"]
    }
  },
  "include": [
    "bin",
    "core",                                // new — the public API directory
    "../../../*/server",                    // was: "../plugins/*/server" — 1 level of nesting
    "../../../*/core",
    "../../../*/shared",

    "../../../*/plugins/*/shared",
    // ... same pattern for 3–6 levels of nesting
  ],
  "exclude": ["../../../framework/plugins/web-sdk/**"]
}
```

Path verification: from `plugins/framework/plugins/server/`, `../../../` = `plugins/`.

## Gateway — one-line change

The gateway's server spawn is data-driven — it discovers backends through spec JSON, not hardcoded paths:

1. `cli/build.ts` writes `~/.singularity/worktrees/<name>.json` with `{ "server": "<abs-path>" }`
2. Gateway reads this via fsnotify (`gateway/registry.go:87–131`), stores it in `Spec.Server`
3. `startBackend()` (`gateway/worktree.go:521–563`) runs `exec.Command("bun", ...)` with `cmd.Dir = spec.Server`

The only gateway code change: `gateway/worktree.go:527` updates the spawn command from `"src/index.ts"` to `"bin/index.ts"`. This applies to all backends (server and central use the same spawn path).

## Boundary system changes

### `boundary.config.ts`

**Zones** — remove `zone("server", { match: "server" })` (line 5). The plugin is auto-discovered as `plugin.framework.server` via `plugin-tree` discovery on `zone("plugin", { match: "plugins" })`.

**Edges** — remove two entries:
- `allow("plugin.** -> server")` (line 38) — after move, covered by `allow("plugin.** -> plugin.**")` (line 44)
- `allow("server -> plugin.**")` (line 41) — same: server becomes `plugin.framework.server`, covered by the `plugin.** -> plugin.**` rule

**Exclude list** — update three entries:

| Current | After |
|---------|-------|
| `server/src/plugins.ts` | `plugins/framework/plugins/server/bin/plugins.ts` |
| `server/src/plugins.generated.ts` | `plugins/framework/plugins/server/bin/plugins.generated.ts` |
| `server/src/index.ts` | `plugins/framework/plugins/server/bin/index.ts` |

### `tooling/src/boundaries/resolve.ts` (lines 111–113)

```typescript
// Before:
if (specifier.startsWith("@server/")) {
  return { zone: "server", runtime: null };
}

// After:
if (specifier.startsWith("@server/")) {
  return { zone: "plugin.framework.server", runtime: "core" };
}
```

Zone name matches the plugin-tree hierarchy. Runtime becomes `"core"` because `@server/*` resolves to the `core/` barrel — this enables runtime isolation (e.g., a plugin's `web/` code cannot import `@server/types`).

### `tooling/src/boundaries/check.ts` (line 14)

Remove `"server/src"` from `SOURCE_ROOTS`. After the move, `plugins/framework/plugins/server/{core,bin}/` are under the `"plugins"` source root and discovered automatically.

## Module identity — singleton safety

`resources.ts` holds module-level singletons (`registry: Map<string, RegistryEntry>`, notification batching state, DAG sort cache). All 76 consumer imports use `@server/resources`. After the alias retarget, every import resolves to the same physical file (`plugins/framework/plugins/server/core/resources.ts`). Bun's module cache is path-keyed — same file = same module instance = singleton preserved.

The `no-relative-server-imports` check is the primary guard against module identity splits. After the move, the check's grep pattern must target the new path to catch regressions.

## Phase 4b: `central/` → `plugins/framework/plugins/central/`

`central/` is structurally identical to `server/`: same entry point contract (`bun bin/index.ts` + `SOCKET_PATH` after gateway update), same `types.ts`/`resources.ts`/`topo.ts` pattern, own `CentralPluginDefinition`. But much smaller — only 8 consumer imports across 8 files (all in `plugins/auth/` and `plugins/infra/plugins/secrets/`), and fewer source files (no contributions.ts, error-reporter.ts, profiler.ts, or paths.ts). Central's `src/` also renames to `bin/` as part of the same gateway change.

| Concern | Phase 4 (server) | Phase 4b (central) |
|---------|-------------------|---------------------|
| Consumer scale | 196 imports / 163 files | 8 imports / 8 files |
| `@central/*` modules used | — | `@central/types` (7), `@central/resources` (1) |
| `cli/build.ts` path | `resolve(root, "server")` | `resolve(root, "central")` — note: `central.json` always points at **main's** copy, not the worktree's |
| Gateway | spec-driven, zero changes | spec-driven, zero changes; `"central"` name is pinned (never swept), but name != path |
| `boundary.config.ts` | Remove `zone("server", ...)` | Remove `zone("central", ...)` |
| `resolve.ts` | `@server/` → `plugin.framework.server` | `@central/` → `plugin.framework.central` |
| `plugin-registry-gen.ts` | `RUNTIMES.server` paths + typeImport | `RUNTIMES.central` paths + typeImport — same pattern |

Phase 4b can run immediately after Phase 4 or in parallel (independent directory, independent alias, independent zone). The only coupling is shared tooling files both phases modify (`boundary.config.ts`, `resolve.ts`, `SOURCE_ROOTS` arrays) — concurrent edits need deconflicting.

## Risk register

| Risk | Severity | Mitigation |
|------|----------|------------|
| **Module identity split** — relative imports bypass alias, duplicating `resources.ts` singleton registry | High | `no-relative-server-imports` check (updated for new path). All 196 existing imports already use the alias. |
| **tsconfig depth errors** — `@plugins/*` → `../../../*` resolves incorrectly | Medium | Verification gate at step 5: `bunx tsc --noEmit` from the new directory catches all resolution failures. |
| **Stale spec JSON** — running worktrees have `server: <old-path>` in `~/.singularity/worktrees/*.json` | Low | `./singularity build` rewrites the spec. Gateway fsnotify picks up the change on next spawn. Already-running backends are unaffected (path is only used at spawn time). |
| **Generated file colocation** — `plugins.generated.ts` has wrong `typeImport` after `types.ts` moves to `core/` | Medium | Fixed by step 6 (change `./types` → `@server/types`). `./singularity build` regenerates the file. Build failure is immediate and obvious. |
| **CLAUDE.md examples drift** — agent-facing docs show old import paths | Low | Updated in step 12. Stale examples cause style drift, not build failures. |
| **bun workspace identity** — package.json `name: @singularity/server` under new path | None | No internal code imports via workspace name. `plugins/**` glob covers the new location. Name is cosmetic. |
