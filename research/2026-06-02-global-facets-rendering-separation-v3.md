# Facets v3 — Finish the Rendering Separation

> Supersedes `2026-05-20-global-unified-facet-docgen.md` (v1) and
> `2026-05-25-plugin-meta-facets-rendering-separation-v2.md` (v2). Read this one.
> v1 is archived (its model predates rendering separation and its dependency graph
> cements the wrong ownership). v2 is current but leaves three things undecided —
> this doc decides them.

## Context

A **facet** is a self-contained slice of plugin metadata (exports, slots, commands,
routes, resources, db-schema, cross-refs, contributions, registrations). Each facet
runs against every plugin and produces structured data consumed by four surfaces:
**doc** (markdown), **detail** (Forge plugin pane), **diff** (PR review), and
**catalog** (Forge aggregated tables).

**Where we are:**
- Extraction is migrated. All 9 facets exist with `core/` (type) + `facet/`
  (`extract`/`relate`/`renderDoc`), discovered by `loadFacets()`.
- Docgen's per-plugin facts already go through the generic `getFacet` + `facet.renderDoc()`
  loop (`docgen.ts:55-63`).
- Diff infrastructure already landed: `DiffRenderer` interface
  (`plugins/review/plugins/plugin-changes/core/diff-renderer.ts`) and the
  `review.plugin-changes.diff-renderer` slot (`.../web/slots.ts:18-21`) exist — with
  **zero contributors**.

**What's still broken / monolithic:**
- `buildPluginTree` gates the entire facet pipeline behind `!skipBarrelImport`
  (`plugin-tree.ts:301`). Both `tree-handler` (detail + catalog) and `compute-plugin-diff`
  pass `skipBarrelImport: true`, so **detail, catalog, and diff all render empty data
  today** — the live bug.
- Rendering for diff/detail/catalog still lives in the *consumers*: 7 hardcoded
  `*Strings()` helpers in `compute-plugin-diff.ts:39-88`, `public-api-section.tsx`, and
  hardcoded catalog tables reading `node.publicApi?.*`.
- A `populateCompatFields()` shim (`plugin-tree.ts:358-422`) + 20+ legacy flat fields on
  `PluginNode` keep the old path alive, and the shim re-couples the orchestrator to every
  facet by name.
- `docs/routes.md` is generated by reading compat fields directly — a leak.

**Goal:** finish the separation so that **adding a facet = adding one folder subtree,
touching no consumer**, and delete the compat shim.

## Mental model: separate by *runtime*, not by "rendering"

The load-bearing principle, stated once:

> A facet's **build-time** operations — `extract`, `relate`, `renderDoc` — live in
> `facet/`, discovered by `loadFacets()` (dynamic import; runs inside `./singularity build`,
> browser-less). Its **browser** renderers — diff, detail, catalog — live in `web/`
> sub-plugins, discovered by web slots. **Pure data transforms** (`toComparable`) live in
> `core/` so either runtime can call them. The discriminator is *the runtime that executes
> the code* — identical to the web/server/core split used everywhere else in the repo.

This is why `renderDoc` stays on the facet and is **not** an asymmetry: doc rendering is a
build-time string operation co-located with the facet's other build-time ops. Diff/detail/
catalog are browser React. Same data, different runtimes, different homes.

Four invariants:
1. **Vertical ownership.** Everything about one metadata aspect lives in that facet's subtree.
2. **Generic collection API.** Consumers iterate registered facets/renderers and call generic
   methods. No consumer names a facet. A consumer may read `node.facets[id]` only via an
   `id` carried by a *contribution* (e.g. `renderer.facetId`), never a hardcoded literal.
3. **One data surface, always computed.** The node carries `facets: Record<string, unknown>`;
   `extract`/`relate` run unconditionally.
4. **Discovery follows runtime.** Build-time → `loadFacets()`. Browser → web slots.

```
facets/plugins/<name>/
  core/                     types + facetDef + toComparable (pure, runtime-agnostic)
  facet/                    extract + relate + renderDoc      [build-time, loadFacets()]
  plugins/
    render-diff/web/        contributes DiffRenderer          [browser, web slot]
    render-detail/web/      contributes PluginView.Section    [browser, web slot]
    render-catalog/web/     contributes Catalog.FacetTable    [browser, web slot]
```

## Decisions

- **D1 — Registration mechanism.** Keep `loadFacets()` (the `CollectedDir` dynamic-import
  registry) for build-time facet discovery; use web slots for the three browser renderers.
  **No new `cli/` runtime barrel.** A `cli/` runtime would duplicate `loadFacets()` for zero
  gain and cost full new-runtime infra. Unifying all build-time discovery (facets, `check/`,
  `lint/`, codegen) under one slot-based registry is a **separate, optional** initiative
  (the existing "Generalize slot registration pattern" task) — explicitly **out of scope** here.
- **D2 — `renderDoc` stays on the `Facet` interface.** Runtime-coherence (above). Docgen is
  unchanged.
- **D3 — `toComparable` lives in `core/`; diff is computed client-side.** It is pure
  (`data → string[]`), so it belongs in the facet's `core/`. The server sends raw facet data
  for both sides; the client iterates `DiffRenderer` contributions and computes diffs. This
  keeps the server blind to facets and resolves v2's unresolved server-vs-client fork.
- **D4 — Facet-vs-core boundary rule.** Structural identity stays as core `PluginNode` fields:
  `path`, `name`, `hierarchyId`, `description`, `runtimes`, `collapsed`, `loadBearing`,
  `children`. Everything describing *what a plugin contributes* is a facet. New metadata that
  isn't structural identity becomes a facet, never a new flat field. (Document in
  `facets/CLAUDE.md`.)
- **D5 — Catalog is a first-class symmetric surface.** Each facet contributes a
  `Catalog.FacetTable` describing its aggregated cross-plugin table. The catalog host iterates
  contributions generically. (The `catalog/plugins/tables/*` live-SQL detail sub-plugins are
  NOT facet consumers and are untouched.)

## Surfaces & ownership

| Surface | Owner plugin | Interface | Runtime | Discovery |
|---|---|---|---|---|
| Doc | `codegen` (via facets) | `facet.renderDoc()` | build-time | `loadFacets()` |
| Detail | `plugin-view` | `PluginView.Section` (exists) | browser | web slot |
| Diff | `plugin-changes` | `DiffRenderer` (exists) | browser | web slot `review.plugin-changes.diff-renderer` |
| Catalog | `forge/catalog` | `Catalog.FacetTable` (**new**) | browser | web slot |

## Interfaces

**`Facet` — unchanged** (`facets/core/facets.ts`): `{ def, extract, relate?, renderDoc }`.

**`DiffRenderer` — exists** (`plugin-changes/core/diff-renderer.ts`):
```ts
interface DiffRenderer { facetId: string; label: string; toComparable: (data: unknown) => string[]; }
```
Each facet's `render-diff/web` contributes one, importing `toComparable` from its parent
`core/`. The host slices `node.facets[renderer.facetId]` and feeds it in.

**`Catalog.FacetTable` — new**, owned by `forge/catalog`. Declarative so it feeds the existing
`data-table` primitive; the host stays facet-blind:
```ts
interface CatalogFacetTable<T = unknown> {
  facetId: string;             // host slices node.facets[facetId] generically
  label: string;               // tab label ("Routes", "Slots", ...)
  columns: ColumnDef[];
  rows: (entries: { node: PluginNode; data: T }[]) => CatalogRow[];  // each row links to its plugin
}
```

**Detail — `PluginView.Section`**, already the slot used by `public-api`. Each facet's
`render-detail/web` contributes a per-plugin section reading its own `node.facets[id]`.

**Wire `PluginNode` type change** (`plugin-view/core/types.ts:68-81`): replace
`publicApi?: PublicApi` with `facets: Record<string, unknown>`; delete `PublicApi` and its
sub-types (`SlotInfo`, `RouteInfo`, `BarrelExport`, …).

**`buildPluginTree` change** (`plugin-tree.ts`): facet extract/relate run **always**; barrel
import only when not skipped; compat shim removed at the end.
```
4a: if (!skipBarrelImport) importAllBarrels()     // runtime facets partial when skipped — acceptable
4b: loadFacets(); for each node/facet: setFacet(extract({dir, importedModules}))   // ALWAYS
4c: for each facet: facet.relate({ tree })                                          // ALWAYS
4d: (removed in cleanup) populateCompatFields(tree)
```
7 of 9 facets are static-only and need no barrels; only `contributions` (runtime part) and
`registrations` are partial under `skipBarrelImport` — acceptable for detail/catalog/diff.

## Phases

> Phase numbering restarts clean (v2's "Phase 1" infra is already done). These phases are the
> intended task list.

### Phase 1 — Catalog slot infrastructure
Define `CatalogFacetTable` + the `Catalog.FacetTable` web slot in `forge/catalog`. (Diff slot
and detail slot already exist.)
- NEW: `plugins/apps/plugins/forge/plugins/catalog/core/facet-table.ts`
- EDIT: `catalog/web/index.ts` (define + export slot)
- Verify: `./singularity build` succeeds.

### Phase 2 — Reference vertical slice: the `exports` facet
Prove all surfaces originate from one facet folder. Doc already works via `renderDoc`.
- Move `toComparable` (the `exportStrings` projection, `compute-plugin-diff.ts:50-58`) into
  `facets/plugins/exports/core/`.
- NEW `exports/plugins/render-diff/web` → contributes `DiffRenderer { facetId: "exports", label: "Exports", toComparable }`.
- NEW `exports/plugins/render-detail/web` → move `RuntimeGroup`, `SymbolRow`, `CATEGORY_STYLES`,
  `categorize`, `ConsumerList` from `public-api-section.tsx`; reads `node.facets["exports"]`.
- NEW `exports/plugins/render-catalog/web` → contributes the Exports aggregation table.
- Verify: `./singularity build`; the exports section renders in detail/catalog/diff with real data.

### Phase 3 — Replicate for the remaining 8 facets
First extract shared web components (`SubHeading`, `PluginLink`, `ConsumerList`) to a shared
home (`plugin-view/web/components/` or a small primitives plugin). Then, per facet, create
`render-diff` / `render-detail` / `render-catalog` sub-plugins. Sources to port:
- **render-diff** (from `compute-plugin-diff.ts`): `slots` (39-41), `contributions` (43-48),
  `cross-refs` (69-78), `routes` (60-67), `resources` (80-84), `db-schema` (86-88).
  `commands`/`registrations` need new `toComparable`s.
- **render-detail** (from `public-api-section.tsx`): `cross-refs` (ImportedByBanner), `slots`
  (SlotsGroup), `routes` (RoutesGroup + METHOD_COLORS), `resources`; new for `contributions`,
  `commands`, `db-schema`, `registrations`.
- **render-catalog** (from `catalog/web/index.ts` + `tables/web`): `routes`, `contributions`,
  `slots`, `resources`, `db-schema` (table names — detail pane's live-SQL sub-plugins stay);
  new where catalog has no existing tab.
- Parallelizable across facets.

### Phase 4 — Flip pipeline + consumers to generic iteration
- **4.1** `buildPluginTree`: run facet extract/relate always (see "buildPluginTree change").
  Static facets populate even under `skipBarrelImport: true`. (`plugin-tree.ts`)
- **4.2** Wire `PluginNode` type → `facets: Record<string, unknown>`; `tree-handler.toApiNode`
  passes `node.facets` through; delete `buildSymbolConsumers` (19-35) and the compat-field reads
  in `toApiNode` (37-107). (`tree-handler.ts`, `plugin-view/core/types.ts`)
- **4.3** Delete the `public-api` plugin (replaced by `render-detail` sub-plugins).
  (`plugins/plugin-meta/plugins/plugin-view/plugins/public-api/`)
- **4.4** `compute-plugin-diff`: send `currentFacets`/`mainFacets` (raw `node.facets`) for both
  sides; delete the 7 `*Strings()` helpers; client iterates `DiffRenderer` contributions +
  `diffSets`. Update `api-changes-section.tsx` to iterate `facetDiffs` generically and
  `api-changes-summary.tsx` totals. (`compute-plugin-diff.ts`, `core/protocol.ts`,
  `api-changes/web/components/*`)
- **4.5** Catalog: iterate contributed `Catalog.FacetTable`; delete the hardcoded
  `publicApi?.*` reads in `catalog/web/index.ts` (47,57,66,70,74), `routes-table.tsx:85`,
  `tables-table.tsx:22`. `tables/*` live-SQL detail untouched.
- Verify: detail pane, catalog, and PR diff all show **real** data (previously empty);
  `docs/*.md` byte-identical.

### Phase 5 — Completeness check (fail loudly)
Add a `./singularity check` that, for every facet from `loadFacets()`, asserts a contributor
exists in each of the three web render slots (diff/detail/catalog) with a matching `facetId`.
Prevents a facet silently vanishing from a surface. (`facets/check/index.ts` or a built-in check)

### Phase 6 — Cleanup (incremental)
Removable per-facet as each lands green; final state:
- Remove `populateCompatFields()` + all 20+ legacy flat fields + `RuntimeDetail` from internal
  `PluginNode`. (`plugin-tree.ts:358-422`)
- Remove compat type re-exports from `plugin-tree/core/index.ts` (`SlotDef`, `CommandDef`,
  `RouteDef`, `ResourceDef`, …) — keep only structural types.
- Remove `docs/routes.md` + its generators (`docgen.ts`: `pluginHasRoutesDeep` 271,
  `groupHttpRoutes` 281, `renderRoutesPluginTree` 293, `renderRoutesDocFromTree` 319,
  `renderRoutesDoc` 160, `pluginRoutesDocPath` 219) + the routes.md validation in
  `plugins-doc-in-sync/check/index.ts:64-78`.
- Update `facets/CLAUDE.md`: the runtime-coherence principle (D2), the facet-vs-core rule (D4),
  and the "how to add a facet" guide (4-folder shape).

## Follow-on (tracked as Phase 7–8, after the core migration)

These are not part of the rendering separation but are the natural next initiatives in the
discovery/registration evolution. Chained after Phase 6; decide each on its own merits.

- **Phase 7 — Generic plugin discovery.** Remove hardcoded folder names from `findAllPluginDirs`;
  every `plugins/<x>/` is a plugin regardless of sub-folders. UI flags non-standard folders
  generically. Adjacent infra; not a blocker for Phases 1–6.
- **Phase 8 — Generalize the build-time registration mechanism** (facets/checks/lint under one
  slot registry, retiring the dynamic-import `CollectedDir`/`loadFacets` pattern). If pursued,
  doc rendering *could* migrate to a slot then, but the facet stays the discovery unit.

## Out of scope

- `catalog/plugins/tables/*` live-SQL detail panes — independent of facets, unchanged.
